1과목 · 소프트웨어 설계

⭐ 요구사항 (기능 vs 비기능)

  • 기능적(Functional): 시스템이 무엇을 하는가 — 입력·출력·처리·DB·통신 등 수행 기능
  • 예) "금융 시스템은 조회·인출·입금·송금 기능이 있어야 한다"
  • 비기능적(Non-functional): 기능 외 품질·제약 — 성능·보안·품질·안정성·사용성·이식성
  • 예) "모든 화면이 3초 이내 표시" → 성능 = 비기능
함정: "처리량(Throughput)·반응시간·보안·안전"은 비기능. 성능/보안을 기능적 요구라 하면 틀린 보기.

⭐ 요구공학 프로세스 (개발 5단계)

순서: 도출 → 분석 → 명세 → 확인(검증) → 관리
단계내용
도출(Elicitation)이해관계자에게서 요구 수집 — 인터뷰·설문·브레인스토밍·유스케이스·프로토타이핑
분석(Analysis)요구 분류·개념 모델링·할당·협상·정형 분석, 상충·모호성 정리
명세(Specification)분석 결과를 문서화 (요구사항 명세서 작성)
확인/검증(Validation)명세가 옳게 작성됐는지 검토 (아래 검토기법)
관리(Management)변경 추적·영향 평가·버전 관리 (전 단계에 걸쳐 지속)
요구사항 분석에 속하는 것: 타당성 조사·비용/일정 제약설정·요구 문서화. 설계 명세서 작성은 분석 다음 단계(명세)라 분석에 넣으면 함정.

요구사항 검토(검증) 방법 — 자주 나옴

방법핵심
동료 검토(Peer Review)작성자가 직접 설명, 참석자가 들으며 결함 발견
워크스루(Walkthrough)회의 전에 명세서 미리 배포 → 사전 검토 후 짧은 회의로 오류 조기 검출
인스펙션(Inspection)작성자를 제외한 다른 검토 전문가가 확인하며 결함 발견
단골 함정: "회의 전 배포·사전 검토" = 워크스루 (인스펙션 아님). 워크스루 ≠ 인스펙션 (2022년 4월 기출).

요구사항 명세 기법 (정형 vs 비정형)

정형(Formal) 명세
  • 수학적 원리·표기법 기반
  • Z, VDM, Petri-Net, CSP, LOTOS
  • 정확·간결·일관 / 이해 어려움
비정형(Informal) 명세
  • 자연어 기반, 상태·기능·객체 중심
  • FSM, 결정표, ER, Use Case, SADT
  • 작성·의사전달 쉬움 / 모호·불일치
함정: "비정형 명세에서 Z를 쓴다" → 틀림. Z·VDM은 정형 명세.

⭐ 객체지향 핵심 개념

캡슐화(Encapsulation)데이터(속성)+함수(메소드)를 하나로 묶음. 결합도↓·재사용↑·인터페이스 단순화, 변경 시 파급효과↓
정보은닉(Information Hiding)내부 구현·데이터를 숨기고 연산(인터페이스)으로만 접근 허용. 캡슐화의 핵심 개념 (예: IP·상세자료구조 은닉)
상속(Inheritance)상위(부모)의 속성·연산을 하위(자식)가 물려받음. 재사용성↑
다형성(Polymorphism)같은 메시지에 객체마다 다르게 응답. 오버로딩(이름同·매개변수異)·오버라이딩(재정의, 상속)
추상화(Abstraction)공통 성질 추출 → 슈퍼클래스. 기법: 과정·자료(데이터)·제어 추상화
함정: "상위 클래스 속성/연산을 하위가 물려받음" = 상속이지 캡슐화 아님. 오버로딩은 "매개변수 타입/개수가 다름"(메소드명은 같음) — "메소드명을 다르게"라 하면 틀림.

객체지향 구성 요소 (용어 구별)

클래스(Class)유사 객체를 묶은 틀·집합. 데이터 추상화 단위. 공통 속성·연산을 가진 객체의 타입
객체(Object)클래스로 실체화된 것. 상태(속성)+동작(연산)+식별자를 가짐
인스턴스(Instance)같은 클래스에 속한 각각의 개별 객체
메시지(Message)객체에게 행위를 지시하는 명령/요구 (객체 간 상호작용 수단)
메소드(Method)객체의 연산(함수). 메시지를 받으면 정해진 기능 수행

객체지향 설계 원칙 SOLID

S O L I D
  • SRP 단일 책임: 클래스는 하나의 책임만
  • OCP 개방-폐쇄: 확장에 열림, 수정에 닫힘
  • LSP 리스코프 치환: 하위 타입은 상위 타입으로 언제든 교체 가능
  • ISP 인터페이스 분리: 안 쓰는 인터페이스에 의존 X
  • DIP 의존 역전: 변하기 쉬운 것 말고 변하기 어려운(추상) 것에 의존

⭐ 럼바우(Rumbaugh) 3모델링

객 동 기 — 객체2 / 동상 / 기자
  • 객체(Object) 모델링: 객체 다이어그램. 정적 구조·관계 (= 정보 모델링, 가장 먼저·가장 중요)
  • 동적(Dynamic) 모델링: 상태 다이어그램. 시간 흐름·제어·상태 변화
  • 기능(Functional) 모델링: 자료흐름도(DFD). 프로세스 간 자료 흐름
키워드 암기 → "정보·요구되는 객체"=객체 / "제어·흐름·동작·상태"=동적 / "DFD·자료흐름"=기능. 정적 모델링은 럼바우 3모델에 없음(함정).

⭐ UML — 구조(정적) vs 행위(동적)

구조 다이어그램 (정적)
클 객 컴 배 복 패
  • 클래스 / 객체 / 컴포넌트
  • 배치(Deployment) / 복합체 / 패키지
행위 다이어그램 (동적)
유 순 커 상 활 타
  • 유스케이스 / 순차(Sequence) / 커뮤니케이션
  • 상태 / 활동(Activity) / 타이밍
★최다 함정: 순차(Sequence)=행위(동적). "정적 측면 모델링"이라 하면 틀림. 활동(Activity)·상태도 행위. 배치(Deployment)만 구조인데 행위에 섞어 냄.

UML 관계 6종

관계의미 / 표기
연관(Association)두 사물의 구조적 연결 (has-a). 실선
집합(Aggregation)전체-부분, 부분이 독립적 존재. 속 빈 마름모(◇)
포함(Composition)전체-부분, 부분이 전체에 종속(생명주기 같이). 채운 마름모(◆)
일반화(Generalization)상속(is-a). 하위→상위 속 빈 삼각 화살표(△)
의존(Dependency)한쪽 명세 바뀌면 영향. 매개변수로 잠깐 사용. 점선 화살표
실체화(Realization)한 객체가 다른 객체의 오퍼레이션 수행 지정(인터페이스 구현). 점선 삼각
단골: "명세 바뀌면 영향·매개변수로 사용" = 의존. "오퍼레이션 수행하도록 지정" = 실체화. "부분-전체(is-a-part-of)" = 집합/집단화. 스테레오타입 표기 = « » (겹화살괄호).

⭐ 응집도 (Cohesion) — 강할수록 좋음

약 → 강 : 우 논 시 절 교(통) 순 기
순서응집도뜻
1 (약)우연적(Coincidental)관련 없는 요소끼리 우연히 모임
2논리적(Logical)유사 성격(입력/출력 등)끼리 묶음
3시간적(Temporal)같은 시간대 실행(초기화 등)
4절차적(Procedural)여러 기능을 순서대로 수행
5교환/통신적(Communicational)같은 자료를 사용
6순차적(Sequential)앞 출력이 뒤 입력
7 (강)기능적(Functional)단일 기능만 수행 (가장 이상적)

⭐ 결합도 (Coupling) — 약할수록 좋음

약 → 강 : 자 스 제 외 공 내
순서결합도뜻
1 (약)자료(Data)파라미터로 값만 전달 (가장 좋음)
2스탬프(Stamp)자료구조 전체를 전달
3제어(Control)제어 플래그를 전달
4외부(External)외부 전역변수/장치 공유
5공통(Common)전역변수 공유
6 (강)내용(Content)다른 모듈 내부를 직접 참조 (가장 나쁨)
단골: "가장 좋은 결합도 = 자료(Data)", "가장 좋은 응집도 = 기능적". 반대로 "가장 나쁜 = 내용 / 우연적". 좋은 설계 = 응집도↑ 결합도↓.

팬인(Fan-In) / 팬아웃(Fan-Out)

⭐ 자료구조 — 선형 vs 비선형

선형 : 배 리 스 큐 덱 (배열·리스트·스택·큐·데크)
비선형 : 트 그 (트리·그래프)
  • 스택(Stack): 한쪽 끝(Top)에서만 삽입/삭제 → LIFO(후입선출). 함수호출·인터럽트·수식(후위)계산·재귀
  • 큐(Queue): 한쪽(rear) 삽입, 다른쪽(front) 삭제 → FIFO(선입선출)
  • 데크(Deque): 양쪽 끝 모두 삽입/삭제 가능
  • 연결리스트: 노드(데이터+포인터)로 연결, 삽입/삭제 용이·접근 느림
  • 오버플로(Overflow): 꽉 찼는데 삽입(push) 시도
  • 언더플로(Underflow): 비었는데 삭제(pop) 시도
  • 스택 포인터: Top(Bottom과 짝). 큐: front·rear 2개 포인터
  • 파일구조: 순차·색인·직접 파일
단골 함정: "큐는 비선형이다" → 큐는 선형! / "스택은 Head·Tail 2포인터" → 스택은 Top 하나(그건 큐 설명).

⭐ 스택 push/pop 출력 순서

A B C D 입력을 push→pop 규칙대로: 전부 push 후 pop이면 역순 D C B A. push·pop을 섞으면 그 순서대로 하나씩 그려서 판정. (예: push,push,pop,push,push,pop,pop,pop → B D C A)

⭐ 트리 순회 (전위/중위/후위)

기준 = Root를 언제 방문? 전=앞 / 중=가운데 / 후=뒤
순회순서암기
전위 PreorderRoot → Left → Right루트 먼저
중위 InorderLeft → Root → Right루트 가운데 (= 중위식/infix)
후위 PostorderLeft → Right → Root루트 마지막 (= 후위식/postfix)
수식표기 변환: 전위(Prefix)=연산자 앞 / 중위(Infix)=연산자 안 / 후위(Postfix)=연산자 뒤. 후위식 계산은 연산자 바로 앞 두 피연산자를 계산. 변환 요령: ①연산자 기준 괄호 묶기 →②연산자를 괄호 뒤(후위)/앞(전위)로 →③괄호 제거.

트리 용어 — 차수·단말노드·간선수

  • 차수(Degree): 한 노드의 자식 수. 트리의 차수 = 전체 중 가장 큰 차수
  • 단말(Terminal/Leaf) 노드: 자식이 없는 노드
  • 루트는 차수 계산엔 포함, 자기 자신은 자식 아님
  • 이진트리: 각 노드 자식 최대 2개
  • 이진탐색트리(BST): 왼쪽<루트<오른쪽
  • 힙(Heap): 완전이진트리, 최대/최소값이 루트 (힙정렬에 사용)
그래프 최대 간선수: 무방향 = n(n-1)/2, 방향(유향) = n(n-1). n개 정점 기준. (무방향에서 ×2가 방향)

그래프 탐색 DFS / BFS

  • DFS(깊이우선): 한 분기를 끝까지 파고든 뒤 되돌아옴(백트래킹). 스택 이용
  • BFS(너비우선): 가까운 노드부터 넓게. 큐 이용
  • 하향식 통합에도 깊이 우선/너비 우선 통합법이 쓰임

⭐⭐ 정렬 알고리즘 & 시간복잡도

O(n²) 삼형제 : 버 선 삽 (버블·선택·삽입)
정렬평균/최악동작 핵심
버블(Bubble)O(n²)인접 두 개 비교·교환. 1 PASS마다 최댓값이 맨 뒤 고정
선택(Selection)O(n²)최솟값 찾아 맨 앞과 교환 반복 (위치 자체를 스왑)
삽입(Insertion)O(n²) 최선 O(n)2번째부터 앞쪽 정렬부와 비교해 알맞은 자리에 끼워넣기
쉘(Shell)O(n1.5) 최악 O(n²)삽입정렬 확장. 매개변수 h 간격 서브파일 정렬
퀵(Quick)O(n log n) 최악 O(n²)분할정복 + 피벗(pivot) 기준 분할. 평균 최고속
힙(Heap)O(n log n)완전이진트리(힙) 구성, 루트(최대) 제거 반복

⭐ "1회전(PASS 1) 결과" 문제는 반드시 직접 굴리기. 선택은 최솟값과 맨앞 교환, 버블은 한 PASS에 여러 번 인접 교환+맨뒤 1개 확정, 삽입은 왼쪽으로 밀어 끼워넣기 — 셋의 1회전 결과가 다르다!

검색 — 순차 / 이진 / 보간

  • 순차(선형)검색: 처음부터 하나씩. 정렬 불필요. O(n)
  • 이진(Binary)검색: 정렬 필수. 중간값 비교 후 절반씩 버림. O(log n)
  • 보간(Interpolation): 찾는 값 위치를 비례로 추정
  • 피보나치 검색: 피보나치 수열로 비교 대상 선정
  • 이진검색: 중간 인덱스 = (low+high)/2. "14 찾기 비교횟수" = mid 갱신 횟수를 직접 세기

해싱(Hashing) & 충돌 해결

  • 해시함수: 제산법(나눗셈)·제곱법·폴딩(중첩)·기수변환·숫자분석·무작위 등. 해시탐색은 이상적 O(1)
  • 충돌(Collision): 서로 다른 키가 같은 주소 → 해결 필요. 같은 주소 겹친 = 동의어(Synonym)
  • 개방주소법(Open Addressing): 충돌 시 다음 빈 버킷 탐사 (선형/제곱/이중해싱)
  • 체이닝(Chaining): 같은 주소를 연결리스트로 매달기
  • 재해싱(Rehashing): 다른 해시함수로 재계산

알고리즘 설계기법 & 빅오

  • 분할정복(Divide & Conquer): 작은 문제로 쪼개 해결 (퀵·합병)
  • 동적계획법(DP): 부분해 저장·재사용(중복 계산 제거)
  • 그리디(탐욕): 매 순간 최적 선택
  • 백트래킹: 가지치기하며 해 탐색, 아니면 되돌아감
  • 분기한정(Branch & Bound): 한계값으로 가지치기
  • O(1) 상수 (해시)
  • O(log n) 로그 (이진탐색)
  • O(n) 선형 (순차탐색)
  • O(n log n) 선형로그 (퀵·합병·힙)
  • O(n²) 제곱 (버블·선택·삽입)
  • O(2ⁿ) 지수, O(n!) 팩토리얼
McCabe 순환복잡도 V(G) = 간선(E) − 노드(N) + 2. (제어흐름 그래프에서 화살표 수 − 원 수 + 2) 예: 6−4+2 = 4

⭐⭐ 블랙박스 vs 화이트박스 테스트

블랙박스 : 동 경 원 오 비 (동치·경계값·원인효과·오류예측·비교)
화이트박스 : 기 조 루 데 (기초경로·조건·루프·데이터흐름)
  • 블랙박스 = 기능/명세 기반, 내부구조 모름, 사용자 관점
  • 동치(동등)분할, 경계값 분석, 원인-효과 그래프, 오류 예측, 비교 검사
  • 경계값: 경계·바로 옆 값이 오류 잘 남 (예: 0,59,60,79,80,100,101)
  • 화이트박스 = 내부 구조/소스 기반, 개발자 관점, 동적 테스트
  • 기초경로(Base Path), 조건, 루프, 데이터흐름 검사
  • 기본경로: 시작→종료 서로 독립 경로, 사이클 최대 1회
함정: "경계값 분석/오류예측은 화이트박스" ❌ → 블랙박스. "조건·루프·기초경로는 블랙박스" ❌ → 화이트박스. 서로 바꿔치기가 매회 나옴.

검증(Verification) vs 확인(Validation)

  • 검증(Verification): 제대로 만들고 있나 — 개발 과정·산출물 검토 (Are we building it right?)
  • 확인(Validation): 제대로 된 걸 만들었나 — 최종 결과가 요구 충족? (Are we building the right thing?)

⭐ 테스트 레벨 (V모델) & 통합 방식

레벨 : 단 통 시 인 (단위→통합→시스템→인수)
레벨내용
단위(Unit)모듈 단위. 구조기반(화이트박스) 위주. 스텁/드라이버 필요
통합(Integration)모듈 간 인터페이스·결합 검증
시스템(System)전체 시스템 (성능·강도(Stress)·회복·보안 등)
인수(Acceptance)사용자 요구 충족 확인 → 알파·베타 테스트
  • 상향식(Bottom-up): 하위→상위. 드라이버(Driver) 필요. 클러스터
  • 하향식(Top-down): 상위→하위. 스텁(Stub) 필요. 깊이/너비우선
  • 빅뱅(Big Bang): 한꺼번에 통합(비점진적)
  • 샌드위치: 상향+하향 혼합
  • 회귀(Regression): 변경 후 기존 기능 재테스트
암기 못 하면 반드시 틀림 → 상향식=드라이버, 하향식=스텁. (드라이버=상위 대역, 스텁=하위 대역)

알파 / 베타 테스트

  • 알파(Alpha): 개발자 장소에서, 개발자 앞에서 사용자가 → 통제된 환경
  • 베타(Beta): 사용자 장소에서, 선정된 최종 사용자가 자유롭게

⭐ 소프트웨어 테스트 기본 원칙

  • 결함 집중 (파레토 법칙): 오류의 80%는 전체 모듈의 20%에서 발견

  • 물리적 독립성: 내부(저장) 스키마 변경이 개념 스키마에 영향 X

데이터 모델 구성요소

데이터 모델 = 구조 · 연산 · 제약조건

  • 연산(Operation): 데이터 처리·조작 방법(검색·삽입 등)
  • 제약조건(Constraint): 실제 저장될 수 있는 데이터의 논리적 제약

⭐ 관계형 용어 (필수 암기)

용어의미
릴레이션(Relation)2차원 테이블
튜플(Tuple)행(Row) = 레코드
속성(Attribute)열(Column) = 필드, 데이터 최소 논리 단위
카디널리티(Cardinality)튜플(행)의 수 = 기수·대응수
차수(Degree)속성(열)의 수
도메인(Domain)한 속성이 가질 수 있는 원자값들의 집합
최다 함정: 카디널리티=행(튜플) 수, 차수(Degree)=열(속성) 수. 헷갈리면 반드시 틀림. "원자값들의 집합=도메인" 통암기.
릴레이션 특성: 튜플은 모두 상이(중복 X)·튜플 간 순서 없음·속성 간 순서 없음·속성명 유일·속성값은 원자값. "튜플 사이에 순서가 있다"→오답.

⭐ 키(Key) 종류

유일성=슈퍼키 / 유일성+최소성=후보키
키성질
슈퍼키(Super)유일성 O, 최소성 X
후보키(Candidate)유일성 O, 최소성 O
기본키(Primary)후보키 중 선정. NULL·중복 불가
대체키(Alternate)후보키 − 기본키
외래키(Foreign)다른 릴레이션의 기본키를 참조하는 속성
외래키는 참조 릴레이션의 기본키 값과 같거나 NULL이어야 함(참조 무결성).

⭐ 무결성(Integrity) 제약

개체 무결성기본키는 NULL·중복 불가 (널 아닌 원자값·유일)
참조 무결성외래키 값 = 참조 릴레이션 기본키 값 또는 NULL
도메인 무결성속성값이 정의된 도메인(타입·범위) 내
"NULL"·"기본키"·"원자값"이라는 단어가 보이면 개체 무결성 국룰. 외래키·참조 관련이면 참조 무결성.
DROP/삭제 옵션: CASCADE=참조 튜플 함께 삭제·변경 / RESTRICT(ED)=참조 중이면 삭제 취소 / SET NULL=관련 속성 NULL로.

⭐ 관계대수 vs 관계해석

관계대수 (절차적, "어떻게")
순수 관계연산SELECT(σ, 행 추출)·PROJECT(π, 열 추출)·JOIN(⋈)·DIVISION(÷)
일반 집합연산합집합 ∪ · 교집합 ∩ · 차집합 − · 카티션곱 ×
관계해석 (비절차적, "무엇을")
  • 수학의 술어 논리(predicate) 기반
  • 튜플 관계해석 · 도메인 관계해석
  • 표현력은 관계대수와 동등
함정: 카티션곱(Cartesian Product)·차집합은 순수 관계연산 아님 → 일반 집합연산. 순수=SELECT·PROJECT·JOIN·DIVISION 4개.
SELECT σ=수평 절단(행), PROJECT π=수직 절단(열). JOIN=공통 속성으로 두 릴레이션 결합. DIVISION=S의 모든 튜플과 관련된 R의 튜플. 관계해석 정량자: ∀(전칭, for all) · ∃(존재, there exists).
⭐ 카티션곱 계산: 차수(Degree)는 덧셈(4+6=10), 카디널리티는 곱셈(5×7=35). 계산 문제 단골.

⭐ 정규화 (최다 출제)

순서: 도 부 이 결 다 조
단계제거 대상 / 조건
1NF (도메인)모든 속성값이 원자값(반복그룹 제거)
2NF (부분)부분 함수 종속 제거(→ 완전 함수 종속)
3NF (이행)이행 함수 종속 제거 (A→B, B→C ⇒ A→C 제거)
BCNF (결정자)모든 결정자가 후보키
4NF (다치)다치(다중값) 종속 제거
5NF (조인)조인 종속 제거
  • 함수 종속(FD): X→Y, X가 결정자(determinant)
  • 이행 종속: A→B, B→C이면 A→C
  • 이상현상(Anomaly): 삽입·삭제·갱신 이상
  • 정규화 목적 = 이상현상·중복 제거
함정: BCNF="모든 결정자가 후보키". 2NF="부분 함수 종속 제거=완전 함수 종속". 3NF="이행 종속 제거". 순서 뒤섞기 단골.
반정규화(Denormalization)=성능 위해 일부러 중복 허용. 중복 테이블 추가 유형: 집계·진행·부분(특정부분) 테이블 등. 정규화의 역과정.

⭐ SQL 분류

DDL (정의)CREATE · ALTER · DROP · TRUNCATE
DML (조작)SELECT · INSERT · UPDATE · DELETE
DCL (제어)GRANT · REVOKE · COMMIT · ROLLBACK · (SAVEPOINT)
DML 두문자: S I U D (셀·인·업·딜)
최다 함정: UPDATE·DELETE=DML(DDL 아님!), DROP·ALTER·TRUNCATE=DDL. GRANT/REVOKE=권한, COMMIT/ROLLBACK=트랜잭션. SELECT는 DCL 아님(DML).

SQL 상세 (조인·서브쿼리·그룹·집계)

  • 조인: INNER(내부) · OUTER(LEFT/RIGHT/FULL) · CROSS(카티션곱) · SELF(자기 참조)

  • 집계(집합)함수: COUNT · SUM · AVG · MAX · MIN  (CREATE는 집계함수 아님)
  • 집합 연산: UNION(합집합, 중복 제거) · UNION ALL(중복 유지) · INTERSECT(교집합) · EXCEPT/MINUS(차집합)

  • 사용자 데이터 관리 간단
단점
  • 독립적 인덱스 못 가짐
  • 정의 변경 불가(ALTER X, DROP·CREATE만)
  • 삽입·삭제·갱신 연산에 제약
함정: 뷰는 물리적으로 존재하지 않는 가상 테이블. "저장장치에 물리적으로 존재"·"뷰 자체가 인덱스를 가짐"→오답. 생성 CREATE / 제거 DROP.

인덱스 · 트리거

  • 인덱스(Index): 검색 속도 향상. 대표 자료구조 B-트리 · 해시. 삽입·갱신 시 부담↑
  • 트리거(Trigger): 삽입·갱신·삭제 이벤트 발생 시 자동 실행되는 절차형 SQL
  • 프로시저(Procedure): 특정 기능 수행 절차형 SQL 블록

⭐ 트랜잭션 ACID

원 일 격 지 = 원자성·일관성·격리성·지속성
원자성(Atomicity)전부 반영(Commit) or 전무(Rollback). Commit/Rollback → 원자성
일관성(Consistency)실행 후에도 DB 일관성 유지
격리성(Isolation)실행 중 다른 트랜잭션이 끼어들 수 없음(독립성)
지속성(Durability)완료 결과는 고장에도 영구 반영
상태 전이: 활동(Active) → 부분완료(Partially Committed) → 완료(Committed) / 실패(Failed) → 철회(Aborted). "Commit·Rollback이 보장하는 특성"이면 무조건 원자성.

병행제어(Concurrency Control)

  • 로킹(Locking): 접근 데이터를 잠가 다른 트랜잭션 접근 차단. 로킹 단위=필드·레코드·테이블·파일·DB
  • 2단계 로킹(2PL): 확장 단계(Lock만)·축소 단계(Unlock만)
  • 타임스탬프(Timestamp) 순서 기법 · 낙관적 검증 기법
  • 교착상태(Deadlock): 서로 상대 자원 대기하며 무한 정지
⭐ 로킹 단위 비례관계: 단위 커지면 → 로크 수↓·오버헤드↓·병행성(공유도)↓·관리 쉬움 / 단위 작아지면 → 오버헤드↑·병행성(공유도)↑·관리 복잡. "작아지면 오버헤드 감소"는 오답.

회복(Recovery) 기법

로그 기반지연 갱신(Deferred, 완료 후 반영·Undo 불필요) / 즉시 갱신(Immediate, 즉시 반영·Undo·Redo 모두 필요)
검사점(Check Point)검사점까지만 회복, 전체 취소 불필요
그림자 페이징복사본(그림자 페이지) 이용, 로그·Undo·Redo 불필요
  • REDO(재시도): 완료된 트랜잭션 재반영
  • UNDO(취소): 미완료 트랜잭션 되돌림
함정: "로그를 필요로 하는 회복 기법"=즉시 갱신·지연 갱신. 그림자 페이징은 로그 불필요.

⭐ 분산 DB — 투명성(Transparency) 5종

위 중 병 장 분 = 위치·중복·병행·장애·분할
위치(Location)실제 저장 위치 몰라도 논리적 명칭으로 접근
중복/복제(Replication)여러 곳 중복돼도 하나처럼 사용
병행(Concurrency)동시 트랜잭션 결과 일관성 유지
장애(Failure)일부 장애에도 트랜잭션 정확 처리(무결성 보장)
분할(Division/Fragmentation)하나의 릴레이션이 여러 단편으로 나뉜 걸 몰라도 됨
함정: "Media Access Transparency"는 없음 (오답 보기 단골). 물리 환경·네트워크 장애에도 무결성 보장→장애 투명성.
4대 목표로는 위치·중복·병행·장애를 꼽기도 함(+분할=5종). 분산 DB 구성요소: 분산 처리기 · 분산 데이터베이스 · 통신 네트워크. "분산 투명성을 배제한다"→오답(제공해야 함).

시스템 카탈로그 (데이터 사전) — 자주 나옴

함정: "사용자가 SQL로 시스템 카탈로그를 직접 갱신한다"→오답. 시스템이 자동 갱신한다.

⭐⭐ 응집도·결합도 (4과목 61번 단골 — 8회 전부 출제)

응집도 약→강: 우 논 시 절 교(통) 순 기 · 결합도 약→강: 자 스 제 외 공 내
  • 응집도(Cohesion) = 모듈 내부 요소의 연관성 → 강할수록 좋음
  • 우연적 < 논리적 < 시간적 < 절차적 < 교환(통신)적 < 순차적 < 기능적(최고)
  • 결합도(Coupling) = 모듈 간 상호의존도 → 약할수록 좋음
  • 자료(최선) < 스탬프 < 제어 < 외부 < 공통 < 내용(최악)
함정: "모듈 간 상호의존 정도"=결합도(응집도 아님). "가장 좋은 = 기능적 응집도 / 자료 결합도". 내용 결합도=다른 모듈 내부를 직접 참조(최악). 모듈은 응집도↑ 결합도↓로 설계.

언어 기초 — 연산자 우선순위 & 제어문

우선순위 높→낮: 단항 → 산술 → 시프트 → 관계 → 비트 → 논리 → 삼항 → 대입
  • 단항(!, ~, ++, --, 부호) > 산술(*,/,% > +,-) > 시프트(<<,>>) > 관계(<,>,<=,>=) > 동등(==,!=) > 비트(& > ^ > |) > 논리(&& > ||) > 삼항(?:) > 대입(=)
  • 결합방향: 대부분 좌→우, 단항·삼항·대입은 우→좌
  • 제어문: 선택(if / switch-case), 반복(for / while / do-while), 분기(break / continue / goto)
삼항 연산자 단골: k = (i>j) ? (i-j) : (i+j); → i>j 참이면 i-j, 거짓이면 i+j. if-else와 1:1 대응시켜 읽을 것.

⭐ CPU 스케줄링 — 선점 vs 비선점

  • 선점(Preemptive): 실행 중 뺏김
  • RR(라운드로빈, 타임슬라이스)
  • SRT(최소잔여시간)
  • 다단계 큐 / 다단계 피드백 큐
  • 비선점(Non-preemptive): 끝까지 실행
  • FCFS(선입선출) · SJF(최단작업)
  • HRN · 우선순위 · 기한부

상호배제 · 임계구역

  • 임계구역(Critical Section): 한 번에 한 프로세스만 접근 가능한 공유자원 영역

⭐ SW 생명주기 모델

폭포수(Waterfall)가장 오래됨·선형 순차. 단계 완료 후 다음, 요구변경 어려움. (타당성→계획→분석→설계→구현→테스트→유지보수)
프로토타입시제품으로 요구 확인, 폭포수 단점 보완
나선형(Spiral, Boehm)위험분석 중심·반복. 대규모 적합
반복점진 / V모델V모델=폭포수 확장, 단계별 테스트 대응
나선형 4활동: 수 분 개 평 = 계획수립 → 위험분석 → 개발·검증 → 고객평가 (반복)
함정: 나선형은 매 주기마다 위험분석(마지막 한 번 아님), 여러 번 반복(한 번 아님). "각 단계 결과 확인 후 진행"은 폭포수/애자일 아님.

⭐ 비용 산정 (하향식 vs 상향식 · 최다급)

  • 하향식: 델파이(전문가 다수 익명 조정) · 전문가 감정
  • 상향식: LOC · FP · COCOMO · Putnam
  • FP(기능점수): 입력·출력·질의·파일·인터페이스 5요소 가중치 (클래스 인터페이스 아님)
  • Putnam: Rayleigh-Norden 노력분포곡선. 자동화도구 SLIM

COCOMO 유형 (5만줄 문제 거의 매회)

유형규모예시
조직형 Organic5만 라인 ↓일괄처리·과학기술계산·업무용 중소
반분리형 Semi-detached30만 라인 ↓컴파일러·인터프리터·트랜잭션 처리
내장형 Embedded30만 라인 ↑미사일·실시간·대형 OS
COCOMO=Boehm 제안, LOC(원시코드 라인수) 기반, 결과는 man-month(인월). 종류: 기본(Basic)·중간(Intermediate)·상세(Detailed). "object/dynamic/function 3모드"는 럼바우이지 COCOMO 아님(함정).
📐 LOC 계산 공식 (계산문제 단골)
  • 노력(인월) = LOC ÷ 1인당 월생산성  =  생산성 = LOC ÷ (인원 × 기간)
  • 개발기간 = 노력(인월) ÷ 투입인원
  • 예: 50000줄 ÷ 200 = 250인월, ÷ 10명 = 25개월  /  36000 ÷ (6×300) = 20개월

일정 관리 (WBS · PERT · CPM · Gantt)

WBS작업 분해 구조(업무를 계층적으로 세분)
PERT불확실·3점 추정(비관·낙관·기대). 작업 순서·상호관계 네트워크. 임계경로·자원할당 표시

프로세스 성숙도 (CMM/CMMI · SPICE)

CMMI 5단계 "초 관 정 정 최"
  1. 초기(Initial)
  2. 관리(Managed)
  3. 정의(Defined)
  4. 정량적 관리(Quantitatively Managed)
  5. 최적화(Optimizing)
SPICE = ISO/IEC 15504
  • 프로세스 수행능력 6수준(0~5)
  • 0 불완전 · 1 수행 · 2 관리 · 3 확립 · 4 예측 · 5 최적화
  • "수준 7"은 없음(함정)
CMMI/SPICE는 프로세스 평가·개선 표준. ISO 12207=SW 생명주기 프로세스(기본: 획득·공급·개발·운영·유지보수). 코드 오류: 생략(Omission)·필사(Transcription)·전위(Transposition, 자리바뀜)·이중전위.
