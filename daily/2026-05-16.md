## 🎯 오늘의 기출 8문제

### 문제 1 [유사 기출: 정보처리기사 필기 — 형상관리(SCM) 절차 반복 출제 정형]
소프트웨어 형상관리(SCM, Software Configuration Management)의 절차에 해당하지 않는 것은?

① 형상 식별(Configuration Identification)
② 형상 통제(Configuration Control)
③ 형상 감사(Configuration Audit)
④ 형상 컴파일(Configuration Compile)

---

### 문제 2 [유사 기출: 정보처리기사 필기 — 데이터 교환 포맷 JSON/XML 반복 출제 정형]
다음에서 설명하는 데이터 교환 포맷으로 가장 적절한 것은?

> 속성-값(attribute-value) 쌍과 배열 자료형을 사용하는 경량(lightweight)의 데이터 교환 형식으로, 사람이 읽고 쓰기 쉬우며 비동기 통신(AJAX)에서 XML을 대체하는 용도로 널리 쓰인다.

① XML(eXtensible Markup Language)
② JSON(JavaScript Object Notation)
③ CSV(Comma-Separated Values)
④ YAML(YAML Ain't Markup Language)

---

### 문제 3 [유사 기출: 정보처리기사 필기 — REST 특성 반복 출제 정형]
REST(REpresentational State Transfer) 방식에 대한 설명으로 옳지 않은 것은?

① 자원(Resource)을 URI(Uniform Resource Identifier)로 표현한다.
② HTTP Method(GET, POST, PUT, DELETE)를 이용해 자원에 대한 행위를 표현한다.
③ 서버는 클라이언트의 상태(session)를 항상 유지해야 하는 Stateful 구조를 원칙으로 한다.
④ 자원의 상태(표현)를 주고받으며, JSON·XML 등의 형식으로 데이터를 전달한다.

---

### 문제 4 [유사 기출: 정보처리기사 필기 — 형상관리 도구(Git/SVN) 특징 반복 출제 정형]
형상관리(버전 관리) 도구에 대한 설명으로 옳지 않은 것은?

① Git은 분산형(Distributed) 버전 관리 시스템으로, 각 개발자가 로컬에 전체 저장소 이력의 복제본을 가진다.
② SVN(Subversion)은 중앙 집중형(Centralized) 버전 관리 시스템이다.
③ CVS(Concurrent Versions System)는 초기 중앙 집중형 버전 관리 도구이다.
④ Git은 중앙 서버에 연결되어 있을 때만 커밋(commit)이 가능한 온라인 전용 도구이다.

---

### 문제 5 [유사 기출: 정보처리기사 필기 — 인터페이스 보안 프로토콜 반복 출제 정형]
인터페이스 보안을 위한 보안 프로토콜과 그 동작 계층(Layer)의 연결이 옳지 않은 것은?

① IPSec(IP Security) — 네트워크 계층(Network Layer)
② SSL/TLS(Secure Socket Layer / Transport Layer Security) — 전송 계층(Transport Layer)
③ S-HTTP(Secure-HTTP) — 응용 계층(Application Layer)
④ IPSec — 응용 계층(Application Layer)

---

### 문제 6 [유사 기출: 정보처리기사 필기 — 소프트웨어 패키징/DRM 반복 출제 정형]
디지털 저작권 관리(DRM, Digital Rights Management)에 대한 설명으로 옳지 않은 것은?

① 콘텐츠의 불법 복제·유통을 막고 사용 권한을 통제하기 위한 기술이다.
② 패키저(Packager)는 콘텐츠를 메타데이터와 함께 배포 단위로 묶는 역할을 한다.
③ 클리어링 하우스(Clearing House)는 라이선스 발급과 사용 권한·결제 관리를 담당한다.
④ DRM은 콘텐츠의 무결성만 보장할 뿐, 사용자별 사용 권한(횟수·기간)은 제어하지 못한다.

---

### 문제 7 [유사 기출: 정보처리기사 필기 — 미들웨어 종류 반복 출제 정형]
다음에서 설명하는 미들웨어(Middleware)로 가장 적절한 것은?

> 클라이언트가 원격에 있는 서버의 함수(프로시저)를 마치 로컬 함수처럼 호출할 수 있도록 지원하는 미들웨어로, 분산 환경에서 프로그램 간 호출을 처리한다.

① DBMS(Database Management System)
② RPC(Remote Procedure Call)
③ MOM(Message Oriented Middleware)
④ WAS(Web Application Server)

---

### 문제 8 [유사 기출: 정보처리기사 필기 — 릴리즈 노트/AJAX 반복 출제 정형]
인터페이스 구현 및 소프트웨어 패키징에 대한 설명으로 옳지 않은 것은?

① AJAX(Asynchronous JavaScript and XML)는 웹 페이지 전체를 갱신하지 않고 비동기로 서버와 데이터를 주고받는 기법이다.
② 릴리즈 노트(Release Note)는 최종 사용자에게 전달되는 문서로, 버전·수정 내역·알려진 버그 등을 기록한다.
③ 소프트웨어 패키징은 개발자 중심으로 수행하며, 사용자의 실행 환경은 고려하지 않는다.
④ 인터페이스 명세서는 송수신 데이터의 식별, 데이터 형식, 통신 방식 등을 정의한 문서이다.

## 💡 상세 해설 (자기 설명과 대조)

### 문제 1
정답: ④ 형상 컴파일(Configuration Compile)
- 제공 해설: 형상관리(SCM)의 4대 절차는 형상 식별(Identification) → 형상 통제(Control) → 형상 감사(Audit) → 형상 기록(Status Accounting, 상태 보고)이다. "형상 컴파일"이라는 절차는 존재하지 않으며, 컴파일은 형상관리 절차가 아닌 빌드 작업에 해당한다.
- 오답 이유 (실제 절차들):
  - ① 형상 식별: 형상 관리 대상(베이스라인)을 정의하고 이름·번호를 부여.
  - ② 형상 통제: 변경 요청을 검토·승인하고 변경을 관리(변경 통제 위원회 CCB).
  - ③ 형상 감사: 변경이 요구사항대로 올바르게 반영되었는지 검증·확인.

### 문제 2
정답: ② JSON(JavaScript Object Notation)
- 제공 해설: 속성-값 쌍과 배열을 사용하는 경량 데이터 교환 형식 = JSON. JavaScript 객체 문법에서 유래했으며, AJAX에서 XML을 대체해 비동기 통신 데이터 포맷으로 널리 쓰인다.
- 오답 이유:
  - ① XML: 태그(tag) 기반 마크업 언어. 구조 표현력은 크지만 태그 중복으로 JSON보다 무겁다.
  - ③ CSV: 쉼표로 값을 구분하는 단순 표 형식. 속성-값 쌍·중첩 구조 표현에는 부적합.
  - ④ YAML: 들여쓰기(indent) 기반 포맷. 사람이 읽기 쉬우나 "속성-값 쌍과 배열을 사용하는 JS 유래 경량 포맷"이라는 설명의 직접 정의에는 JSON이 더 정확.

### 문제 3
정답: ③ (옳지 않은 것)
- 제공 해설: REST의 핵심 원칙 중 하나는 무상태성(Stateless)이다. 즉 서버는 클라이언트의 상태(session)를 유지하지 않으며, 각 요청은 처리에 필요한 모든 정보를 포함해야 한다. ③은 Stateful이 원칙이라 했으므로 잘못된 설명이다.
- 오답 이유 (옳은 보기들):
  - ① 자원을 URI로 표현 — 옳음(REST의 자원 식별 원칙).
  - ② HTTP Method로 행위 표현 — 옳음(GET/POST/PUT/DELETE).
  - ④ 자원의 표현(상태)을 JSON·XML로 주고받음 — 옳음.

### 문제 4
정답: ④ (옳지 않은 것)
- 제공 해설: Git은 분산형(Distributed) 도구로, 각 개발자가 로컬에 전체 저장소 복제본을 갖기 때문에 중앙 서버에 연결되지 않은 오프라인 상태에서도 커밋(commit)이 가능하다. "온라인 전용"이라는 ④는 명백히 거짓이다.
- 오답 이유 (옳은 보기들):
  - ① Git은 분산형, 로컬에 전체 이력 복제본 보유 — 옳음.
  - ② SVN은 중앙 집중형 — 옳음.
  - ③ CVS는 초기 중앙 집중형 버전 관리 도구 — 옳음.

### 문제 5
정답: ④ (옳지 않은 것)
- 제공 해설: IPSec(IP Security)은 네트워크 계층(Network Layer, IP 계층)에서 동작하는 보안 프로토콜이다. ④는 IPSec을 응용 계층이라 했으므로 잘못된 연결이다.
- 오답 이유 (옳은 연결들):
  - ① IPSec — 네트워크 계층 — 옳음.
  - ② SSL/TLS — 전송 계층 — 옳음.
  - ③ S-HTTP — 응용 계층 — 옳음(HTTP 메시지 단위 암호화).

### 문제 6
정답: ④ (옳지 않은 것)
- 제공 해설: DRM(Digital Rights Management)은 무결성 보장에 그치지 않고 사용자별 사용 권한(재생 횟수·기간·복사 허용 여부 등)을 라이선스로 통제하는 것이 핵심 목적이다. ④는 권한 제어를 못한다고 했으므로 거짓이다.
- 오답 이유 (옳은 보기들):
  - ① 불법 복제·유통 차단, 사용 권한 통제 기술 — 옳음.
  - ② 패키저는 콘텐츠를 메타데이터와 함께 배포 단위로 묶음(암호화 패키징) — 옳음.
  - ③ 클리어링 하우스는 라이선스 발급·사용 권한·결제 관리 담당 — 옳음.

### 문제 7
정답: ② RPC(Remote Procedure Call)
- 제공 해설: 원격 서버의 함수(프로시저)를 로컬 함수처럼 호출하도록 지원하는 미들웨어 = RPC. 분산 환경에서 프로그램 간 함수 호출을 처리한다.
- 오답 이유:
  - ① DBMS: 데이터베이스 접근을 처리하는 미들웨어(DB 연동). 원격 프로시저 호출과는 다름.
  - ③ MOM(Message Oriented Middleware): 메시지 큐 기반 비동기 메시지 전달 미들웨어.
  - ④ WAS(Web Application Server): 웹 기반 동적 콘텐츠/비즈니스 로직 처리 서버.

### 문제 8
정답: ③ (옳지 않은 것)
- 제공 해설: 소프트웨어 패키징은 개발자가 아닌 사용자(고객) 중심으로 수행하며, 사용자의 실행 환경(OS·하드웨어 등)을 반드시 고려해 배포 단위를 구성해야 한다. ③은 사용자 환경을 고려하지 않는다고 했으므로 거짓이다.
- 오답 이유 (옳은 보기들):
  - ① AJAX는 페이지 전체 갱신 없이 비동기로 데이터를 주고받는 기법 — 옳음.
  - ② 릴리즈 노트는 버전·수정 내역·알려진 버그 등을 기록한 최종 사용자 전달 문서 — 옳음.
  - ④ 인터페이스 명세서는 송수신 데이터·형식·통신 방식을 정의 — 옳음.

## 📚 역추적 참조 (사후 검증용 — 학습 자료 아님)

### 핵심 용어
| 용어 | 정의 |
|---|---|
| 형상관리(SCM) | 소프트웨어 변경 사항을 체계적으로 추적·통제하는 활동. 식별/통제/감사/기록의 4절차. |
| 베이스라인(Baseline) | 변경 통제의 기준이 되는 공식 승인된 형상 항목 집합. |
| Git | 분산형(Distributed) 버전 관리 시스템. 로컬 저장소에 전체 이력 보유, 오프라인 커밋 가능. |
| SVN(Subversion) | 중앙 집중형(Centralized) 버전 관리 시스템. 중앙 서버에 단일 저장소. |
| JSON | 속성-값 쌍·배열 기반 경량 데이터 교환 포맷. JS 객체 문법 유래. |
| XML | 태그 기반 마크업 언어. 구조 표현력 크나 상대적으로 무거움. |
| YAML | 들여쓰기(indent) 기반 데이터 직렬화 포맷. 설정 파일에 자주 사용. |
| CSV | 쉼표로 값을 구분하는 단순 표 형식. |
| REST | 자원을 URI로, 행위를 HTTP Method로 표현하는 무상태(Stateless) 아키텍처 스타일. |
| SOAP | XML 기반 메시지 규약. 표준이 엄격하고 메시지가 무거움. |
| AJAX | 비동기 JavaScript+XML. 페이지 전체 갱신 없이 서버와 데이터 교환. |
| IPSec | 네트워크(IP) 계층 보안 프로토콜. |
| SSL/TLS | 전송 계층 보안 프로토콜. |
| S-HTTP | 응용 계층에서 HTTP 메시지를 암호화하는 프로토콜. |
| DRM | 디지털 콘텐츠의 저작권 보호·사용 권한 통제 기술. |
| 미들웨어 | 클라이언트와 서버(또는 분산 시스템) 사이를 연결·중계하는 소프트웨어. |

### 핵심 개념 (사후 검증용)

**형상관리(SCM) 4대 절차**
1. 형상 식별(Configuration Identification): 관리 대상(형상 항목, 베이스라인)을 정의·명명.
2. 형상 통제(Configuration Control): 변경 요청 검토·승인/거부, 변경 통제 위원회(CCB) 운영.
3. 형상 감사(Configuration Audit): 변경이 요구대로 올바르게 반영됐는지 검증.
4. 형상 기록/상태 보고(Status Accounting): 형상 항목의 변경 이력·상태를 기록·보고.

**형상관리 도구 비교**
| 도구 | 유형 | 로컬 저장소 | 특징 |
|---|---|---|---|
| Git | 분산형 | O (전체 이력) | 오프라인 커밋·브랜치 자유, 빠름 |
| SVN | 중앙 집중형 | X (작업 사본만) | 중앙 단일 저장소, 순차 리비전 번호 |
| CVS | 중앙 집중형 | X | SVN 이전 세대, 파일 단위 버전 관리 |

**데이터 교환 포맷 비교**
- XML: `<tag>값</tag>` 태그 기반, 구조 명확하나 무거움.
- JSON: `{"key": "value"}` 속성-값 쌍·배열, 경량, AJAX 표준.
- YAML: 들여쓰기 기반, 사람이 읽기 쉬움, 설정(config)에 강세.
- CSV: `a,b,c` 쉼표 구분, 표 형태 단순 데이터.

**REST 핵심 특성**
- 자원(Resource): URI로 식별.
- 행위: HTTP Method(GET/POST/PUT/DELETE).
- 표현(Representation): JSON·XML 등으로 자원 상태 전달.
- 무상태(Stateless): 서버는 클라이언트 세션을 유지하지 않음.
- 균일 인터페이스(Uniform Interface), 캐시 가능(Cacheable), 계층 구조(Layered System).

**REST vs SOAP**
| 항목 | REST | SOAP |
|---|---|---|
| 데이터 형식 | JSON·XML 등 자유 | XML 고정 |
| 표준/규약 | HTTP 기반, 가벼움 | 엄격한 표준(WS-*), 무거움 |
| 상태 | Stateless | 상태 유지 가능 |

**인터페이스 보안 프로토콜 — 계층**
| 프로토콜 | 풀네임 | 계층 |
|---|---|---|
| IPSec | IP Security | 네트워크 계층 |
| SSL/TLS | Secure Socket Layer / Transport Layer Security | 전송 계층 |
| S-HTTP | Secure-HTTP | 응용 계층 |

**DRM 구성요소**
- 콘텐츠 제공자(Contents Provider) / 분배자(Distributor)
- 패키저(Packager): 콘텐츠를 메타데이터와 함께 암호화·패키징.
- 클리어링 하우스(Clearing House): 라이선스 발급·사용 권한·결제 관리.
- DRM 컨트롤러(Controller): 사용자 단말에서 사용 권한을 통제·실행.

**미들웨어 종류**
- DBMS: 데이터베이스 연동 미들웨어.
- RPC(Remote Procedure Call): 원격 프로시저를 로컬 함수처럼 호출.
- MOM(Message Oriented Middleware): 비동기 메시지 큐 기반 전달.
- TP-Monitor(Transaction Processing Monitor): 트랜잭션 처리·관리.
- ORB(Object Request Broker): 분산 객체 간 통신(CORBA).
- WAS(Web Application Server): 웹 기반 동적 처리·비즈니스 로직 수행.

**소프트웨어 패키징·릴리즈 노트**
- 패키징: 사용자(고객) 중심, 실행 환경 고려, 배포 단위 구성.
- 릴리즈 노트 구성: 헤더(제품명·버전·날짜), 개요, 목적, 문제 요약, 수정·개선 내역, 알려진 버그(known issues), 사용자 영향도, 연락처 등.
