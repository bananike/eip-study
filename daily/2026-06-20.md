### 문제 2 [유사 기출: 정처기 필기 2021년 2회 · 2023년 1회 · 2024년 3회 반복 — DELETE/TRUNCATE/DROP 비교 정형 패턴]
다음 SQL 명령어에 대한 설명으로 가장 옳지 **않은** 것은?

① **DROP TABLE 학생;** — 테이블 구조와 데이터를 모두 제거한다.
② **TRUNCATE TABLE 학생;** — 모든 행을 삭제하고 테이블 구조는 유지하며, ROLLBACK으로 복구할 수 없다(DDL).
③ **DELETE FROM 학생 WHERE 학번 = 1001;** — WHERE 조건에 해당하는 행만 삭제하며, ROLLBACK으로 복구 가능하다(DML).
④ **TRUNCATE TABLE 학생;** 명령은 **DML**에 속하며, COMMIT 없이 자동으로 트랜잭션이 종료되지 않는다.

### 문제 3 [유사 기출: 정처기 필기 2020년 2회 · 2022년 3회 · 2024년 1회 반복 — SELECT 절 작성 순서 정형 패턴]
SQL의 **SELECT 문 작성(기술) 순서**가 옳게 나열된 것은? (실행 순서가 아닌 SQL 문장 작성 순서)

① SELECT - WHERE - FROM - GROUP BY - HAVING - ORDER BY
② **SELECT - FROM - WHERE - GROUP BY - HAVING - ORDER BY**
③ FROM - WHERE - SELECT - GROUP BY - HAVING - ORDER BY
④ SELECT - FROM - WHERE - HAVING - GROUP BY - ORDER BY

### 문제 4 [유사 기출: 정처기 필기 2021년 1회 · 2023년 2회 · 2025년 1회 반복 — WHERE vs HAVING 정형 패턴]
SQL의 **WHERE 절**과 **HAVING 절**에 대한 설명으로 가장 옳은 것은?

① WHERE는 그룹화 후 그룹을 필터링하고, HAVING은 그룹화 전 행을 필터링한다.
② WHERE 절에는 집계함수(SUM, AVG 등)를 사용할 수 있고, HAVING은 사용할 수 없다.
③ **WHERE는 그룹화 전 개별 행을 필터링(집계함수 사용 불가), HAVING은 GROUP BY 후 그룹을 필터링(집계함수 사용 가능)한다.**
④ WHERE와 HAVING은 기능이 완전히 동일하며 상호 대체 가능하다.

### 문제 5 [유사 기출: 정처기 필기 2020년 1·4회 · 2022년 2회 · 2024년 3회 반복 — JOIN 종류 정형 패턴]
다음 JOIN에 대한 설명 중 가장 옳지 **않은** 것은?

① **INNER JOIN** — 두 테이블의 조인 조건을 모두 만족하는 행만 결과에 포함한다(교집합).
② **LEFT OUTER JOIN** — 왼쪽 테이블의 모든 행을 결과에 포함하고, 오른쪽에 매칭되는 행이 없으면 NULL로 채운다.
③ **FULL OUTER JOIN** — 두 테이블의 모든 행을 결과에 포함하고, 매칭되지 않는 부분은 NULL로 채운다(합집합).
④ **CROSS JOIN** — 두 테이블의 조인 조건을 만족하는 행만 모두 결합하며, 결과 행 수는 항상 두 테이블 행 수의 **합(m+n)** 과 같다.

### 문제 6 [유사 기출: 정처기 필기 2021년 3회 · 2023년 3회 · 2025년 2회 반복 — 서브쿼리 연산자 정형 패턴]
SQL 서브쿼리에서 사용되는 **다중행(Multiple-Row) 연산자**에 해당하지 **않는** 것은?

① IN
② ANY (또는 SOME)
③ EXISTS
④ **= (등호)**

### 문제 7 [유사 기출: 정처기 필기 2020년 3회 · 2022년 1회 · 2024년 2회 반복 — 뷰(View) 특성 정형 패턴]
관계형 데이터베이스의 **뷰(View)** 에 대한 설명으로 가장 옳지 **않은** 것은?

① 뷰는 물리적으로 데이터를 저장하지 않는 **가상 테이블(Virtual Table)** 이다.
② 뷰는 CREATE VIEW로 생성하고 DROP VIEW로 제거하며, **ALTER VIEW**로 컬럼 구조를 직접 변경할 수 있다.
③ 뷰가 그룹함수(SUM, AVG), DISTINCT, JOIN 등을 포함하면 일반적으로 **INSERT·UPDATE·DELETE가 제한**된다.
④ 뷰는 보안성(원본 테이블의 일부 컬럼만 노출), 독립성, 편의성을 제공한다.

### 문제 8 [유사 기출: 정처기 필기 2021년 2회 · 2023년 1회 · 2025년 1회 반복 — 인덱스(Index) 특성 정형 패턴]
관계형 데이터베이스의 **인덱스(Index)** 에 대한 설명으로 가장 옳지 **않은** 것은?

① 인덱스는 검색(SELECT) 속도를 향상시키지만, 인덱스 자체의 유지 관리로 **INSERT·UPDATE·DELETE 성능은 저하**될 수 있다.
② **B-Tree 인덱스**는 범위 검색(BETWEEN, <, >)에 효율적이며 RDBMS의 기본 인덱스 구조로 가장 널리 쓰인다.
③ **Hash 인덱스**는 등가 비교(=)에 매우 빠르지만 **범위 검색에는 적합하지 않다**.
④ 인덱스는 별도 저장 공간을 차지하지 **않으며**, 무조건 많이 만들수록 모든 쿼리 성능이 향상된다.

## 🔍 오답 역추적 개념 발췌

### 1. SQL 3분류 — 두음 "정·조·제"
| 분류 | 두음 | 명령어 | 역할 |
|---|---|---|---|
| **DDL**(Data Definition) | 정의 | CREATE / ALTER / DROP / **TRUNCATE** / RENAME | 스키마·객체(테이블·뷰·인덱스) 정의 |
| **DML**(Data Manipulation) | 조작 | SELECT / INSERT / UPDATE / DELETE | 데이터 조회·삽입·수정·삭제 |
| **DCL**(Data Control) | 제어 | GRANT / REVOKE | 권한 부여·회수 |
| **TCL**(Transaction Control) | (트랜잭션) | COMMIT / ROLLBACK / SAVEPOINT | 트랜잭션 제어 (DCL에 포함하기도) |

> **함정 단골**: SELECT를 DDL에 끼워넣기 / TRUNCATE를 DML에 끼워넣기 / COMMIT을 DDL에 끼워넣기. **SELECT=DML, TRUNCATE=DDL, COMMIT=TCL(또는 DCL)** 1:1 매칭.

### 2. DROP vs TRUNCATE vs DELETE — 가장 빈출 비교표
| 항목 | DROP | TRUNCATE | DELETE |
|---|---|---|---|
| 분류 | **DDL** | **DDL** | **DML** |
| 대상 | 테이블 **구조+데이터** 모두 제거 | 데이터만 **일괄** 제거(구조 유지) | 조건(WHERE) 만족하는 행만 |
| WHERE 절 | X (테이블 통째로) | X (전체 삭제) | **O** (조건부) |
| 로깅·롤백 | 불가(Auto-COMMIT) | **불가**(Auto-COMMIT) | **가능**(트랜잭션) |
| 속도 | 빠름 | **매우 빠름** | 느림(행 단위 로깅) |

> **함정 단골**: "TRUNCATE는 DML" / "TRUNCATE는 ROLLBACK 가능" → 모두 X. **TRUNCATE = DDL = ROLLBACK 불가** 1자 매칭.

### 3. CASCADE vs RESTRICT — DROP/REVOKE 시 종속 처리
- **CASCADE** = 종속 객체·권한까지 **연쇄 제거**. 예: `DROP TABLE 학생 CASCADE;` → 학생을 참조하는 외래키, 뷰까지 함께 삭제.
- **RESTRICT** = 종속 객체·권한이 **존재하면 거부**(기본값인 DBMS 많음). 예: 외래키로 참조 중이면 DROP 실패.

> 참조 무결성 위반 옵션 4종(CASCADE/SET NULL/SET DEFAULT/NO ACTION)과 헷갈리지 말 것. **CASCADE/RESTRICT는 DDL·DCL의 옵션**, NO ACTION 등은 **외래키 제약의 ON DELETE/UPDATE 옵션**.

### 4. SELECT 절 — 작성 순서 vs 실행 순서
| 단계 | **작성 순서** (SQL 문법) | **실행 순서** (논리적 처리) |
|---|---|---|
| 1 | **SELECT** 컬럼 | **FROM** 테이블 |
| 2 | **FROM** 테이블 | **WHERE** 행 필터 |
| 3 | **WHERE** 행 필터 | **GROUP BY** 그룹화 |
| 4 | **GROUP BY** 그룹화 | **HAVING** 그룹 필터 |
| 5 | **HAVING** 그룹 필터 | **SELECT** 컬럼 추출 |
| 6 | **ORDER BY** 정렬 | **ORDER BY** 정렬 |

> 시험은 **작성 순서**를 묻는 게 일반적: **SELECT-FROM-WHERE-GROUP BY-HAVING-ORDER BY**. 두음 **"셀-프-웨-그-해-오"** 또는 그냥 키워드 그대로 외움.

### 5. WHERE vs HAVING — 그룹화 기준 분리
| 항목 | WHERE | HAVING |
|---|---|---|
| 시점 | GROUP BY **전** | GROUP BY **후** |
| 대상 | 개별 **행** 필터 | **그룹** 필터 |
| 집계함수 | **사용 불가** (SUM, AVG, COUNT 등) | **사용 가능** |
| 예시 | `WHERE 점수 >= 60` | `HAVING AVG(점수) >= 70` |

> 단골 함정: "WHERE에 집계함수 사용 가능" → X. "WHERE와 HAVING은 동일" → X. **HAVING은 GROUP BY 동반 시에만 의미**(GROUP BY 없이 HAVING만 쓰면 전체 1그룹으로 처리).

### 6. JOIN 5종 — 결과 행 수 & 매칭 규칙
| JOIN | 결과 | 매칭 규칙 |
|---|---|---|
| **INNER JOIN** | 교집합 | 조건 만족 행만 |
| **LEFT OUTER JOIN** | 좌 전체 + 우 매칭 | 좌측 모든 행 유지, 우측 매칭 없으면 NULL |
| **RIGHT OUTER JOIN** | 우 전체 + 좌 매칭 | 우측 모든 행 유지, 좌측 매칭 없으면 NULL |
| **FULL OUTER JOIN** | 합집합 (좌∪우) | 양쪽 모두 유지, 매칭 없으면 NULL |
| **CROSS JOIN** | **데카르트 곱 (m × n)** | 조건 없이 모든 조합 — 행 수 **m×n** |

> **함정 단골**: "CROSS JOIN 결과는 m+n" → **X**. CROSS는 **곱(m×n)** 으로 폭증. WHERE 없는 다중 FROM과 동일.

### 7. 서브쿼리 연산자 — 단일행 vs 다중행
| 구분 | 연산자 | 의미 |
|---|---|---|
| **단일행** | =, <>, >, <, >=, <= | 서브쿼리 결과가 **정확히 1행** |
| **다중행** | **IN** | 결과 집합에 포함 여부 |
| | **ANY / SOME** | **하나라도** 만족 (`> ANY` = 최솟값보다 크면 됨) |
| | **ALL** | **모두** 만족 (`> ALL` = 최댓값보다 커야 함) |
| | **EXISTS / NOT EXISTS** | 결과 행 **존재 여부**(true/false만 반환) |

> **함정 단골**: "=는 다중행 연산자" → **X** (단일행). "EXISTS는 단일행" → **X** (다중행). **ANY=하나라도 / ALL=모두** 의미 뒤바꿈 단골.

### 8. 뷰(View) — 가상 테이블 핵심 5가지
1. **물리 저장 X** — 정의(SELECT 문)만 카탈로그에 저장.
2. **CREATE VIEW**로 생성, **DROP VIEW**로 제거. **ALTER VIEW로 컬럼 구조 변경 불가** — 재정의는 `CREATE OR REPLACE VIEW`.
3. **INSERT·UPDATE 제한 케이스**: 그룹함수(SUM/AVG)·DISTINCT·GROUP BY·JOIN·UNION·서브쿼리·기본키 미포함·계산컬럼·DISTINCT 포함 시.
4. **장점**: 보안(원본 일부만 노출), 논리적 독립성, 사용 편의(복잡 쿼리 단순화).
5. **단점**: 인덱스 직접 생성 X(머티리얼라이즈드 뷰는 예외), 정의 변경 어려움, 성능 저하 가능.

> 시험 정형 함정: **"ALTER VIEW로 뷰 컬럼을 변경할 수 있다"** → **X**. 반드시 DROP 후 재생성 또는 `CREATE OR REPLACE`.

### 9. 인덱스(Index) — 4핵심 + 종류
| 핵심 | 설명 |
|---|---|
| 검색 ↑ | SELECT(특히 WHERE·JOIN) 성능 향상 |
| 변경 ↓ | INSERT/UPDATE/DELETE 시 인덱스도 갱신 → 성능 ↓ |
| 저장 공간 ↑ | 별도 자료구조 저장 |
| 카디널리티 高 컬럼 효율 | 중복 적은 컬럼(이메일·학번)이 효과적 |

| 종류 | 적합 시나리오 |
|---|---|
| **B-Tree** | 범위 검색(BETWEEN, <, >) — RDBMS 기본 |
| **Hash** | 등가 비교(=) — 범위 검색 부적합 |
| **Bitmap** | 카디널리티 **낮은** 컬럼(성별·국적) — DW에 강함 |
| Clustered | 데이터 자체를 키 순으로 저장 — 테이블당 1개 |
| Non-Clustered | 별도 인덱스 페이지 — 여러 개 가능 |

> **함정 단골**: "인덱스를 많이 만들수록 모든 성능 향상" → **X** (변경 성능 ↓). "인덱스는 저장 공간을 차지하지 않는다" → **X**. "Hash 인덱스가 범위 검색에 강하다" → **X** (B-Tree가 강함).

### 10. GRANT / REVOKE — 권한 부여·회수
- **GRANT** 권한리스트 ON 객체 TO 사용자 [**WITH GRANT OPTION**];
  - WITH GRANT OPTION: 받은 사용자가 다시 **재부여** 가능.
- **REVOKE** [GRANT OPTION FOR] 권한리스트 ON 객체 FROM 사용자 [**CASCADE | RESTRICT**];
  - CASCADE: 종속(재부여된) 권한까지 연쇄 회수.
  - RESTRICT: 종속 권한 존재 시 회수 거부.

## 💡 상세 해설

### 문제 1
**정답 ④ SELECT**
- **DDL(Data Definition Language)** = **스키마·객체(테이블/뷰/인덱스)의 정의·변경·제거** 명령어. 두음 **"정의"**.
- 표준 DDL: **CREATE / ALTER / DROP / TRUNCATE / RENAME** — 모두 객체 구조에 작용.
- **SELECT는 DML(Data Manipulation Language)** — 데이터 **조회**가 본질. INSERT/UPDATE/DELETE와 한 묶음.
- 시험 정형 함정: SELECT를 DDL에 끼워넣는 4지선다. **"SELECT = DML"** 1자 매칭으로 즉답.

### 문제 2
**정답 ④ TRUNCATE TABLE 학생; 명령은 DML에 속하며, COMMIT 없이 자동으로 트랜잭션이 종료되지 않는다.**
- **TRUNCATE는 DDL** — 데이터를 일괄 삭제하지만 본질은 테이블 자료 영역 초기화(자료 페이지 해제). **Auto-COMMIT**으로 자동 종료 → **ROLLBACK 불가**.
- ① **DROP TABLE**: 구조+데이터 모두 제거(DDL). ✔
- ② **TRUNCATE TABLE**: 데이터만 일괄 제거, 구조 유지, ROLLBACK 불가(DDL). ✔
- ③ **DELETE FROM … WHERE …**: 조건 행만 삭제, 트랜잭션 일부로 ROLLBACK 가능(DML). ✔
- 함정 단골: TRUNCATE를 DML로 착각 / ROLLBACK 가능으로 착각. **TRUNCATE = DDL = ROLLBACK 불가 = Auto-COMMIT** 1자 매칭.

### 문제 3
**정답 ② SELECT - FROM - WHERE - GROUP BY - HAVING - ORDER BY**
- SQL **문장 작성 순서**(문법) = SELECT - FROM - WHERE - GROUP BY - HAVING - ORDER BY.
- **실행(논리적 처리) 순서**는 다름: FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY. 시험에 둘 다 출제되므로 문제 지문 **"작성/기술"** vs **"실행/처리"** 키워드 구분 필수.
- ①은 SELECT가 WHERE보다 앞 → 정상이지만 WHERE-FROM 순서 뒤바뀜 함정.
- ③은 실행 순서를 작성 순서로 착각하게 만드는 함정.
- ④는 HAVING이 GROUP BY 앞 — **GROUP BY 없이 HAVING은 무의미**, 항상 GROUP BY 뒤.

### 문제 4
**정답 ③ WHERE는 그룹화 전 개별 행을 필터링(집계함수 사용 불가), HAVING은 GROUP BY 후 그룹을 필터링(집계함수 사용 가능)한다.**
- **WHERE** = GROUP BY **이전**에 동작 → 개별 **행** 필터 → 집계함수 **사용 불가**.
- **HAVING** = GROUP BY **이후**에 동작 → 그룹화된 결과(그룹) 필터 → 집계함수 **사용 가능**.
- 예: `SELECT 학과, AVG(점수) FROM 성적 WHERE 점수>=60 GROUP BY 학과 HAVING AVG(점수)>=70;` — WHERE는 행, HAVING은 그룹 평균을 필터.
- ① 시점 뒤바꿈, ② 집계함수 사용 가능 여부 뒤바꿈, ④ "완전 동일·대체 가능" → X (역할 분리).
- 정형 함정: **시점/대상/집계함수 사용여부** 3가지 중 1~2개 뒤바꾸기. 표 한 줄 외우면 즉답.

### 문제 5
**정답 ④ CROSS JOIN — 두 테이블의 조인 조건을 만족하는 행만 모두 결합하며, 결과 행 수는 항상 두 테이블 행 수의 합(m+n)과 같다.**
- **CROSS JOIN = 데카르트 곱(Cartesian Product)** — **조건 없이** 모든 조합 결합. 결과 행 수는 **m × n** (합 X, **곱** O).
- ① INNER JOIN: 교집합. ✔
- ② LEFT OUTER JOIN: 좌측 전체 + 우측 매칭(없으면 NULL). ✔
- ③ FULL OUTER JOIN: 양측 모두 + 매칭 없는 부분 NULL. ✔
- 함정 단골: **CROSS의 결과 수가 m+n** → X, **m×n**. "조건을 만족하는 행만" → X (조건 자체가 **없음**).
- 표준 4지선다: CROSS의 행 수 공식(m+n vs m×n) 또는 정의(조건 있음 vs 없음)에서 1개만 뒤바꿈.

### 문제 6
**정답 ④ = (등호)**
- **다중행 연산자**: 서브쿼리가 **여러 행** 반환할 때 사용 — **IN / ANY(SOME) / ALL / EXISTS** 4종.
- **단일행 연산자**: =, <>, >, <, >=, <= 6종. 서브쿼리 결과가 **정확히 1행**일 때만 사용 가능.
- ① IN — 다중행. ✔
- ② ANY (또는 SOME) — 다중행 ("**하나라도** 만족"). ✔
- ③ EXISTS — 다중행 (존재 여부, true/false만 반환). ✔
- 단골 함정: "=는 다중행" → X. "EXISTS는 단일행" → X. "ANY=모두 만족" → X (ANY=하나라도, ALL=모두).
- 정형 표현 즉답: **`> ANY (서브쿼리)`** = 서브쿼리 결과 중 **최솟값보다 크면** 참 / **`> ALL (서브쿼리)`** = **최댓값보다 커야** 참.

### 문제 7
**정답 ② 뷰는 CREATE VIEW로 생성하고 DROP VIEW로 제거하며, ALTER VIEW로 컬럼 구조를 직접 변경할 수 있다.**
- 표준 SQL에는 **ALTER VIEW로 뷰의 컬럼 구조를 변경하는 기능이 없음** — 재정의는 `CREATE OR REPLACE VIEW` 또는 DROP 후 재생성. (일부 DBMS의 ALTER VIEW는 권한/주석 변경 한정)
- ① 뷰는 가상 테이블(데이터 미저장, 정의만 카탈로그에). ✔
- ③ 그룹함수·DISTINCT·JOIN 등 포함 시 갱신 제한. ✔ — **기본키 미포함 뷰는 INSERT 불가**(NOT NULL 컬럼 값 채울 수 없음).
- ④ 뷰의 장점(보안·독립성·편의성). ✔
- 정형 함정 단골: **"ALTER VIEW로 컬럼 추가/변경 가능"** → **X**. 1자 매칭으로 즉답.

### 문제 8
**정답 ④ 인덱스는 별도 저장 공간을 차지하지 않으며, 무조건 많이 만들수록 모든 쿼리 성능이 향상된다.**
- 인덱스는 **별도 자료구조(B-Tree/Hash/Bitmap)** 를 저장 → **추가 저장 공간 필요**.
- 인덱스를 많이 만들면 **INSERT/UPDATE/DELETE 성능 저하**(인덱스도 함께 갱신해야 함) — "무조건 향상" **X**.
- ① 검색 ↑ / 변경 ↓ — 정확한 trade-off 설명. ✔
- ② B-Tree는 범위 검색 강함(균형 트리 구조, 인접 노드 탐색 효율) — RDBMS 기본. ✔
- ③ Hash는 정확히 일치(=)에 O(1) 빠름, 범위 검색은 해시 특성상 부적합. ✔
- 정형 함정 단골: **"인덱스 무조건 많을수록 좋음"** → X (변경 비용·저장 공간 증가). **"인덱스는 저장 공간 차지 X"** → X. 1자 매칭으로 즉답.
