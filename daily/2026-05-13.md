## 🔑 핵심 용어 (10개)

| 용어 | 한 줄 정의 | 핵심 키워드 |
|---|---|---|
| 요구공학(RE, Requirements Engineering) | 사용자 요구사항을 도출·분석·명세·확인하는 구조화된 활동 | 도출/분석/명세/확인 |
| 기능 요구사항(Functional Requirement) | 시스템이 "무엇(what)"을 해야 하는지 | 입출력·조회·인출·송금 |
| 비기능 요구사항(Non-Functional Requirement) | 시스템이 "어떻게(how well)" 동작해야 하는지 | 성능·보안·가용성·이식성 |
| 자료흐름도(DFD, Data Flow Diagram) | 데이터의 흐름과 변환을 도형으로 표현, 버블 차트라고도 함 | 프로세스(○)·자료흐름(→)·자료저장소(═)·단말(□) |
| 자료사전(DD, Data Dictionary) | DFD에 등장하는 자료를 정의하는 사전 | `=` 정의, `+` 연결, `{}` 반복, `[]` 선택, `()` 생략, `**` 주석 |
| 소단위 명세서(Mini-Spec) | DFD 최하위 처리(Process)의 처리 절차를 1~2쪽으로 기술 | 구조적 언어·표·결정표 |
| HIPO(Hierarchy + Input/Process/Output) | 모듈을 계층 구조로 표현한 하향식 문서화 도구 | 가시적 도표·총체적 도표·세부적 도표 |
| 정형 명세 기법(Formal Specification) | 수학적 기호·표기법으로 요구사항을 표현 | VDM·Z·Petri-Net·CSP |
| 비정형 명세 기법(Informal Specification) | 자연어·다이어그램으로 표현 | FSM·ER모델·SADT·Use Case |
| 요구사항 검토(Validation) | 명세서의 결함을 찾는 활동 | 동료 검토 / 워크스루 / 인스펙션 |

## 📖 핵심 개념 정리

### 1. 요구공학 4단계 (★최빈출★)
**"도-분-명-확"** 으로 암기.
1. **도출(Elicitation)** — 이해관계자 식별, 인터뷰·설문·브레인스토밍·델파이·롤플레잉·워크숍·프로토타이핑·유스케이스로 요구사항 수집
2. **분석(Analysis)** — 모호·충돌하는 요구를 명확화. 분류 → 개념 모델링 → 할당 → 협상 → 정형 분석
3. **명세(Specification)** — 체계적으로 문서화 (SRS, Software Requirements Specification). 정형/비정형 명세 기법
4. **확인(Validation)** — 요구사항이 올바른지 검증. 동료 검토·워크스루·인스펙션·프로토타이핑·요구사항 추적표(RTM)

### 2. 기능 vs 비기능 요구사항 비교
| 구분 | 기능 요구사항 | 비기능 요구사항 |
|---|---|---|
| 정의 | 시스템이 수행할 **기능** | 시스템 동작의 **품질 속성** |
| 관점 | What | How well |
| 예시 | 회원가입, 송금, 검색 | 응답 3초 이내, AES-256 암호화, 99.9% 가용성 |
| 분류 | 입출력·처리·인터페이스 기능 | 성능·보안·신뢰성·가용성·이식성·사용성·확장성·유지보수성 |

암기법: **비기능 = "이성가신유보확"** (이식성·성능·가용성·신뢰성·유지보수성·보안·확장성)

### 3. DFD 4가지 구성요소와 표기 (★빈출 함정★)
| 요소 | 표기 | 의미 |
|---|---|---|
| Process(처리) | **원(○)** 또는 둥근 사각형 | 자료 변환 |
| Data Flow(자료 흐름) | **화살표(→)** | 자료 이동 |
| Data Store(자료 저장소) | **평행선(═)** ← **삼각형 아님!** | 자료 보관 |
| Terminator(단말, 외부 실체) | **사각형(□)** | 시스템 경계 외부 |

**DFD의 특징**
- "자료 흐름 그래프" 또는 **버블(bubble) 차트**라고도 함
- 구조적 분석 기법에 사용
- ❌ **시간 흐름은 명확히 표현 불가** (시퀀스 다이어그램과 구별)
- 최하위 Process는 **소단위 명세서(Mini-Spec)** 를 가짐
- 배경도(Context Diagram, Level 0)에도 명칭·번호 부여

### 4. 자료사전(DD) 기호 (암기 필수)
| 기호 | 의미 | 예시 |
|---|---|---|
| `=` | is composed of (정의) | 회원 = 이름 + 생년월일 |
| `+` | and (연결) | 이름 + 주소 |
| `{ }` | iteration of (반복) | {주문상품} |
| `[ \| ]` | or (선택) | [현금 \| 카드] |
| `( )` | optional (생략 가능) | (중간이름) |
| `**` | comment (주석) | **회원 식별자** |

### 5. 명세 기법
- **정형(Formal)** — VDM, **Z**, Petri-Net, CSP, LOTOS → 수학적·정확·일관성 보장, 사용자 이해 어려움
- **비정형(Informal)** — FSM, ER 모델, SADT, Use Case, 결정표, 자연어 → 이해 쉬움, 작성자별 일관성 ↓

암기법: **"정형 = ZVPC(즈비피씨)"** = Z·VDM·Petri·CSP

### 6. CASE 도구 (요구사항 분석 자동화)
- **SADT**(SoftTech) — 블록 다이어그램 기반
- **SREM(RSL/REVS)**(TRW) — 실시간 시스템용
- **PSL/PSA**(미시간大) — 문제기술서 분석
- **TAGS** — 시스템 공학 전 주기 통합

### 7. 요구사항 검토 방법 비교
| 방법 | 핵심 |
|---|---|
| 동료 검토(Peer Review) | 작성자가 직접 설명, 동료가 들으며 결함 발견 |
| 워크스루(Walkthrough) | 명세서를 **사전 배포** 후 짧은 회의로 검토 |
| 인스펙션(Inspection) | 작성자 **제외**, 다른 전문가가 결함 검사 (가장 공식적) |

### 8. 요구사항이 갖춰야 할 특성 (SRS 품질 기준)
완전성, 일관성, 명확성, 검증 가능성, 추적성, 수정 용이성, 개발 후 이용성

## 💡 상세 해설

### 문제 1 해설 — 정답 ③
Data Store의 표기는 **평행선(═, 두 줄)** 또는 끝이 열린 직사각형이다. **삼각형은 ERD의 약한 개체 표기**로 혼동을 유도한 함정. Process는 원, Data Flow는 화살표, Terminator는 사각형이 맞다.

### 문제 2 해설 — 정답 ③
DFD는 **시간(순서) 흐름을 명확히 표현할 수 없다.** 시간 표현이 필요하면 시퀀스 다이어그램이나 상태 다이어그램을 사용한다. 나머지(버블 차트, 구조적 분석, 4요소 표기)는 모두 옳다.

### 문제 3 해설 — 정답 ④
"필요한 자료가 반드시 입력되어야 한다"는 **올바른 지침**이므로 보기 ①②③④ 중 작성지침이 아닌 것을 찾는 문제로 출제되며, 통상 ④번 보기로 "최하위 처리는 소단위명세서를 갖지 않아도 된다"와 같은 변형이 오답으로 출제된다(통합 빈출 변형 답: 자료 흐름을 임의로 분기시켜도 된다 류). 본 문제에서는 ④가 함정으로 처리된 회차로, **출력자료 산출에 입력 자료가 반드시 필요하다는 진술은 옳지만 작성 지침과 거리가 있다**고 본 회차 가답안 처리.
※ 회차에 따라 답안이 다를 수 있으니, "작성 지침이 아닌 것"의 키워드 — 즉 "임의 변경/생략 가능"이 들어간 보기가 오답임을 기억할 것.

### 문제 4 해설 — 정답 ④
비정형 명세기법은 자연어 기반이라 **작성자에 따라 결과가 달라 일관성과 완전성 검증이 어렵다.** 일관성·완전성 검증이 가능한 것은 수학 기반의 **정형 명세 기법**(Z, VDM 등)이다.

### 문제 5 해설 — 정답 ③
비기능 요구사항은 **시스템의 품질 속성(성능·보안·가용성·이식성 등)** 을 정의한다. ①②④는 모두 "무엇을 하는가"를 기술하는 기능 요구사항이다.

### 문제 6 해설 — 정답 ③
모듈을 **계층 구조**로 표현하는 도구는 **HIPO(Hierarchy + Input/Process/Output) Chart**. DFD는 자료 흐름, ERD는 개체 관계, Use Case는 액터-기능 관계를 표현한다.

### 문제 7 해설 — 정답 ②
요구공학 4단계는 "도-분-명-확", 즉 **도출(Elicitation) → 분석(Analysis) → 명세(Specification) → 확인(Validation)** 순.

### 문제 8 해설 — 정답 ④
정형 명세는 **수학적 표기**를 사용 — VDM, Z, Petri-Net, CSP, LOTOS. **Use Case는 비정형 명세** (자연어·다이어그램 기반).

### 문제 9 해설 — 정답 ③
애자일은 "계획에 따른 엄격한 단계"가 아니라 **변화에 유연하게 대응**한다. 보기 ③은 폭포수 모델의 특징.

### 문제 10 해설 — 정답 ②
객체와 속성, 클래스와 멤버는 **객체지향 분석(OOA, Object-Oriented Analysis)** 의 핵심 단위. FOA는 기능 중심(DFD), 자료 흐름 지향은 DFD 기반 구조 설계, 구조적 분석은 절차 중심이다.

### 문제 11 해설 — 정답 ③
자료사전에서 **반복**은 `{ }` (iteration of). `=`는 정의, `+`는 연결(and), `[ ]`는 선택(or).

### 문제 12 해설 — 정답 ②
자료사전에서 **선택**은 `[ | ]`. `( )`는 생략 가능(optional), `{ }`는 반복, `**`는 주석.

### 문제 13 해설 — 정답 ④
**JUnit은 단위 테스트 도구**로 요구사항 분석용 CASE가 아니다. SADT·SREM·PSL/PSA·TAGS가 대표적 요구사항 분석 자동화 도구.

### 문제 14 해설 — 정답 ③
**인스펙션(Inspection)** 은 작성자를 제외하고 별도 전문가가 체크리스트 기반으로 결함을 찾는 가장 공식적 검토. 동료 검토는 작성자가 직접 설명, 워크스루는 사전 배포 후 짧은 회의.

### 문제 15 해설 — 정답 ②
ㄴ(3초 이내 표시 = 성능), ㄷ(SHA-256 암호화 = 보안)이 비기능. ㄱ(카드 결제 가능)과 ㄹ(주문내역 조회)은 시스템이 수행하는 동작이므로 기능 요구사항.

### 문제 16 해설 — 정답 ②
**소단위 명세서(Mini-Spec)** 는 DFD 최하위 Process의 처리 절차를 1~2쪽으로 기술. 구조적 언어·결정표·결정 트리 등으로 작성.

### 문제 17 해설 — 정답 ③
유스케이스는 **외부 사용자 관점의 기능**을 표현하며, **시스템 내부 알고리즘은 다루지 않는다.** 내부 알고리즘은 활동 다이어그램·시퀀스 다이어그램 등이 표현.

### 문제 18 해설 — 정답 ③
요구사항 명세서는 오히려 **모호성을 제거**해야 한다 — 명확성(Unambiguity)이 핵심 품질 기준. 완전성·일관성·검증가능성·추적성 등이 SRS의 표준 품질 특성이다.

### 문제 19 해설 — 정답 ③
처리는 **반드시 입력 자료가 있어야 출력 자료를 산출**할 수 있다. "입력 없는 출력"은 DFD 균형 원칙 위반.

### 문제 20 해설 — 정답 ④
워크스루는 **요구사항 검토(Validation)** 기법이지 도출 기법이 아니다. 도출에는 인터뷰·설문·브레인스토밍·델파이·롤플레잉·워크숍·프로토타이핑·유스케이스가 포함된다.

### 문제 21 해설 — 정답 ④
요구사항 분석 단계는 분류·개념 모델링·할당·협상·정형 분석을 수행. **코드 컴파일**은 구현 단계의 활동.

### 문제 22 해설 — 정답 ③
**Z는 정형 명세 기법.** 결정표·FSM·Use Case는 모두 비정형 명세에 속한다.

### 문제 23 해설 — 정답 ④
"인수 테스트의 기준이 되며 사용자가 원하는 바와 일치하는지 검토"하는 단계는 **확인(Validation)** = 요구공학 4단계 중 마지막.

### 문제 24 해설 — 정답 ③
"회계 처리"는 시스템이 수행하는 업무 → **기능 요구사항**이다. 성능·보안·가용성은 모두 품질 속성이므로 비기능.

## 📌 시험 직전 1분 요약
- **요구공학 순서**: 도출 → 분석 → 명세 → 확인 (도-분-명-확)
- **기능 vs 비기능**: "무엇을 하는가" vs "성능·보안·가용성" 등 품질 속성
- **DFD 4요소**: Process(○) / Data Flow(→) / Data Store(**═ 평행선** — 삼각형 아님!) / Terminator(□). 시간 흐름 표현 ❌
- **자료사전**: `=`정의 `+`연결 `{}`반복 `[]`선택 `()`생략 `**`주석
- **정형 명세 = Z·VDM·Petri-Net·CSP** / 검토 = 동료·워크스루·**인스펙션(가장 공식적)**
