## 🎯 오늘의 기출문제 — 2025년 정처기 필기 20문제 (5과목 균등 · 회독 2/4 신규 비중복)

### 문제 1 [유사 기출: 2025년 1회 1과목 — GraphQL vs REST 정형 패턴]
GraphQL과 REST 아키텍처 비교에 대한 설명으로 옳지 않은 것은?
① REST는 리소스별로 다수의 엔드포인트를 사용하는 반면 GraphQL은 일반적으로 단일 엔드포인트(/graphql)를 사용한다.
② GraphQL은 클라이언트가 필요한 필드만 지정 조회할 수 있어 REST의 오버페칭(Over-fetching)·언더페칭(Under-fetching) 문제를 완화한다.
③ GraphQL은 스키마 우선(Schema-First) 개발·타입 시스템·인트로스펙션(Introspection)을 지원한다.
④ REST는 GET/POST/PUT/DELETE 등 HTTP 메서드를 활용하고 GraphQL은 오직 GET 메서드만 사용해야 하며 캐싱은 GraphQL이 REST보다 항상 유리하다.

### 문제 2 [유사 기출: 2025년 2회 1과목 — gRPC(HTTP/2 · ProtoBuf) 정형 패턴]
gRPC에 대한 설명으로 옳은 것은?
① gRPC는 HTTP/1.1 기반의 텍스트 프로토콜로 JSON을 기본 직렬화 포맷으로 사용한다.
② gRPC는 HTTP/2 기반이며 Protocol Buffers(ProtoBuf)를 IDL(인터페이스 정의 언어) 겸 직렬화 포맷으로 사용하고 단항·서버 스트리밍·클라이언트 스트리밍·양방향 스트리밍 4가지 통신 방식을 지원한다.
③ gRPC는 브라우저의 자바스크립트 코드에서 별도의 프록시 없이 자유롭게 호출 가능하다.
④ gRPC는 언어별 종속성이 강해 서버·클라이언트 모두 반드시 Java로 구현해야 한다.

### 문제 3 [유사 기출: 2025년 3회 1과목 — API 게이트웨이 정형 패턴]
API 게이트웨이(API Gateway)의 주요 역할로 옳지 않은 것은?
① 요청 라우팅(Routing) 및 서비스 디스커버리(Service Discovery)와 연동
② 인증(Authentication)·인가(Authorization)·Rate Limiting·Throttling 등 횡단 관심사(Cross-cutting Concern) 처리
③ 프로토콜 변환(예: REST ↔ gRPC)·응답 캐싱·로깅·모니터링
④ 마이크로서비스 내부 비즈니스 로직 실행 및 각 서비스의 데이터베이스 스키마 직접 관리

### 문제 4 [유사 기출: 2025년 1회 1과목 — 헥사고날/클린 아키텍처 · 이벤트 소싱/CQRS/BFF 정형 패턴]
아키텍처 패턴에 대한 설명으로 옳지 않은 것은?
① 헥사고날 아키텍처(Hexagonal · Ports and Adapters)는 도메인을 중심에 두고 외부(UI·DB·메시징)와의 결합을 포트(인터페이스)와 어댑터(구현)로 분리한다.
② 클린 아키텍처(Clean Architecture)의 의존성 규칙은 "의존성은 항상 안쪽(도메인)을 향한다"이며 바깥 계층이 안쪽 계층을 알 수 있지만 그 반대는 허용되지 않는다.
③ 이벤트 소싱(Event Sourcing)은 상태의 최종값만 저장하며 이벤트 로그·재현(Replay)·시간 여행(Time Travel) 조회는 지원하지 않는다.
④ CQRS(Command Query Responsibility Segregation)는 명령(쓰기)과 조회(읽기)의 모델을 분리하고, BFF(Backend For Frontend)는 클라이언트(웹·모바일)별로 최적화된 게이트웨이를 별도로 두는 패턴이다.

### 문제 5 [유사 기출: 2025년 2회 2과목 — 병행성(뮤텍스 · 세마포어 · 데드락 4조건) 정형 패턴]
병행성(Concurrency) 제어에 대한 설명으로 옳지 않은 것은?
① 뮤텍스(Mutex)는 상호 배제를 위한 이진 락으로, 소유권(Owner)이 있는 스레드만이 해제할 수 있다.
② 세마포어(Semaphore)는 카운팅 형태로 여러 자원의 동시 접근 제한이 가능하며, wait(P)·signal(V) 연산으로 제어한다.
③ 데드락 발생 4대 필요조건은 상호 배제(Mutual Exclusion)·점유 대기(Hold and Wait)·비선점(No Preemption)·환형 대기(Circular Wait)이며, 이 중 하나라도 깨면 데드락은 발생하지 않는다.
④ 락을 사용하지 않는 무잠금(Lock-free) 알고리즘은 원자적 연산(CAS 등)을 활용하며 항상 락 기반보다 성능이 우수하고 구현이 단순하다.

### 문제 6 [유사 기출: 2025년 3회 2과목 — 트라이(Trie) 정형 패턴]
트라이(Trie) 자료구조에 대한 설명으로 옳지 않은 것은?
① 문자열 집합을 저장하는 다진 트리로, 접두사(Prefix)를 공유하는 문자열들이 같은 경로를 사용한다.
② 길이 m의 문자열 검색을 O(m) 시간에 수행할 수 있어 사전(Dictionary)·자동완성·IP 라우팅(라딕스 트라이)에 활용된다.
③ 공간 효율을 위해 압축 트라이(Compressed Trie·Radix Tree·PATRICIA)로 확장할 수 있다.
④ 트라이의 시간 복잡도는 저장된 문자열의 개수 n에 비례하여 검색이 항상 O(n)이 소요된다.

### 문제 7 [유사 기출: 2025년 1회 2과목 — KMP · 라빈-카프 문자열 검색 정형 패턴]
문자열 검색 알고리즘에 대한 설명으로 옳지 않은 것은?
① 단순(Naive) 문자열 검색은 최악 O(n·m)이며, n은 텍스트 길이·m은 패턴 길이이다.
② KMP(Knuth-Morris-Pratt)는 실패 함수(Failure Function·부분 일치 테이블)를 전처리하여 텍스트 문자를 되돌리지 않고 O(n+m)에 검색한다.
③ 라빈-카프(Rabin-Karp)는 롤링 해시(Rolling Hash)를 사용하여 여러 패턴을 동시에 탐색하는 데 유리하며, 평균 O(n+m)·최악 O(n·m)이다.
④ 보이어-무어(Boyer-Moore)는 패턴의 왼쪽부터 검사하여 실패 시 오른쪽으로 한 칸만 이동하며, 실무에서 KMP보다 항상 느리다.

### 문제 8 [유사 기출: 2025년 2회 2과목 — TDD · BDD · Mutation Testing 정형 패턴]
테스트 방법론에 대한 설명으로 옳지 않은 것은?
① TDD(Test-Driven Development)는 Red(실패 테스트)→Green(최소 코드 통과)→Refactor(리팩토링) 사이클을 반복한다.
② BDD(Behavior-Driven Development)는 Given(사전 조건)-When(행위)-Then(결과) 시나리오로 요구사항을 표현하며 Cucumber·SpecFlow 등이 대표 도구이다.
③ Mutation Testing(변이 테스트)은 원본 코드에 인위적 결함(Mutant)을 주입한 뒤, 테스트가 그 결함을 감지(Kill)하는지로 테스트 스위트의 품질을 측정한다. PIT·Stryker가 대표 도구이다.
④ TDD의 목적은 커버리지 100% 달성 자체이며, 리팩토링 단계는 시간 부족 시 생략해도 무방하다.

### 문제 9 [유사 기출: 2025년 3회 3과목 — 데이터 웨어하우스(DW) 정형 패턴]
데이터 웨어하우스(Data Warehouse)의 특성(Inmon 정의)으로 옳지 않은 것은?
① 주제 지향적(Subject-Oriented) — 업무 프로세스가 아닌 주제 영역(고객·매출·상품 등) 중심으로 구성
② 통합적(Integrated) — 여러 원천 시스템의 이질적 데이터를 표준화·정제(ETL)하여 통합
③ 시계열적(Time-Variant) — 특정 시점의 스냅샷을 유지하며 이력(History) 데이터를 장기간 축적
④ 휘발성(Volatile) — 최신 트랜잭션 반영을 위해 데이터를 실시간으로 갱신·삭제하며 이력을 유지하지 않는다.

### 문제 10 [유사 기출: 2025년 1회 3과목 — 데이터 레이크 · 레이크하우스 정형 패턴]
데이터 레이크(Data Lake)와 레이크하우스(Lakehouse)에 대한 설명으로 옳지 않은 것은?
① 데이터 레이크는 정형·반정형·비정형 원시 데이터(Raw Data)를 그대로 저장하며 Schema-on-Read 방식을 취한다.
② 데이터 웨어하우스가 Schema-on-Write(사전 정형)를 요구하는 것과 대비되어, 데이터 레이크는 저장 시점에 스키마를 강제하지 않는다.
③ 레이크하우스(Lakehouse)는 데이터 레이크의 확장성과 데이터 웨어하우스의 트랜잭션·품질(ACID·스키마 강제)을 결합하며, Delta Lake·Apache Iceberg·Apache Hudi가 대표 구현이다.
④ 데이터 레이크는 저장 비용이 매우 높고 정형 데이터만 취급하므로 반정형(JSON)·비정형(이미지·로그)은 저장할 수 없다.

### 문제 11 [유사 기출: 2025년 2회 3과목 — OLTP vs OLAP 정형 패턴]
OLTP(Online Transaction Processing)와 OLAP(Online Analytical Processing) 비교로 옳지 않은 것은?
① OLTP는 짧은 트랜잭션·CRUD·정규화된 스키마를 특징으로 하며, 초당 트랜잭션(TPS)·응답 시간이 핵심 지표이다.
② OLAP는 대량 집계·분석·비정규화(스타·스노우플레이크 스키마)를 특징으로 하며, 쿼리 응답 시간·복잡도가 핵심 지표이다.
③ OLTP는 행 지향(Row-oriented) 저장이 유리하고, OLAP는 열 지향(Column-oriented) 저장이 유리하다.
④ OLAP는 짧은 실시간 CRUD 트랜잭션에 최적화되어 있어 OLTP의 상위 호환이며 OLTP를 대체하기 위한 시스템이다.

### 문제 12 [유사 기출: 2025년 3회 3과목 — 인메모리 DB(Redis · Memcached · HANA) 정형 패턴]
인메모리(In-Memory) DB에 대한 설명으로 옳은 것은?
① Redis는 문자열·리스트·해시·셋·정렬셋(Sorted Set)·스트림·비트맵 등 다양한 자료구조를 지원하며, RDB 스냅샷·AOF 로그로 영속성을 제공한다.
② Memcached는 다양한 자료구조와 영속성·복제·클러스터링을 기본 제공하는 반면, Redis는 단순 Key-Value 캐시 전용이다.
③ SAP HANA는 디스크 기반 저장만 지원하며 실시간 분석(HTAP)에는 부적합하다.
④ 인메모리 DB는 항상 RAM에만 저장하므로 어떠한 방식으로도 재시작 후 데이터를 복구할 수 없다.

### 문제 15 [유사 기출: 2025년 3회 4과목 — Python 데코레이터 정형 패턴]
다음 Python 코드의 출력 결과로 옳은 것은?
```python
def deco(fn):
    def wrapper(*args, **kwargs):
        result = fn(*args, **kwargs)
        return result * 2
```

**정답 ④** — WebSocket은 **풀 듀플렉스(Full-Duplex) 양방향 프로토콜**로 서버가 클라이언트로 능동 푸시 가능(HTTP 폴링과 대비되는 핵심 강점). "요청-응답 반이중·서버 능동 전송 불가"는 프로토콜 정의 정반대. **함정 키워드**: "WebSocket 반이중·서버 푸시 불가"는 존재 이유 자체 부정.

### 문제 17
**정답 ③** — 12-Factor 6번째 원칙은 **"프로세스(Processes) — 무상태(Stateless)로 실행"** 이다. **세션 상태는 반드시 외부 스토어(Redis·DB)에 저장**해야 하며, **로컬 파일 시스템·메모리는 신뢰할 수 없다**(스케일 아웃·재시작 시 유실). "로컬 파일에 상태 유지"는 12-Factor 철학 정반대. **함정 키워드**: "Stateful·로컬 파일 세션 유지"는 12-Factor 6번(Processes) 정면 부정 초빈출 함정.

### 문제 18
**정답 ④** — GitOps의 핵심은 **모든 변경이 Git PR·머지를 통해서만 이루어지고, 자동화된 컨트롤러(ArgoCD·Flux)가 이를 클러스터에 지속적으로 동기화**한다는 점. **수동 kubectl 직접 적용은 Drift 발생·GitOps 원칙 정면 위반**. **함정 키워드**: "수동 kubectl만·Git과 분리"는 GitOps 정의 완전 정반대.

### 문제 19
**정답 ④** — 에러 예산(Error Budget) = **1 − SLO** 로 정의되는 **"허용 가능한 실패의 총량"**. 예산이 **남으면 신기능·위험한 변경을 배포할 여유가 있다는 뜻**이고, **예산을 소진하면 신기능 배포를 중단하고 안정화에 집중**한다. "예산이 남으면 배포 중단"은 방향 정확히 정반대. **함정 키워드**: "에러 예산 = SLO 초과 여유·남으면 중단"은 개념 정반대 초빈출 함정.

### 문제 20
**정답 ④** — **쇼어(Shor) 알고리즘은 RSA·ECC 등 공개키 암호(정수 인수분해·이산 로그)를 다항 시간에 무력화**한다. **AES 대칭키는 쇼어의 영향을 직접 받지 않으며**, 그로버(Grover)로 유효 키 길이가 절반이 되는 정도(AES-256 → AES-128 수준)라 **키 길이 확장으로 대응 가능**. "쇼어가 AES 무력화·RSA/ECC는 그로버 영향 없어 안전"은 두 알고리즘의 타깃을 완전히 뒤바꾼 오답. **함정 키워드**: "쇼어=AES·그로버=RSA/ECC 무영향"은 양자 알고리즘 타깃 정반대 뒤바꾸기 초빈출 함정.

## 📊 자주 틀리는 함정 5가지 (2025년 회독 2/4 · 최신 아키텍처·통신·데이터·클라우드·양자 트렌드 영역 위주)

1. **GraphQL vs REST & gRPC & API 게이트웨이 경계** — GraphQL은 오버페칭 해결·단일 엔드포인트·**GET/POST 모두 가능**이며 **캐싱은 REST가 URL 단위로 오히려 유리**. gRPC = **HTTP/2 + ProtoBuf + 4가지 스트리밍**·브라우저는 gRPC-Web 필요. API 게이트웨이는 **횡단 관심사만 처리**, 비즈니스 로직·서비스 내부 DB 스키마 관리는 배제. 각 개념의 방향·강점·책임 경계를 정반대로 뒤바꾸는 함정.
2. **헥사고날·클린 아키텍처·이벤트 소싱·CQRS·BFF** — 헥사고날 = 도메인 중심·포트/어댑터. 클린 = **의존성은 항상 안쪽 도메인 향함**. **이벤트 소싱은 최종값이 아닌 이벤트 시퀀스 저장·Replay·시간 여행 필수**. CQRS = 명령·조회 모델 분리. BFF = 클라이언트별 게이트웨이. "이벤트 소싱이 최종값만 저장" 오답 초빈출.
3. **트라이·KMP·라빈-카프·보이어-무어 방향/복잡도** — **트라이 검색은 O(m)** (m=문자열 길이, n 무관). KMP는 **실패 함수·되돌림 없이 O(n+m)**. 라빈-카프는 **롤링 해시·다중 패턴 유리**. **보이어-무어는 오른쪽부터 검사·큰 점프·실무 최속**. "트라이 O(n)·보이어-무어 왼쪽부터·한 칸 이동"은 알고리즘 원리 정반대.
4. **DW 4대 특성·데이터 레이크·레이크하우스·OLTP vs OLAP·인메모리** — DW = 주제 지향·통합·시계열·**비휘발성**("휘발성·실시간 갱신" 완전 정반대). 데이터 레이크 = **정형·반정형·비정형 모두·저비용 스토리지·Schema-on-Read**. 레이크하우스 = 레이크 + 웨어하우스(Delta Lake·Iceberg·Hudi). **OLAP는 OLTP를 대체하지 못한다**(상호 보완). Redis는 다양 자료구조 + RDB/AOF 영속성(Memcached는 단순 캐시). 특성·능력·역할을 정반대로 뒤바꾸는 함정 반복 출제.
5. **12-Factor & GitOps & SRE 에러 예산 & 양자 알고리즘 타깃** — 12-Factor 6번 = **프로세스는 무상태(Stateless)·세션은 외부 스토어**("로컬 파일 상태 유지"는 정반대). GitOps는 **Git = SSoT·자동 동기화**(수동 kubectl은 정반대). **에러 예산이 남으면 배포 여유·소진 시 배포 중단**("남으면 중단"은 방향 정반대). **쇼어는 RSA/ECC 파괴·그로버는 대칭키 절반 축소**("쇼어=AES·그로버=RSA" 뒤바꾸기 초빈출). 최신 클라우드·SRE·양자 개념의 방향성을 정확히 정반대로 왜곡하는 함정 초빈출.
