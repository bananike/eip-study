### 문제 15 [유사 기출: 2023년 3회 4과목 — 세마포어 vs 뮤텍스 정형 패턴]
운영체제의 동기화 기법 중 **세마포어(Semaphore)** 와 **뮤텍스(Mutex)** 의 차이점으로 옳지 **않은** 것은?

① 세마포어는 정수 카운터를 가지지만 뮤텍스는 0/1(잠금/해제) 두 상태만 가진다.
② 뮤텍스는 소유(Ownership) 개념이 있어 잠근 스레드만 풀 수 있지만, 세마포어는 소유 개념이 없다.
③ 세마포어는 여러 개의 자원(N개)을 동시에 관리할 수 있지만, 뮤텍스는 하나의 자원에 대한 상호배제만 관리한다.
④ 세마포어와 뮤텍스는 모두 P(wait)·V(signal) 연산으로만 동작하며 기능적으로 동일하다.

## 💡 상세 해설

### 문제 1
**정답 ③**. **프레임워크(Framework) vs 라이브러리(Library) 결정적 차이 = 제어의 역전(IoC, Inversion of Control)**.
- **라이브러리**: 개발자 코드가 라이브러리 함수를 **호출**(개발자 → 라이브러리). 예) `Math.pow(2, 10)`, jQuery, lodash. **제어권은 개발자**에게 있음.
- **프레임워크**: 프레임워크가 개발자 코드를 **호출**(프레임워크 → 개발자). 예) Spring, Django, React 컴포넌트 라이프사이클, JUnit `@Test`. **제어권은 프레임워크**에게 있음. **"Don't call us, we'll call you"(할리우드 원칙)**.
- **미들웨어(Middleware)**: OS와 응용 사이에 위치하여 표준 통신·데이터 처리를 제공(WAS·MOM·DB미들웨어·RPC).
- 함정: ①/② 뒤바꾸기, ④ 라이선스 조건은 IoC 판별 기준 아님. **호출 방향(개발자↔코드) = IoC 여부**가 유일한 판별 기준.

### 문제 2
**정답 ② Information Expert(정보 전문가)**. GRASP(General Responsibility Assignment Software Patterns) **9패턴**:
- **① Creator(생성자)**: B가 A를 포함/사용/기록하면 B가 A 생성 책임
- **② Information Expert(정보 전문가)**: **정보를 가장 잘 아는 객체에게 책임 할당** — 정답
- **③ Controller(제어자)**: UI 이벤트를 도메인 계층에 위임하는 첫 번째 객체
- **④ Low Coupling(낮은 결합도)** / **⑤ High Cohesion(높은 응집도)**
- **⑥ Polymorphism(다형성)**: 타입 기반 분기 대신 다형성
- **⑦ Pure Fabrication(순수 조작)**: 도메인에 없는 인공 클래스로 결합도↓·응집도↑ 확보(예: 서비스·리포지토리)
- **⑧ Indirection(간접화)**: 중간자를 두어 결합도 완화(예: 어댑터)
- **⑨ Protected Variations(변경 보호)**: 변경 가능한 부분을 인터페이스·추상화로 캡슐화(OCP와 유사)

- 함정: **Creator vs Information Expert 뒤바꾸기 초빈출**. "생성 책임 = Creator" / "처리 책임 = Info Expert". Controller는 UI-도메인 경계, Pure Fabrication은 인공 클래스 도입. **GOF 23 패턴과 다른 체계**(GRASP는 책임 할당 원칙).

### 문제 3
**정답 ④**. CBD(Component Based Development)의 특징:
- **컴포넌트(Component)**: 독립적으로 배포 가능한 실행 단위. **바이너리 형태로 배포 가능**(소스 공개 X). 인터페이스만 공개.
- **CBD 프로세스**: 요구분석 → 컴포넌트 식별 → 아키텍처 설계 → 컴포넌트 개발/획득 → 조립·통합 → 테스트.
- **장점**: 재사용성↑, 개발 생산성↑, 유지보수↓, 표준 인터페이스로 결합도↓, 병렬 개발 가능.
- **핵심 원칙**: **"블랙박스(Black Box) 재사용"** — 내부 구현 몰라도 인터페이스로 사용. 소스 공개 강제하지 않음(예: COM/DCOM·EJB·.NET Assembly).

- 함정: **④ 소스 공개 강제는 CBD의 특징이 아니라 오히려 반대**. CBD는 블랙박스 재사용을 지향하므로 소스 공개 불필요. **오픈소스 재사용과 CBD는 별개 개념**. ①~③은 모두 CBD 정통 특징.

### 문제 4
**정답 ④ 재공학(Re-engineering)**. 소프트웨어 재사용 4대 방법:
- **① 합성(Composition) 중심 재사용**: 기존 컴포넌트를 **부품 조립**하듯 결합(GUI 위젯, 라이브러리 함수 조합)
- **② 생성(Generation) 중심 재사용**: 응용 프로그램 생성기(코드 생성기, 스캐폴딩)로 재사용
- **③ 재구성(Restructuring)**: 프로그램 **동작을 유지하면서 구조만 개선**(리팩토링·모듈화)
- **④ 재공학(Re-engineering)**: **기존 시스템의 인터페이스는 유지하되 내부 구현을 새로 개발**하여 성능·품질 개선. 프로세스 = 분석(Analysis) → 재구성(Restructuring) → 역공학(Reverse Engineering) → 이식(Migration). 정답
- 관련: **역공학(Reverse Engineering)** = 실행 파일→소스/명세 복원 / **정공학(Forward Engineering)** = 명세→구현(일반적 개발)

- 함정: **재구성 vs 재공학 뒤바꾸기 초빈출**. "구조만 개선 = 재구성" / "성능·품질 개선(내부 다시 만듦) = 재공학". 재공학은 인터페이스 유지가 핵심(외부에서 봤을 때 호환).

### 문제 5
**정답 ② A → B → D → E → F → C**. **DFS(Depth First Search) 스택(재귀) 트레이싱** — 인접 리스트 사전순:
- A=[B,C], B=[A,D,E], C=[A,F], D=[B], E=[B,F], F=[C,E]
- Visit A → 인접 B,C 중 사전순 첫 B
- Visit B → 인접 A(방문완료), D, E 중 D 먼저
- Visit D → 인접 B(방문완료) → 되돌아옴(pop D) → B로 복귀
- Visit E → 인접 B(방문완료), F
- Visit F → 인접 C, E(방문완료) → C 방문
- Visit C → 인접 A(방문완료), F(방문완료) → 종료
- 최종 순서: **A → B → D → E → F → C**

**BFS 순회(참고)** 로 하면 큐 사용: A → (B,C 인큐) → B → (D,E 인큐) → C → (F 인큐) → D → E → F 로 A → B → C → D → E → F 순서가 나온다. **③(A→B→D→E→C→F)** 은 DFS에서 F 이전에 C를 방문한 잘못된 트레이싱.

- 함정: **DFS(스택·깊이 우선) vs BFS(큐·너비 우선) 뒤바꾸기 초빈출**. DFS는 재귀 또는 명시적 스택 사용, BFS는 큐 사용. 인접 정점 방문 순서 규칙(사전순/역순) 명시 시 그대로 따를 것. ④는 A→C 먼저 잡은 경우(사전순 규칙 위반).

### 문제 6
**정답 ④ 체이닝(Chaining, 분리 연결법)**. 해시 충돌 해결 2대 계열:
- **개방 주소법(Open Addressing)**: 충돌 시 **다른 빈 슬롯**을 찾아 저장
  - **선형 조사법(Linear Probing)**: h(k), h(k)+1, h(k)+2, ... — 1차 클러스터링 문제
  - **이차 조사법(Quadratic Probing)**: h(k), h(k)+1², h(k)+2², ... — 2차 클러스터링
  - **이중 해싱(Double Hashing)**: h(k), h(k)+h₂(k), h(k)+2·h₂(k), ... — 클러스터링 최소
- **분리 연결법(Separate Chaining / Chaining)**: 각 버킷에 **연결 리스트**를 두고 충돌 원소를 연결. Java `HashMap`(JDK8+는 버킷 크기 8 초과 시 Red-Black Tree로 승격), Python `dict`(Open Addressing).

- 함정: **Chaining vs Open Addressing 개념 뒤바꾸기 초빈출**. "연결 리스트 = Chaining", "빈 슬롯 탐색 = Open Addressing". Linear/Quadratic/Double Hashing은 모두 Open Addressing의 하위 유형. **Java HashMap 내부 구조**(Chaining + Tree) 자주 출제.

### 문제 7
**정답 ③**. 시스템 통합 3형 비교:
- **Point-to-Point**: 시스템 간 1:1 직접 연결. N개 시스템 → 최대 N(N-1)/2 회선. **배선 폭발 문제**.
- **EAI(Enterprise Application Integration) — Hub & Spoke**: 중앙 **허브**에 모든 시스템이 연결. 허브가 라우팅·변환. 허브 장애 = 전체 마비(단일 장애점).
- **ESB(Enterprise Service Bus) — Bus**: 표준화된 **버스(메시지 브로커)** 를 통해 서비스 간 **느슨하게 결합**. SOA/MSA와 자연스럽게 결합. 확장성·유연성 최고. 대표: MuleSoft, Apache Camel, Kafka(브로커).
- **ETL(Extract-Transform-Load)**: **데이터 웨어하우스(DW)** 구축용. 원본 → 추출 → 변환 → 적재. 배치성 통합(실시간 X). 대표: Informatica, Talend, Airflow.

- 함정: **EAI(Hub&Spoke) vs ESB(Bus) 구조 뒤바꾸기 초빈출**. Hub&Spoke는 중앙 집중(허브 병목), Bus는 분산·느슨한 결합. **ETL은 통합 방식이 아니라 데이터 이관 방식**(오답 ④). ① Point-to-Point, ② EAI, ③ ESB(정답), ④ ETL 4대 분류 필수.

### 문제 8
**정답 ③ 회귀(Regression) 테스트**. 테스트 유형 비교:
- **스모크(Smoke) 테스트**: 빌드가 최소한 실행 가능한지 **간단·짧게** 확인. 빌드 초기 단계. "빌드 검증 테스트(BVT, Build Verification Test)".
- **새니티(Sanity) 테스트**: 특정 기능·모듈에 국한된 **얕고 좁은** 회귀 검증(제한적 회귀). 스모크와 유사하나 대상이 좁고 깊이는 얕음.
- **회귀(Regression) 테스트**: **변경·수정이 기존 기능에 부작용을 일으키지 않는지** 확인. **이미 통과했던 테스트 케이스를 재수행**. CI/CD 파이프라인의 핵심.
- **알파(Alpha) 테스트**: 개발사 내부 환경에서 실 사용자가 수행(사내). / **베타(Beta) 테스트**: 실 고객 환경에서 수행(사외 배포 전).

- 함정: **회귀 vs 새니티 뒤바꾸기 초빈출**. "부작용 방지 = 회귀(넓고 깊음)" / "특정 모듈 얕은 재검증 = 새니티(좁고 얕음)" / "빌드 자체 실행 확인 = 스모크(최소)". 알파/베타는 인수 테스트 계열로 별도.

### 문제 9
**정답 ③**. 뷰(View) 특성:
- ✅ CREATE VIEW / DROP VIEW로 생성·삭제 가능
- ✅ SELECT는 자유, INSERT/UPDATE/DELETE는 **단일 테이블 기반·집계함수·DISTINCT·GROUP BY 없는 경우에만 제한적 가능**
- ✅ **ALTER VIEW 불가**(일부 DBMS는 CREATE OR REPLACE VIEW 지원) — 정의 변경 시 DROP 후 재생성
- ❌ **뷰 자체에는 인덱스 생성 불가**(뷰는 저장된 SELECT 문 정의일 뿐 실제 데이터 없음). Oracle의 **Materialized View(구체화된 뷰)** 는 물리 데이터를 가지므로 인덱스 가능.
- ✅ 원본 테이블의 특정 컬럼/행만 노출하여 **보안·논리적 독립성** 강화
- ✅ 뷰를 통해 복잡한 조인·집계를 캡슐화(재사용성)

- 함정: **③ 뷰에 인덱스 생성 = 오답 초빈출**. 일반 뷰(Virtual View)는 정의만 저장하므로 인덱스 대상 아님. **Materialized View 는 예외**(용어 구분 필수). ② ALTER VIEW 불가 = 정답 특성. 뷰의 **WITH CHECK OPTION**(뷰 조건 위반 INSERT/UPDATE 방지), **WITH READ ONLY** 옵션도 자주 함께 출제.

### 문제 10
**정답 ② CURRVAL**. Oracle Sequence 의사 컬럼:
- **NEXTVAL**: 시퀀스를 **다음 값으로 증가시키고 반환**. 세션 최초 호출 필수(CURRVAL 사전 조건).
- **CURRVAL**: **현재 세션에서 마지막으로 발급된** 시퀀스 값 반환. 세션에서 NEXTVAL을 한 번도 호출 안 했으면 오류.
- **ROWNUM**: SELECT 결과 행에 부여되는 순번(1부터). WHERE ROWNUM <= N 페이징.
- **ROWID**: 실제 물리 저장 위치를 나타내는 16진수 문자열(블록·행 정보).

시퀀스 생성 예: `CREATE SEQUENCE seq_a START WITH 1 INCREMENT BY 1 MAXVALUE 9999 NOCYCLE NOCACHE;`
사용: `INSERT INTO t VALUES(seq_a.NEXTVAL, 'A'); SELECT seq_a.CURRVAL FROM DUAL;`

- 함정: **NEXTVAL vs CURRVAL 뒤바꾸기 초빈출**. "다음 값 발급 = NEXTVAL", "현재(마지막) 값 조회 = CURRVAL". ROWNUM/ROWID와 혼동 X. **MySQL에는 SEQUENCE 없이 AUTO_INCREMENT 사용**(DBMS별 차이).

### 문제 11
**정답 ① 테이블 통합(Merging Tables)**. 역정규화(Denormalization) 5대 유형:
- **① 테이블 통합**: 두 개 이상의 테이블을 **하나로 합쳐** 조인 비용 절감. 1:1 관계이거나 자주 함께 조회되는 경우.
- **② 테이블 분할**: 하나의 테이블을 **수직 분할(컬럼)** 또는 **수평 분할(행, 파티셔닝)** 로 나눠 성능 확보.
- **③ 중복 속성 추가**: 자주 조회되는 컬럼을 **다른 테이블에 복사** 저장하여 조인 회피.
- **④ 파생 속성 추가**: 계산 결과(합계·평균)를 미리 컬럼으로 저장(예: total_amount = sum).
- **⑤ 이력 테이블(History Table)**: 변경 이력을 별도 테이블에 저장하여 이력 조회 최적화.

- 함정: **① 통합 vs ② 분할 정반대 뒤바꾸기 초빈출**. "합침 = 조인 비용↓·중복↑" / "쪼갬 = 조회 성능↑·복잡도↑". 중복 속성 vs 파생 속성 구분: **중복 = 다른 테이블에 같은 값 복사** / **파생 = 계산 결과**. **역정규화는 성능을 위해 정규형(3NF+)에서 의도적 후퇴**하는 것 — 무결성 위험을 감수함.

### 문제 12
**정답 ③**. 저장 프로시저(Procedure) vs 함수(Function) vs 트리거(Trigger) 4대 차이:

| 구분 | 프로시저(Procedure) | 함수(Function) | 트리거(Trigger) |
|---|---|---|---|
| **호출 방식** | EXECUTE / CALL 명시 | SELECT/DML 내에서 호출 | **자동 실행** (이벤트 기반) |
| **반환값** | 없음(OUT 매개변수) | **반드시 RETURN 값** | 없음 |
| **매개변수** | IN/OUT/INOUT 가능 | IN만 가능 | 없음(NEW/OLD 참조) |
| **주 용도** | 업무 로직·트랜잭션 처리 | 계산·값 반환 | 무결성·감사·자동 갱신 |

- 함정: ① 트리거는 반환값 없음, SELECT에서 호출 안 함 / ② EXECUTE는 프로시저 방식 / ④ 트리거는 매개변수 없음. **"자동 실행 + 이벤트 기반" = 트리거의 결정적 특성**. **DDL 트리거(스키마 변경 감지)**·**Instead-Of 트리거(뷰에 대한 DML 대체)** 등 응용 유형도 자주 출제.

### 문제 13
**정답 ② 10 20 0 40 쓰레기**. C 동적 할당 4함수:
- **malloc(size)**: size 바이트 할당, **초기화 안 함**(쓰레기값)
- **calloc(n, size)**: n×size 바이트 할당, **0으로 초기화**
- **realloc(ptr, new_size)**: 기존 블록을 new_size로 조정. 기존 값은 유지, **확장 부분은 초기화 안 함**(쓰레기값). 이동 발생 가능(포인터 재할당).
- **free(ptr)**: 할당 해제. **NULL free는 안전, 이중 free는 UB**.

코드 트레이싱:
1. `calloc(3, sizeof(int))` → p = [0, 0, 0] (**0 초기화**)
2. `p[0]=10; p[1]=20` → p = [10, 20, 0]
3. `realloc(p, 5*sizeof(int))` → p = [10, 20, 0, **쓰레기**, **쓰레기**] (기존 3칸은 유지, 확장 2칸은 초기화 안 됨)
4. `p[3]=40` → p = [10, 20, 0, 40, **쓰레기**]
5. 출력: `10 20 0 40 (쓰레기)`

- 함정: **malloc(초기화 X) vs calloc(0 초기화) vs realloc(확장부 초기화 X)** 3자 차이가 핵심. **realloc의 확장 부분은 malloc과 같이 쓰레기값** — "realloc도 0으로 초기화"라는 오답(①번)이 초빈출. **free 후 dangling pointer**·**메모리 릭(leak)** 개념도 함께 출제.

**정답 ④**. 세마포어(Semaphore) vs 뮤텍스(Mutex) 4대 차이:

| 구분 | 세마포어(Semaphore) | 뮤텍스(Mutex) |
|---|---|---|
| **상태** | 정수 카운터 (0~N) | 0(잠금)/1(해제) 이진 |
| **관리 자원 수** | **N개 동시 관리** | **1개 상호배제만** |
| **소유(Ownership)** | 소유 개념 **없음** — 다른 스레드가 signal 가능 | **잠근 스레드만 해제 가능**(소유권) |
| **연산** | P(wait)·V(signal) 이진/일반 세마포어 | lock()·unlock() |
| **주 용도** | 자원 카운팅·생산자-소비자 | 임계 구역 상호배제 |

- **이진 세마포어(Binary Semaphore)** 는 뮤텍스와 유사하지만 **소유 개념 없음**이 결정적 차이. **뮤텍스만 우선순위 상속(Priority Inheritance)** 가능(우선순위 역전 방지).
- **P(wait)**: 카운터--, 0이면 대기 / **V(signal)**: 카운터++, 대기 스레드 깨움.

- 함정: **④ "기능적으로 동일" = 대표 오답**. 세마포어와 뮤텍스는 **소유 개념·관리 자원 수·용도가 다름**. "이진 세마포어 = 뮤텍스" 오답 유의. **모니터(Monitor)**·**조건 변수(Condition Variable)** 도 자주 함께 출제(모니터는 언어 레벨 동기화 구조체).

### 문제 17
**정답 ④**. 대칭키 알고리즘 4종:
- **DES(Data Encryption Standard)**: 64bit 블록, **56bit 키**, 페이스텔 16라운드. **현재는 취약**(1997 크랙). **3DES(Triple DES)** = 168bit 유효 키로 확장.
- **AES(Advanced Encryption Standard)**: 128bit 블록, **128/192/256bit 키**, SPN 구조(Rijndael), 10/12/14 라운드. **현행 표준**.
- **SEED**: 128bit 블록, 128bit 키. **KISA(한국인터넷진흥원) 개발**, 페이스텔 16라운드, 국가 표준.
- **ARIA(Academy, Research Institute, Agency)**: 128bit 블록, **128/192/256bit 키**(AES와 동일), SPN 12/14/16 라운드. **국가정보원 개발 국산 표준**, 공공기관 필수.
- **④ "ARIA = RSA 기반 공개키" = 명백한 오답**. ARIA는 국산 **대칭키**, RSA는 미국 **비대칭키**.

- 함정: **ARIA vs RSA 계열 뒤바꾸기 = 오답 초빈출**. 국산 알고리즘 3대: **SEED(1998)·ARIA(2004)·LEA(2013, 경량)**. **HIGHT**(경량 IoT용)도 국산. 키 길이(DES 56·AES 128~256·SEED/ARIA 128~256)는 반드시 암기. **HAS-160**(국산 해시 함수)과 혼동 X.

### 문제 18
**정답 ④**. 비대칭키(공개키) 알고리즘 안전성 근거:
- **RSA(Rivest-Shamir-Adleman, 1977)**: **큰 정수의 소인수분해(Integer Factorization)** 문제. n=p·q, n에서 p,q 복원 어려움. 2048~4096bit 키 사용.
- **ECC(Elliptic Curve Cryptography)**: **타원곡선 상의 이산대수(ECDLP)** 문제. **동일 보안 강도에서 RSA보다 훨씬 짧은 키**(RSA 2048bit ≈ ECC 224bit). IoT·모바일 유리.
- **ElGamal**: **이산대수(DLP)** 문제. Diffie-Hellman과 같은 기반.
- **DSA(Digital Signature Algorithm)**: 이산대수 기반 서명 전용.
- **Diffie-Hellman(1976)**: 최초의 공개키 개념·키 교환. 이산대수 문제.
- **AES**: **대칭키** — 소인수분해 근거 X. AES 안전성은 **키 공간 탐색의 계산 복잡성**(현실적으로 무차별 대입 불가).

- **④ "AES = 소인수분해" = 명백한 오답**. AES는 대칭키이므로 수학적 어려움 기반이 아닌 **연산 복잡도** 기반.

- 함정: **비대칭키의 3대 수학 문제 = 소인수분해(RSA) / 이산대수(ElGamal·DSA·DH) / 타원곡선 이산대수(ECC)** 매칭. **AES·DES·SEED·ARIA는 대칭키**이므로 이 분류에 속하지 않음. **양자 컴퓨팅(Shor 알고리즘)** 은 소인수분해·이산대수를 모두 다항 시간에 해결 가능 → **PQC(Post-Quantum Cryptography)** 필요성.

## 📊 자주 틀리는 함정 5가지 (2023년 회독 3/4 정리)

1. **프레임워크 vs 라이브러리 = IoC(제어의 역전) 방향으로만 판별** — "프레임워크가 개발자를 호출"(할리우드 원칙), 라이브러리는 "개발자가 함수를 호출". 오픈소스 여부·컴파일 여부는 판별 기준 아님. Spring/Django/React가 프레임워크의 대표. GRASP는 GOF 23 패턴과 다른 체계(책임 할당 원칙).
2. **역정규화 통합(Merge) vs 분할(Split) 정반대 뒤바꾸기** — "조인 비용↓ = 통합" / "성능↑·복잡도↑ = 분할(수직 컬럼 분할/수평 파티셔닝)". 중복 속성(다른 테이블 복사) vs 파생 속성(계산 결과) 구분. 역정규화는 **의도적 3NF 후퇴** — 무결성 위험 감수.
3. **realloc 확장 부분은 초기화 안 됨(쓰레기값)** — malloc(초기화 X)·calloc(0 초기화)·realloc(확장부 초기화 X)·free(해제). "realloc이 0으로 초기화한다"는 초빈출 오답. C 언어 문제는 **초기화·경계·NULL free** 3대 함정 반드시 확인.
4. **세마포어 vs 뮤텍스 = 소유(Ownership) 개념 유무가 결정적** — 뮤텍스는 잠근 스레드만 해제(소유권 O)·세마포어는 아무나 signal 가능(소유권 X). 이진 세마포어 ≠ 뮤텍스. 세마포어는 N개 자원 카운팅·뮤텍스는 1개 상호배제. 모니터(언어 레벨)와 혼동 X.
5. **RIP(홉 15) vs OSPF(다익스트라·링크상태) vs BGP(AS간·경로벡터·인터넷 백본) 3자 매칭** — RIP·OSPF는 IGP(내부), BGP는 EGP(외부). 대칭키(DES 56·AES 128~256·SEED/ARIA 128~256) vs 비대칭키(RSA 소인수분해·ECC 이산대수·ElGamal 이산대수) 수학 근거 매칭. AES는 대칭키이므로 소인수분해 문제 아님(연산 복잡도 기반).
