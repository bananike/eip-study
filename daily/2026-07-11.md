## 🎯 오늘의 기출문제 — 2024년 정처기 필기 20문제 (5과목 균등 · 회독 4/4 최종)

### 문제 1 [유사 기출: 2024년 1회 1과목 — SOLID 5원칙 정형 패턴]
객체지향 설계 5대 원칙 **SOLID**에 관한 설명 중 **가장 옳지 않은 것**은?

① **SRP(Single Responsibility Principle · 단일 책임 원칙)** — 하나의 클래스는 오직 하나의 책임(변경 이유)만 가져야 하며, 응집도를 높이고 결합도를 낮추는 기반이 된다.
② **OCP(Open-Closed Principle · 개방-폐쇄 원칙)** — 소프트웨어 개체는 확장에는 열려 있고 수정에는 닫혀 있어야 하며, 추상화·다형성(인터페이스·상속)을 통해 실현된다.
③ **LSP(Liskov Substitution Principle · 리스코프 치환 원칙)** — 자식 클래스는 언제나 부모 클래스로 대체(치환)될 수 있어야 하며, 부모의 계약(사전·사후 조건)을 위반하지 않아야 한다.
④ **DIP(Dependency Inversion Principle · 의존 역전 원칙)** — 상위 모듈이 하위 모듈의 구체 클래스에 직접 의존해야 하며, 추상(인터페이스)에 의존하지 않고 세부 구현에 의존하는 것이 원칙이다.

### 문제 2 [유사 기출: 2024년 2회 1과목 — 컴포넌트 기반 개발(CBD) 정형 패턴]
**컴포넌트 기반 개발(CBD: Component Based Development)**에 관한 설명 중 **가장 옳지 않은 것**은?

① 재사용 가능한 독립적 컴포넌트(Component)를 조립(Assembly)하여 시스템을 구축하는 방법론으로, 개발 생산성·유지보수성을 높인다.
② 컴포넌트는 명확한 **인터페이스(Interface)**를 통해서만 상호작용하며, 내부 구현은 캡슐화되어 블랙박스로 제공된다.
③ **CBD 개발 프로세스**는 요구분석 → 컴포넌트 식별 → 컴포넌트 검색·조립 → 사용자 요구 검증 → 유지보수 순으로 진행되며, 컴포넌트 저장소(Repository)를 활용한다.
④ CBD는 **개별 컴포넌트 간 강한 결합(Tight Coupling)과 낮은 응집도**를 지향하며, 컴포넌트 간 내부 데이터를 자유롭게 공유하도록 설계하는 것이 원칙이다.

### 문제 3 [유사 기출: 2024년 3회 1과목 — 마이크로서비스 아키텍처(MSA) 정형 패턴]
**마이크로서비스 아키텍처(MSA: Micro Service Architecture)**에 관한 설명 중 **가장 옳지 않은 것**은?

① 대규모 애플리케이션을 작고 독립적으로 배포 가능한 여러 개의 서비스로 분할하는 아키텍처 스타일로, 모놀리식(Monolithic) 구조의 한계를 극복하기 위해 등장했다.
② 각 마이크로서비스는 **독립된 프로세스**로 실행되고 **REST/gRPC/메시지 큐** 등 경량 프로토콜로 통신하며, 서비스별 **독립 배포·독립 확장·독립 기술 스택** 선택이 가능하다.
③ 각 서비스는 **자체 데이터베이스**를 가지며(Database per Service), 서비스 간 데이터 일관성은 **Saga 패턴·이벤트 소싱(Event Sourcing)** 등으로 관리한다.
④ MSA는 **모든 서비스가 하나의 공유 데이터베이스**를 사용하고, **단일 배포 파일(WAR/EAR)**로 통합 배포되는 것이 특징이며, 서비스 간 강한 결합을 통해 트랜잭션 처리를 단순화한다.

### 문제 4 [유사 기출: 2024년 1회 1과목 — 리팩토링(Refactoring) 정형 패턴]
**리팩토링(Refactoring)**에 관한 설명 중 **가장 옳지 않은 것**은?

① 리팩토링은 소프트웨어의 **외부 동작(기능)은 그대로 유지**하면서 **내부 구조(코드 품질)를 개선**하는 활동을 의미한다.
② 리팩토링의 **목적**은 코드 가독성 향상·유지보수성 개선·중복 제거·설계 개선이며, 코드 스멜(Code Smell)을 식별하여 리팩토링 대상으로 삼는다.
③ 리팩토링 대표 기법으로 **메서드 추출(Extract Method)·클래스 추출(Extract Class)·이름 변경(Rename)·매개변수 리스트 정리** 등이 있으며, 각 단계마다 **회귀 테스트(Regression Test)로 검증**한다.
④ 리팩토링은 **새 기능 추가와 성능 최적화**를 목적으로 하며, 외부 사용자에게 새로운 기능을 제공하기 위해 코드의 동작 방식을 대폭 변경하는 활동이다.

### 문제 5 [유사 기출: 2024년 2회 2과목 — AVL 트리 회전 4종 정형 패턴]
**AVL 트리(자가 균형 이진 탐색 트리)**의 회전 연산에 관한 설명 중 **가장 옳지 않은 것**은?

① AVL 트리는 모든 노드의 **좌·우 서브트리 높이 차(Balance Factor)가 -1, 0, +1** 이내로 유지되며, 초과 시 회전(Rotation)으로 재균형한다.
② **LL 회전(Right Rotation · 우회전)** — 왼쪽 자식의 왼쪽 서브트리에 삽입되어 왼쪽으로 치우칠 때, **1회 우회전**으로 균형을 회복한다.
③ **LR 회전 · RL 회전(이중 회전)** — 지그재그로 삽입될 때 **2회 회전(좌+우 또는 우+좌)**을 순차 수행하여 균형을 회복한다.
④ AVL 트리는 **어떤 상황에서도 회전 없이 균형**이 유지되며, 삽입·삭제 시 회전 연산이 필요하지 않아 일반 이진 탐색 트리(BST)와 동일한 성능을 보인다.

### 문제 6 [유사 기출: 2024년 3회 2과목 — 해싱(Hashing) 충돌 해결 정형 패턴]
**해싱(Hashing)**의 충돌(Collision) 해결 방법에 관한 설명 중 **가장 옳지 않은 것**은?

① 해싱은 **해시 함수(Hash Function)**를 통해 키(Key)를 해시 테이블의 인덱스로 변환하여 O(1) 평균 시간에 데이터를 검색·삽입한다.
② **체이닝(Chaining · 개별 연결법)** — 동일 해시 값에 매핑된 항목들을 **연결 리스트(Linked List)**로 이어 관리하며, 적재율(Load Factor) 1을 초과해도 저장 가능하다.
③ **개방 주소법(Open Addressing)** — 충돌 시 다른 빈 슬롯을 순차 탐색(선형 조사·이차 조사·이중 해싱)하며, 삭제 처리 시 **삭제 마커(Tombstone)** 관리가 필요하다.
④ **완전 해싱(Perfect Hashing)이 실무에서 항상 보장**되므로 충돌은 이론적으로만 존재하며, 해시 테이블 크기와 무관하게 O(1)이 항상 성립한다.

### 문제 7 [유사 기출: 2024년 1회 2과목 — Big-O 시간 복잡도 비교 정형 패턴]
알고리즘의 **시간 복잡도(Big-O Notation)** 크기 비교에 관한 설명 중 **가장 옳지 않은 것**은?

① 대표 정렬 알고리즘의 평균 시간 복잡도 — **버블·선택·삽입 정렬 = O(n²)**, **병합·힙·퀵(평균) 정렬 = O(n log n)**, 퀵 정렬 최악 = O(n²).
② 이진 탐색(Binary Search) = **O(log n)**, 선형 탐색 = **O(n)**, 해시 테이블 검색(평균) = **O(1)**, 최악 = **O(n)**.
③ 시간 복잡도의 증가율 순서(느림 → 빠름 관점, 즉 **더 나쁨 → 더 좋음**)는 **O(n!) > O(2ⁿ) > O(n³) > O(n²) > O(n log n) > O(n) > O(log n) > O(1)**이다.
④ **O(1)이 O(n²)보다 나쁜 알고리즘**이며, 입력 크기 n이 커질수록 O(1) 알고리즘의 실행 시간이 O(n²)보다 훨씬 빠르게 증가한다.

### 문제 8 [유사 기출: 2024년 2회 2과목 — 형상 관리 도구(Git·SVN) 정형 패턴]
**형상 관리 도구(SCM Tool)** Git과 SVN의 비교에 관한 설명 중 **가장 옳지 않은 것**은?

① **Git** — **분산 버전 관리 시스템(DVCS)**으로, 모든 개발자가 전체 저장소(로컬 리포지토리)를 가지며 오프라인 작업·빠른 브랜치·병합이 가능하다.
② **SVN(Subversion)** — **중앙 집중식 버전 관리 시스템(CVCS)**으로, 단일 중앙 서버에 저장소가 있고 모든 커밋이 서버에 즉시 반영되며 오프라인 작업이 제한적이다.
③ Git의 주요 명령: **clone·pull·push·commit·branch·merge·rebase·checkout·stash·log**, SVN의 주요 명령: **checkout·update·commit·revert·branch·merge·log**.
④ **Git은 중앙 집중식이며 오프라인 작업이 불가능**하고, **SVN은 완전한 분산형으로 서버 없이도 개발자 간 P2P 병합**이 표준으로 지원된다.

### 문제 9 [유사 기출: 2024년 3회 3과목 — 뷰(View) 특징 정형 패턴]
**관계형 DB의 뷰(View)**에 관한 설명 중 **가장 옳지 않은 것**은?

① 뷰는 하나 이상의 기본 테이블에서 유도된 **가상 테이블(Virtual Table)**로, 실제 데이터를 저장하지 않고 **SELECT 정의(질의)만 저장**한다.
② 뷰의 **장점**: 논리적 독립성·보안(민감 컬럼 은닉)·복잡한 쿼리 단순화·사용자별 맞춤 데이터 제공.
③ 뷰의 **제약**: 정의된 뷰는 삽입·갱신·삭제에 제약이 있으며(특히 조인·집계·DISTINCT 포함 뷰), 뷰의 뷰 정의는 가능하지만 인덱스 생성은 원칙적으로 불가하다(구체화 뷰 예외).
④ 뷰는 **자체 물리적 저장 공간을 반드시 확보**해야 하며, **원본 테이블이 삭제되어도 뷰는 독립적으로 유지**되어 안정성을 보장한다.

### 문제 10 [유사 기출: 2024년 1회 3과목 — 인덱스(B-Tree·Hash·Bitmap) 정형 패턴]
데이터베이스 **인덱스(Index)** 유형에 관한 설명 중 **가장 옳지 않은 것**은?

① **B-Tree 인덱스** — 범위 검색·정렬 검색에 유리하며 대부분의 RDBMS 기본 인덱스로 사용되고, **높이가 낮은 균형 트리**로 O(log n) 검색을 보장한다.
② **Hash 인덱스** — 등가 비교(=)에 매우 빠르지만 **범위 검색(<, >, BETWEEN)에 사용 불가**하며, MySQL Memory 엔진 등 특수 엔진에서만 지원된다.
③ **Bitmap 인덱스** — **카디널리티(구분값 수)가 낮은 컬럼**(성별·부서 코드 등)에 유리하며, DW·OLAP 환경에서 대량 데이터 집계에 효과적이다.
④ **B-Tree 인덱스는 카디널리티가 낮은 컬럼(성별 등)에 가장 적합**하며, **Bitmap 인덱스는 카디널리티가 높은 컬럼(주민번호·유일ID 등)에서 가장 효율적**이다.

### 문제 11 [유사 기출: 2024년 2회 3과목 — 트리거(Trigger)·스토어드 프로시저 정형 패턴]
DB의 **트리거(Trigger)와 스토어드 프로시저(Stored Procedure)**에 관한 설명 중 **가장 옳지 않은 것**은?

① **트리거** — 특정 이벤트(INSERT·UPDATE·DELETE)가 발생할 때 **자동으로 실행**되는 저장 프로그램으로, 데이터 무결성 유지·감사(Audit) 로그 자동화 등에 활용된다.
② **스토어드 프로시저** — **명시적 호출(CALL·EXEC)**로 실행되는 이름 있는 SQL 프로그램으로, 반복 로직 캡슐화·네트워크 부하 감소·보안 강화에 활용된다.
③ 두 객체 모두 **PL/SQL(Oracle)·T-SQL(SQL Server)·SQL/PSM(표준)** 등 절차형 SQL로 작성되며, DBMS 내부에서 컴파일·저장되어 실행된다.
④ **트리거는 사용자가 명시적으로 CALL 명령을 통해 호출**하며, **스토어드 프로시저는 INSERT·UPDATE·DELETE 이벤트에 자동으로 발동**되어 서로 반대 방식으로 동작한다.

### 문제 12 [유사 기출: 2024년 3회 3과목 — SQL 명령어 분류(DDL·DML·DCL·TCL) 정형 패턴]
SQL 명령어 **4대 분류(DDL·DML·DCL·TCL)**에 관한 설명 중 **가장 옳지 않은 것**은?

① **DDL(Data Definition Language · 데이터 정의어)** — 스키마 정의·객체 생성. 대표 명령: **CREATE·ALTER·DROP·TRUNCATE·RENAME**.
② **DML(Data Manipulation Language · 데이터 조작어)** — 데이터 조회·삽입·수정·삭제. 대표 명령: **SELECT·INSERT·UPDATE·DELETE·MERGE**.
③ **DCL(Data Control Language · 데이터 제어어)** — 권한 부여·회수. 대표 명령: **GRANT·REVOKE**, **TCL(Transaction Control Language)** = **COMMIT·ROLLBACK·SAVEPOINT**.
④ **TRUNCATE·GRANT·COMMIT은 모두 DML로 분류**되며, **SELECT·INSERT·UPDATE·DELETE는 모두 DDL**에 속한다.

### 문제 14 [유사 기출: 2024년 2회 4과목 — Java 컬렉션 프레임워크 정형 패턴]
Java **컬렉션 프레임워크**에 관한 설명 중 **가장 옳지 않은 것**은?

① **List 인터페이스** — 순서 있는 데이터 저장, **중복 허용**. 구현체: **ArrayList·LinkedList·Vector**.
② **Set 인터페이스** — 순서 없는 데이터 저장, **중복 불허**. 구현체: **HashSet(순서 무보장)·LinkedHashSet(삽입 순서)·TreeSet(정렬 순서)**.
③ **Map 인터페이스** — Key-Value 쌍 저장, **Key 중복 불허·Value 중복 허용**. 구현체: **HashMap·LinkedHashMap·TreeMap·Hashtable**.
④ **List는 중복이 금지되고, Set은 중복을 허용**하며, **Map은 Key 대신 Value에 유일성 제약**이 있어 List·Set·Map 모두 중복 처리 방식이 서로 반대이다.

### 문제 16 [유사 기출: 2024년 1회 4과목 — 라우팅 프로토콜(RIP·OSPF·BGP) 정형 패턴]
**라우팅 프로토콜(Routing Protocol)** 3종에 관한 설명 중 **가장 옳지 않은 것**은?

① **RIP(Routing Information Protocol)** — **거리 벡터(Distance Vector)** 방식, **홉(Hop) 수**를 메트릭으로 사용하고 **최대 15홉** 제한(16홉 = 무한대), 소규모 네트워크용, 30초마다 라우팅 테이블 브로드캐스트.
② **OSPF(Open Shortest Path First)** — **링크 상태(Link State)** 방식, **Dijkstra 알고리즘**으로 최단 경로 계산, **대역폭·지연 등 종합 비용(Cost)** 사용, 대규모 네트워크용, 계층 구조(Area) 지원.
③ **BGP(Border Gateway Protocol)** — **경로 벡터(Path Vector)** 방식, **AS(Autonomous System) 간 라우팅**에 사용되는 **외부 게이트웨이 프로토콜(EGP)**, 인터넷 백본 표준.
④ **RIP는 링크 상태 방식이며 무제한 홉을 지원**하고, **OSPF는 거리 벡터 방식으로 홉 수만 계산**하며, **BGP는 AS 내부 라우팅에 사용되는 IGP** 프로토콜이다.

### 문제 17 [유사 기출: 2024년 2회 5과목 — PMBOK 5 프로세스 그룹 정형 패턴]
**PMBOK(Project Management Body of Knowledge) 5 프로세스 그룹**에 관한 설명 중 **가장 옳지 않은 것**은?

① **착수(Initiating)** — 프로젝트 헌장(Project Charter) 작성·이해관계자 식별 등 프로젝트 승인·시작 단계.
② **계획(Planning)** — 범위·일정·비용·품질·자원·의사소통·리스크·조달·이해관계자 관리 계획 수립. WBS·간트 차트·예산 산정 포함.
③ **실행(Executing)·감시 및 통제(Monitoring & Controlling)·종료(Closing)** — 계획 수행·성과 측정·변경 관리·최종 인수·교훈(Lessons Learned) 문서화.
④ **PMBOK는 3개 프로세스 그룹만 정의**하며(**착수·실행·종료** 3단계), 계획·감시통제는 별도 방법론(PRINCE2)에서만 다루는 독립 활동으로 분류된다.

### 문제 18 [유사 기출: 2024년 3회 5과목 — PERT/CPM 임계 경로 정형 패턴]
**프로젝트 일정 관리 기법 PERT·CPM**에 관한 설명 중 **가장 옳지 않은 것**은?

① **CPM(Critical Path Method · 임계 경로 기법)** — 프로젝트의 **가장 긴 경로(임계 경로)**를 찾아 전체 완료 기간을 결정하며, **결정론적 확정 시간**을 사용한다.
② **PERT(Program Evaluation and Review Technique)** — 각 작업의 **낙관치(a)·기대치(m)·비관치(b)**를 사용해 기대 시간 = **(a + 4m + b) / 6**로 산출하며, **확률론적** 방식이다.
③ 임계 경로 상의 작업은 **여유 시간(Slack/Float)이 0**이며, 임계 경로 상의 작업이 지연되면 **전체 프로젝트가 지연**된다.
④ **CPM은 확률적 시간을 사용**하며 **PERT는 결정론적 확정 시간만 사용**하고, **여유 시간이 있는 작업이 임계 경로에 포함**되어 지연되면 전체 프로젝트가 조기 완료된다.

### 문제 19 [유사 기출: 2024년 1회 5과목 — 클라우드 서비스 모델(IaaS·PaaS·SaaS) 정형 패턴]
**클라우드 서비스 모델(IaaS·PaaS·SaaS)**에 관한 설명 중 **가장 옳지 않은 것**은?

① **IaaS(Infrastructure as a Service)** — 서버·스토리지·네트워크 등 **인프라 자원**을 서비스 형태로 제공. 예: AWS EC2·Azure VM·GCP Compute Engine.
② **PaaS(Platform as a Service)** — OS·미들웨어·런타임·개발 환경 등 **개발 플랫폼**을 제공하여 개발자가 애플리케이션 개발·배포에 집중. 예: AWS Elastic Beanstalk·Google App Engine·Heroku.
③ **SaaS(Software as a Service)** — **완성된 소프트웨어**를 웹 브라우저 등으로 이용. 예: Gmail·Salesforce·Microsoft 365·Slack.
④ **IaaS는 완성된 소프트웨어를 즉시 이용**하는 최상위 추상화 서비스이며, **SaaS는 서버·스토리지 등 인프라 자원**만 제공하고 나머지는 사용자가 직접 설치·관리해야 한다.

### 문제 20 [유사 기출: 2024년 2회 5과목 — SSO/OAuth 2.0/SAML/Kerberos 정형 패턴]
**인증(Authentication) 프로토콜** SSO·OAuth 2.0·SAML·Kerberos에 관한 설명 중 **가장 옳지 않은 것**은?

① **SSO(Single Sign-On · 통합 인증)** — 한 번의 로그인으로 여러 시스템·서비스에 접근 가능하게 하는 인증 아키텍처. Delegation·Propagation 모델.
② **OAuth 2.0** — 사용자의 자원(리소스)에 대한 **제3자 애플리케이션의 위임 접근**을 허용하는 인증·인가 프레임워크. 4대 승인 유형(Authorization Code·Implicit·Password·Client Credentials).
③ **SAML(Security Assertion Markup Language)** — **XML 기반**의 인증·인가 정보 교환 표준으로, 엔터프라이즈 SSO에 널리 사용되며 IdP-SP 간 어설션(Assertion) 교환.
④ **Kerberos는 공개키(비대칭키) 암호 기반 프로토콜**이며 티켓(TGT/ST) 대신 인증서(X.509)를 사용하고, **KDC 없이 클라이언트-서버 간 직접 P2P 인증**을 수행한다.

## 💡 상세 해설

### 문제 1
**정답 ④** — **DIP(의존 역전 원칙)**는 정확히 반대로, **상위 모듈은 하위 모듈(구체 클래스)에 의존해서는 안 되며, 둘 다 추상(인터페이스)에 의존**해야 한다. 추상은 세부 구현에 의존해서는 안 되고, 세부 구현이 추상에 의존해야 함. SOLID = **S**RP·**O**CP·**L**SP·**I**SP(인터페이스 분리)·**D**IP. **함정 키워드**: DIP를 "상위 모듈이 하위 구체에 의존" 완전 뒤바꾸기, "세부 구현에 의존이 원칙" 정반대 함정.

### 문제 2
**정답 ④** — CBD는 정확히 반대로 **컴포넌트 간 느슨한 결합(Loose Coupling)과 높은 응집도(High Cohesion)**를 지향하며, 컴포넌트는 인터페이스로만 상호작용하고 **내부 데이터·구현은 캡슐화되어 외부에서 접근 불가**. 내부 데이터 공유는 캡슐화 위반이자 CBD 원칙 정반대. **함정 키워드**: "강한 결합·낮은 응집도·내부 데이터 자유 공유"는 CBD 완전 정반대(오히려 모놀리식의 나쁜 특성).

### 문제 3
**정답 ④** — MSA는 정확히 반대로, **서비스마다 독립된 데이터베이스**(Database per Service)를 가지고 **서비스별 독립 배포**를 원칙으로 하며 **느슨한 결합(Loose Coupling)**을 지향. "공유 DB + 단일 배포 파일(WAR/EAR) + 강한 결합"은 **모놀리식(Monolithic) 아키텍처의 특징**으로 MSA와 정확히 정반대. **함정 키워드**: MSA에 모놀리식 특성(공유 DB·단일 WAR·강한 결합) 덧씌우기 완전 오답.

### 문제 4
**정답 ④** — 리팩토링의 정의상 **외부 동작(기능)은 절대 변경하지 않으며**, 새 기능 추가·성능 최적화는 리팩토링의 목적이 **아니라 별개 활동**(새 기능 = Feature Add, 성능 = Performance Tuning). 리팩토링은 순수히 **내부 구조 개선**만 목적. **함정 키워드**: 리팩토링을 "새 기능 추가·동작 대폭 변경"으로 뒤바꾸는 완전 정반대 오답.

### 문제 5
**정답 ④** — AVL 트리의 **본질은 회전(Rotation)으로 균형을 강제 유지**하는 자가 균형 BST. 삽입·삭제 시 균형 팩터가 -1, 0, +1 범위를 벗어나면 **반드시 회전 연산 수행**. "회전 없이 균형 유지, BST와 동일 성능"은 완전 오답. AVL의 검색 성능은 항상 O(log n)이지만 일반 BST는 최악 O(n)(편향 트리). **함정 키워드**: AVL이 "회전 없음·일반 BST와 동일"이라는 정반대 함정.

### 문제 6
**정답 ④** — **완전 해싱(Perfect Hashing)은 실무에서 항상 보장되지 않으며**, 정적 키 집합에 대해서만 이론적으로 가능. 일반 해싱에서 충돌은 **비둘기집 원리**에 의해 필연이고, 해시 테이블 크기·해시 함수 품질·적재율에 따라 성능이 O(1)~O(n) 범위 변동. **함정 키워드**: "완전 해싱 항상 보장·O(1) 무조건 성립"은 실무 오해 유도 완전 오답.

### 문제 7
**정답 ④** — 시간 복잡도는 **입력 크기가 커질수록 실행 시간이 얼마나 빠르게 증가하는지**를 표현. **O(1)은 상수 시간(입력과 무관)으로 가장 빠르고 최상**이며 O(n²)보다 훨씬 좋음. O(n²)은 입력 크기 제곱에 비례해 실행 시간이 급증. "O(1)이 나쁘고 실행 시간이 O(n²)보다 빠르게 증가"는 완전 정반대. **함정 키워드**: O(1)을 "가장 나쁨"으로, O(n²)을 "더 빠름"으로 뒤바꾸기.

### 문제 8
**정답 ④** — 정확히 반대로, **Git이 분산형(DVCS)** · **SVN이 중앙 집중식(CVCS)**. Git은 각 개발자가 완전한 로컬 저장소를 가져 오프라인 커밋·브랜치·로그 조회 가능. SVN은 서버 없이는 커밋 불가. "Git이 중앙 집중·SVN이 분산"은 완전 뒤바꾸기 오답. **함정 키워드**: Git·SVN의 CVCS/DVCS 방식을 정반대로 뒤바꾸는 초빈출 함정.

### 문제 9
**정답 ④** — 뷰는 **물리적 저장 공간을 확보하지 않는 가상 테이블**이며, **SELECT 정의(질의문)만 데이터 사전에 저장**. 원본 테이블이 삭제되면 **뷰도 사용 불가(무효화)**되며 독립 유지 불가. 물리 저장이 있는 것은 **구체화 뷰(Materialized View)** 별개 개념. **함정 키워드**: 일반 뷰를 "물리 저장·원본 삭제 시 독립 유지"로 왜곡(구체화 뷰와 혼동 유도).

### 문제 10
**정답 ④** — 정확히 반대로, **B-Tree 인덱스는 카디널리티가 높은 컬럼**(주민번호·이메일 등 유일 값이 많은 컬럼)에 적합하고, **Bitmap 인덱스는 카디널리티가 낮은 컬럼**(성별 2종·부서 코드 소수 등)에 적합. Bitmap은 비트맵 압축으로 저카디널리티에서 효율 극대화. **함정 키워드**: B-Tree/Bitmap의 카디널리티 적합도를 정반대로 뒤바꾸기 초빈출.

### 문제 11
**정답 ④** — 정확히 반대로, **트리거는 이벤트에 자동 발동**(INSERT/UPDATE/DELETE 발생 시), **스토어드 프로시저는 명시적 CALL/EXEC로 호출**. 두 개념의 실행 방식을 뒤바꾸는 완전 오답. 트리거 예: BEFORE INSERT ON emp FOR EACH ROW … / 프로시저 예: CALL sp_calc_bonus(1000). **함정 키워드**: 트리거/프로시저의 자동·수동 실행 방식을 정반대로 뒤바꾸기.

### 문제 12
**정답 ④** — SQL 분류 기준: **TRUNCATE = DDL** (테이블 구조 유지하지만 데이터 전체 삭제 + 자동 커밋), **GRANT = DCL** (권한 부여), **COMMIT = TCL** (트랜잭션 확정). **SELECT·INSERT·UPDATE·DELETE = DML** (데이터 조작). 4대 분류를 뒤섞은 완전 오답. **함정 키워드**: TRUNCATE를 DML로 오분류(주의: TRUNCATE는 DDL이므로 롤백 불가), GRANT를 DML로 오분류.

### 문제 13
**정답 ③** — fact(5) = 5 × fact(4) = 5 × 4 × fact(3) = 5 × 4 × 3 × fact(2) = 5 × 4 × 3 × 2 × fact(1) = 5 × 4 × 3 × 2 × 1 = **120**. 재귀 종료 조건 n<=1일 때 1 반환. **함정 키워드**: 5! = 120인데 6! = 720으로 오답 유도, 5+4+3+2+1=15 등 합산 오해.

### 문제 14
**정답 ④** — 정확히 반대로, **List는 중복 허용·순서 있음**, **Set은 중복 불허·순서 무관(구현체에 따라 순서 유지 가능)**, **Map은 Key 중복 불허·Value 중복 허용**. "List 중복 금지, Set 중복 허용"은 완전 정반대. **함정 키워드**: List·Set의 중복 허용/금지를 정반대로 뒤바꾸기 초빈출.

### 문제 15
**정답 ③** — 단계별 추적: 초기 d = {'a':1, 'b':2, 'c':3} → d['b']=20 대입 → d = {'a':1, 'b':20, 'c':3} → d['d']=4 추가 → d = {'a':1, 'b':20, 'c':3, 'd':4} → del d['a'] → d = {'b':20, 'c':3, 'd':4} → sum(d.values()) = 20 + 3 + 4 = **27**. **함정 키워드**: 초기 값의 합(1+2+3=6)이나 b를 갱신 안 함(2+3+4=9)으로 오답 유도.

### 문제 16
**정답 ④** — 정확히 반대로, **RIP = 거리 벡터·홉 수 메트릭·15홉 제한**, **OSPF = 링크 상태·Dijkstra·Cost 메트릭·계층 구조**, **BGP = 경로 벡터·AS 간(외부) 라우팅·EGP(IGP 아님)**. IGP(Interior Gateway Protocol) = RIP·OSPF·IGRP·EIGRP / EGP = BGP. "RIP 링크 상태·OSPF 거리 벡터·BGP IGP"는 3중 뒤바꾸기 완전 오답. **함정 키워드**: 3개 프로토콜의 특성을 서로 뒤바꾸는 초빈출 함정.

### 문제 17
**정답 ④** — PMBOK은 **5개 프로세스 그룹**을 명확히 정의: **착수·계획·실행·감시 및 통제·종료**. 착수·실행·종료 3단계만 있고 계획·감시통제를 PRINCE2로만 분리한다는 것은 완전 오답. PRINCE2는 별개 프로젝트 관리 방법론(영국 정부 표준)으로 7원칙·7테마·7프로세스 기반. **함정 키워드**: PMBOK 프로세스 그룹 수를 3개로 축소하거나 계획/감시통제를 다른 방법론으로 분리하는 오답.

### 문제 18
**정답 ④** — 정확히 반대로, **CPM = 결정론적(확정 시간)** · **PERT = 확률론적(3점 추정 = 낙관·기대·비관)**. 임계 경로 = 여유 시간(Slack) 0인 최장 경로이며, 임계 경로 상 작업 지연 시 프로젝트 지연·조기 완료 불가능. "여유 있는 작업이 임계 경로에 포함되어 지연되면 조기 완료"는 완전 논리 모순. **함정 키워드**: CPM/PERT의 결정/확률 방식을 정반대로 뒤바꾸기, 임계 경로 여유 시간 개념 오해 유도.

### 문제 19
**정답 ④** — 정확히 반대로, **IaaS는 인프라(최하위 계층)만 제공하고 OS·미들웨어·앱은 사용자 책임**, **SaaS는 완성된 소프트웨어(최상위 계층)를 즉시 이용**. 추상화 수준: **IaaS < PaaS < SaaS** (아래일수록 사용자 관리 범위 큼). "IaaS 완성 SW·SaaS 인프라 자원"은 완전 정반대. **함정 키워드**: IaaS/SaaS의 추상화 수준·관리 범위를 정반대로 뒤바꾸기 초빈출.

### 문제 20
**정답 ④** — **Kerberos는 대칭키(비밀키) 암호 기반** 프로토콜이며 **KDC(Key Distribution Center) 중심**의 **티켓(TGT: Ticket Granting Ticket / ST: Service Ticket)** 방식으로 인증. 공개키·X.509 인증서·P2P 인증은 Kerberos 방식이 아님(공개키 인증은 PKI/SSL/TLS 영역). KDC = AS(인증 서버) + TGS(티켓 승인 서버). **함정 키워드**: Kerberos를 공개키·인증서·P2P로 왜곡하는 완전 정반대 오답.

## 📊 자주 틀리는 함정 5가지 (2024년 회독 4/4 · 최종회독 · 신규 영역 위주)

1. **SOLID 원칙 이름-정의 매칭** — SRP(단일 책임)·OCP(개방-폐쇄)·LSP(리스코프 치환)·ISP(인터페이스 분리)·DIP(의존 역전). 특히 **DIP는 "상위·하위 모두 추상(인터페이스)에 의존"**이 핵심(구체 의존 X). 원칙 이름을 뒤바꾸거나 정의를 정반대로 왜곡하는 함정 초빈출.
2. **B-Tree vs Bitmap 인덱스 카디널리티** — **B-Tree = 고카디널리티(주민번호·이메일 등)**, **Bitmap = 저카디널리티(성별·부서 등)**. 정반대로 뒤바꾸는 초빈출 함정. Hash 인덱스는 등가 비교 전용(범위 검색 불가) 별개 특성.
3. **SQL 명령어 4대 분류** — **TRUNCATE는 DDL**(자동 커밋, 롤백 불가), **DELETE는 DML**(트랜잭션 롤백 가능). **GRANT/REVOKE = DCL**, **COMMIT/ROLLBACK/SAVEPOINT = TCL**. TRUNCATE를 DML로 오분류하면 롤백 가능성 판단에서 실전 오답 유도.
4. **RIP·OSPF·BGP 라우팅 프로토콜** — **RIP(거리 벡터·15홉)** · **OSPF(링크 상태·Dijkstra·Cost)** · **BGP(경로 벡터·AS 간·EGP)**. 3개 특성을 서로 뒤바꾸는 함정과 함께 IGP(내부 = RIP·OSPF) vs EGP(외부 = BGP) 구분 필수.
5. **IaaS·PaaS·SaaS 추상화 수준 & Kerberos 대칭키** — 클라우드 서비스는 **IaaS(인프라) < PaaS(플랫폼) < SaaS(SW 완성)** 순으로 추상화 상승·사용자 관리 감소. Kerberos는 **대칭키 + KDC + 티켓(TGT/ST)** 조합이 핵심(공개키·인증서 X). 두 개념의 방식·구성 요소를 정반대로 왜곡하는 함정 초빈출.
