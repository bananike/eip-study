① 검증은 "올바른 제품을 만들고 있는가"를, 확인은 "제품을 올바르게 만들고 있는가"를 평가한다.
② 검증은 개발 단계 산출물이 그 단계의 초기 설정 조건을 만족하는지 평가하고, 확인은 최종 시스템이 명시된 요구사항을 만족하는지 평가한다.
③ 검증과 확인은 동일한 개념이며 상호 교환적으로 사용 가능하다.
④ 검증은 사용자가, 확인은 개발자가 수행하는 활동이다.

## 💡 상세 해설 (자기 설명과 대조)
 
### 문제 1
정답: ④
- 제공 해설: PERT는 작업 간 상호 관련성·결정 경로·경계 시간·자원 할당을 네트워크로 표현하는 차트다. 노드(작업)와 간선(선후관계)으로 구성되고, 각 작업에 낙관치/기대치/비관치를 기재하는 확률적 모형.
- 오답 이유:
  - ①②③: 모두 Gantt(간트) 차트의 설명. 막대 도표·시간선 차트·수평 막대 길이=작업 기간은 Gantt의 특징.
 
### 문제 2
정답: ①
- 제공 해설: Putnam 모형은 Rayleigh-Norden 곡선(시간 함수로 표현된 노력 분포)을 사용하는 대규모 프로젝트용 산정 기법. 자동화 도구로 SLIM이 있다.
- 오답 이유:
  - ②: COCOMO — Boehm이 제안, LOC 기반 수학적 모형.
  - ③: Function-Point — 사용자 관점의 기능 단위(입력/출력/조회/파일/인터페이스) 수치화.
  - ④: LOC — 원시 코드 라인 수의 낙관치·비관치·기대치로 산정하는 상향식.
 
### 문제 3
정답: ④
- 제공 해설: COCOMO 프로젝트 유형은 Organic(조직형) / Semi-detached(반분리형) / Embedded(내장형) 3종. Sequential은 COCOMO 분류에 없는 함정 보기.
- 오답 이유: ①②③은 모두 정상 분류명.
- 함정: Basic/Intermediate/Detailed는 "COCOMO 모델 종류(발전형)"이지 "프로젝트 유형"이 아니다. 두 개념을 섞어 묻는 변형 출제 다발.
 
### 문제 4
정답: ③
- 제공 해설: 하향식(Top-down)은 시스템 전체를 보고 추정 → 세부로 내려가는 방식. 전문가 감정·델파이 기법이 대표적. 델파이는 전문가 감정의 주관 편향을 다수 전문가 의견 종합으로 보완.
- 오답 이유:
  - ①: LOC — 상향식. 라인 수 추정으로 비용 산출.
  - ②: 개발 단계별 인월수(Effort Per Task) — 상향식. LOC 보완 기법.
  - ④: COCOMO — 수학적 산정(Boehm). 별도 분류로 보지만, 일부 교재에서는 "상향식"으로 분류.
 
### 문제 5
정답: ②
- 제공 해설: CPM의 임계 경로(Critical Path)는 시작에서 종료까지 경로 중 가장 "긴" 경로다. 가장 긴 경로의 길이가 프로젝트 최단 완료 시간을 결정한다(병렬 경로 중 최장이 율속).
- 오답 이유:
  - ①③④: 모두 CPM의 옳은 설명. 노드/간선 네트워크, 선후 관계와 소요 기간 표시, 일정 단축 시 임계 경로 우선 검토.
- 함정: "가장 짧은" / "가장 긴" 한 글자 차이로 오답 유도 빈번.
 
### 문제 6
정답: ③
- 제공 해설: SPICE(ISO/IEC 15504) 능력 수준 6단계는 0 불완전(Incomplete) → 1 수행(Performed) → 2 관리(Managed) → 3 확립(Established) → 4 예측(Predictable) → 5 최적화(Optimizing). "정형(Formal)" 단계는 존재하지 않는 함정 보기.
- 오답 이유: ①②④는 모두 실제 SPICE 단계명.
 
### 문제 7
정답: ②
- 제공 해설: CMMI 성숙도는 1 초기(Initial) → 2 관리(Managed) → 3 정의(Defined) → 4 정량적 관리(Quantitatively Managed) → 5 최적화(Optimizing). 정의(Defined) 이전에 관리(Managed)가 온다.
- 오답 이유:
  - ①: 관리와 정의 순서가 뒤바뀜.
  - ③: 정량적 관리가 정의보다 앞에 오면 안 됨.
  - ④: 관리 단계가 4단계로 가면 안 됨.
- 함정: SPICE와 순서가 다름에 주의. SPICE는 2 관리 → 3 확립, CMMI는 2 관리 → 3 정의. 단계 명칭과 순서 모두 변별 필요.
 
### 문제 8
정답: ②
- 제공 해설: 검증(Verification)은 "각 개발 단계의 산출물이 그 단계의 초기에 설정된 조건(명세)을 만족하는가"를 평가. 확인(Validation)은 "최종 산출물이 명시된 사용자 요구사항을 만족하는가"를 평가.
- 오답 이유:
  - ①: 검증/확인 문구가 뒤바뀜. 올바른 짝은 검증="Are we building the product right?" / 확인="Are we building the right product?"
  - ③: V&V는 명확히 구별되는 개념.
  - ④: 수행 주체 기준으로 정의되는 개념이 아님.

## 📚 역추적 참조 (사후 검증용 — 학습 자료 아님)
 
### 핵심 용어
| 용어 | 정의 |
|---|---|
| PERT | Program Evaluation and Review Technique. 불확실 프로젝트용. 3점 추정(낙관/기대/비관)으로 가중평균 = (낙관 + 4×기대 + 비관)/6 |
| CPM | Critical Path Method. 확정 시간 추정. 임계 경로(시작→종료 최장 경로) 식별, 일정 단축 시 임계 경로 우선 |
| Gantt | 막대(수평) 차트, 시간선 차트라고도 함. 작업의 시작·종료를 막대로 시각화 |
| LOC | Lines Of Code. 라인 수의 낙관/기대/비관치로 산정하는 상향식 |
| COCOMO | Constructive Cost Model. Boehm 제안. LOC 기반 수학적 모형 |
| Putnam | Rayleigh-Norden 곡선 기반 동적 모형. 자동화 도구 SLIM |
| Function Point | 사용자 관점 기능 단위(입력/출력/조회/내부논리파일/외부연계파일) 수치화. UFP = 데이터FP + 트랜잭션FP |
| 델파이 | 전문가 감정의 주관 편향 보완. 다수 전문가 의견 종합 |
| CMMI | Capability Maturity Model Integration. 5단계(초기/관리/정의/정량적 관리/최적화) |
| SPICE | ISO/IEC 15504. 6단계(불완전/수행/관리/확립/예측/최적화) |
| V&V | Verification(단계 산출물 검증) & Validation(요구사항 확인) |
| SLIM | Putnam 모형 기반 자동화 추정 도구 |
 
### 핵심 개념
- **COCOMO 프로젝트 유형 3종 (LOC 규모별):**
  - Organic(조직형): ≤ 5만 라인. 일괄 자료 처리, 소규모 사무용
  - Semi-detached(반분리형): ≤ 30만 라인. 트랜잭션 처리, OS, DB
  - Embedded(내장형): ≥ 30만 라인. 최대형 OS, 실시간 제어
- **COCOMO 모델 종류 3종 (정밀도별):**
  - Basic: 라인 수만 고려
  - Intermediate: 4가지 특성·15가지 요인 추가 고려
  - Detailed: 단계별 노력 승수까지 세분화
- **비용 산정 기법 분류:**
  - 하향식: 전문가 감정, 델파이
  - 상향식: LOC, 개발 단계별 인월수
  - 수학적: COCOMO, Putnam, Function Point
- **CMMI 5단계 vs SPICE 6단계 매핑 (대응되지만 명칭/단계수 다름):**
  - CMMI 1 초기 ↔ SPICE 0 불완전 + 1 수행
  - CMMI 2 관리 ↔ SPICE 2 관리
  - CMMI 3 정의 ↔ SPICE 3 확립
  - CMMI 4 정량적 관리 ↔ SPICE 4 예측
  - CMMI 5 최적화 ↔ SPICE 5 최적화
- **V 모델:** 폭포수 모델 좌측 날개(분석·설계·구현) ↔ 우측 날개(단위·통합·시스템·인수 테스트) 대응. V&V를 시각화한 모델.
