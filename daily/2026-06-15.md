### 문제 5 [유사 기출: 정처기 필기 2020년 4회 · 2022년 3회 · 2024년 반복 — 유스케이스 관계 정형 패턴]
유스케이스(Use Case) 다이어그램의 관계(Relationship)에 대한 설명으로 **틀린** 것은?

① **포함(Include)** 관계는 한 유스케이스가 다른 유스케이스의 기능을 **반드시 포함**하여 실행한다는 의미이며, `<<include>>` 스테레오타입으로 표기한다.
② **확장(Extend)** 관계는 **조건에 따라 선택적으로** 다른 유스케이스의 기능을 사용하며, `<<extend>>` 스테레오타입으로 표기한다.
③ **일반화(Generalization)** 관계는 자식 유스케이스가 부모 유스케이스를 상속받는 관계이다.
④ **연관(Association)** 관계는 두 액터가 **서로의 기능을 호출**하는 관계를 의미한다.

### 문제 6 [유사 기출: 정처기 필기 2020·2021·2022·2024 반복 — SRP 정의 정형 패턴]

> "클래스(객체)는 **단 하나의 책임**만 가져야 하며, **변경의 이유도 오직 하나**여야 한다."

① **단일 책임 원칙 (SRP, Single Responsibility Principle)**
② 개방-폐쇄 원칙 (OCP, Open-Closed Principle)
③ 리스코프 치환 원칙 (LSP, Liskov Substitution Principle)
④ 의존 역전 원칙 (DIP, Dependency Inversion Principle)

### 문제 7 [유사 기출: 정처기 필기 2021·2022·2023 반복 — OCP 정형 키워드 패턴]
"소프트웨어 개체(클래스·모듈·함수 등)는 **확장에는 열려(open) 있어야 하고, 변경에는 닫혀(closed) 있어야 한다**"는 객체지향 설계 원칙은?

① SRP (Single Responsibility Principle)
② **OCP (Open-Closed Principle)**
③ LSP (Liskov Substitution Principle)
④ ISP (Interface Segregation Principle)

### 문제 8 [유사 기출: 정처기 필기 2020·2022·2023 반복 — DIP / 추상 의존 정형 패턴]
SOLID 원칙 중 **의존 역전 원칙(DIP, Dependency Inversion Principle)**에 대한 설명으로 가장 옳은 것은?

① 클래스는 자신이 사용하지 않는 메서드에 의존하도록 강요받지 않아야 한다.
② 자식 클래스는 부모 클래스를 완전히 대체할 수 있어야 한다.
③ **상위 모듈은 하위 모듈에 의존해서는 안 되며, 둘 다 추상화(인터페이스)에 의존해야 한다.**
④ 클래스는 단 하나의 책임만 가지며, 변경의 이유도 하나여야 한다.

## 🔍 오답 역추적 개념 발췌

### A. UML 다이어그램 14종 분류 (★ 최빈출 — 표 통째 암기)
| 구분 | 다이어그램 | 1줄 정의 |
|---|---|---|
| **구조 7** | **클래스(Class)** | 시스템의 정적 구조 — 클래스·속성·연산·관계 |
| | **객체(Object)** | 특정 시점의 클래스 인스턴스 스냅샷 |
| | **컴포넌트(Component)** | 물리적 컴포넌트(파일/라이브러리)와 의존 관계 |
| | **배치(Deployment)** | 하드웨어 노드 + 소프트웨어 배포 구조 |
| | **복합체 구조(Composite Structure)** | 클래스 내부 구조·포트·협력 관계 |
| | **패키지(Package)** | 관련 요소들의 그룹화 |
| | **프로파일(Profile)** | UML 메타모델 확장(스테레오타입 정의) |
| **행위 7** | **유스케이스(Use Case)** | 시스템 기능 + 액터 관점 요구사항 |
| | **시퀀스(Sequence)** | **시간 순서**대로 객체 간 메시지 흐름(수직 생명선) |
| | **통신(Communication)** | 객체 간 메시지 + 객체 관계 강조(공간 구조) |
| | **상태(State)** | 객체의 **상태 변화 + 이벤트**(동적 모델링) |
| | **활동(Activity)** | 처리 흐름·업무 절차(워크플로우) |
| | **상호작용 개요(Interaction Overview)** | 활동D + 시퀀스D 결합 |
| | **타이밍(Timing)** | 시간 축 + 상태 변화 그래프 |

> ★ "**플로우차트/순서도/절차/자료흐름도(DFD)/E-R/HIPO**"는 UML이 **아님** — 함정 출제 빈도 최고

### B. 구조 vs 행위 한 줄 식별
- **구조(Structural)** — **명사형·정적·시스템이 어떻게 구성**되어 있는가 (Class/Object/Component/Deployment/Composite/Package/Profile)
- **행위(Behavioral)** — **동사형·동적·시스템이 어떻게 동작**하는가 (Use Case/Sequence/Communication/State/Activity/Interaction Overview/Timing)

### C. 럼바우(Rumbaugh) 3대 모델링 (★ 정답 매칭 통째)
| 모델링 | 다이어그램 | 키워드 |
|---|---|---|
| **객체(Object)** 모델링 | **객체 다이어그램** | 시스템 정적 구조·클래스·관계 |
| **동적(Dynamic)** 모델링 | **상태 다이어그램(State)** | 상태 변화·이벤트·시간 흐름 |
| **기능(Functional)** 모델링 | **자료 흐름도(DFD)** | 입력→처리→출력·기능 분해 |

> 함정: "동적 모델링"에 **시퀀스 다이어그램**을 답으로 고르면 오답 — 럼바우 기법에서는 **상태(State)**가 정답.

### D. 유스케이스 4관계 (★ 빈출)
| 관계 | 표기 | 의미 |
|---|---|---|
| **연관(Association)** | 실선 | **액터 ↔ 유스케이스** 연결 (액터끼리 X) |
| **포함(Include)** | 점선 + `<<include>>` | 기능 **항상 포함** (필수 호출) — 화살표는 포함하는 → 포함되는 방향 |
| **확장(Extend)** | 점선 + `<<extend>>` | **조건부 선택적** 확장 — 화살표는 확장하는 → 확장되는(기준) 방향 |
| **일반화(Generalization)** | 실선 + 빈 삼각형 화살표 | 부모-자식 **상속** 관계(액터·유스케이스 모두 가능) |

> 함정: ④ "연관 = 액터끼리 서로의 기능을 호출" → 액터끼리의 관계는 **일반화**만 가능 (연관 X). 연관은 **액터-유스케이스 사이의 연결선**.

### E. SOLID 5원칙 (★ 단·개·리·인·의)
| 약어 | 풀네임 | 정의 | 함정 문구 |
|---|---|---|---|
| **S — SRP** | **Single Responsibility** Principle | 클래스는 **단 하나의 책임**만 / 변경의 이유는 하나 | "한 가지 일만 잘하라" |
| **O — OCP** | **Open-Closed** Principle | **확장에는 열려, 변경에는 닫혀** 있어야 | "기존 코드 수정 없이 새 기능 추가" |
| **L — LSP** | **Liskov Substitution** Principle | **자식이 부모를 완전히 대체** 가능해야 | "부모 자리에 자식 넣어도 정상 동작" |
| **I — ISP** | **Interface Segregation** Principle | **클라이언트는 사용하지 않는 메서드에 의존 X** / 인터페이스 작게 분리 | "뚱뚱한 인터페이스 → 여러 개로 쪼개라" |
| **D — DIP** | **Dependency Inversion** Principle | **상위·하위 모듈 모두 추상화(인터페이스)에 의존** / 구체 X | "구체 클래스가 아닌 **추상**에 의존" |

> ★ 두음 "**단개리인의**" — 단일/개방폐쇄/리스코프/인터페이스분리/의존역전

### F. SOLID 5원칙 — 1줄 매칭 정답 키워드
- "변경 이유 하나" / "단 하나의 책임" → **SRP**
- "확장 open / 수정 closed" → **OCP**
- "자식이 부모 대체" / "치환 가능" → **LSP**
- "사용하지 않는 메서드 의존 X" / "뚱뚱한 인터페이스 분리" → **ISP**
- "추상(인터페이스)에 의존" / "구체 클래스 의존 X" / "상위가 하위에 의존 X" → **DIP**

### G. UML 스테레오타입 표기 정리 (`<<...>>` = guillemet)
- `<<include>>` — 유스케이스 **필수 포함**
- `<<extend>>` — 유스케이스 **조건부 확장**
- `<<interface>>` — 클래스D에서 인터페이스 표기
- `<<abstract>>` — 추상 클래스
- `<<actor>>` — 비인간 액터(외부 시스템 등)

### H. UML 4+3+1 뷰(view) 모델 (Kruchten) — 보너스
- **논리(Logical) 뷰** — Class/Object 다이어그램
- **개발(Development) 뷰** — Component/Package 다이어그램
- **프로세스(Process) 뷰** — Sequence/Communication/Activity/State
- **물리(Physical) 뷰** — Deployment 다이어그램
- **+1 시나리오(Use Case) 뷰** — Use Case 다이어그램

## 💡 상세 해설

### 문제 1
**정답 ④ 순서도(Flowchart) 다이어그램**
- 순서도(Flowchart)는 UML이 **아닌** 별도 표기법. UML은 **구조 7 + 행위 7 = 14종**(시험상 13종 표기 다회 출제).
- **① 시퀀스 / ② 클래스 / ③ 유스케이스** — 모두 UML 표준 다이어그램.
- 빈출 함정 후보 (UML 아닌 것): **순서도(Flowchart) / 자료흐름도(DFD) / E-R 다이어그램 / HIPO / 절차 다이어그램**. 출제 시 이 5개 중 하나가 보기에 끼어 있으면 그게 정답.

### 문제 2
**정답 ③ 활동 다이어그램(Activity Diagram)**
- **활동(Activity)**은 처리 흐름·워크플로우를 동적으로 표현 → **행위(Behavioral)** 다이어그램.
- **① 클래스 / ② 객체 / ④ 배치** — 모두 정적 구조 표현 → **구조(Structural)** 다이어그램.
- 한 줄 식별: **구조 = 명사형·정적**, **행위 = 동사형·동적**.
> 시험 함정 보기 (구조로 착각하기 쉬운 행위): **시퀀스 / 통신 / 상태 / 활동 / 유스케이스 / 타이밍 / 상호작용 개요** — 모두 행위.

### 문제 3
**정답 ③ 시퀀스 다이어그램(Sequence Diagram)**
- 핵심 키워드 **"메시지 + 시간 순서 + 수직 생명선(lifeline)"** → 시퀀스 정형 문구.
- **① 클래스**: 정적 구조(메시지·시간 X).
- **② 컴포넌트**: 물리적 컴포넌트·의존.
- **④ 배치**: 하드웨어 노드·배포 토폴로지.
- 시퀀스와 헷갈리는 **통신(Communication)** 다이어그램: 같은 상호작용을 표현하지만 **공간 구조(객체 관계)**를 강조, 시간 순서는 시퀀스 번호로 부수적 표시.

### 문제 4
**정답 ③ 상태 다이어그램(State Diagram)**
- 럼바우(Rumbaugh) **OMT 3모델링**:
  - **객체** 모델링 = **객체 다이어그램**
  - **동적** 모델링 = **상태 다이어그램(State)** ← 본 문제 정답
  - **기능** 모델링 = **자료 흐름도(DFD)**
- **① 객체D** → 객체 모델링 (정적), **② DFD** → 기능 모델링, **④ E-R** → 럼바우와 무관(DB 모델링).
> 함정: "동적 모델링 = 시퀀스 다이어그램"으로 고르기 쉬우나, **럼바우 기법에서는 상태(State)가 정답**. UML 일반 동적 다이어그램은 시퀀스 포함이지만, 럼바우 OMT 기법에서는 상태 다이어그램으로 정해져 있음.

### 문제 5
**정답 ④** (틀린 설명)
- **연관(Association)** 관계는 **액터 ↔ 유스케이스** 사이의 **단순 연결선(실선)**. "두 액터가 서로의 기능을 호출"하는 관계가 아님.
- 액터끼리의 관계는 **일반화(Generalization)**만 가능 (한 액터가 다른 액터를 상속).
- **① 포함(Include)**: 정확한 설명. `<<include>>` 스테레오타입, 기능을 **반드시 포함**(필수).
- **② 확장(Extend)**: 정확한 설명. `<<extend>>` 스테레오타입, **조건부 선택**.
- **③ 일반화**: 정확한 설명. 자식이 부모를 상속받음.

### 문제 6
**정답 ① 단일 책임 원칙(SRP)**
- "**단 하나의 책임 / 변경의 이유도 하나**" → SRP 정형 정의.
- **② OCP**: 확장 open / 변경 closed.
- **③ LSP**: 자식이 부모를 대체.
- **④ DIP**: 추상에 의존, 구체 X.
> 두음 **"단개리인의"** — SRP는 첫 글자 **단**.

### 문제 7
**정답 ② OCP (Open-Closed Principle)**
- "**확장에는 열려(open), 변경에는 닫혀(closed)**" → OCP 정형 키워드. 한 글자도 안 바꾸고 출제됨.
- **① SRP**: 단 하나의 책임.
- **③ LSP**: 자식 ↔ 부모 치환.
- **④ ISP**: 인터페이스 분리.
> 함정: "수정 없이 새 기능 추가 가능"이라는 문구도 OCP의 동의어 — 같이 출제됨.

### 문제 8
**정답 ③**
- **DIP(Dependency Inversion Principle)** 정형 정의: "**상위 모듈은 하위 모듈에 의존해서는 안 되며, 둘 다 추상화(인터페이스)에 의존해야 한다**". 또한 "**추상화는 세부사항(구체 클래스)에 의존해서는 안 되며, 세부사항이 추상화에 의존해야 한다**".
- **① 사용하지 않는 메서드에 의존 X** → **ISP** (Interface Segregation).
- **② 자식이 부모를 완전히 대체** → **LSP** (Liskov Substitution).
- **④ 단 하나의 책임 / 변경 이유 하나** → **SRP** (Single Responsibility).
> 5문제 중 4문제가 SOLID 분류 매칭 패턴 — 보기 4개에 SOLID 정의를 흩뿌리는 형태가 다회 출제됨. 정의 ↔ 약어 1:1 매칭 통째 암기 필수.
