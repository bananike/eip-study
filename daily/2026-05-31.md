## 🔍 오답 역추적 개념 발췌

- DB 설계 5단계: 요구사항 분석 / 개념적 설계 / 논리적 설계 / 물리적 설계 / 구현
- 개념적 설계: E-R 다이어그램, 트랜잭션 모델링, 개념 스키마
- 논리적 설계: 테이블 설계, 정규화, 트랜잭션 인터페이스 설계, 스키마 평가·정제
- 물리적 설계: 저장 레코드 양식, 레코드 집중, 접근 경로, 응답 시간·디스크 용량 고려
- 분산 DB 6투명성: 분할 / 위치 / 지역사상 / 중복(복제) / 장애 / 병행
- 함정 투명성 후보: "응답 투명성", "이주 투명성", "성능 투명성" 등 — 분산 DB 6투명성에 없음
- 이산 단편화: 수평 분할(튜플 단위) / 수직 분할(속성 단위)
- NoSQL: 스키마-less, BASE, 수평 확장, 비정형 데이터, 분산 환경 최적화
- CAP: Consistency / Availability / Partition tolerance (3중 동시 만족 불가)
- BASE: Basically Available / Soft state / Eventually consistent
- NoSQL 4분류: Key-Value / Document / Column-family(Wide-column) / Graph
- 대표 제품 매핑(외움):
  - KV: Redis, DynamoDB, Riak
  - Document: MongoDB, CouchDB
  - Column-family: Cassandra, HBase, Bigtable
  - Graph: Neo4j, OrientDB

## 💡 상세 해설 (자기 설명과 대조)

### 문제 1 — 정답 ④
"트랜잭션 인터페이스 설계"는 **논리적 설계** 단계의 활동.
- ① 저장 레코드 양식 설계 → 물리적 설계 O
- ② 레코드 집중의 분석 및 설계 → 물리적 설계 O
- ③ 접근 경로(인덱스) 설계 → 물리적 설계 O
- ④ 트랜잭션 인터페이스 설계 → **논리적 설계** (오답 보기로 출제)

### 문제 2 — 정답 ②
분산 DB 6투명성: 분할 / 위치 / 지역사상 / 중복 / 장애 / 병행. "응답 투명성"은 없는 용어.
- ① 분할 투명성 O
- ② **응답 투명성** — 6투명성에 없음
- ③ 위치 투명성 O
- ④ 중복(복제) 투명성 O

### 문제 3 — 정답 ①
표준 순서: 요구사항 분석 → 개념적 → 논리적 → 물리적 → 구현. 개념(추상화)에서 물리(저장)로 가는 흐름. ②③④는 순서가 어긋남.

### 문제 4 — 정답 ④
개념적 설계의 산출물은 E-R 다이어그램·개념 스키마·트랜잭션 모델링.
- ①②③ 모두 개념적 설계 활동
- ④ **트랜잭션 인터페이스 설계**는 논리적 설계 활동

### 문제 5 — 정답 ④
NoSQL은 ACID 대신 **BASE**(Basically Available / Soft state / Eventually consistent)를 따른다. 분산·확장성·가용성을 위해 강한 일관성을 포기.
- ① 스키마 유연성 O
- ② Scale-out 용이 O
- ③ 비정형 처리 O
- ④ **ACID 엄격 보장 X** (BASE)

### 문제 6 — 정답 ④
CAP = Consistency(일관성) / Availability(가용성) / Partition Tolerance(분할 내성). **원자성(Atomicity)은 ACID 속성**으로 CAP과 다름.

### 문제 7 — 정답 ④
Graph DB 대표 제품은 Neo4j·OrientDB·ArangoDB 등. **MySQL·Oracle은 관계형 DBMS**이며 NoSQL Graph 저장소가 아님.

### 문제 8 — 정답 ②
저장 위치를 몰라도 논리명만으로 접근 = **위치 투명성**.
- ① 분할 투명성: 분할 여부를 모름
- ② 위치 투명성: 저장 위치를 모름 ← 정답
- ③ 중복 투명성: 사본이 여러 곳에 있음을 모름
- ④ 장애 투명성: 장애 발생을 모름(트랜잭션 원자성 유지)

## 📚 역추적 참조 (사후 검증용 — 학습 자료 아님)

### 핵심 용어
| 용어 | 정의 |
|---|---|
| DB 설계 (Database Design) | 사용자 요구사항을 분석하여 효율적·일관적 DB 구조를 만드는 과정 |
| 개념적 설계 (Conceptual Design) | 현실 세계를 추상화한 개념 스키마·E-R 모델 작성 |
| 논리적 설계 (Logical Design) | 개념 모델을 특정 DBMS에 맞춘 논리 스키마(테이블)로 변환·정규화 |
| 물리적 설계 (Physical Design) | 저장 매체에 어떻게 저장할지(파일·인덱스·접근 경로) 결정 |
| 분산 DB (Distributed Database) | 물리적으로 여러 사이트에 분산된 DB를 사용자에게 단일 DB처럼 보이게 하는 시스템 |
| 분할 투명성 (Fragmentation T.) | 데이터가 분할되어 있음을 사용자가 알 필요 없음 |
| 위치 투명성 (Location T.) | 데이터의 물리적 저장 위치를 알 필요 없음 |
| 지역사상 투명성 (Local Mapping T.) | 지역 DBMS와 물리 DB 간 매핑 보장, 지역 시스템 이름과 무관 |
| 중복(복제) 투명성 (Replication T.) | 데이터가 여러 사이트에 복제됨을 알 필요 없음 |
| 장애 투명성 (Failure T.) | 구성요소 장애에 무관하게 트랜잭션 원자성 유지 |
| 병행 투명성 (Concurrency T.) | 다수 트랜잭션 동시 수행 시 결과 일관성 유지 |
| NoSQL | "Not Only SQL". 비관계형·분산·확장형 DBMS의 총칭 |
| CAP 이론 | 분산 시스템은 C·A·P 중 동시에 둘만 만족 가능 |
| BASE | NoSQL이 따르는 완화된 일관성 모델 (BA·S·E) |

### 핵심 개념 (사후 검증용)

**1) DB 설계 5단계 — 활동·산출물 매핑**

| 단계 | 활동 | 산출물 |
|---|---|---|
| 요구사항 분석 | 사용자 인터뷰, 업무 분석 | 요구사항 명세서 |
| 개념적 설계 | E-R 모델링, 트랜잭션 모델링 | 개념 스키마 (E-R 다이어그램) |
| 논리적 설계 | 정규화, 테이블 설계, 트랜잭션 인터페이스 설계 | 논리 스키마 (관계 스키마) |
| 물리적 설계 | 저장 레코드 양식, 접근 경로(인덱스), 레코드 집중 | 물리 스키마 (저장 구조) |
| 구현 | DDL 작성, 트랜잭션 작성 | 실제 DB·응용 프로그램 |

**2) 분산 DB — 4대 목표 vs 6투명성**
- 4대 목표(전통적 분류): 위치·중복·병행·장애 투명성
- 확장 6투명성: + 분할 / + 지역사상
- 시험 출제는 "6투명성에 해당 안 되는 것" 형태가 빈출. 함정 보기: **응답 투명성 / 이주 투명성 / 성능 투명성 / 접근 투명성** (모두 분산 DB 6투명성에 없음)

**3) 분산 DB의 단편화 (Fragmentation)**
- 수평 분할(Horizontal): 튜플 단위로 행을 나눔 (지역별 고객 등)
- 수직 분할(Vertical): 속성 단위로 열을 나눔
- 혼합 분할(Mixed): 둘의 결합

**4) CAP 이론**
- C (Consistency): 모든 노드가 같은 시점에 같은 데이터
- A (Availability): 모든 요청이 응답 받음 (성공/실패 무관)
- P (Partition Tolerance): 네트워크 분단 발생 시에도 동작
- 분산 시스템은 동시에 2개만 선택. (예: CP — HBase, AP — Cassandra/Dynamo, CA — 전통 RDBMS)

**5) BASE 모델**
- BA (Basically Available): 항상 가용
- S (Soft state): 상태가 시간에 따라 변할 수 있음 (외부 입력 없이도)
- E (Eventually consistent): 결국 일관성에 도달

**6) NoSQL 4분류 상세**

| 유형 | 데이터 구조 | 대표 제품 | 강점 |
|---|---|---|---|
| Key-Value | (key, value) 쌍 | Redis, DynamoDB, Riak | 단순·고속, 캐시·세션 |
| Document | JSON/BSON 문서 | MongoDB, CouchDB | 반정형·중첩 구조 |
| Column-family | 행 키 + 컬럼 그룹 | Cassandra, HBase, Bigtable | 대용량 분산·시계열 |
| Graph | 노드·간선·속성 | Neo4j, OrientDB | 관계 탐색·소셜·추천 |

**7) RDBMS vs NoSQL 비교**

| 항목 | RDBMS | NoSQL |
|---|---|---|
| 스키마 | 고정 | 유연(스키마-less) |
| 트랜잭션 | ACID | BASE |
| 확장 | 수직(Scale-up) | 수평(Scale-out) |
| 쿼리 | SQL | 제품별 API |
| 일관성 | 강한 일관성 | 결과적 일관성 |
