### 문제 1 [유사 기출: 2021년 1회 5과목 — 폭포수(Waterfall) 모델 정형]
**폭포수(Waterfall) 모델**에 관한 설명 중 **가장 옳지 않은 것**은?

① **선형 순차 모델(Linear Sequential Model)** 로, 각 단계(**타당성 조사 → 요구 분석 → 설계 → 구현 → 시험 → 유지보수**)를 완료 후 다음 단계로 진행하며 위 단계로의 되돌아감(피드백)을 원칙적으로 제한한다.
② 각 단계 종료 시점에 **문서(산출물)** 를 완성해 승인 절차를 거치며, 요구사항이 **초기에 명확하게 정의**된 프로젝트·소규모 하드웨어 개발·안전이 중요한 임베디드에 적합하다.
③ 단점으로 **초기 요구사항 오류가 후반에 발견되면 수정 비용이 기하급수적으로 증가**하며, **동작하는 소프트웨어가 프로젝트 후반부에야 확인** 가능하고, **변화에 대한 대응력이 낮다**.
④ 폭포수 모델은 **위험 분석을 4단계로 반복**하며 **프로토타입을 지속적으로 정제**하는 **위험 주도(Risk-driven)** 접근법으로, **요구사항이 불명확한 대형 프로젝트**에 가장 적합하고 **역방향 피드백**을 원칙적으로 장려한다.

### 문제 2 [유사 기출: 2022년 2회 5과목 — 프로토타입(Prototype) vs 나선형(Spiral) 정형]
**프로토타입 모델**과 **나선형 모델**에 관한 설명 중 **가장 옳지 않은 것**은?

① **프로토타입 모델** — 사용자의 요구사항을 명확히 하기 위해 **시제품(prototype)** 을 만들어 사용자와 함께 검증·수정하는 모델. **사용자 요구 확정 후 폐기(throwaway prototyping)** 하거나 **점진 확장(evolutionary prototyping)** 하는 두 방식이 있다.
② **나선형 모델(Spiral Model, Boehm 1988)** — **계획 → 위험 분석 → 개발 → 고객 평가**의 4단계를 나선형으로 반복하며, **위험 관리(Risk Management)** 를 핵심 축으로 두어 **대규모·고위험 프로젝트**에 적합하다.
③ **나선형은 반복(iteration)마다 프로토타입을 만들어 위험을 조기 식별**하고, 사용자 평가를 통해 다음 나선(spiral)의 범위·목표를 재설정한다. 반복 횟수·비용 관리가 어렵다는 단점이 있다.
④ 프로토타입 모델은 **위험 분석을 4단계로 매 회 반복**하는 **위험 주도** 모델이고, **나선형 모델은 시제품 없이 문서만 순차 완성**하는 **폭포수의 별칭**이며 사용자 평가를 원칙적으로 배제한다.

### 문제 3 [유사 기출: 2020년 3회 5과목 — V-모델 검증 vs 확인 정형]
**V-모델(V-Model)** 과 **검증(Verification) · 확인(Validation)** 에 관한 설명 중 **가장 옳지 않은 것**은?

① **V-모델** — 폭포수 모델을 개선하여 **개발 단계(왼쪽 하강선)** 와 **시험 단계(오른쪽 상승선)** 를 **V자 형태로 대응**시킨 모델. 요구 분석 ↔ 인수 시험, 시스템 설계 ↔ 시스템 시험, 상세 설계 ↔ 통합 시험, 코딩 ↔ 단위 시험이 각각 짝을 이룬다.
② **검증(Verification)** — **"제품을 올바르게 만들고 있는가(Are we building the product right?)"**. 명세서·설계서 대비 산출물의 일치성을 확인하는 **개발자·정적 관점**의 활동(코드 리뷰·인스펙션·워크스루·정적 분석).
③ **확인(Validation)** — **"올바른 제품을 만들고 있는가(Are we building the right product?)"**. 사용자의 실제 요구·업무 목적 충족 여부를 확인하는 **사용자·동적 관점**의 활동(사용자 인수 시험·베타 테스트·필드 시험).
④ **V-모델의 오른쪽 시험은 왼쪽 개발 단계와 무관하게 무작위 순서로 진행**하며, **검증(Verification)은 사용자의 요구 충족 여부**를, **확인(Validation)은 개발자의 명세서 일치 여부**를 확인하는 활동으로 정의된다.

### 문제 4 [유사 기출: 2023년 1회 5과목 — 애자일(Agile) 4가치 정형]
**애자일 선언문(Agile Manifesto, 2001)** 의 **4가지 핵심 가치**에 관한 설명 중 **가장 옳은 것**은?

① **공정과 도구**보다 **개인과 상호작용**을, **포괄적인 문서**보다 **작동하는 소프트웨어**를, **계약 협상**보다 **고객과의 협력**을, **계획을 따르기**보다 **변화에 대응하기**를 더 가치 있게 여긴다.
② **개인과 상호작용**보다 **공정과 도구**를, **작동하는 소프트웨어**보다 **포괄적인 문서**를, **고객과의 협력**보다 **계약 협상**을, **변화에 대응하기**보다 **계획을 따르기**를 더 가치 있게 여긴다.
③ 애자일 4가치는 **초기 요구사항의 완전한 확정**과 **문서 우선주의**를 강조하며, 사용자와의 협력 없이 **개발팀 내부 결정**만으로 프로젝트를 진행할 것을 권장한다.
④ 애자일 4가치의 우측 항목(공정·도구·문서·계약·계획)은 **가치가 없다**는 의미이며, 좌측 항목만 사용해야 한다는 **극단적 배타 원칙**을 규정한다.

### 문제 5 [유사 기출: 2021년 2회 5과목 — XP(eXtreme Programming) 5가치·12실천 정형]
**XP(eXtreme Programming, Kent Beck)** 의 **5가지 핵심 가치**와 **12가지 실천 방법**에 관한 설명 중 **가장 옳지 않은 것**은?

① **5가치** — **의사소통(Communication) · 단순성(Simplicity) · 피드백(Feedback) · 용기(Courage) · 존중(Respect)**. 초기 4가지에 2004년 2판에서 **존중**이 추가되었다.
② **핵심 실천 방법** — **짝 프로그래밍(Pair Programming) · 테스트 주도 개발(TDD) · 지속적 통합(CI) · 리팩터링(Refactoring) · 소규모 릴리즈(Small Release) · 계획 게임(Planning Game) · 공동 소유(Collective Ownership) · 온-사이트 고객(On-Site Customer) · 40시간 근무(Sustainable Pace)** 등 12개.
③ **짝 프로그래밍** — 두 명이 하나의 컴퓨터 앞에서 **드라이버(코드 작성)** 와 **내비게이터(리뷰·전략)** 역할을 교대하며 즉시 코드 리뷰·지식 공유 효과를 얻는 방법.
④ XP의 5가치는 **문서화(Documentation) · 계획 준수(Plan-following) · 개인 소유(Individual Ownership) · 위험 회피(Risk Aversion) · 계층화(Hierarchy)** 이며, 짝 프로그래밍·TDD·CI는 XP가 아닌 **스크럼(Scrum)의 12실천 방법**이다.

### 문제 6 [유사 기출: 2022년 3회 5과목 — 스크럼(Scrum) 프레임워크 정형]
**스크럼(Scrum)** 프레임워크의 **3역할 · 5이벤트 · 3산출물**에 관한 설명 중 **가장 옳지 않은 것**은?

① **3역할** — **제품 책임자(Product Owner, PO)** = 제품 백로그 우선순위·가치 관리 / **스크럼 마스터(Scrum Master, SM)** = 스크럼 프로세스 촉진·장애 제거 / **개발 팀(Development Team)** = 스프린트 목표 달성·자기 조직화.
② **5이벤트** — **스프린트(Sprint, 1~4주) · 스프린트 계획(Sprint Planning) · 일일 스크럼(Daily Scrum, 15분) · 스프린트 리뷰(Sprint Review) · 스프린트 회고(Sprint Retrospective)**. 모두 **타임박스(time-box)** 로 제한된다.
③ **3산출물** — **제품 백로그(Product Backlog·전체 요구사항 우선순위 목록) · 스프린트 백로그(Sprint Backlog·해당 스프린트 작업 목록) · 증분(Increment·스프린트 종료 시 완성된 잠재 배포 가능 제품)**.
④ **스크럼의 스크럼 마스터(SM)** 는 **팀에 작업을 지시·통제하는 관리자 역할**을 수행하며, **일일 스크럼**은 **1시간 이상 상세 상태 보고**를 원칙으로 하고, **스프린트 기간은 6개월~1년 이상**으로 설정하는 것이 표준이다.

### 문제 7 [유사 기출: 2020년 2회 5과목 — CBD(Component Based Development) 정형]
**CBD(Component Based Development, 컴포넌트 기반 개발 방법론)** 에 관한 설명 중 **가장 옳지 않은 것**은?

① **재사용 가능한 컴포넌트(component)** — 독립적으로 배포·조립할 수 있는 소프트웨어 단위 — 를 **조립(assembly)** 하여 시스템을 개발하는 방법론으로, **개발 생산성·품질·유지보수성**을 동시에 높이는 것을 목표로 한다.
② 컴포넌트는 **인터페이스(interface)** 를 통해서만 외부와 상호작용하며, **내부 구현은 캡슐화(encapsulation)** 되어 있어 대체·확장이 용이하다. 대표 표준은 **EJB(Enterprise JavaBeans) · CORBA · COM/DCOM · .NET Assembly**.
③ CBD의 특징 — **재사용성·독립성·이식성·상호운용성** — 은 컴포넌트 간 **표준 인터페이스**와 **인터페이스 명세(IDL 등)** 에 의해 보장되며, 개발 프로세스는 **컴포넌트 식별 → 컴포넌트 설계 → 컴포넌트 구현·검증 → 컴포넌트 통합** 순으로 진행된다.
④ CBD는 **컴포넌트 재사용을 금지**하고 매번 **처음부터 절차지향(procedural)** 으로 새롭게 개발하는 방법론이며, **인터페이스·캡슐화 개념이 없고** 개발 생산성·품질과는 무관한 **하드웨어 조립 전용 방법론**이다.

### 문제 11 [유사 기출: 2020년 4회 5과목 — COCOMO 3유형 정형]
**COCOMO(COnstructive COst MOdel, Boehm 1981)** 의 **3가지 프로젝트 유형** 분류 중 **가장 옳지 않은 것**은?

① **Organic(조직형·단순형)** — **5만 LOC(50 KDSI) 이하**의 소규모·단순 프로젝트. 사무 처리·과학 계산·소규모 인력. 개발 노력(PM) = **2.4 × (KDSI)^1.05**.
② **Semi-detached(준분리형·중간형)** — **5만~30만 LOC** 규모의 중간 프로젝트. 트랜잭션 처리·DBMS·컴파일러. 개발 노력(PM) = **3.0 × (KDSI)^1.12**.
③ **Embedded(임베디드형·내장형)** — **30만 LOC 초과** 대규모·복잡한 실시간 시스템(운영체제·항공/미사일/실시간 통제). 개발 노력(PM) = **3.6 × (KDSI)^1.20**. 지수·상수가 세 유형 중 가장 크다.
④ COCOMO의 3유형은 **Organic(30만 LOC 초과·대규모)·Semi-detached(5만 LOC 이하·소규모)·Embedded(중간 규모)** 로 분류되며, 지수 상수의 크기 순서는 **Organic > Semi-detached > Embedded** 로 소규모일수록 노력 지수가 크다.

### 문제 12 [유사 기출: 2021년 1회 5과목 — 비용 산정 기법 분류 정형]
**소프트웨어 비용 산정 기법**의 분류와 대표 기법 매칭 중 **가장 옳지 않은 것**은?

① **하향식 산정(Top-down)** — **전문가 판단(Expert Judgement) · 델파이(Delphi) 기법** 등 프로젝트 전체를 개괄적으로 추정 후 세부 항목으로 분할하는 방식. **델파이 기법**은 익명 전문가 다수의 설문을 **반복(iteration)** 하여 합의를 도출한다.
② **상향식 산정(Bottom-up)** — **LOC(Line Of Code)·FP(Function Point)·COCOMO** 등 세부 항목(모듈·기능)의 비용을 추정 후 합산하는 방식. **LOC** 는 코드 라인 수 기반 3점 추정, **FP** 는 기능 관점(입출력·조회·파일·인터페이스 5요소)의 가중치 기반이다.
③ **수학적 산정 모형** — **COCOMO(Boehm·회귀 모델) · Putnam(Rayleigh-Norden 곡선·SLIM 도구) · FP(Albrecht·ESTIMACS/CHECKPOINT 도구)** 등 통계·수학 모델 기반. Putnam의 노력 곡선은 프로젝트 수명 주기에 걸쳐 Rayleigh 분포를 따른다.
④ **델파이(Delphi) 기법**은 **한 명의 최고 전문가가 단독 판단**하여 즉시 결정하는 기법이며, **FP(Function Point)** 는 코드 라인 수만을 계산하여 **LOC와 완전히 동일**하고, **Putnam 모델**은 회귀 분석 기반의 **COCOMO의 별칭**이다.

### 문제 13 [유사 기출: 2022년 2회 5과목 — PERT vs CPM 정형]
**PERT(Program Evaluation and Review Technique)** 와 **CPM(Critical Path Method)** 의 비교 중 **가장 옳지 않은 것**은?

① **PERT** — **확률적(probabilistic)** 일정 추정 기법. 각 작업의 소요 시간을 **낙관치(a)·기대치(m)·비관치(b)** 의 **3점 추정** 으로 산정하고, 기대 시간 = **(a + 4m + b) / 6** 로 계산한다. **연구·개발·신제품 프로젝트**처럼 소요 시간이 불확실한 경우에 적합.
② **CPM** — **확정적(deterministic)** 일정 추정 기법. 각 작업의 소요 시간을 단일 값으로 산정하고 **임계 경로(Critical Path)** 를 계산한다. **건설·플랜트·반복 경험이 있는 프로젝트**처럼 소요 시간이 확실한 경우에 적합.
③ **임계 경로(Critical Path)** = 프로젝트 시작에서 종료까지의 **가장 긴(longest) 경로** = 프로젝트 최소 완료 기간. 임계 경로 상의 활동은 **여유 시간(Slack·Float) = 0** 이며 지연 시 프로젝트 전체가 지연된다.
④ PERT는 **확정적** 기법으로 소요 시간을 단일 값으로 산정하며, CPM은 **확률적** 3점 추정을 사용하고, **임계 경로**는 프로젝트에서 **가장 짧은(shortest) 경로**를 의미하며 이 경로의 지연은 프로젝트 전체에 영향을 주지 않는다.

### 문제 14 [유사 기출: 2023년 3회 5과목 — Gantt 차트·프로젝트 관리 정형]
**Gantt 차트**와 **소프트웨어 프로젝트 관리(SPM, Software Project Management)** 에 관한 설명 중 **가장 옳지 않은 것**은?

① **Gantt 차트(Henry Gantt, 1917)** — 프로젝트의 각 작업(task)을 **가로 막대(bar)** 로 표현하여 **일정(시작·종료)·기간·진행률**을 시각적으로 나타내는 도구. 작업 간 **의존성 표현이 약한 것**이 단점.
② **PERT/CPM 네트워크 다이어그램**은 작업 간 **선후 관계(의존성)** 를 노드·화살표로 명시하고 **임계 경로**를 계산할 수 있지만, **일정 진행률의 시각적 확인**은 Gantt 차트가 더 직관적이다.
③ **소프트웨어 프로젝트 관리 3P** — **People(인력)·Problem(문제 정의·범위)·Process(프로세스·방법론)**. 세 요소의 균형이 프로젝트 성공의 핵심 요건이며, Brooks의 법칙("지연 프로젝트에 인력 추가 시 더 지연")이 대표 원칙.
④ Gantt 차트는 **작업 간 의존성·임계 경로를 자동 계산**하는 **확률적 네트워크 기법**이며, PERT/CPM은 **막대 그래프**로 일정을 표현할 뿐 임계 경로 개념이 없다. 프로젝트 3P는 **Program·Product·Process** 로 인력 요소는 포함되지 않는다.

## 💡 상세 해설

### 문제 1 — 폭포수(Waterfall) 모델
**핵심 한 줄**: 폭포수는 **선형 순차·문서 중심·되돌아감 제한**의 고전 모델. 요구사항이 초기에 확정된 소규모·안전 임계 프로젝트에 적합하나 후반부 오류 수정 비용·변화 대응력이 약점.
**오답 함정 키워드(④)**: "위험 분석 4단계 반복·프로토타입 정제·위험 주도" 는 **나선형(Spiral) 모델**의 특징이며, "요구사항이 불명확한 대형 프로젝트·역방향 피드백 장려"도 폭포수의 정의와 정면 반대. 폭포수 ↔ 나선형 개념 완전 뒤바꾸기 함정.
**실전 암기**: **"폭포수 = 위→아래 단방향·문서 완성·요구 확정 후 착수"**. 폭포수의 6단계 = **타 → 요 → 설 → 구 → 시 → 유** (타당성·요구·설계·구현·시험·유지보수).

### 문제 2 — 프로토타입 vs 나선형
**핵심 한 줄**: **프로토타입 = 시제품으로 요구 명확화** (throwaway/evolutionary 두 방식) · **나선형(Boehm 1988) = 계획·위험분석·개발·평가 4단계 나선 반복 · 위험 주도**. 대규모 고위험 프로젝트에 적합.
**오답 함정 키워드(④)**: 프로토타입과 나선형의 특성을 **완전 뒤바꿈**(위험 4단계 반복은 나선형 · 문서 순차는 폭포수). "나선형 = 폭포수의 별칭" 은 정의 자체 붕괴.
**실전 암기**: **"프로토타입 = 시제품·사용자 참여"** · **"나선형 = 위험분석 4단계 계속 회전"** · **"폭포수 = 문서 순차"**. Boehm 4단계 = **Plan → Risk Analysis → Development → Evaluation**.

### 문제 3 — V-모델 검증(Verification) vs 확인(Validation)
**핵심 한 줄**: **V-모델 = 폭포수 개선·개발과 시험을 V자로 대응**(요구↔인수·시스템설계↔시스템시험·상세설계↔통합시험·코딩↔단위시험). **Verification = "제품을 올바르게 만들고 있는가"(개발자·명세 일치)** · **Validation = "올바른 제품을 만들고 있는가"(사용자·요구 충족)**.
**오답 함정 키워드(④)**: V-모델의 좌우 대응을 "무작위"라 왜곡, 검증(Verification)과 확인(Validation)의 정의를 **정확히 반대로** 뒤바꿈. 시험 = 좌우 대응 필수.
**실전 암기**: **"V-erification = 명세대로 (개발자 관점 · Are we building the product Right?)"** · **"V-alidation = 사용자 원하는대로 (사용자 관점 · Are we building the Right product?)"**. V는 W-모델(양방향 W자·테스트 강화)로 확장됨.

### 문제 4 — 애자일 4가치
**핵심 한 줄**: **애자일 4가치** = ① 개인과 상호작용 > 공정·도구 ② 작동하는 SW > 포괄적 문서 ③ 고객과의 협력 > 계약 협상 ④ 변화 대응 > 계획 준수. **우측도 가치 있으나 좌측이 더 가치 있다**는 표현.
**오답 함정 키워드(②③④)**: ② 좌우 완전 반전 · ③ 문서 우선·사용자 배제로 왜곡 · ④ "우측은 가치 없음" 극단 배타 원칙은 애자일 선언문 명시 부정("that is, while there is value in the items on the right"). 4가치는 우선순위지 배제 원칙 아님.
**실전 암기**: **"개-작-고-변 > 공-문-계-계"** 로 좌·우 각 4단어 통암기. 애자일 12원칙은 4가치에서 파생.

### 문제 5 — XP(eXtreme Programming) 5가치·12실천
**핵심 한 줄**: XP 5가치 = **의사소통·단순성·피드백·용기·존중**(초기 4 + 2판 존중 추가). 12실천 = **짝 프로그래밍·TDD·CI·리팩터링·소규모 릴리즈·계획 게임·공동 소유·온-사이트 고객·40시간 근무·설계 개선·메타포어·코딩 표준**.
**오답 함정 키워드(④)**: XP 5가치를 **문서화·계획 준수·개인 소유·위험 회피·계층화** 로 완전 반대 왜곡(오히려 XP가 배격하는 요소들). 짝 프로그래밍·TDD·CI는 **XP 12실천의 대표 3가지**이지 스크럼의 것이 아님.
**실전 암기**: **"의-단-피-용-존"** 5가치 5글자 통암기. XP = 코드 중심·2주 이하 짧은 반복. Kent Beck·1996 Chrysler C3 프로젝트에서 시작.

### 문제 6 — 스크럼(Scrum) 프레임워크
**핵심 한 줄**: **3역할** = PO(제품 백로그 관리) · SM(프로세스 촉진·장애 제거·**서번트 리더**) · Dev Team(자기 조직화). **5이벤트** = 스프린트(1~4주)·계획·일일 스크럼(15분)·리뷰·회고. **3산출물** = 제품 백로그·스프린트 백로그·증분.
**오답 함정 키워드(④)**: SM을 "관리자·통제자"로 왜곡(SM은 서번트 리더·촉진자), 일일 스크럼을 "1시간 이상 상세 보고"로 왜곡(15분 타임박스), 스프린트를 "6개월~1년"으로 왜곡(1~4주). 세 개념 모두 정면 위배.
**실전 암기**: **"스크럼 3역할 = PO·SM·Team"** · **"5이벤트 = Sprint(1~4주)·Planning·Daily(15분)·Review·Retro"** · **"3산출물 = Product·Sprint·Increment 백로그+증분"**. SM = 서번트 리더, 관리자 아님.

### 문제 7 — CBD(Component Based Development)
**핵심 한 줄**: CBD = **재사용 가능한 컴포넌트를 조립**하여 시스템 개발. **인터페이스로만 상호작용·캡슐화·독립 배포**. 표준 = **EJB·CORBA·COM/DCOM·.NET Assembly**. 특징 4C = **재사용성·독립성·이식성·상호운용성**.
**오답 함정 키워드(④)**: CBD의 핵심(재사용·인터페이스·캡슐화)을 완전 부정, "절차지향으로 새로 개발·하드웨어 조립 전용"으로 정의 왜곡. CBD는 소프트웨어 개발 방법론이며 재사용성이 핵심.
**실전 암기**: **"CBD = Component + 재사용 + 조립 + 인터페이스"**. 프로세스 4단계 = **식별 → 설계 → 구현·검증 → 통합**. 대표 예: Java Spring Bean, .NET Assembly, Web Service.

### 문제 8 — CMMI 5단계 성숙도
**핵심 한 줄**: **CMMI 5단계 = 초·관·정·정·최** = **Initial(1) → Managed(2) → Defined(3) → Quantitatively Managed(4) → Optimizing(5)**. 미국 SEI·카네기멜론대. CMM의 후속(2000년 CMMI로 통합).
**오답 함정 키워드(②③④)**: ② 관리·정의 순서 반전, ③ 초기·정의 뒤바꾸기, ④ 5단계 완전 무작위 배열. 순서 암기가 정답의 핵심.
**실전 암기**: **"초·관·정·정·최"** = **"초기 → 관리 → 정의 → 정량적 관리 → 최적화"**. 1단계 예외적으로 활동 없음(카오스), 2~5는 프로세스 성숙도 점증. CMMI-DEV(개발)·CMMI-SVC(서비스)·CMMI-ACQ(획득) 3분야.

### 문제 9 — SPICE(ISO/IEC 15504) 6단계
**핵심 한 줄**: **SPICE 6단계 = 불·수·관·확·예·최** = **0.Incomplete(불완전) → 1.Performed(수행) → 2.Managed(관리) → 3.Established(확립) → 4.Predictable(예측) → 5.Optimizing(최적화)**. **0단계부터 시작**하는 것이 CMMI(1부터)와의 차이.
**오답 함정 키워드(②③④)**: ② CMMI 5단계와 헷갈리게 왜곡("6.완전" 추가), ③ 수행·불완전 순서 반전, ④ 5→0 완전 역순 배열. **SPICE는 0.불완전부터 시작**이 최대 트랩.
**실전 암기**: **"불·수·관·확·예·최"** 6글자 통암기. **CMMI = 1~5단계 · SPICE = 0~5단계**. SPICE는 유럽 주도 ISO 표준·CMMI는 미국 SEI·현재는 ISO/IEC 33000 시리즈로 발전.

### 문제 10 — LOC 3점 추정 공식 계산
**핵심 한 줄**: LOC 3점 추정 공식 = **`(a + 4m + b) / 6`** (Beta 분포 기대치). 대입 = **(2000 + 4×5000 + 8000) / 6 = (2000 + 20000 + 8000) / 6 = 30000 / 6 = 5000 LOC** = ②.
**오답 함정 키워드(①③④)**: 공식을 (a+m+b)/3(단순 평균 = (2000+5000+8000)/3 = 5000, 우연 일치도 있으나 공식 자체 오류), (a+2m+b)/4, (a+6m+b)/8 등으로 왜곡한 계산 함정. 반드시 **4를 m(기대치)에 곱한다** 는 것이 핵심.
**실전 암기**: **"a + 4m + b / 6"** (기대치에 4를 곱한 뒤 6으로 나눔·PERT와 동일 공식). 노력(인월) = **LOC / 1인당 월 생산성**. 개발 기간(월) = **노력 / 인원**.

### 문제 11 — COCOMO 3유형
**핵심 한 줄**: **COCOMO 3유형** = **Organic(5만 LOC 이하·PM=2.4×KDSI^1.05) · Semi-detached(5만~30만·PM=3.0×KDSI^1.12) · Embedded(30만↑·PM=3.6×KDSI^1.20)**. 임계 = **5만·30만 LOC**. 지수·상수 = Organic < Semi < Embedded.
**오답 함정 키워드(④)**: 3유형의 규모 범위를 **완전 뒤바꾸기**(Organic ↔ Embedded 역전), 지수 순서도 정반대로 왜곡. Organic이 5만 이하·Embedded가 30만 초과 임을 반드시 암기.
**실전 암기**: **"오르가닉 5만·세미 30만·임베디드 초과"** = **"5-30-∞"**. Boehm(1981) 원저·COCOMO II(1997)로 개정(Application Composition·Early Design·Post-Architecture 3모델).

### 문제 12 — 비용 산정 분류
**핵심 한 줄**: **하향식 = 전문가 판단·델파이**(전체 → 세부) · **상향식 = LOC·FP·COCOMO**(세부 → 합산) · **수학적 = COCOMO·Putnam·FP**. **델파이 = 익명 전문가 다수·반복 설문 합의** · **Putnam = Rayleigh 곡선·SLIM 도구** · **FP = 기능 관점·Albrecht** · **LOC = 코드 라인 기반**.
**오답 함정 키워드(④)**: 델파이를 "한 명의 최고 전문가 단독 판단"으로 왜곡(핵심은 **다수·익명·반복**), FP와 LOC를 동일시(FP는 코드 무관·기능 관점), Putnam을 COCOMO의 별칭으로 왜곡(둘 다 수학 모델이나 별개).
**실전 암기**: **"델파이 = 다수·익명·반복·합의"** · **"FP = 기능(입출력·조회·파일·인터페이스 5요소)"** · **"Putnam = Rayleigh·SLIM"** · **"COCOMO = 회귀·Boehm"**. 두 도구 매칭 필수.

### 문제 13 — PERT vs CPM
**핵심 한 줄**: **PERT = 확률적·3점 추정 (a+4m+b)/6·R&D 신제품** · **CPM = 확정적·단일 추정·건설/반복 프로젝트**. **임계 경로(Critical Path) = 가장 긴 경로 = 최소 완료 기간 · Slack = 0 · 지연 시 전체 지연**.
**오답 함정 키워드(④)**: PERT/CPM의 확률적/확정적 특성을 **완전 뒤바꿈**, 임계 경로를 **"가장 짧은 경로"** 로 정의 정반대 왜곡(핵심 트랩). 임계 경로 = 가장 긴 경로·Slack=0 반드시 암기.
**실전 암기**: **"PERT = 확률적·3점·R&D"** · **"CPM = 확정적·단일·건설"** · **"임계 경로 = 가장 긴 경로·Slack=0·지연=전체 지연"**. Slack = 여유 시간(EF-LF), 0이면 임계.

### 문제 14 — Gantt·SPM 3P
**핵심 한 줄**: **Gantt 차트 = 막대 시각화·일정·진행률·의존성 표현 약함** · **PERT/CPM = 네트워크·의존성·임계 경로**. **SPM 3P = People(인력)·Problem(문제)·Process(프로세스)**. **Brooks 법칙 = 지연 프로젝트에 인력 추가 시 더 지연** (『Mythical Man-Month』).
**오답 함정 키워드(④)**: Gantt와 PERT/CPM의 기능을 **완전 뒤바꿈**(Gantt = 막대·PERT = 네트워크), 3P를 **Program·Product·Process**로 왜곡(People이 핵심). SPM 3P는 P(사람)-P(문제)-P(프로세스) 순.
**실전 암기**: **"Gantt = 막대 그래프·시각화·의존성 약함"** · **"PERT/CPM = 네트워크·의존성·임계 경로"** · **"3P = People·Problem·Process"** · **"Brooks 법칙 = 늦은 프로젝트에 인력 = 더 늦어짐"**.

### 문제 15 — 임계 경로 손추적
**핵심 한 줄**: 경로별 소요 = **① S–A–B–E–F: 3+5+4 = 12일** · **② S–A–C–D–E–F: 3+2+6+4 = 15일** · **③ S–A–C–G–F: 3+2+7 = 12일**. 최댓값 = **15일**·임계 경로 = **S–A–C–D–E–F** = ②.
**오답 함정 키워드(①③④)**: ① B 경로만 12일로 선택(가장 긴 경로 아님) · ③ G 경로 12일 선택(15일보다 짧음) · ④ 9일은 세 경로 모두 계산 오류. **최소 완료 기간 = 가장 긴 경로 = 병렬 작업이라도 가장 오래 걸리는 경로가 결정**.
**실전 암기**: **"임계 경로 찾기 3단계"** = ① 모든 경로 열거 → ② 각 경로 소요 합산 → ③ **최댓값** 선택. 계산 후 **모든 경로 소요를 다 더하지 말 것**(합이 아니라 최댓값). Slack = 임계 경로 소요 - 해당 활동의 최장 경로 = 여유.
