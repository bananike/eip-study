### 문제 3 [유사 기출: 정처기 필기 2020년 3회·2022년 2회·2024년 3회 반복 — COCOMO 유형 매칭 정형 패턴]
**COCOMO(COnstructive COst MOdel)** 의 프로젝트 **유형(Mode)** 에 대한 설명으로 가장 옳지 **않은** 것은?

① **Organic(조직형)** — 일반 업무용 응용 소프트웨어로, **5만 라인 이하**의 소규모 프로젝트에 적합하다.
② **Semi-detached(반결합형·중간형)** — 트랜잭션 처리·DBMS 등 **30만 라인 이하**의 중간 규모 프로젝트에 적합하다.

④ **Functional(기능형)** — 사용자 요구 기능을 단위로 산정하며, **알브레히트(Albrecht)** 가 제안한 **Function Point** 기법과 동일한 COCOMO 유형이다.

### 문제 4 [유사 기출: 정처기 필기 2021년 2회·2023년 1회·2025년 2회 반복 — Putnam·FP 자동화도구 매칭 정형 패턴]
소프트웨어 비용 산정 **수학적 기법**과 그 **자동화 도구(Tool)** 의 매칭으로 가장 옳지 **않은** 것은?

① **Putnam 모형** — **Rayleigh-Norden 곡선**의 노력 분포를 이용하며, 자동화 도구로 **SLIM(Software Life-cycle Management)** 을 사용한다.

③ **COCOMO 모형** — **보엠(Boehm)** 이 제안하였으며, **LOC 기반** 노력·기간·인력을 산정하는 수학적 모형이다.

### 문제 7 [유사 기출: 정처기 필기 2020년 1회·2022년 2회·2024년 3회 반복 — V&V 정의 비교 정형 패턴]
소프트웨어 **검증(Verification)** 과 **확인(Validation)** 에 대한 설명으로 가장 옳은 것은?

① **검증(Verification)** = "올바른 제품을 만들고 있는가(Building the **right** product)" 평가, **확인(Validation)** = "제품을 올바르게 만들고 있는가(Building the product **right**)" 평가.
② **검증(Verification)** = 개발 단계 산출물이 그 단계의 명세 조건을 만족하는지를 평가(개발자 관점), **확인(Validation)** = 최종 시스템이 사용자 요구사항을 만족하는지를 평가(사용자 관점).
③ 검증과 확인은 동일한 개념이며 IEEE 표준에서도 상호 교환적으로 사용된다.
④ 검증은 시스템 인수 단계에서 사용자가 수행하고, 확인은 단위 테스트 단계에서 개발자가 수행한다.

### 문제 8 [유사 기출: 정처기 필기 2021년 2회·2023년 1회·2025년 2회 반복 — PERT/CPM/Gantt 비교 정형 패턴]
프로젝트 **일정 관리 기법**에 대한 설명으로 가장 옳지 **않은** 것은?

① **PERT(Program Evaluation and Review Technique)** 는 **낙관치·기대치·비관치 3점 추정**을 사용하며, **불확실성이 높은** 프로젝트에 적합한 **확률적** 일정 모형이다.
② **CPM(Critical Path Method)** 의 **임계 경로(Critical Path)** 는 네트워크상의 경로 중 **가장 긴 경로**를 의미하며, 프로젝트 일정을 단축할 때 **우선 검토 대상**이다.
③ **Gantt 차트(간트 차트)** 는 작업 일정을 **막대 도표**로 표시한 시간선 차트로, 각 작업의 시작·종료·소요 기간을 한눈에 보여준다.
④ CPM의 **임계 경로**는 네트워크상의 경로 중 **가장 짧은 경로**를 의미하며, 이 경로상의 작업은 **여유 시간(Slack)이 가장 많아** 일정 변경의 우선 대상이 된다.

## 🔍 오답 역추적 개념 발췌

### 1. 비용 산정 3분류 — 두음 "하·상·수"
| 분류 | 기법 | 특징 |
|---|---|---|
| **하향식(Top-Down)** | **전문가 감정 / 델파이(Delphi)** | 경영진·전문가가 전체 비용 먼저 산정 후 단위 분배 / 델파이 = **익명 라운드 + 중재자** 의견 수렴 |
| **상향식(Bottom-Up)** | **LOC / 단계별 인월수(M/M) / 표준 작업 분배** | 모듈·작업 단위 산정 후 합산 |
| **수학적(Algorithmic)** | **COCOMO / Putnam / Function Point(FP)** | 통계·수식 기반 자동 산정 |

> **함정 단골**: **델파이를 상향식·수학적**으로 분류 → **X**(델파이는 **하향식**). **COCOMO를 상향식**으로 → X(수학적). **LOC를 하향식**으로 → X(상향식). 1:1 매칭 외워 즉답.

### 2. LOC 노력(M/M) 산정 공식
- **추정 LOC = (낙관치 + 4×기대치 + 비관치) / 6** (3점 추정 가중평균 — PERT와 동일 형식)
- **노력(M/M) = 추정 LOC ÷ (개발자 1인당 월간 생산 LOC)**
- **개발 비용 = 노력(M/M) × 개발자 1인당 월 인건비**
- **개발 기간 = 노력(M/M) ÷ 투입 인력(명)**

> **계산형 정형 함정**: 낙관/비관 가중치를 잘못 적용(예: 단순 산술평균 ÷ 3) → X. **반드시 (a + 4m + b) / 6** 공식.

### 3. COCOMO 3유형 — 라인 임계치 + 영역
| 유형 (Mode) | 라인 임계 | 적용 영역 |
|---|---|---|
| **Organic (조직형·단순)** | **5만 라인 이하** | 일반 업무용·과학 응용·트랜잭션 단순 |
| **Semi-detached (반결합형·중간)** | **30만 라인 이하** | 트랜잭션·DBMS·컴파일러 |
| **Embedded (내장형·복잡)** | **30만 라인 초과** | 신호처리·실시간 OS·항공우주·미사일 |

**COCOMO 3종 (상세도)**: **Basic / Intermediate / Detailed** — 모형 정밀도 단계. 유형(Mode)과 혼동 금지.

> **함정 단골**: **Functional·Sequential·Procedural·Distributed** 같은 가짜 4번째 유형 끼워넣기 → **X**. COCOMO는 **3개 유형 + 3개 상세도** 고정.

### 4. 수학적 산정 기법 ↔ 자동화 도구 1:1 매칭
| 기법 | 제안자 | 핵심 | 자동화 도구 |
|---|---|---|---|
| **COCOMO** | **Boehm(보엠·1981)** | LOC 기반 노력·기간·인력 산정 | (COCOMO II 워크북·USC 도구) |
| **Putnam** | **Putnam(1978)** | **Rayleigh-Norden 곡선** 노력 분포(생명주기 시간) | **SLIM (Software Life-cycle Management)** |
| **Function Point (FP)** | **Albrecht(알브레히트·1979)** | 기능 5종 가중치 측정 | **ESTIMACS** |

> **함정 단골**: **델파이를 수학적·Rayleigh 기반**으로 → **X**(델파이는 하향식 전문가 의견 수렴). **SLIM을 FP에 매칭** → X(SLIM은 Putnam). **ESTIMACS를 COCOMO에 매칭** → X(ESTIMACS는 FP).

### 5. Function Point(FP) — 알브레히트 5종 기능 단위
| 기능 단위 | 약어 | 정의 |
|---|---|---|
| **외부 입력 (External Input)** | EI | 외부에서 시스템으로 데이터 입력 |
| **외부 출력 (External Output)** | EO | 시스템에서 외부로 결과 출력 |
| **외부 조회 (External inQuiry)** | EQ | 입력 + 즉시 응답 출력 |
| **내부 논리 파일 (Internal Logical File)** | ILF | 시스템 내부 유지 데이터 |
| **외부 인터페이스 파일 (External Interface File)** | EIF | 외부 시스템과 공유 데이터 |

> **함정 단골**: "FP는 LOC 기반" → **X**(FP는 **기능 기반·언어 독립적**). "FP는 알고리즘 복잡도 측정" → X(기능 가중치 합산).

### 6. CMMI 5단계 — 두음 "초·관·정·정·최"
| 단계 | 영문 | 핵심 키워드 |
|---|---|---|
| **1. Initial (초기)** | Initial | 임시방편·개인 영웅주의·예측 불가 |
| **2. Managed (관리)** | Managed | 프로젝트 단위 관리·기본 프로세스 |
| **3. Defined (정의)** | Defined | 조직 표준 프로세스 정의·일관성 |
| **4. Quantitatively Managed (정량적 관리)** | Quantitatively Managed | 통계적 관리·정량 지표 |
| **5. Optimizing (최적화)** | Optimizing | 지속 개선·혁신·결함 예방 |

> **함정 단골**: **Managed ↔ Defined 순서 뒤바꿈** (Initial 다음은 **Managed**, 그 다음 Defined). "Initial → Defined → Managed" 순서 보이면 즉시 X.

### 7. SPICE(ISO/IEC 15504) 6단계 — 두음 "불·수·관·확·예·최"
| 수준 | 단계 | 영문 | 핵심 |
|---|---|---|---|
| **0** | **불완전(Incomplete)** | Incomplete | 미구현·미달성 |
| **1** | 수행(Performed) | Performed | 기본 수행 |
| **2** | 관리(Managed) | Managed | 계획·통제 |
| **3** | 확립(Established) | Established | 표준 프로세스 확립 |
| **4** | 예측(Predictable) | Predictable | 정량적 예측 가능 |
| **5** | 최적화(Optimizing) | Optimizing | 지속 개선 |

> **함정 단골**: **"정형(Formal) 단계"** → **X**(SPICE에 없는 가짜 단계). "정의(Defined)" → X(CMMI 용어). SPICE는 **0단계부터 시작** + 6단계.

### 8. CMMI vs SPICE 비교
| 구분 | CMMI | SPICE |
|---|---|---|
| 출처 | 미국 SEI(소프트웨어 공학 연구소) | ISO/IEC 15504 (국제표준) |
| 단계 수 | **5단계 (1~5)** | **6단계 (0~5)** |
| 시작 단계 | Initial(1) | **Incomplete(0)** |
| 최고 단계 | Optimizing(5) | Optimizing(5) |

> **함정 단골**: CMMI를 6단계·SPICE를 5단계로 → X. SPICE만 0단계(불완전) 보유.

### 9. V&V — Verification vs Validation 1:1 매칭
| 구분 | Verification(검증) | Validation(확인) |
|---|---|---|
| 영문 슬로건 | Building the product **right** | Building the **right** product |
| 의미 | "올바르게 만들고 있는가" | "올바른 것을 만들고 있는가" |
| 평가 대상 | 각 **단계 산출물 ↔ 명세** 일치 | **최종 시스템 ↔ 요구사항** 만족 |
| 관점 | **개발자 관점** | **사용자 관점** |
| 시점 | 개발 전 과정(단위·통합 테스트 등) | 시스템·인수 테스트 |

> **함정 단골**: **두 정의 슬로건 뒤바꿈**("검증=올바른 제품" / "확인=올바르게 만들기") → X. **right product vs product right** — "**검증=right로 끝**(부사·올바르**게**), 확인=**right** product(형용사·올바른)" 으로 외움.

> **함정 단골**: "임계 경로 = **가장 짧은** 경로" → **X**(정확히 반대). "임계 경로상 작업은 여유 시간이 가장 **많다**" → **X**(여유 시간 **0**). "Gantt = 확률적 3점 추정" → X(Gantt는 시간선 표시일뿐 추정 기법 X).

### 문제 1
**정답 ④ 델파이(Delphi) 기법 — 상향식 산정 기법으로, 개발자가 작성한 단위 모듈의 LOC를 합산하여 노력을 추정한다.**
- **델파이(Delphi) 기법은 하향식(Top-Down) 산정 기법** — **전문가 감정 기법의 단점(개인 편향)을 보완**하기 위해 **여러 전문가의 익명 의견을 중재자(Coordinator)가 라운드로 수렴**하는 방식. LOC 합산과는 무관.
- ① 하향식 = **전문가 감정 + 델파이**. ✔
- ② 상향식 = **LOC + 단계별 인월수(M/M)**. ✔
- ③ 수학적 = **COCOMO + Putnam + FP**. ✔
- 정형 함정: **델파이를 상향식·수학적**으로 분류 / **LOC를 하향식**으로 분류 / **COCOMO를 상향식**으로 분류 — 모두 X. 분류 3대 그룹 1:1 매칭 즉답.

### 문제 2
**정답 ③ 추정 LOC = 1,700 · 노력 = 약 5.67 M/M**
- **추정 LOC** = (낙관치 + 4×기대치 + 비관치) / 6 = (1,200 + 4×1,500 + 2,400) / 6 = (1,200 + 6,000 + 2,400) / 6 = **10,200 / 6 = 1,700 라인**.
- **노력(M/M)** = 추정 LOC ÷ 1인당 월 생산 LOC = **1,700 ÷ 300 ≈ 5.67 M/M**.
- ① **1,500**: 단순히 기대치만 사용(공식 무시). X
- ② **1,600**: 산술평균 (1,200 + 1,500 + 2,400) / 3 = 1,700인데 잘못 계산. X
- ④ **1,800**: 가중치 잘못 적용(낙관·비관에 4 적용 등). X
- 정형 계산 함정: **반드시 (a + 4m + b) / 6** — PERT 3점 추정과 동일 공식. 가중평균 분모 6 고정.

### 문제 3
**정답 ④ Functional(기능형) — 알브레히트 FP 기법과 동일한 COCOMO 유형이다.**
- **COCOMO에 "Functional" 유형은 존재하지 않음**. COCOMO 유형(Mode)은 **Organic / Semi-detached / Embedded 3가지뿐**. Function Point는 **별개의 비용 산정 기법**으로 알브레히트가 제안한 **기능 기반·언어 독립적** 모형 — COCOMO와 무관.
- ① **Organic**: 일반 업무용, **5만 라인 이하** 소규모. ✔
- ② **Semi-detached**: 트랜잭션·DBMS·컴파일러, **30만 라인 이하**. ✔
- ③ **Embedded**: 실시간 OS·항공우주, **30만 라인 초과**. ✔
- 정형 함정: **Functional·Sequential·Procedural·Distributed** 같은 가짜 4번째 유형 끼워넣기 — 모두 X. COCOMO는 **유형 3 + 상세도 3(Basic·Intermediate·Detailed)** 고정.

### 문제 4
**정답 ④ 델파이(Delphi) 기법 — Rayleigh-Norden 곡선 기반 수학적 산정 기법으로, 자동화 도구 COCOMO II 를 사용한다.**
- **델파이는 하향식 전문가 의견 수렴 기법**으로, **Rayleigh-Norden 곡선과 무관**하며 **자동화 도구 매칭 없음**. **Rayleigh-Norden 곡선 = Putnam 모형**, **자동화 도구 SLIM**.
- ① **Putnam — Rayleigh-Norden + SLIM**. ✔
- ② **FP — Albrecht 제안 + ESTIMACS**. ✔
- ③ **COCOMO — Boehm 제안 + LOC 기반**. ✔
- 정형 함정: **델파이를 수학적·Rayleigh 기반**으로 분류 → X. **SLIM·ESTIMACS·COCOMO II** 자동화 도구 1:1 매칭 외워 즉답.

### 문제 5
**정답 ② Initial(초기) → Managed(관리) → Defined(정의) → Quantitatively Managed(정량적 관리) → Optimizing(최적화)**
- **CMMI 5단계 두음 "초·관·정·정·최"** 외워 즉답 — Initial(1) → **Managed(2)** → **Defined(3)** → Quantitatively Managed(4) → Optimizing(5).
- **Managed가 Defined보다 먼저**(2단계 = 프로젝트 단위 관리 시작, 3단계 = 조직 표준 프로세스 정의·일관성).
- ①·④: **Initial 다음에 Defined** 끼워넣음 — X(Initial 다음은 Managed).
- ③: **Quantitatively Managed가 Defined보다 먼저** — X(Defined → Quantitatively Managed 순서).
- 정형 함정: **Managed ↔ Defined 순서 뒤바꿈** 단골. "**관리 먼저 → 정의**" 두음 고정.

### 문제 6
**정답 ③ 정형(Formal) 단계**
- **SPICE에 "정형(Formal)" 단계는 존재하지 않음** — 가짜 항목. SPICE 6단계: **불완전(0) → 수행(1) → 관리(2) → 확립(3) → 예측(4) → 최적화(5)**.
- ① **불완전(Incomplete) — 0** 수준. ✔ (SPICE만 0단계 보유)
- ② **확립(Established) — 3** 수준. ✔
- ④ **최적화(Optimizing) — 5** 수준. ✔
- 정형 함정: **"정형(Formal)"·"정의(Defined·CMMI 용어)"·"초기(Initial·CMMI 용어)"** 같은 가짜·타 모델 용어 끼워넣기 — 모두 X. SPICE 두음 **"불·수·관·확·예·최"** 6단계 고정.

### 문제 7
**정답 ② 검증(Verification) = 개발 단계 산출물이 그 단계의 명세 조건을 만족하는지를 평가(개발자 관점), 확인(Validation) = 최종 시스템이 사용자 요구사항을 만족하는지를 평가(사용자 관점).**
- **Verification(검증)** = "Building the **product right**"(올바르**게** 만드는가) = **개발 단계 산출물 ↔ 명세** 일치 = **개발자 관점**.
- **Validation(확인)** = "Building the **right** product"(올바**른** 제품 만드는가) = **최종 시스템 ↔ 사용자 요구사항** 일치 = **사용자 관점**.
- ① 슬로건이 **정확히 뒤바뀜** — "검증 = 올바른 제품"·"확인 = 올바르게 만들기" → X.
- ③ **별개 개념** — IEEE 표준에서도 명확히 구분(IEEE 1012 등). X
- ④ 수행자·시점 뒤바뀜 — **검증은 개발자**(개발 전 과정), **확인은 사용자**(인수 단계). X
- 정형 함정: 두 정의 슬로건 뒤바꿈 단골. **"검증=right로 끝(부사·올바르게)" / "확인=right product(형용사·올바른)"** 으로 외움.

### 문제 8
**정답 ④ CPM의 임계 경로(Critical Path)는 네트워크상 가장 짧은 경로를 의미하며, 여유 시간(Slack)이 가장 많아 일정 변경 우선 대상이다.**
- **임계 경로 = 네트워크상 가장 긴 경로** = 프로젝트 **최소 소요 시간** = **여유 시간(Slack/Float) = 0** — 한 작업이라도 지연되면 전체 일정 지연. 일정 단축 시 **우선 검토 대상**.
- ① **PERT**: **3점 추정(낙관·기대·비관) + 확률적** + 불확실 프로젝트(R&D 등). ✔
- ② **CPM**: 임계 경로 = **가장 긴 경로** + 일정 단축 우선 검토. ✔
- ③ **Gantt 차트**: **막대 도표 시간선** + 작업 시작·종료·기간 시각화. ✔
- 정형 함정: **"임계 경로 = 가장 짧은 경로"** → **X** (정반대). **"임계 경로 작업은 여유 시간이 가장 많다"** → X (여유 시간 = 0). **"Gantt = 확률적 3점 추정"** → X (Gantt는 시각화일 뿐 추정 기법 X). **임계 경로 = 가장 긴 경로 = Slack 0** 핵심 3박자.
