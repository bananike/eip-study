# Day 12 · [소프트웨어 설계] 요구사항 분석 (1과목)
> 날짜: 2026-05-23 (KST) | D-76 (목표 2026-08-07)
> 암기 목표:
> - 기능 요구사항 ↔ 비기능 요구사항 키워드 식별
> - 요구공학(Requirements Engineering) 5단계 **순서** 암기 (도출→분석→명세→확인→관리)
> - DFD(Data Flow Diagram) 4구성요소 + 기호 1:1 암기

> - HIPO(Hierarchy Input Process Output) 3종 도표 명칭 암기
> - 정형 명세(Formal) ↔ 비정형 명세(Informal) 기법 분류 암기
> - 요구사항 검토 기법 3종(워크스루/인스펙션/동료검토) 암기

## 💡 상세 해설

### 문제 1
정답: ③
- 제공 해설: ③의 "도서를 검색하고 대출 신청을 수행할 수 있어야 한다"는 시스템이 수행해야 할 **동작·기능**을 기술하므로 **기능 요구사항**이다. 비기능 요구사항은 시스템이 "어떻게(How well)" 동작해야 하는지를 규정한다.
- 오답 이유:
  - ① 응답시간 → **성능(Performance)** = 비기능
  - ② 가용성(Availability) = 비기능
  - ④ 암호화 / 보안(Security) = 비기능

### 문제 2
정답: ①
- 제공 해설: 요구공학 프로세스의 표준 순서는 **도출(Elicitation) → 분석(Analysis) → 명세(Specification) → 확인(Validation)** 이며, 전 과정에 걸쳐 **관리(Management)**가 병행된다.
- 오답 이유: 분석이 도출보다 먼저 올 수 없고, 명세는 분석 이후이며, 확인은 항상 마지막.

### 문제 3
정답: ④
- 제공 해설: DFD의 4구성요소는 **프로세스(원·둥근사각형), 자료흐름(화살표), 자료저장소(평행선·열린사각형), 외부엔티티(사각형)**이다. 자료 사전(Data Dictionary)은 DFD의 자료흐름·자료저장소에 등장하는 데이터를 정의·설명하는 **보완 도구**이며 DFD 자체의 구성요소가 아니다.
- 오답 이유: ①②③ 모두 DFD의 구성요소.

### 문제 4
정답: ④
- 제공 해설: 자료사전에서 `( )`는 **자료의 생략(optional)**, `[ ]`가 **자료의 선택(selection)**이다. 이 둘을 바꿔치기하는 함정이 자주 출제된다.
- 표기 기호 7종 (전체):
  - `=` 자료의 정의 (is composed of)
  - `+` 자료의 연결 (and)
  - `( )` 자료의 생략 가능 (optional)
  - `[ ]` 자료의 선택 (selection, OR)
  - `{ }` 자료의 반복 (iteration)
  - `**` 자료의 주석 (comment)
  - `|` 선택 항목의 구분자

### 문제 5
정답: ④
- 제공 해설: HIPO 3종 도표 = **가시적 도표(VTOC) / 총체적 도표(Overview) / 세부적 도표(Detail)**. 상태 전이 도표(STD)는 객체·시스템의 동적 모델링 기법으로 HIPO와 무관.
- 오답 이유: ①②③ 모두 HIPO 3종 도표.

### 문제 6
정답: ①
- 제공 해설: 정형 명세(Formal Specification)는 수학·논리 기반으로 명확하고 무모순적 기술이 가능하며 대표 기법은 **Z, VDM(Vienna Development Method), Petri Net, CSP**이다.
- 오답 이유:
  - ② Decision Table: 비정형 명세 (표 기반)
  - ③ Use Case Diagram: 비정형 명세 (UML, 도형 기반)
  - ④ DFD: 비정형 명세 (도형 기반)

### 문제 7
정답: ④
- 제공 해설: 단위 테스트(Unit Testing)는 **구현·테스트 단계** 활동이며 요구사항 도출 기법이 아니다. 요구사항 도출 대표 기법: 인터뷰, 워크샵(JAD), 설문, 브레인스토밍, 프로토타이핑, 유스케이스, 관찰, 시나리오 분석, BPR.
- 오답 이유: ①②③ 모두 표준 도출 기법.

### 문제 8
정답: ④
- 제공 해설: 요구사항 확인(Validation) 단계의 검토 기법은 **동료검토(Peer Review), 워크스루(Walkthrough), 인스펙션(Inspection)** 3종이다. 리팩토링(Refactoring)은 구현된 코드의 구조 개선이며 요구사항 검토 기법이 아니다.
- 3종 검토 기법 차이:
  - 동료검토: 비공식, 동료에게 설명
  - 워크스루: 비공식, 검토 모임에서 작성자 주도 발표
