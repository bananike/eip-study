## 🔑 핵심 용어 (10개)

| 용어 | 한 줄 정의 | 핵심 키워드 |
|---|---|---|
| 디자인 패턴(Design Pattern) | 자주 발생하는 설계 문제에 대한 재사용 가능한 해결책 | 재사용·생산성·의사소통 |
| GoF(Gang of Four, 4인방) | Erich Gamma, Richard Helm, Ralph Johnson, John Vlissides 4인이 1995년 발표한 23개 패턴 | 23종·3분류 |
| 생성 패턴(Creational, 5종) | 객체 **인스턴스 생성** 과정에 관여하는 패턴 | "**생추빌프싱**"(생성-추상Factory-빌더-프로토타입-싱글톤)·Factory Method 포함 |
| 구조 패턴(Structural, 7종) | 클래스·객체를 **결합·합성**하여 더 큰 구조를 만드는 패턴 | "**구애브컴데퍼플프**"(어댑터-브리지-컴포지트-데코레이터-퍼사드-플라이웨이트-프록시) |
| 행위 패턴(Behavioral, 11종) | 객체 간 **상호작용·책임 분배** 방식을 정의하는 패턴 | "**행 책 커 인 반 미 옵 상 전 템 비 메**"(11종) |
| 싱글톤(Singleton) | 클래스의 **인스턴스가 단 하나**만 존재하도록 보장하고 전역 접근점 제공 | private 생성자·getInstance |
| 어댑터(Adapter) | **호환되지 않는 인터페이스**를 변환하여 함께 동작하게 함 | 인터페이스 변환·Wrapper |
| 옵저버(Observer) | 한 객체의 상태 변화를 **종속 객체들에게 자동 통지** | publish-subscribe·1:N |
| 스트래티지(Strategy, 전략) | 동일 계열의 **알고리즘을 캡슐화**하여 런타임에 교체 가능 | 알고리즘 교체·캡슐화 |
| 데코레이터(Decorator) | 객체에 **동적으로 새 기능을 덧붙임** (상속 대안) | 기능 동적 추가·래핑 |

## 📖 핵심 개념 정리

### 1. 디자인 패턴의 정의·장점 (★빈출★)

**(1) 정의**
- 소프트웨어 설계 과정에서 **반복적으로 발생하는 문제**에 대한 **재사용 가능한 해결책**
- 코드가 아닌 **설계 노하우**를 정리한 것 → 같은 패턴도 언어·상황에 따라 다르게 구현됨

**(2) 사용 장점 (5가지)**
1. **재사용성·생산성 향상** (개발 시간 단축)
2. **유지보수성 향상** (구조 파악 용이)
3. **개발자 간 의사소통 원활** (공통 어휘)
4. **소프트웨어 품질·신뢰성 향상**
5. **객체지향 설계의 모범 답안 제공**

**(3) 단점**
- 객체지향 기반이므로 **절차지향에는 적용 어려움**
- 초기 학습 비용 / 패턴 오용 시 과설계(over-engineering)

### 2. GoF 23종 한 표 정리 (★★★ 최핵심 ★★★)

| 분류 | 개수 | 패턴 | 한 줄 정의 / 키워드 |
|---|---|---|---|
| **생성** | 5 | **Factory Method**(팩토리 메서드) | 객체 생성 인터페이스 정의, **서브클래스가 어떤 객체를 만들지 결정** |
| | | **Abstract Factory**(추상 팩토리) | **관련된 객체 군(family)** 을 함께 생성, 구체 클래스 의존X |
| | | **Builder**(빌더) | **복합 객체의 생성 과정과 표현을 분리** → 동일 절차로 다양한 표현 |
| | | **Prototype**(프로토타입) | **원형 객체를 복제(clone)** 하여 새 객체 생성 |
| | | **Singleton**(싱글톤) | **인스턴스 단 하나** 보장 + 전역 접근점 |
| **구조** | 7 | **Adapter**(어댑터) | **호환되지 않는 인터페이스 변환**, Wrapper |
| | | **Bridge**(브리지) | **추상화와 구현을 분리** → 독립적 확장 |
| | | **Composite**(컴포지트) | **트리 구조**로 객체와 객체 그룹을 동일하게 다룸 (부분-전체) |
| | | **Decorator**(데코레이터) | 객체에 **동적으로 기능 추가** (상속 대안) |
| | | **Facade**(파사드) | 서브시스템에 대한 **통합된 단순 인터페이스** 제공 |
| | | **Flyweight**(플라이웨이트) | **공유**를 통해 다수의 작은 객체 메모리 절약 |
| | | **Proxy**(프록시) | 다른 객체에 대한 **대리자(접근 제어)** |
| **행위** | 11 | **Chain of Responsibility**(책임 연쇄) | 처리 객체들을 **체인**으로 연결, 차례로 처리 시도 |
| | | **Command**(커맨드, 명령) | **요청을 객체로 캡슐화** → 매개변수화·취소(undo) |
| | | **Interpreter**(인터프리터) | **언어의 문법을 클래스화** 하여 해석 |
| | | **Iterator**(반복자) | 컬렉션의 **내부 구조 노출 없이 순차 접근** |
| | | **Mediator**(중재자) | 객체 간 **상호작용을 캡슐화** → 결합도 ↓ |
| | | **Memento**(메멘토) | 객체의 **이전 상태 저장·복원** (undo·snapshot) |
| | | **Observer**(옵저버) | 상태 변경 시 **종속 객체에 자동 통지** (1:N pub-sub) |
| | | **State**(상태) | **객체의 내부 상태에 따라** 행위 변경 |
| | | **Strategy**(전략) | **알고리즘 군을 캡슐화**하여 런타임 교체 |
| | | **Template Method**(템플릿 메서드) | **알고리즘 골격은 상위 클래스**, 세부는 하위 |
| | | **Visitor**(방문자) | 객체 구조 변경 없이 **연산을 외부 클래스(방문자)** 로 분리 |

### 3. 두문자 암기법 (★시험 직전 외울 것★)

**(1) 생성 5개 → "생추빌프싱"**
- **추**상 팩토리 / **빌**더 / **프**로토타입 / **싱**글톤 / + **F**actory Method

**(2) 구조 7개 → "애브컴데퍼플프"** (또는 "어브컴퍼데플프")
- **A**dapter / **B**ridge / **C**omposite / **D**ecorator / **F**acade / F**l**yweight / **P**roxy

**(3) 행위 11개 → "책 커 인 반 미 옵 상 전 템 비 메"**
- Chain / Command / Interpreter / Iterator / Mediator / Observer / State / Strategy / Template / Visitor / Memento

> 💡 시험에서 가장 헷갈리는 것: **Prototype(생성)** vs **Iterator(행위)** vs **Adapter(구조)**. 매번 출제됩니다.

### 5. 시험에서 자주 묻는 함정 (★)

1. **MVC**(Model-View-Controller)는 **GoF 디자인 패턴이 아니라 아키텍처 패턴**입니다.
2. **Repository, DAO**도 GoF 23개에 속하지 않습니다.
3. **Factory Method ≠ Abstract Factory**:
   - Factory Method = **한 개** 객체 생성 (서브클래스가 결정)
   - Abstract Factory = **객체 군(family)** 생성
4. **Adapter는 생성 패턴이 아니라 구조 패턴** (가장 많이 틀리는 함정).
5. **Iterator는 구조 패턴이 아니라 행위 패턴** (컬렉션을 떠올려 헷갈림).
6. **Prototype은 행위 패턴이 아니라 생성 패턴** (이름 때문에 헷갈림).

## 💡 상세 해설

### 문제 1 해설 — 정답 ④
**Prototype**(프로토타입)은 **생성 패턴**입니다. 원형 객체를 복제(clone)해서 새 객체를 만들기 때문에 "객체 생성"을 다루는 생성 패턴으로 분류됩니다. 나머지 Observer/Iterator/Strategy는 모두 행위 패턴. **암기 함정**: 이름이 "원형/모형"이라 행위처럼 느껴지지만 **생성**입니다.

### 문제 2 해설 — 정답 ③
**Adapter**(어댑터)는 **구조 패턴**입니다. 호환되지 않는 인터페이스를 변환해 "서로 다른 모양의 객체를 결합"하는 것이 핵심이므로 **구조**로 분류. Singleton·Factory Method·Abstract Factory는 모두 객체 생성에 관여하는 **생성 패턴** 5종에 속합니다.

### 문제 3 해설 — 정답 ①
생성 패턴 5종은 **Factory Method, Abstract Factory, Builder, Prototype, Singleton**. Adapter(구조), Decorator(구조), Bridge(구조)는 모두 구조 패턴.

### 문제 4 해설 — 정답 ④
**Command**(커맨드)는 **행위 패턴**입니다. 요청을 객체로 캡슐화하는 것. 구조 패턴 7종은 "**애브컴데퍼플프**" — Adapter·Bridge·Composite·Decorator·Facade·Flyweight·Proxy.

### 문제 5 해설 — 정답 ④
"객체를 하나만 생성" + "어디서든 참조" = **Singleton**의 정의 그 자체. private 생성자 + static getInstance()로 구현. Builder는 단계적 객체 조립, Factory Method는 객체 생성 위임이 키워드.

### 문제 6 해설 — 정답 ③
디자인 패턴은 **코드 자체가 아니라 설계 노하우**입니다. 같은 패턴도 언어·상황에 따라 구현이 달라지므로 "그대로 복사" 표현은 틀림. 나머지 보기는 모두 디자인 패턴의 올바른 설명.

### 문제 7 해설 — 정답 ①
"호환되지 않는 인터페이스 변환" → **Adapter**. Bridge는 추상화-구현 분리, Facade는 복잡한 서브시스템의 단순 인터페이스, Proxy는 접근 제어 대리자.

### 문제 8 해설 — 정답 ②
"생성 과정과 표현 분리" + "동일한 생성 절차로 서로 다른 표현" = **Builder**의 표준 정의. 복잡한 객체를 단계적으로 조립할 때 사용 (예: 햄버거 빌더, StringBuilder).

### 문제 9 해설 — 정답 ④
GoF는 **생성·구조·행위 3분류**입니다. "추상 패턴(Abstract)"이라는 분류는 존재하지 않습니다. (Abstract Factory는 생성 패턴의 하나일 뿐, 별도 분류가 아님.)

### 문제 10 해설 — 정답 ①
"한 객체 상태 변경 → 의존 객체들에 자동 알림" = **Observer**(옵저버, pub-sub, 1:N). 신문 구독 모델로 비유됨. Mediator는 객체 간 상호작용 캡슐화, State는 상태에 따라 행위 변경.

### 문제 11 해설 — 정답 ①
"알고리즘 군 캡슐화 + 교환 가능" = **Strategy**(전략). Template Method는 알고리즘 골격이 상위에 고정되고 세부만 하위에서 정의하는 차이점 (상속 기반 vs 위임 기반).

### 문제 12 해설 — 정답 ②
"동적으로 책임 추가" + "서브클래싱 대안" = **Decorator**. 객체를 감싸는 래퍼로 기능을 덧붙임 (예: Java I/O Stream의 BufferedReader가 FileReader를 감싸는 구조).

### 문제 13 해설 — 정답 ②
"서브시스템 복잡함 감추기" + "단순 인터페이스 제공" = **Facade**(파사드, 건물의 정면). 복잡한 내부를 가리고 클라이언트에게는 깔끔한 진입점만 제공.

### 문제 14 해설 — 정답 ③
"접근 제어" + "대리자(Surrogate)" = **Proxy**. 가상 프록시(지연 로딩), 보호 프록시(접근 제어), 원격 프록시(원격 객체 접근), 캐싱 프록시 등 변형.

### 문제 15 해설 — 정답 ①
"객체 간 상호작용을 캡슐화" + "결합도 ↓" = **Mediator**(중재자). 객체들이 서로 직접 참조하지 않고 중재자를 통해 통신 (예: 항공 관제탑이 비행기들 사이의 중재자).

### 문제 16 해설 — 정답 ①
"내부 상태 저장 → 나중에 복원" + "Undo" = **Memento**(메멘토, 기념품/기억). 객체의 캡슐화를 깨지 않으면서 스냅샷을 저장.

### 문제 17 해설 — 정답 ③
"원형 객체 복제(clone)" = **Prototype**. Java의 `Object.clone()` / `Cloneable` 인터페이스가 이 패턴의 구현. **함정**: 이름 때문에 "초기 모형"으로 헷갈리지만 생성 패턴입니다.

### 문제 18 해설 — 정답 ①
"요청을 객체로 캡슐화" + "큐잉/로깅/undo" = **Command**(커맨드). 메뉴 클릭, 버튼 누름 같은 사용자 동작을 객체로 만들어 큐에 쌓거나 취소·재실행할 수 있게 함.

### 문제 19 해설 — 정답 ①
"내부 표현 노출 없이 순차 접근" = **Iterator**(반복자). Java의 `Iterator` 인터페이스, Python의 `__iter__` 가 이 패턴. **함정**: 컬렉션을 떠올려 구조 패턴으로 착각하기 쉽지만 **행위 패턴**.

### 문제 20 해설 — 정답 ①
"알고리즘 골격 = 상위 클래스, 세부 = 하위 클래스" = **Template Method**(템플릿 메서드). 추상 메서드/훅 메서드로 변경점만 하위에서 구현. Strategy와 비교: Strategy는 위임(컴포지션), Template Method는 상속.

### 문제 21 해설 — 정답 ①
"요청 처리 객체를 체인" + "차례로 다음 객체로 전달" = **Chain of Responsibility**(책임 연쇄). 예: 휴가 결재(팀장→부장→사장), 예외 처리의 catch 블록 연쇄.

### 문제 22 해설 — 정답 ①
"부분-전체 트리" + "개별·복합 동일하게 다룸" = **Composite**(컴포지트). 파일 시스템(파일/디렉터리), GUI 컨테이너(위젯/패널)가 대표 예. Decorator와 혼동 주의: Decorator는 기능 추가, Composite는 트리 구성.

### 문제 23 해설 — 정답 ①
"공유로 메모리 절약" + "다수의 작은 객체" = **Flyweight**(플라이웨이트, 권투의 경량급). 예: 워드프로세서의 글자 객체(같은 문자는 객체 공유). Singleton은 1개 인스턴스, Flyweight는 N개를 공유.

### 문제 24 해설 — 정답 ①
"상태에 따라 행위 변경" = **State**(상태). 객체 내부 상태 객체를 교체해서 마치 클래스가 바뀐 듯 동작 (예: 신호등 - 빨강/노랑/초록 상태 객체). Strategy와 구조 유사하나 **의도**가 다름: Strategy=알고리즘 교체, State=상태 변화.

### 문제 25 해설 — 정답 ④
디자인 패턴은 **객체지향 설계**에 기반하므로 **절차지향 언어에는 적용이 어렵습니다**. 패턴의 장점은 ①②③ 외에도 구조 파악 용이, 유지보수성 향상 등. ④는 **단점**에 해당.

## 📌 시험 직전 1분 요약 (5줄)

1. **GoF 23 = 생성 5 + 구조 7 + 행위 11**. "생추빌프싱 / 애브컴데퍼플프 / 책커인반미옵상전템비메"로 외운다.
2. **함정 3대장**: Adapter는 구조(❌생성), Iterator는 행위(❌구조), Prototype은 생성(❌행위).
3. **패턴 → 키워드 직매칭**: Singleton="유일", Adapter="변환", Observer="알림 1:N", Strategy="알고리즘 교체", Facade="단순 인터페이스", Composite="트리 부분-전체", Proxy="대리자", Memento="undo 복원", Command="요청 객체화", Decorator="동적 기능 추가".
4. **헷갈림 페어**: Factory Method(객체1개) ↔ Abstract Factory(객체군), Strategy(알고리즘 교체) ↔ State(상태에 따른 행위), Decorator(기능 추가) ↔ Composite(트리 구성), Template Method(상속) ↔ Strategy(위임).
5. MVC는 GoF가 아니라 **아키텍처 패턴**. 디자인 패턴 장점=재사용/품질/의사소통, 단점=절차지향 부적합·학습비용.
