## 🔍 오답 역추적 개념 발췌

오늘 문제에서 헷갈리기 쉬운 키워드만 발췌. 정의·설명은 사후 검증용 섹션 참조.

- 블랙박스(명세 기반) vs 화이트박스(구조 기반) — 기법 소속 구분
- 블랙박스 기법: 동치 분할, 경계값 분석, 원인-효과 그래프, 오류 예측, 비교(Comparison)
- 화이트박스 기법: 기초 경로 검사(Basis Path), 제어 구조 검사(조건/루프/데이터 흐름)
- 커버리지 4종: 문장(Statement) / 분기(Branch=Decision) / 조건(Condition) / 경로(Path)
- McCabe 순환 복잡도(Cyclomatic Complexity) V(G) = E − N + 2
- 테스트 레벨 4종: 단위(Unit) → 통합(Integration) → 시스템(System) → 인수(Acceptance)
- V-모델: 좌측 개발단계 ↔ 우측 테스트단계 대응
- 통합 방식: 하향식(Top-down)→스텁(Stub), 상향식(Bottom-up)→드라이버(Driver)
- 스텁(Stub, 하위 가상 모듈) ↔ 드라이버(Driver, 상위 가상 모듈)
- 테스트 오라클: 참 / 샘플링 / 휴리스틱 / 일관성 검사

## 💡 상세 해설 (자기 설명과 대조)

### 문제 1
정답: ③ 기초 경로 검사(Basis Path Testing)
- 제공 해설: 기초 경로 검사는 프로그램의 제어 흐름(논리 구조)을 분석하여 독립적인 실행 경로를 도출하는 화이트박스(구조 기반) 기법이다. McCabe의 순환 복잡도를 이용해 기본 경로 수를 산정한다.
- 오답 이유 (모두 블랙박스 기법):
  - ① 동치 분할 검사: 입력 도메인을 유효/무효 등가 클래스로 나눠 대표값을 테스트 — 명세 기반.
  - ② 경계값 분석: 등가 분할의 경계(최소/최대 및 그 ±1) 근처를 집중 테스트 — 명세 기반.
  - ④ 원인-효과 그래프 검사: 입력 조건(원인)과 출력(효과)의 논리 관계를 그래프화 — 명세 기반.

### 문제 2
정답: ② 분기 커버리지(Branch / Decision Coverage)
- 제공 해설: 각 결정문(분기점)의 참/거짓 결과를 모두 적어도 한 번씩 수행시키는 것이 분기(결정) 커버리지다. 결정 커버리지(Decision Coverage)와 동의어.
- 오답 이유:
  - ① 문장 커버리지: 모든 실행 가능한 "문장(statement)"을 한 번 이상 실행. 분기의 참/거짓 모두를 보장하지는 못함(분기보다 약함).
  - ③ 조건 커버리지: 결정문 내부 "개별 조건(condition)" 각각의 참/거짓을 보장. 설명은 분기 단위라 부적절.
  - ④ 경로 커버리지: 모든 가능한 "경로(path)" 조합을 수행하는 가장 강한 커버리지. 분기보다 강함.

### 문제 3
정답: ③ 5
- 제공 해설: McCabe 순환 복잡도 V(G) = E − N + 2 (E=에지 수, N=노드 수). E=11, N=8 → 11 − 8 + 2 = 5. (또는 영역(region) 수 = 5, 또는 분기 노드 수 + 1로도 동일.)
- 오답 이유:
  - ① 3, ② 4: E − N 계산 또는 +2 누락 오류.
  - ④ 6: +2 대신 +3을 더하거나 에지 수 오산 시 발생하는 함정.

### 문제 4
정답: ④ 인수 테스트(Acceptance Test)
- 제공 해설: 인수 테스트는 사용자(고객)가 요구사항 충족 여부를 검증하여 인수 여부를 결정하는 최종 테스트. 알파 테스트(개발자 환경)·베타 테스트(실 사용자 환경)가 포함된다.
- 오답 이유:
  - ① 단위 테스트: 최소 단위(모듈/함수) 단독 검증. 주로 개발자가 수행.
  - ② 통합 테스트: 모듈 간 인터페이스(연계) 결함 검증.
  - ③ 시스템 테스트: 통합된 전체 시스템이 기능·비기능 요구를 만족하는지 검증(개발 조직 주도).

### 문제 5
정답: ② 스텁(Stub)
- 제공 해설: 하향식 통합은 상위 모듈부터 통합하므로, 아직 없는 "하위 모듈"을 대신할 가상 모듈인 스텁(Stub)이 필요하다. 스텁은 호출에 대해 미리 정해진 값만 반환하는 단순 더미.
- 오답 이유:
  - ① 드라이버(Driver): 상향식 통합에서 아직 없는 "상위 모듈"을 대신해 하위 모듈을 호출·구동하는 가상 모듈.
  - ③ 테스트 케이스: 입력·실행조건·기대결과의 집합(개념이 다름).
  - ④ 테스트 슈트(Suite): 여러 테스트 케이스의 묶음(개념이 다름).

### 문제 6
정답: ③ (옳지 않은 것) 시스템 테스트 ↔ 요구사항 분석
- 제공 해설: V-모델에서 시스템 테스트는 "시스템(아키텍처) 설계"가 아닌, 일반적으로 요구사항 정의/기능 요구사항 단계에 대응되지만, 4단계 표준 대응에서 "요구사항 분석"에 직접 대응되는 것은 인수 테스트다. 시스템 테스트는 시스템 설계 단계에 대응하는 것으로 보며(아키텍처/기능 명세 검증), "시스템 테스트 ↔ 요구사항 분석" 연결은 옳지 않다.
- 오답 이유 (옳은 보기들):
  - ① 단위 테스트 ↔ 상세 설계 — 옳음(모듈 상세 설계를 검증).
  - ② 통합 테스트 ↔ 시스템(아키텍처) 설계 — 옳음(모듈 간 인터페이스 설계 검증).
  - ④ 인수 테스트 ↔ 요구사항 분석 — 옳음(사용자 요구사항 충족 검증).

### 문제 7
정답: ② 문장 → 분기 → 조건/분기 → 다중 조건
- 제공 해설: 일반적인 커버리지 강도(약→강)는 문장(Statement) < 분기(결정, Decision) < 조건/분기(Condition/Decision) < 다중 조건(Multiple Condition) 순이다. 문장 커버리지가 가장 약하고, 다중 조건이 가장 강하다.
- 오답 이유:
  - ① 분기를 문장보다 앞에 둠 — 문장이 더 약하므로 순서 오류.
  - ③ 조건을 문장보다 앞에 둠 — 시작이 잘못됨.
  - ④ 강→약 역순으로 나열 — 문제 요구(약→강)와 반대.

### 문제 8
정답: ② 샘플링(Sampling) 오라클
- 제공 해설: 샘플링 오라클은 모든 입력이 아니라 "선별된 일부(샘플) 입력값"에 대해서만 기대 결과를 제공하는 오라클이다.
- 오답 이유:
  - ① 참(True) 오라클: 모든 입력값에 대해 기대 결과를 제공(가장 이상적, 비용 큼).
  - ③ 휴리스틱(Heuristic) 오라클: 일부 입력은 참값, 나머지는 추정(경험적)값으로 처리 — 샘플링을 개선한 형태.
  - ④ 일관성 검사(Consistent) 오라클: 변경 전/후 실행 결과가 동일한지(일관성)를 비교하는 오라클(회귀 테스트용).

## 📚 역추적 참조 (사후 검증용 — 학습 자료 아님)

### 핵심 용어
| 용어 | 정의 |
|---|---|
| 블랙박스 테스트(Black-box) | 내부 구조를 보지 않고 명세(입력→출력)만으로 검증. 기능/명세 기반. |
| 화이트박스 테스트(White-box) | 프로그램 내부 논리(제어 흐름)를 보고 검증. 구조 기반. |
| 동치 분할(Equivalence Partitioning) | 입력을 유효/무효 등가 클래스로 나눠 대표값 테스트. |
| 경계값 분석(Boundary Value Analysis) | 등가 클래스의 경계(최소·최대 및 ±1) 집중 테스트. |
| 원인-효과 그래프(Cause-Effect Graph) | 입력 조건(원인)·출력(효과)의 논리 관계를 그래프화. |
| 오류 예측(Error Guessing) | 경험·직관으로 결함이 있을 법한 부분을 추정 테스트. |
| 기초 경로 검사(Basis Path Testing) | 순환 복잡도로 독립 경로 수를 도출해 검증(화이트박스). |
| 순환 복잡도(Cyclomatic Complexity) | V(G) = E − N + 2. 독립 경로의 최대 수. |
| 스텁(Stub) | 하향식 통합에서 미완성 하위 모듈을 대신하는 가상 모듈. |
| 드라이버(Driver) | 상향식 통합에서 미완성 상위 모듈을 대신하는 가상 모듈. |
| 테스트 오라클(Test Oracle) | 테스트 결과의 참/거짓을 판단하는 기대값 제공 메커니즘. |

### 핵심 개념 (사후 검증용)

**블랙박스 vs 화이트박스 기법 분류**
- 블랙박스(명세 기반): 동치 분할, 경계값 분석, 원인-효과 그래프, 오류 예측, 비교 검사
- 화이트박스(구조 기반): 기초 경로 검사(Basis Path), 제어 구조 검사 — 조건/루프/데이터 흐름

**화이트박스 커버리지 강도 (약 → 강)**
- 문장(Statement) < 분기=결정(Branch/Decision) < 조건(Condition) / 조건·분기(Condition/Decision) < 다중 조건(Multiple Condition) < 경로(Path)
- 문장: 모든 문장 1회 이상 실행
- 분기: 각 분기점 참/거짓 모두 실행
- 조건: 결정문 내 개별 조건의 참/거짓 모두 실행
- 경로: 가능한 모든 경로 조합 실행(가장 강함, 현실적으로 어려움)

**McCabe 순환 복잡도 V(G)**
- V(G) = E − N + 2 (E=에지 수, N=노드 수, 연결 그래프 1개 기준)
- = 평면 그래프의 영역(region) 수
- = 분기(조건) 노드 수 + 1
- 예) E=11, N=8 → 11 − 8 + 2 = 5

**테스트 레벨 4종 (V-모델 우측)**
| 레벨 | 대상 | 대응 개발단계 |
|---|---|---|
| 단위 테스트 | 개별 모듈/함수 | 상세 설계 |
| 통합 테스트 | 모듈 간 인터페이스 | 시스템(아키텍처) 설계 |
| 시스템 테스트 | 통합 전체 시스템 | 시스템 설계/요구 명세 |
| 인수 테스트 | 사용자 요구 충족 | 요구사항 분석 |

**통합 테스트 방식**
| 항목 | 하향식(Top-down) | 상향식(Bottom-up) |
|---|---|---|
| 방향 | 상위→하위 | 하위→상위 |
| 가상 모듈 | 스텁(Stub) | 드라이버(Driver) |
| 역할 | 하위 모듈 대체 | 상위 모듈 대체 |
| 장점 | 초기에 골격/상위 결함 확인 | 하위 모듈 검증 용이 |
| 단점 | 스텁 다수 필요 | 상위 구조 늦게 확인 |
- 혼합식(Sandwich): 상향·하향 절충. 빅뱅(Big-bang): 한 번에 전체 결합(결함 추적 어려움).

**테스트 오라클 4종**
- 참(True): 모든 입력에 기대 결과 제공
- 샘플링(Sampling): 일부 입력에만 기대 결과 제공
- 휴리스틱(Heuristic): 일부는 참값, 나머지는 추정값
- 일관성 검사(Consistent): 변경 전/후 결과 일관성 비교
