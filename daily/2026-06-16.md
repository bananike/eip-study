### 문제 5 [유사 기출: 정처기 필기 2021·2023 반복 — Factory Method vs Abstract Factory 정형 패턴]
다음 설명에 해당하는 디자인 패턴은?

> 서로 **관련 있는 객체들의 집합(제품군, family of products)**을 생성하기 위한 인터페이스를 제공하며, **구체적인 클래스를 지정하지 않고도** 호환되는 객체군을 통째로 생성할 수 있게 한다.

① 팩토리 메서드(Factory Method)
② 빌더(Builder)

### 문제 6 [유사 기출: 정처기 필기 2020년 3회 · 2022·2024 반복 — Observer 정의 정형 패턴]
다음 설명에 해당하는 디자인 패턴은?

> 한 객체의 상태가 변경되면 그 객체에 의존하는 다른 모든 객체들에게 **자동으로 통보(notify)되어 갱신**되도록, 객체 사이에 **일대다(1:N) 의존 관계**를 정의한다. 발행-구독(publish-subscribe) 모델로도 알려져 있다.

① **옵저버(Observer)**
② 컴포지트(Composite)
③ 스트래티지(Strategy)
④ 데코레이터(Decorator)

### 문제 7 [유사 기출: 정처기 필기 2021·2023·2024 반복 — Adapter vs Bridge 식별 정형 패턴]
다음 중 디자인 패턴에 대한 설명으로 **틀린** 것은?

① **어댑터(Adapter)** 패턴은 호환되지 않는 인터페이스를 가진 클래스를 함께 동작하도록 **인터페이스를 변환**해 주는 구조 패턴이다.
② **브리지(Bridge)** 패턴은 **추상(abstraction)과 구현(implementation)을 분리**하여 두 부분이 서로 **독립적으로 변화**할 수 있도록 하는 구조 패턴이다.
③ **프록시(Proxy)** 패턴은 다른 객체에 대한 **접근을 제어**하기 위해 그 객체의 **대리자(surrogate)**를 제공하는 구조 패턴이다.
④ **데코레이터(Decorator)** 패턴은 **객체의 생성 과정을 단계별로 분리**하여 동일한 생성 절차에서 서로 다른 표현을 만들 수 있는 **생성 패턴**이다.

### 문제 8 [유사 기출: 정처기 필기 2022·2023·2024 반복 — Strategy vs Template Method 식별 정형 패턴]
다음 설명에 해당하는 디자인 패턴은?

> 동일 목적의 **알고리즘 군(family of algorithms)**을 각각 클래스로 캡슐화하여, 클라이언트가 **런타임에 알고리즘을 교체**해서 사용할 수 있도록 한다. 알고리즘 변경이 이를 사용하는 클라이언트와 **독립적**이다.

① **스트래티지(Strategy)**
② 템플릿 메서드(Template Method)
③ 커맨드(Command)
④ 상태(State)

### B. 헷갈리기 쉬운 쌍 — 정확한 구별

| 쌍 | 차이의 핵심 |
|---|---|
| **Factory Method vs Abstract Factory** | 전자는 **단일 제품 생성**을 서브클래스에 위임. 후자는 **제품군(family) 전체**를 통째 생성 — 키워드 "관련 객체군" |
| **Adapter vs Bridge** | Adapter는 **이미 만들어진** 호환 안 되는 인터페이스를 **사후 변환**. Bridge는 **설계 시점**에 추상-구현을 **나누어** 서로 독립적 변화 |
| **Decorator vs Proxy** | Decorator는 **기능 추가**(투명한 래핑, 책임 첨가). Proxy는 **접근 제어/대리**(원격·가상·보호 프록시) |
| **Composite vs Decorator** | Composite는 **트리 구조(부분-전체)**를 동일 처리. Decorator는 **단일 객체 래핑**으로 기능 첨가 |
| **Strategy vs Template Method** | Strategy는 **위임(composition)**으로 알고리즘 통째 교체. Template Method는 **상속**으로 알고리즘 골격 고정, 일부 단계만 오버라이드 |
| **State vs Strategy** | 구조 유사하나 의도가 다름. State는 **객체의 상태 변화**에 따른 행동 변경. Strategy는 **알고리즘 선택** |
| **Mediator vs Facade** | Mediator는 **객체들 간 다대다 통신을 중재**(상호 작용). Facade는 서브시스템에 **단일 진입점** 제공(단방향 단순화) |
| **Observer vs Mediator** | Observer는 **1:N 발행-구독**(주체→관찰자). Mediator는 **N:N 중재**(중재자 경유) |
| **Iterator vs Visitor** | Iterator는 컬렉션 **순회 방법 캡슐화**. Visitor는 **구조 변경 없이 새 연산 추가** |
| **Command vs Chain of Responsibility** | Command는 요청을 **객체로 캡슐화**(실행 취소·로깅). Chain은 요청을 **여러 처리자에 차례로 전달** |

### C. 자주 출제되는 정형 키워드 ↔ 패턴

- "인스턴스 하나 / 전역 접근점" → **Singleton**
- "서브클래스가 어떤 인스턴스를 생성할지 결정" → **Factory Method**
- "관련 객체군(제품군) 통째 생성 / 호환 객체군" → **Abstract Factory**
- "복잡한 객체를 단계별 분리 생성 / 동일 절차 다른 표현" → **Builder**
- "원본 객체를 복제(clone)" → **Prototype**
- "호환 안 되는 인터페이스를 변환" → **Adapter**
- "추상과 구현을 분리하여 독립 변화" → **Bridge**
- "부분-전체 / 트리 구조 / 단일·복합 동일 처리" → **Composite**
- "기능을 동적으로 첨가 / 상속 대신 합성" → **Decorator**
- "복잡한 서브시스템에 단일 인터페이스" → **Facade**
- "다수의 작은 객체 공유로 메모리 절약" → **Flyweight**
- "대리자 / 접근 제어 / 원격·가상·보호" → **Proxy**
- "1:N 의존 / 상태 변경 자동 통보 / publish-subscribe" → **Observer**
- "알고리즘 군 캡슐화, 런타임 교체" → **Strategy**
- "알고리즘 골격 고정, 일부 단계만 서브클래스" → **Template Method**
- "요청을 객체로 캡슐화, 실행 취소/큐잉/로깅" → **Command**
- "객체 내부 상태 저장·복원(undo)" → **Memento**
- "객체 상태에 따라 행동 변경" → **State**
- "구조 변경 없이 새 연산 추가, 이중 디스패치" → **Visitor**
- "컬렉션 내부 표현 노출 없이 순차 접근" → **Iterator**
- "객체 간 다대다 통신을 중재자가 단순화" → **Mediator**
- "요청을 처리자 사슬을 따라 전달" → **Chain of Responsibility**
- "문법 표현·해석(SQL/정규식)" → **Interpreter**

## 💡 상세 해설

### 문제 1
**정답 ④ 추상(Abstract) 패턴**
- GoF 분류는 **생성(Creational) / 구조(Structural) / 행위(Behavioral)** 3가지뿐. **"추상 패턴"은 존재하지 않음** — 가짜 분류.
- 함정 단골: "추상 팩토리(Abstract Factory)"라는 **패턴 이름**과 혼동을 노린 보기. Abstract Factory는 **생성 패턴**의 하나일 뿐, 분류명이 아님.
- 동일 유형의 가짜 분류 보기: "기본 패턴 / 변환 패턴 / 객체 패턴 / 클래스 패턴" — 모두 GoF 분류 아님.

### 문제 2
**정답 ③ 데코레이터(Decorator)**
- **Decorator는 구조(Structural) 패턴**. 객체에 동적으로 기능(책임)을 첨가.
- ① Abstract Factory / ② Builder / ④ Singleton — 모두 **생성** 패턴. 생성 5종 두음: **"추·빌·팩·프·싱"** (Abstract Factory / Builder / Factory Method / Prototype / Singleton).
- 빈출 함정 후보 (생성으로 착각): **Adapter / Decorator / Proxy / Composite** — 모두 구조 패턴.

### 문제 3
**정답 ② 비지터(Visitor)**
- **Visitor는 행위(Behavioral) 패턴**. 구조를 변경하지 않고 새로운 연산을 추가(이중 디스패치).
- ① Builder / ③ Prototype — **생성** 패턴 (두음 "추·빌·팩·프·싱"에 포함).
- ④ Bridge — **구조** 패턴 (두음 "어·브·컴·데·퍼·플·프"에 포함).
- 행위 11종 빈출: Observer / Strategy / State / Template Method / Command / Iterator / Visitor / Mediator / Memento / Chain of Responsibility / Interpreter.

### 문제 4
**정답 ② 싱글톤(Singleton)**
- 정형 정의: **"인스턴스 하나(only one)" + "전역 접근점(global access point)"** 두 키워드가 동시에 나오면 무조건 Singleton.
- ① Prototype — 원본을 **복제(clone)**해 새 인스턴스 생성(인스턴스 여러 개).
- ③ Builder — 복잡한 객체를 **단계별 분리**해 생성(인스턴스 수와 무관).
- ④ Factory Method — 서브클래스가 **인스턴스 종류를 결정**(인스턴스 1개 보장 X).

### 문제 5
**정답 ③ 추상 팩토리(Abstract Factory)**
- 정형 키워드: **"관련 객체군(family of products) / 제품군 전체 / 호환 객체군 통째 생성"** → Abstract Factory.
- ① Factory Method — **단일 객체** 생성을 서브클래스에 위임(한 종류만).
- ② Builder — **복잡한 객체를 단계별로** 분리하여 생성(절차 강조).
- ④ Prototype — **원본 복제** 기반 생성.
> 함정: Factory Method와 Abstract Factory를 같은 보기에 동시 배치하는 패턴이 다회 출제. **"제품군 / 관련 객체들의 집합"**이 보이면 Abstract Factory.

### 문제 6
**정답 ① 옵저버(Observer)**
- 정형 키워드: **"1:N 의존 / 상태 변경 자동 통보(notify) / 발행-구독(pub-sub)"** 세 가지가 동시에 나오면 Observer.
- ② Composite — 트리(부분-전체) 동일 처리, 통보와 무관.
- ③ Strategy — 알고리즘 교체, 통보와 무관.
- ④ Decorator — 기능을 동적 첨가, 통보와 무관.
> 함정: "이벤트 리스너 / 게시-구독" 같은 동의어로도 출제됨. 모두 Observer.

### 문제 7
**정답 ④** (틀린 설명)
- **Decorator(데코레이터)**는 **구조(Structural) 패턴**. "객체에 동적으로 책임(기능)을 첨가"하는 패턴. 보기에 적힌 "생성 과정을 단계별로 분리"는 **Builder(빌더)의 정의** — 분류·정의 모두 틀림.
- ① Adapter — 정확. **인터페이스 변환**으로 호환되지 않던 클래스를 함께 동작시킴.
- ② Bridge — 정확. **추상과 구현을 분리**하여 둘이 **독립적으로 변화**. Adapter와 자주 헷갈리지만, Adapter는 사후 변환, Bridge는 설계 시점 분리.
- ③ Proxy — 정확. **대리자(surrogate)** 제공으로 **접근 제어**. 원격·가상·보호 프록시 3종.
> 함정 핵심: Decorator·Adapter·Bridge·Proxy 모두 **구조 패턴**으로 같은 분류. 정의를 서로 섞어 끼워 넣는 패턴이 단골.

### 문제 8
**정답 ① 스트래티지(Strategy)**
- 정형 키워드: **"알고리즘 군(family of algorithms) 캡슐화 + 런타임 교체 + 클라이언트와 독립"** → Strategy.
- ② Template Method — 알고리즘의 **골격(skeleton)을 상위 클래스에 고정**하고, 일부 단계만 서브클래스가 **오버라이드**. **상속 기반**(Strategy는 합성 기반).
- ③ Command — 요청을 **객체로 캡슐화**(실행 취소·큐잉·로깅). "알고리즘 교체"가 아니라 "요청 캡슐화".
- ④ State — 객체의 **내부 상태에 따라 행동 변경**. 구조는 Strategy와 유사하나, **상태 전이**가 핵심.
> 함정: Strategy와 State는 클래스 다이어그램이 거의 동일 — 의도(intent)로만 구별. "**런타임 알고리즘 교체**"는 Strategy, "**상태에 따른 행동 변경**"은 State.

## 🔑 함정 키워드 ↔ 정답 개념 매칭표 (1과목 전용)

| 함정 키워드/문구 | 정답 개념 | 메모 |
|---|---|---|
| "추상 패턴 / 기본 패턴 / 변환 패턴 / 객체 패턴" | **GoF 분류 아님** | GoF는 생성·구조·행위 3분류뿐 |
| **"인스턴스 하나 + 전역 접근점"** | **Singleton** | 생성, 두 키워드 동시 출현 |
| "서브클래스가 인스턴스 종류 결정" | **Factory Method** | 생성, 단일 객체 위임 |
| **"관련 객체군(제품군) 통째 생성"** | **Abstract Factory** | 생성, 호환 제품군 |
| "복잡한 객체를 단계별 분리 생성 / 동일 절차 다른 표현" | **Builder** | 생성, 절차 강조 |
| "원본 객체를 복제(clone)" | **Prototype** | 생성 (행위 아님 — 함정 단골) |
| **"호환되지 않는 인터페이스를 변환"** | **Adapter** | 구조 (사후 변환) |
| **"추상과 구현을 분리, 독립적 변화"** | **Bridge** | 구조 (설계 시점 분리) |
| "트리 / 부분-전체 / 단일·복합 동일 처리" | **Composite** | 구조 |
| "객체에 동적으로 기능(책임) 첨가 / 상속 대신 합성" | **Decorator** | 구조 |
| "복잡 서브시스템에 단일 인터페이스" | **Facade** | 구조, 단방향 단순화 |
| "다수의 작은 객체를 공유" | **Flyweight** | 구조, 메모리 절약 |
| **"대리자 / 접근 제어 / 원격·가상·보호"** | **Proxy** | 구조 |
| **"1:N 의존 / 상태 변경 자동 통보 / pub-sub"** | **Observer** | 행위 |
| **"알고리즘 군 캡슐화, 런타임 교체"** | **Strategy** | 행위, 합성 |
| "알고리즘 골격 고정, 일부 단계만 서브클래스" | **Template Method** | 행위, 상속 |
| "요청을 객체로 캡슐화, 실행 취소/큐잉/로깅" | **Command** | 행위 |
| "객체 내부 상태 저장·복원(undo)" | **Memento** | 행위 |
| "객체 상태에 따라 행동 변경" | **State** | 행위 (Strategy와 구조 유사) |
| "구조 변경 없이 새 연산 추가, 이중 디스패치" | **Visitor** | 행위 (생성으로 착각 함정) |
| "컬렉션 내부 표현 노출 없이 순차 접근" | **Iterator** | 행위 |
| "객체 간 다대다 통신을 중재" | **Mediator** | 행위 (Observer는 1:N, Mediator는 N:N) |
| "요청을 처리자 사슬을 따라 전달" | **Chain of Responsibility** | 행위 |
| "문법 표현·해석(SQL/정규식)" | **Interpreter** | 행위 |
| "이벤트 리스너 / 게시-구독" | **Observer 동의어** | 모두 Observer로 매칭 |
| "Wrapper 패턴" | **Decorator(또는 Adapter) 별칭** | 둘 다 래핑 — 문맥상 "기능 첨가"면 Decorator, "변환"이면 Adapter |

> ★ 디자인 패턴 문제는 **정형 키워드 1:1 매칭**이 절대 다수 — 위 표의 키워드가 보이면 0.5초 안에 정답 패턴이 떠올라야 함. 헷갈리는 쌍(Factory Method/Abstract Factory, Adapter/Bridge, Strategy/Template Method, State/Strategy)은 **구별점**까지 함께 외울 것.

## 📚 역추적 참조 (사후 검증용)
- 디자인 패턴은 함정 키워드 매칭표가 1차 검증 도구. 위 표의 키워드를 보고 정답 패턴이 즉시 떠오르지 않으면 해당 항목 재학습.
- 23패턴 분류가 흔들리면 두음(생성: **추·빌·팩·프·싱** / 구조: **어·브·컴·데·퍼·플·프**)부터 다시. 행위 11종은 "나머지 전부"로 외워두면 분류 문제는 답이 떨어짐.
- 헷갈리는 쌍(Factory Method vs Abstract Factory, Adapter vs Bridge, Strategy vs Template Method, State vs Strategy, Decorator vs Proxy, Composite vs Decorator, Mediator vs Facade)은 **차이의 핵심 한 줄**을 통째로 외우기.
