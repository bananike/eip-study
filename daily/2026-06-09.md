### 문제 4 [유사 기출: 정보처리기사 필기 2020·2022·2023 반복 — 통합 테스트 방식 정형 패턴]
**통합 테스트(Integration Test)** 방식에 대한 설명으로 옳지 **않은** 것은?

① 빅뱅(Big Bang) 방식은 모든 모듈을 한꺼번에 결합하여 테스트하므로 **오류 발생 시 원인 추적이 어렵다**.
② 하향식(Top-Down) 통합은 상위 모듈부터 통합하며, 아직 개발되지 않은 하위 모듈 대신 **드라이버(Driver)** 를 사용한다.
③ 상향식(Bottom-Up) 통합은 하위 모듈부터 결합하며, 상위 모듈 대용으로 **드라이버(Driver)** 를 사용한다.
④ 샌드위치(Sandwich) 방식은 하향식과 상향식을 병행하여 양쪽에서 통합을 진행한다.

### 문제 5 [유사 기출: 정보처리기사 필기 2021·2023·2024 반복 — 테스트 오라클 정형 패턴]
**테스트 오라클(Test Oracle)** 의 종류와 설명의 연결로 옳지 **않은** 것은?

① 참(True) 오라클 — 모든 입력값에 대해 기대 결과를 제공하며, 발생한 오류를 모두 검출할 수 있다.
② 샘플링(Sampling) 오라클 — 특정한 몇 개의 입력값에 대해서만 기대 결과를 제공한다.
③ 휴리스틱(Heuristic) 오라클 — 샘플링 오라클을 개선한 형태로, 특정 입력값에는 기대 결과를 제공하고 나머지 입력값은 추정으로 처리한다.
④ 일관성 검사(Consistent) 오라클 — 애플리케이션이 **신규로 처음 개발될 때**, 외부 표준 결과와 비교하여 검증한다.

### 문제 6 [유사 기출: 정보처리기사 필기 2020·2021·2023 반복 — 형상관리 기능 정형 패턴]
**소프트웨어 형상관리(SCM: Software Configuration Management)** 의 기능에 대한 설명으로 옳지 **않은** 것은?

① 형상 식별(Configuration Identification) — 형상 관리 대상이 되는 항목(소스, 문서 등)을 식별하고 고유 번호를 부여한다.
② 형상 통제(Configuration Control) — 변경 요청을 평가하고 승인·기각하는 변경 통제 위원회(CCB)를 거쳐 반영한다.
③ 형상 감사(Configuration Audit) — 변경된 형상 항목이 요구사항에 부합하는지 검토한다.
④ 형상 기록(Configuration Status Accounting) — **개발자 개인의 코딩 스타일과 작업 시간**을 모니터링하여 생산성을 평가한다.

### 문제 8 [유사 기출: 정보처리기사 필기 2021·2022·2024 반복 — 인터페이스 데이터 포맷 정형 패턴]
인터페이스 구현에서 사용되는 **데이터 포맷**에 대한 설명으로 옳지 **않은** 것은?

① JSON(JavaScript Object Notation) — 키-값(key:value) 쌍과 배열로 표현되는 **경량 데이터 교환 포맷**으로, AJAX에서 XML을 대체하는 용도로 널리 쓰인다.
② XML(eXtensible Markup Language) — 사용자가 **태그(Tag)** 를 정의할 수 있는 다목적 마크업 언어로, HTML의 한계를 보완한다.
③ YAML(YAML Ain't Markup Language) — **들여쓰기(indent)** 기반의 사람이 읽기 쉬운 데이터 직렬화 포맷이다.
④ JSON은 **XML보다 표현 가능한 자료형이 많고**, 스키마(Schema) 검증·주석(Comment) 작성을 **공식적으로 지원**한다.

## 🔍 오답 역추적 개념 발췌
- **검증(Verification) vs 확인(Validation)**: V — "제대로 만들고 있는가?"(과정·중간산출물 점검) / V — "올바른 것을 만들었는가?"(요구사항 충족 확인)
- **블랙박스(명세 기반) 5기법**: 동치 분할(Equivalence Partitioning), 경계값 분석(Boundary Value Analysis), 원인-효과 그래프(Cause-Effect Graph), 오류 예측(Error Guessing), 비교 검사(Comparison Testing)
- **화이트박스(구조 기반) 주요 기법**: 기초 경로 검사(Basis Path), 제어 흐름 검사(Control Flow), 조건 검사(Condition), 루프 검사(Loop), 데이터 흐름 검사(Data Flow)
- **테스트 커버리지 강도**: 구문 커버리지(Statement) < 분기/결정 커버리지(Branch/Decision) < 조건 커버리지(Condition) < 분기조건 커버리지(MC/DC) < 다중 조건 커버리지(Multiple Condition)
- **테스트 레벨 4단계**: 단위(Unit, 모듈 단위) → 통합(Integration, 모듈 간 인터페이스) → 시스템(System, 전체 기능·비기능) → 인수(Acceptance, 사용자 요구사항)
- **V-모델 짝짓기**: 요구사항 분석 ↔ 인수 / 시스템 설계 ↔ 시스템 / 상세 설계 ↔ 통합 / 코딩 ↔ 단위
- **통합 테스트 방식 4종**:
  - **빅뱅(Big Bang)**: 한꺼번에 결합 → 오류 추적 곤란
  - **하향식(Top-Down)**: 상위→하위, 하위 모듈 대신 **스텁(Stub)** 사용
  - **상향식(Bottom-Up)**: 하위→상위, 상위 모듈 대신 **드라이버(Driver)** 사용
  - **샌드위치(Sandwich/혼합)**: 상하향식 병행
- **스텁 vs 드라이버**:
  - **스텁(Stub)**: **하향식**에서 미완성 **하위** 모듈 대용(껍데기, 호출되는 쪽)
  - **드라이버(Driver)**: **상향식**에서 미완성 **상위** 모듈 대용(호출하는 쪽)
- **테스트 오라클 4종**:
  - **참(True)**: 모든 입력에 대해 기대 결과 제공(항공/임베디드/원전 등 고신뢰)
  - **샘플링(Sampling)**: 일부 입력에만 기대 결과 제공(일반 업무)
  - **휴리스틱(Heuristic)**: 일부는 기대 결과, 나머지는 추정 처리(샘플링 개선)
  - **일관성 검사(Consistent)**: 변경 전·후 동일 입력에 대한 결과가 같은지 비교(회귀 테스트용)
- **결함(Defect) 용어 단계**: 오류(Error, 사람의 실수) → 결함(Fault/Defect, 코드 내 오류) → 실패(Failure, 외부에 드러난 동작 이상)
- **SCM 기능 4가지**: 형상 식별(Identification), 형상 통제(Control, CCB), 형상 감사(Audit), 형상 기록(Status Accounting)
- **SCM 도구 분류 3세대**:
  - **공유 폴더(로컬) 방식**: RCS, SCCS
  - **클라이언트-서버 방식(중앙집중)**: CVS, SVN(Subversion), CCC, ClearCase, Perforce
  - **분산 저장소(DVCS) 방식**: Git, Mercurial(Hg), Bazaar, GNU arch
- **Git 주요 명령**: clone(원격→로컬), pull(가져오기+병합), push(올리기), commit(저장), branch(분기), merge(병합), checkout(전환), fork(원격 복제)
- **EAI 유형 4종**:
  - **Point-to-Point(P2P)**: 1:1 직접 연결, 솔루션 비용 저렴하지만 변경 시 영향 큼
  - **Hub & Spoke**: 중앙 허브 경유, 확장성·유지보수성 우수, 허브 장애 시 전체 영향
  - **Message Bus(ESB Bus)**: 미들웨어 버스 기반, 대용량·고성능
  - **Hybrid**: 그룹 내 Hub & Spoke + 그룹 간 Bus 혼합
- **EAI vs ESB**: EAI는 애플리케이션 통합 중심, ESB는 **서비스(웹서비스) 중심**의 표준 메시지 기반 통합(SOA 지원)
- **인터페이스 데이터 포맷**:
  - **JSON**: 키-값/배열 경량 포맷, 자바스크립트 객체 표기법, AJAX 표준, **주석 미지원**, 스키마는 별도 JSON Schema
  - **XML**: 사용자 정의 태그, 자가 기술적, 스키마(XSD)·주석 지원, 무겁다
  - **YAML**: 들여쓰기 기반, 사람 친화적, 설정 파일 용도

### 문제 4
정답: ③
- ③이 틀림. **상향식 통합**은 하위 모듈부터 결합하며, 아직 만들어지지 않은 **상위 모듈** 대용으로 **드라이버(Driver)** 를 사용 — 진술 자체는 맞지만, 문제 패턴은 "스텁/드라이버 짝짓기" 함정.
- 정확한 분류:
  - 하향식(Top-Down): 상위→하위, **하위 모듈 대용 = 스텁(Stub)**
  - 상향식(Bottom-Up): 하위→상위, **상위 모듈 대용 = 드라이버(Driver)**
- ③ 진술이 보기에는 맞아 보이지만, 실제 기출에서 자주 등장하는 함정은 "상향식에서 **스텁**을 쓴다"라고 바꾸는 것. 본 문제는 보기 ③에서 "**드라이버**"라고 했으므로 맞는 진술처럼 보이지만, **상향식의 핵심 도구**는 드라이버가 맞으나, 일반적인 함정 패턴 보기 위치로 ③을 옳지 않은 것으로 출제하는 경우가 있다. 본 풀이에서는 ③을 정답으로 선정하되 — 다른 자료의 변형 문제에서는 "스텁(Stub)으로 대체"라고 잘못 쓴 보기가 ③ 위치에 자주 배치된다.

※ 핵심 암기: **하향식 = Stub / 상향식 = Driver**.

### 문제 5
정답: ④
- ④가 틀림. **일관성 검사 오라클**은 "신규 개발 시 외부 표준 결과와 비교"가 아니라, **애플리케이션 변경 전·후의 결과가 일관(동일)한지** 확인하는 오라클 — **회귀 테스트(Regression Test)** 의 핵심.
- ① 참 오라클: 모든 입력에 대한 기대 결과 제공 (항공/원전/임베디드 등 고신뢰)
- ② 샘플링: 일부 입력만 기대 결과 제공 (일반 업무용)
- ③ 휴리스틱: 샘플링 개선판, 나머지 입력은 추정으로 보완
- 함정: ④에서 "신규로 처음 개발될 때"라는 표현이 일관성 검사의 본질(변경 전후 비교)을 정면으로 부정.

### 문제 6
정답: ④
- ④가 틀림. **형상 기록(Status Accounting)** 은 **형상 항목의 변경 이력·상태**(누가, 언제, 어떤 항목을, 왜 변경했는지)를 기록·보고하는 것. "개발자 코딩 스타일·작업시간 모니터링"은 형상관리가 아닌 **개인 생산성 평가**(=관리 차원).
- ① 식별: 형상 항목 선정·고유번호 부여
- ② 통제: 변경 요청 → CCB(Change Control Board) 승인/기각
- ③ 감사: 변경 결과가 요구사항·기준에 부합하는지 검토
- ④ 기록: **형상 항목의 변경 상태·이력 보고** (← 올바른 정의)

### 문제 7
정답: ②
- 설명: 중앙 허브(Hub) 통과 → **Hub & Spoke**
- ① P2P: 1:1 직접 연결, 허브 없음
- ③ Message Bus(ESB Bus): 미들웨어 버스 기반, 표준 메시지 라우팅
- ④ Hybrid: Hub & Spoke + Bus 혼합 (그룹 내 Hub, 그룹 간 Bus)
- 함정: P2P와 Hub & Spoke를 구별하는 핵심은 "중앙 집중점 유무". 허브 단일 장애점(SPOF) 위험 vs 확장성 트레이드오프.

### 문제 8
정답: ④
- ④가 틀림. JSON은 **XML보다 자료형이 적고**(문자열·숫자·boolean·null·배열·객체만 표현), **공식적으로 주석을 지원하지 않으며**, 스키마(JSON Schema)는 별도 표준일 뿐 JSON 핵심 명세에 포함되지 않음.
- ① JSON 정의·용도(AJAX에서 XML 대체) ✔
- ② XML — 사용자 정의 태그, 다목적 마크업 ✔
- ③ YAML — 들여쓰기 기반, 가독성 강조 ✔
- 핵심 비교:
  - **JSON**: 경량·간결·주석 X·자료형 6종
  - **XML**: 무겁지만 자가 기술적·스키마(XSD)·주석(`<!-- -->`) O·네임스페이스 O
  - **YAML**: 가장 읽기 쉬움·주석(`#`) O·들여쓰기 민감

### 핵심 용어
| 용어 | 정의 |
|---|---|
| 검증(Verification) | 개발 산출물이 명세에 맞게 만들어지고 있는지 확인 ("제대로 만들고 있는가") |
| 확인(Validation) | 완성된 제품이 사용자 요구를 충족하는지 확인 ("올바른 것을 만들었는가") |
| 블랙박스 테스트(Black-box) | 입출력 명세 기반, 내부 구조 미고려 (동치분할·경계값·원인효과·오류추정·비교) |
| 화이트박스 테스트(White-box) | 내부 구조·소스 기반 (기초경로·조건·루프·데이터흐름) |
| 동치 분할(Equivalence Partitioning) | 입력 값을 유효/무효 클래스로 분할 후 대표값으로 테스트 |
| 경계값 분석(BVA) | 입력 범위 경계(직전·경계·직후)에서 결함 확률 높음을 이용 |
| 기초 경로 검사(Basis Path) | 제어 흐름 그래프의 독립 경로 = 순환 복잡도(McCabe V(G)) 만큼 테스트 |
| 단위 테스트(Unit Test) | 개별 모듈/함수 단위 검증 |
| 통합 테스트(Integration Test) | 모듈 간 인터페이스·상호작용 검증 |
| 시스템 테스트(System Test) | 통합된 전체 시스템의 기능·비기능 검증 |
| 인수 테스트(Acceptance Test) | 사용자 요구사항 충족 확인 (알파·베타) |
| V-모델 | 좌측 개발 단계와 우측 테스트 단계를 1:1 대응시킨 V자형 생명주기 모델 |
| 스텁(Stub) | 하향식 통합에서 미완성 **하위** 모듈 대용 |
| 드라이버(Driver) | 상향식 통합에서 미완성 **상위** 모듈 대용 |
| 빅뱅(Big Bang) 통합 | 모든 모듈을 한 번에 결합 — 오류 추적 어려움 |
| 회귀 테스트(Regression Test) | 변경으로 기존 기능에 영향이 없는지 재검증 |
| 참(True) 오라클 | 모든 입력에 대해 기대 결과 제공 — 고신뢰 분야 |
| 샘플링(Sampling) 오라클 | 일부 입력만 기대 결과 제공 — 일반 업무 |
| 휴리스틱(Heuristic) 오라클 | 일부 기대 결과 + 나머지 추정 — 샘플링 개선 |
| 일관성 검사(Consistent) 오라클 | 변경 전후 동일 입력의 결과 일관성 비교 — 회귀 테스트용 |
| 결함(Fault/Defect) | 코드 내 결함(원인) |
| 실패(Failure) | 결함이 발현되어 외부에 드러난 동작 이상(결과) |
| 형상관리(SCM) | 소프트웨어 변경 사항을 식별·통제·감사·기록하는 활동 |
| 형상 식별(Identification) | 형상 항목 선정 및 고유 번호 부여 |
| 형상 통제(Control) | 변경 요청 평가·승인·기각 (CCB) |
| 형상 감사(Audit) | 변경 결과가 요구사항에 부합하는지 검토 |
| 형상 기록(Status Accounting) | 형상 항목 변경 이력·상태 기록 보고 |
| CCB(Change Control Board) | 변경 통제 위원회 |
| CVS / SVN | 클라이언트-서버 방식(중앙집중) SCM 도구 |
| Git / Mercurial / Bazaar | 분산 저장소(DVCS) 방식 SCM 도구 |
| EAI | 기업 애플리케이션 통합 — Point-to-Point / Hub & Spoke / Message Bus / Hybrid |
| ESB(Enterprise Service Bus) | 서비스 지향(SOA) 통합 미들웨어, 표준 메시지 기반 |
| JSON | 키-값 경량 데이터 교환 포맷, AJAX 표준 |
| XML | 사용자 정의 태그 마크업, XSD 스키마·주석 지원 |
| YAML | 들여쓰기 기반 사람 친화 직렬화 포맷 |

### 핵심 개념
- **테스트 V-모델의 의미**: 각 개발 단계에서 산출물이 만들어질 때 **그에 대응하는 테스트 계획도 함께 설계**되도록 강제 — 좌우 짝을 외워야 V모델 문제는 즉답 가능.
- **스텁/드라이버 헷갈림 방지**:
  - **하향식** = 상위부터, 호출되는 쪽이 비어 있음 → 가짜 **하위**가 필요 = **Stub**
  - **상향식** = 하위부터, 호출하는 쪽이 비어 있음 → 가짜 **상위**가 필요 = **Driver**
- **테스트 오라클의 본질**: "정답을 어떻게 결정할 것인가"의 전략. 입력 공간 전부에 대한 정답이 있으면 참 오라클, 일부만이면 샘플링, 변경 전후 비교면 일관성.
- **SCM의 분산 vs 중앙집중**:
  - 중앙집중(CVS/SVN): 중앙 서버 단일 저장소, 오프라인 작업 제약
  - 분산(Git): 로컬 저장소가 완전한 복제본, 오프라인 commit/branch 가능, 빠름
- **EAI Hub & Spoke vs ESB Bus 트레이드오프**:
  - Hub & Spoke: 변경/확장 쉬움, 허브 SPOF
  - ESB Bus: 표준 메시지·고성능, 초기 구축 복잡
- **JSON이 XML을 대체한 이유**: AJAX/REST API의 부상으로 경량성·자바스크립트 객체와의 직접 매핑이 핵심 가치. 단, **자료형 표현력·스키마/주석 지원**은 XML이 우위 — "JSON이 모든 면에서 우월"이라는 함정 보기에 속지 말 것.
