④ **사용자 정의 무결성 — 동일 트랜잭션 내 모든 연산은 ACID 속성을 만족해야 한다.**

## 🔍 오답 역추적 개념 발췌

### DBMS 정의·기능
- **DBMS(DataBase Management System)** — 데이터베이스를 정의·생성·조작·제어하기 위한 통합 소프트웨어. 사용자와 DB 사이의 인터페이스.
- **필수 기능 3** — **정의**(Definition·DDL, 스키마 정의) / **조작**(Manipulation·DML, CRUD) / **제어**(Control·DCL, 권한·동시성·무결성·보안).
- **DBMS 장점 5** — 데이터 ① 중복 최소화 ② 일관성 유지 ③ 무결성 유지 ④ 공유 ⑤ 보안 강화.
- **DBMS 단점 4** — ① 운영비 증가 ② 복잡도(설계·관리) 증가 ③ 백업·복구 난이도 ④ 집중 관리에 따른 시스템 중단 위험(SPOF).

### 3단계 스키마 + 데이터 독립성
| 단계 | 관점 | 정의 |
|---|---|---|
| **외부 스키마(External)** | 개별 사용자·응용 프로그램의 view | 한 DB에 여러 개 존재. 서브 스키마라고도 함. |
| **개념 스키마(Conceptual)** | 조직 전체의 논리적 구조 | DB에 **단 하나만** 존재. 모든 응용의 통합 뷰. |
| **내부 스키마(Internal)** | 물리적 저장 구조 | 레코드 형식·인덱스·블록 배치 등. **단 하나**. |

- **사상(Mapping) 2종** — 외부/개념 사상 + 개념/내부 사상.
- **논리적 데이터 독립성** = 개념 ↔ 외부 (개념 스키마 변경이 외부 스키마·응용에 영향 X).
- **물리적 데이터 독립성** = 개념 ↔ 내부 (저장 구조 변경이 개념 스키마에 영향 X).

### Key 7종 정형 매칭
| 키 | 정의 | 핵심 함정 |
|---|---|---|
| **후보키(Candidate)** | **유일성(Uniqueness) + 최소성(Minimality)** 모두 만족 | 두 조건이 함께 와야 함 |
| **기본키(Primary)** | 후보키 중 **DBA가 선정**, **NULL·중복 불가** | 개체 무결성의 핵심 |
| **대체키(Alternate)** | 후보키 중 **기본키로 선정되지 않은** 나머지 | 보조키(Secondary Key)라고도 |
| **슈퍼키(Super)** | **유일성만** 만족, **최소성 X** | 후보키에 군더더기 속성 추가한 것 |
| **외래키(Foreign)** | 다른 릴레이션의 **기본키를 참조** | NULL 허용, 참조 PK에 존재해야 |
| **유일키(Unique)** | UNIQUE 제약, **NULL 허용** | 기본키와 달리 NULL 가능 |
| **복합키(Composite)** | **두 개 이상 속성**의 결합으로 키 구성 | 단일 속성으론 유일하지 않을 때 |

### 무결성 제약 4종
| 제약 | 대상 | 규칙 |
|---|---|---|
| **개체 무결성(Entity)** | **기본키** | NULL 불가 + 중복 불가 |
| **참조 무결성(Referential)** | **외래키** | 참조 릴레이션의 PK에 존재 or NULL |
| **도메인 무결성(Domain)** | **속성 값** | 정의된 도메인(타입·범위·NULL 허용여부) 내 |
| **사용자 정의 무결성(User-defined)** | **업무 규칙** | "급여는 0 이상" 등 도메인 외 규칙 |

### 참조 무결성 위반 시 옵션 4
- **NO ACTION** / RESTRICT — 거부 (기본값).
- **CASCADE** — 부모 변경/삭제 시 자식도 **연쇄** 변경/삭제.
- **SET NULL** — 자식의 외래키를 NULL로.
- **SET DEFAULT** — 자식의 외래키를 기본값으로.

### E-R 다이어그램 (Peter Chen 표기법)
| 기호 | 의미 |
|---|---|
| **사각형 ▭** | 개체(Entity) |
| **이중 사각형 ▭▭** | **약한 개체(Weak Entity)** — 함정! |
| **마름모 ◇** | 관계(Relationship) |
| **이중 마름모 ◇◇** | 약한 개체의 식별 관계 |
| **타원 ○** | 속성(Attribute) |
| **이중 타원 ○○** | 다중값 속성(Multi-valued) |
| **점선 타원** | 유도(파생) 속성(Derived) |
| **밑줄** | 기본키 속성 |
| **선** | 개체-속성/관계 연결 |

### 릴레이션 용어 매칭표
| 일반 용어 | 관계 모델 용어 | 표 형태 |

## 💡 상세 해설

### 문제 1
**정답 ④ 유지(Maintenance) 기능**
- DBMS의 **필수 기능 3종**: **정의(Definition·DDL)** / **조작(Manipulation·DML)** / **제어(Control·DCL)**.
  - 정의 = CREATE / ALTER / DROP — 스키마 구조 정의.
  - 조작 = SELECT / INSERT / UPDATE / DELETE — 데이터 CRUD.
  - 제어 = GRANT / REVOKE / COMMIT / ROLLBACK — 권한·동시성·무결성·트랜잭션 제어.
- **유지(Maintenance)·백업·보안 강화** 등은 DBMS가 제공하는 **부수적 기능**일 뿐, "필수 3기능"엔 들어가지 않음.
- 두음 정리: **"정·조·제"** — 정의·조작·제어.

### 문제 2
**정답 ③ 내부 스키마(Internal Schema)**
- **3단계 스키마(ANSI/SPARC)**:
  - **외부 스키마** = 개별 사용자·응용 프로그램이 보는 view (서브스키마). 여러 개 존재.
  - **개념 스키마** = 조직 전체의 **논리적** 구조. DB에 단 하나, DBA가 관리.
  - **내부 스키마** = **물리적 저장 구조** — 레코드 형식·인덱스·블록 배치·접근 경로 정의.
- "물리적 저장 / 인덱스 / 레코드 형식" 함정 키워드 → **내부 스키마** 즉시.
- ④ **부속 스키마**는 외부 스키마의 별칭이라고 부르긴 하나, "물리 저장" 정의가 아님 — 함정 보기.

### 문제 3
**정답 ② 유일성(Uniqueness) + 최소성(Minimality)**
- **후보키(Candidate Key)** 정형 2조건:
  - **유일성** = 릴레이션 내 모든 튜플에 대해 키 값이 **중복되지 않음**.
  - **최소성** = 키를 구성하는 속성 중 **하나라도 제거하면 유일성이 깨짐** — 즉, 군더더기 속성이 없음.
- 슈퍼키는 유일성만, 후보키는 **둘 다**, 기본키는 후보키 중 DBA가 선정.
- 함정: ① "유일성+무결성", ④ "원자성+유일성" 등 비슷한 용어 끼워넣기 — **"유일·최소" 짝**으로 외워두면 즉답.

### 문제 4
**정답 ② 기본키를 구성하는 속성은 NULL 값이나 중복 값을 가질 수 없다.**
- **개체 무결성(Entity Integrity)** = 각 릴레이션의 **기본키 속성은 NULL이나 중복 값을 가질 수 없다**는 제약.
- ① = **참조 무결성**의 정의(외래키 관련).
- ③ = **도메인 무결성**의 정의(속성 값 범위 관련).
- ④ = **사용자 정의 무결성**의 정의(업무 규칙 관련).
- 4종 정의가 보기로 모두 나오는 정형 4지선다 패턴. **"기본키 NULL 불가 = 개체"** 1:1 매칭.

### 문제 5
**정답 ③ 슈퍼키(Super Key)**
- **슈퍼키** = **유일성**은 만족(튜플을 유일하게 식별)하지만 **최소성**은 만족하지 않음(불필요한 속성 포함).
- 예: {학번, 이름} — 학번만으로 유일성 OK인데 이름 추가 → 최소성 X. 슈퍼키지만 후보키는 아님.
- ① 후보키 = 유일성 + **최소성**, ② 기본키 = 후보키 중 선정, ④ 외래키 = 참조키.
- 함정: "유일성만 O" 키워드 보이면 **슈퍼키** 즉시.

### 문제 6
**정답 ③ 참조하는 릴레이션의 기본키 값 중 하나와 일치하거나 NULL 값을 가져야 한다.**
- **참조 무결성(Referential Integrity)** = **외래키(FK)** 값은 ① **참조 릴레이션의 기본키(PK) 값 중 하나와 일치**하거나 ② **NULL**이어야 함.
- ① "항상 NULL 아닌 값" — NULL 허용이 핵심이므로 **틀림**.
- ② "동일한 이름" — 이름이 아니라 **값**이 일치해야 함. 이름은 달라도 무방.
- ④ "모든 후보키" — 참조 대상은 **기본키(PK)** 1개임. "모든 후보키"는 과도 조건.
- 위반 시 4옵션 — NO ACTION / CASCADE / SET NULL / SET DEFAULT 함께 암기.

### 문제 7
**정답 ④ 사용자 정의 무결성 — 동일 트랜잭션 내 모든 연산은 ACID 속성을 만족해야 한다.**
- **사용자 정의 무결성(User-defined)** = **업무 규칙**(예: "급여는 0 이상", "주문 수량은 1 이상") — DBMS가 기본 제공하지 않는, 사용자가 정의한 규칙. **ACID와 무관**.
- **ACID**(Atomicity·Consistency·Isolation·Durability)는 **트랜잭션 4속성**으로, 무결성 제약과 다른 개념(Day 40에서 학습).
- ①②③은 정확히 맞는 정의.
- 함정 빈출: "ACID = 사용자 정의 무결성"이라고 끼워넣기.

### 문제 8
**정답 ④ 이중 사각형(▭▭) — 약한 속성(Weak Attribute)**
- **이중 사각형**은 **약한 개체(Weak Entity)** — 다른 개체에 종속되어 식별자 없이 존재 불가한 개체(예: 부양가족).
- 약한 속성이라는 용어는 표준 Chen 표기법에 없음 — 함정 용어.
- **이중 타원(○○)** = **다중값 속성(Multi-valued Attribute)** — 한 개체가 여러 값(예: 전화번호 여러 개).
- **점선 타원** = **유도(파생) 속성(Derived)** — 다른 속성에서 계산되는 값(예: 나이 = 오늘 − 생일).
- 정형 매칭: 사각형=개체 / 마름모=관계 / 타원=속성 / 이중사각형=약한개체 / 이중타원=다중값속성 / 밑줄=기본키.
