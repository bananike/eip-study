## 🔍 오답 역추적 개념 발췌
- **필드(변수) 접근은 정적 바인딩, 메소드는 동적 바인딩**: 부모 타입 참조 변수로 자식 객체를 가리킬 때, `a.x`(필드)는 **컴파일 타입(=A)** 기준으로 부모의 x가 선택되지만, `a.getX()`(메소드)는 **런타임 객체 타입(=B)** 기준으로 자식 메소드가 호출된다. 시험 단골 함정.
- **생성자 호출 순서**:
  1) 자식 생성자 진입
  2) 첫 줄에 `super(...)` 또는 `this(...)` 호출이 명시되지 않으면 **컴파일러가 `super()`를 암묵 삽입** → 부모 생성자 실행 후 자식 생성자 본문 실행
  3) `this(인자)`를 명시하면 같은 클래스의 다른 생성자를 먼저 호출하며, 그 다른 생성자가 다시 부모 생성자를 호출
- **오버로딩 vs 오버라이딩 정리**:
  - **오버로딩(Overloading)**: 같은 이름 + **매개변수 개수/타입이 다름** (반환형만 다른 건 안 됨)
  - **오버라이딩(Overriding)**: 상속 관계에서 **이름·매개변수·반환형이 모두 동일**(반환형은 공변 가능). 접근 제어자는 같거나 더 넓게.
- **try-catch-finally + return 흐름**:
  - try 안의 `return`은 즉시 반환되지 않고, **finally를 먼저 실행한 뒤** 반환값을 들고 돌아온다.
  - finally에서 `return`을 하면 try의 return을 덮어쓴다(권장하지 않음).
  - finally는 **예외 발생 여부와 무관하게 거의 항상 실행**(예외: `System.exit` 등).
- **추상 클래스 vs 인터페이스 (Java 7 기준)**:
  - 추상 클래스: 일반·추상 메소드 혼재 가능, **단일 상속만**(extends 1개), 필드 자유롭게 가능, 생성자 존재
  - 인터페이스: 모든 메소드 `public abstract`(본문 없음), 모든 필드 `public static final` 상수, **다중 구현 가능(implements A, B, C)**, 생성자 없음
  - Java 8부터 default/static 메소드, Java 9부터 private 메소드 추가 — 하지만 시험은 보통 Java 7 기준
- **static 변수의 의미**: 클래스 전체가 **하나의 메모리**를 공유. 인스턴스 수가 늘어도 값은 누적되지 않고 같은 변수를 공유. `static int cnt`는 `클래스명.cnt`로 접근.
- **다형성(Polymorphism) + 동적 바인딩**: `상위타입 변수 = new 하위타입()`에서 `변수.메소드()` 호출 시 **실제 객체의 타입에 정의된 오버라이딩 메소드가 실행**된다. 컴파일러는 메소드 존재 여부를 컴파일 타입에서 확인하지만, 실제 호출 대상은 런타임에 결정.
- **접근 제어자 범위** (좁은 순):
  - `private` — **같은 클래스 내부**에서만
  - `default`(package-private, 키워드 없음) — **같은 패키지** 내에서
  - `protected` — 같은 패키지 + **상속받은 자식 클래스**(다른 패키지여도)
  - `public` — **어디서나**
- **컬렉션 핵심 동작**:
  - `ArrayList`: 순서 유지, 중복 허용, 인덱스 접근 O(1)
  - `HashMap`: key 중복 불가(같은 key put 시 **value 덮어씀**), key 기준 해시
  - `HashSet`: 순서 X, 중복 X, 내부 HashMap 기반
  - `Hashtable`: 동기화 O, null key/value 불가 (vs HashMap은 null 허용)

## 💡 상세 해설

### 문제 1
**정답: ② 10 20**

- `A a = new B();` — 컴파일 타입은 A, 런타임 객체 타입은 B(자식).
- `a.x` (**필드 접근**): **필드는 정적 바인딩** — 참조 변수의 **컴파일 타입(A)** 기준으로 결정 → A의 x인 **10**.
- `a.getX()` (**메소드 호출**): **메소드는 동적 바인딩(Dynamic Dispatch)** — 실제 객체 타입(B)의 오버라이딩 메소드 실행 → B의 getX() = B.x = **20**.
- 출력: `10 20`

**오답 분석**
- ① 10 10 — 메소드도 정적 바인딩으로 본 오류(자바는 메소드는 동적 바인딩)
- ③ 20 10 — 필드와 메소드 결과를 바꿔 본 오류
- ④ 20 20 — 필드도 동적 바인딩으로 본 오류(필드는 컴파일 타입 기준)

> 핵심: **자바에서 필드는 컴파일 타입 기준(정적), 메소드는 런타임 객체 타입 기준(동적).**

### 문제 2
**정답: ① PCc5**

- `new Child(5)` → `Child(int n)` 생성자 호출.
- `Child(int n)` 첫 줄에 `this()`가 있음 → **같은 클래스의 인자 없는 `Child()` 생성자 호출**.
  - `Child()`의 첫 줄에는 명시적 호출이 없으므로 컴파일러가 **`super()` 암묵 삽입** → `Parent()` 실행 → `"P"` 출력.
  - 이어서 `Child()` 본문 실행 → `"C"` 출력.
- `this()` 호출이 끝나면 `Child(int n)` 본문의 나머지 실행 → `"c" + 5` 출력.
- 최종 출력: `P` + `C` + `c5` = **`PCc5`**

**오답 분석**
- ② Pc5 — `this()`로 인한 `Child()` 호출(=Parent()→"P", "C" 출력)을 누락한 오류
- ③ p5Cc5 — `Parent(int)`가 호출됐다고 본 오류(자식 → 부모 인자 자동 전달은 일어나지 않음)
- ④ PCcC5 — "C"가 두 번 출력됐다고 본 오류

> 핵심: **`this(...)`는 같은 클래스의 다른 생성자 호출, `super(...)`는 부모 생성자 호출. 첫 줄에 둘 다 없으면 컴파일러가 `super()` 암묵 삽입.**

### 문제 3
**정답: ③ 반환형만 달라도 오버로딩이 성립한다. (오답이라서 정답)**

- ③이 **틀린 설명**. 자바 컴파일러는 오버로딩 여부를 **메소드 이름 + 매개변수 시그니처(개수·타입·순서)** 로만 판단하며, **반환형은 시그니처에 포함되지 않는다**. 따라서 반환형만 다른 두 메소드는 컴파일 에러.
- ① ② ④는 모두 옳은 설명:
  - ①: 같은 이름이어야 오버로딩
  - ②: 매개변수 개수/타입 차이가 핵심 조건
  - ④: 접근 제어자(public/private 등)나 throws 선언은 오버로딩 성립 여부에 영향 없음(다른 인자 시그니처면 OK)

> 핵심: **반환형만 다른 메소드는 오버로딩 X. 매개변수 개수/타입/순서 중 하나는 달라야 한다.**

### 문제 4
**정답: ② F1**

- `main`에서 `System.out.print(f());` — f() 호출 → 반환값을 받아서 출력. 단, f() 내부에도 출력문이 있으므로 흐름을 정확히 따라가야 함:
  1) `f()` 진입 → try 블록의 `return 1` 평가 → **반환값 1을 임시 저장**(아직 호출자로 돌아가지 않음)
  2) **finally 블록 실행** → `System.out.print("F")` → 콘솔에 `F` 출력
  3) 저장된 값 1을 main으로 반환
  4) main의 `System.out.print(f())` → `1` 출력
- 콘솔에 순서대로 찍힌 결과: `F` + `1` = **`F1`**

**오답 분석**
- ① 1 — finally가 실행 안 된다고 본 오류
- ③ 1F — return이 즉시 반환되고 finally가 그 후 실행된다고 본 오류(실제는 finally가 반환 직전 실행)
- ④ F2 — catch 블록이 실행됐다고 본 오류(예외 미발생)

> 핵심: **try 안의 `return 값`은 평가 후 반환값을 임시 저장, finally를 먼저 실행한 다음에 반환. 따라서 finally의 출력이 먼저 콘솔에 찍힌다.**

### 문제 5
**정답: ④ 한 클래스는 여러 추상 클래스를 다중 상속할 수 있다. (오답이라서 정답)**

- ④가 **틀린 설명**. 자바는 **클래스의 다중 상속을 금지**한다(extends 키워드는 단 하나의 클래스만 가능). 추상 클래스도 결국 클래스이므로 동일하게 단일 상속만 가능. 다중 상속이 필요하면 **인터페이스의 다중 구현(implements A, B, C)** 으로 우회한다.
- ① ② ③은 모두 옳은 설명:
  - ①: 추상 클래스는 일반 메소드 + 추상 메소드 혼재 가능
  - ②: 인터페이스의 모든 메소드는 묵시적 `public abstract`(Java 7 기준, 본문 X)
  - ③: 인터페이스의 모든 필드는 묵시적 `public static final` 상수

> 핵심: **자바 클래스는 단일 상속만, 인터페이스만 다중 구현 가능. 추상 클래스 ≠ 인터페이스.**

### 문제 6
**정답: ② 3 1**

- `static int cnt`: 클래스 전체가 공유하는 변수. 인스턴스가 늘 때마다 누적.
  - `Counter a`: cnt = 1
  - `Counter b`: cnt = 2
  - `Counter c`: cnt = 3
  - → `Counter.cnt` = **3**
- `int local`: 각 인스턴스가 **독립적으로 보유하는 변수**. 생성자에서 0 → 1로 증가.
  - 각 인스턴스의 local 값은 모두 1 (서로 영향 없음)
  - → `c.local` = **1**
- 출력: `3 1`

**오답 분석**
- ① 1 1 — cnt도 인스턴스마다 독립이라고 본 오류(static의 의미 누락)
- ③ 3 3 — local도 static처럼 누적된다고 본 오류
- ④ 1 3 — cnt와 local의 역할을 뒤바꿔 본 오류

> 핵심: **`static` 변수 = 클래스에 1개(공유), 일반 인스턴스 변수 = 인스턴스마다 1개(독립).**

### 문제 7
**정답: ② Woof Meow ??**

- 배열 `arr`는 컴파일 타입이 `Animal[]`이지만, 각 원소의 **런타임 객체 타입은 Dog, Cat, Animal**.
- 향상된 for문에서 `a.sound()` 호출 시 **동적 바인딩** — 각 원소의 실제 타입에 정의된 sound() 호출.
  - `new Dog()` → Dog.sound() = `"Woof"`
  - `new Cat()` → Cat.sound() = `"Meow"`
  - `new Animal()` → Animal.sound() = `"??"`(오버라이딩되지 않은 부모 메소드)
- 출력: `Woof Meow ??`

**오답 분석**
- ① ?? ?? ?? — 모두 컴파일 타입(Animal) 기준으로 호출된다고 본 오류
- ③ Woof Woof Woof — 첫 원소의 결과를 반복한 오류
- ④ 컴파일 오류 — Animal[] 배열에 Dog/Cat를 넣는 것은 다형성으로 정상

> 핵심: **다형성 + 동적 바인딩 = 부모 타입 배열에서도 실제 객체의 메소드가 호출된다.**

### 문제 8
**정답: ② 3 3**

- `HashMap<String, Integer>`은 **key 중복 불가**, 동일 key에 다른 value를 put하면 **이전 값 덮어쓰기**.
  - `put("A", 1)` → {A:1}
  - `put("B", 2)` → {A:1, B:2}
  - `put("A", 3)` → {A:3, B:2} (key "A"의 value가 1→3으로 갱신, size는 그대로 2)
  - `put("C", 4)` → {A:3, B:2, C:4}
- `m.size()` = **3**(서로 다른 key 3개)
- `m.get("A")` = **3**(마지막에 put한 값으로 덮어써짐)
- 출력: `3 3`

**오답 분석**
- ① 3 1 — A의 값이 처음 put한 1로 유지된다고 본 오류(HashMap은 덮어쓰기)
- ③ 4 3 — 같은 key의 put을 별도 항목으로 카운트한 오류(중복 key는 size 증가 X)
- ④ 4 1 — 위 두 오류가 함께 발생한 경우

> 핵심: **HashMap은 같은 key로 put하면 value를 덮어쓴다. size는 고유 key 수.**
