## 🎯 오늘의 기출문제 — 디자인 패턴 GoF 23 약점 집중 15문제 (Phase 4 약점보강 1일차 · Day 14 신규 비중복)

### 문제 1 [유사 기출: 2021년 1회 1과목 — GoF 디자인 패턴 3대 분류 정형 패턴]
**GoF(Gang of Four) 디자인 패턴 23종의 3대 분류**에 관한 설명 중 **가장 옳지 않은 것**은?

① **생성(Creational) 패턴 5종** = **Abstract Factory · Builder · Factory Method · Prototype · Singleton**. 객체 생성 과정을 캡슐화·유연화하며 "무엇을(what)"·"누가(who)"·"언제(when)"·"어떻게(how)" 생성할지의 결합도를 낮춘다.
② **구조(Structural) 패턴 7종** = **Adapter · Bridge · Composite · Decorator · Facade · Flyweight · Proxy**. 클래스·객체를 조합해 더 큰 구조를 형성하며, 인터페이스 변환·기능 확장·부분-전체 계층·대리자 등의 역할을 한다.
③ **행위(Behavioral) 패턴 11종** = **Chain of Responsibility · Command · Interpreter · Iterator · Mediator · Memento · Observer · State · Strategy · Template Method · Visitor**. 객체 간의 상호작용·책임 분배 알고리즘을 캡슐화한다.
④ **GoF 23종은 3대 분류가 없고 모두 "생성" 하나의 카테고리로 통합**되며, **Adapter·Facade는 행위 패턴**, **Observer·Strategy는 구조 패턴**, **Singleton은 행위 패턴**으로 분류된다.

### 문제 2 [유사 기출: 2020년 3회 1과목 — 싱글턴(Singleton) 정형 패턴]
**싱글턴(Singleton) 패턴**에 관한 설명 중 **가장 옳지 않은 것**은?

① **의도** — **오직 하나의 인스턴스**만 생성되도록 보장하고, 그 인스턴스에 대한 **전역 접근점(Global Access Point)** 을 제공한다. 로거·설정 관리자·연결 풀·캐시 관리자 등에서 사용.
② **구현 요소** — ① private 생성자(외부 `new` 금지) ② private static 인스턴스 필드 ③ public static `getInstance()` 팩토리 메서드. 지연 초기화(Lazy) vs 이른 초기화(Eager) 선택 가능.
③ **스레드 안전 이슈** — 지연 초기화 + 다중 스레드 환경에서는 **동기화(synchronized)·DCL(Double-Checked Locking) + volatile·Holder Idiom(초기화 온 디맨드 홀더)·enum Singleton** 등을 사용해야 중복 인스턴스 생성을 방지할 수 있다.
④ **싱글턴 패턴은 인스턴스를 무제한 생성하는 것이 원칙**이며, **전역 접근점을 제공해서는 안 되고**, **생성자를 public으로 열어 외부 `new` 호출을 강제**해야 한다.

### 문제 3 [유사 기출: 2022년 2회 1과목 — Factory Method vs Abstract Factory 정형 패턴]
**팩토리 메서드(Factory Method) · 추상 팩토리(Abstract Factory)** 두 생성 패턴의 차이에 관한 설명 중 **가장 옳지 않은 것**은?

① **Factory Method** — **객체 생성의 인터페이스를 상위 클래스에 정의**하고, 어떤 클래스의 인스턴스를 만들지는 **하위(구체) 클래스가 결정**하도록 위임한다. 상속(Inheritance) 기반의 생성 위임.
② **Abstract Factory** — **관련성이 있는(같은 제품군에 속하는) 여러 객체 계열**을 만들 수 있는 인터페이스를 제공하고, 구체 팩토리 클래스가 실제 제품군을 생성한다. UI 킷(Windows/Mac/GTK), DB 드라이버 팩토리 등이 대표 예.
③ **구분 요약** — Factory Method는 **하나의 팩토리 메서드로 하나의 제품**을 만들고 상속으로 다양성 확보, Abstract Factory는 **하나의 팩토리로 여러 제품군**을 만들며 팩토리 자체를 교체해 다양성 확보. Abstract Factory 내부에서 Factory Method가 자주 함께 사용된다.
④ **Factory Method와 Abstract Factory는 완전히 동일한 패턴**이며, **모두 구체 클래스에서 객체를 직접 `new` 로 생성해 상위 계층에 강한 결합을 만드는 것이 원칙**이고, **제품군(family) 개념은 두 패턴 모두 존재하지 않는다**.

### 문제 4 [유사 기출: 2021년 3회 1과목 — 빌더(Builder) · 프로토타입(Prototype) 정형 패턴]
**빌더(Builder) · 프로토타입(Prototype) 패턴**에 관한 설명 중 **가장 옳지 않은 것**은?

① **Builder** — **복잡한 객체의 생성 과정을 단계별로 분리**하여, 동일한 생성 절차로 서로 다른 표현(representation)의 객체를 만들 수 있게 한다. `StringBuilder`·Lombok `@Builder`·SQL Query Builder·Spring `UriComponentsBuilder` 등이 실무 예.
② **Builder의 강점** — **다수의 매개변수(선택 vs 필수)를 가진 생성자**의 가독성 문제(텔레스코핑 생성자·매개변수 순서 혼동)를 해결하고, **필수/선택 파라미터의 조합**을 안전하게 조립 가능. 불변(Immutable) 객체 생성에도 자주 사용.
③ **Prototype** — **원본 객체를 복제(clone)** 하여 새 객체를 생성. 생성 비용이 큰 객체(예: DB에서 로딩한 초기 데이터)를 매번 새로 생성하는 대신 복제로 성능을 개선. **얕은 복사(shallow copy) vs 깊은 복사(deep copy)** 선택 이슈가 존재.
④ **Builder는 객체 복제 패턴이고 Prototype은 단계별 생성 패턴**이며, **두 패턴은 모두 구조 패턴에 속하고**, **StringBuilder·`clone()` 메서드는 GoF 패턴과 무관한 별개 개념**이다.

### 문제 5 [유사 기출: 2020년 4회 1과목 — Adapter vs Bridge 정형 패턴]
**어댑터(Adapter) · 브리지(Bridge) 두 구조 패턴**의 차이에 관한 설명 중 **가장 옳지 않은 것**은?

① **Adapter** — **호환되지 않는 인터페이스를 가진 클래스들을 함께 동작**하도록, **기존 클래스의 인터페이스를 클라이언트가 기대하는 인터페이스로 변환**한다. **레거시 코드 통합·서드파티 라이브러리 wrapping** 등 이미 존재하는 코드에 사후(after-the-fact) 적용하는 것이 특징. 클래스 어댑터(다중 상속)·객체 어댑터(위임/합성) 두 형태.
② **Bridge** — **추상(Abstraction)과 구현(Implementation)을 분리**해, 각각을 **독립적으로 확장**할 수 있게 한다. 도형(원·사각형·삼각형) × 렌더링(래스터·벡터·API별) 처럼 두 축이 모두 확장 가능한 경우, 상속으로 조합 폭발(2×3=6, 3×3=9…)을 피하고 합성으로 대체.
③ **핵심 차이** — Adapter는 **기존 인터페이스 불일치 해결(사후 적용)**, Bridge는 **처음부터 두 계층을 분리 설계(사전 설계)**. Adapter는 "결과 위주", Bridge는 "설계 원칙 위주"라는 관점이 자주 강조된다.
④ **Adapter는 처음부터 추상·구현 두 계층을 분리 설계하는 사전 설계 패턴**이고, **Bridge는 이미 만들어진 두 인터페이스 사이의 호환성 문제를 사후에 해결하는 통합 패턴**이며, **둘은 실질적으로 같은 목적을 가진 동일 패턴**이다.

### 문제 6 [유사 기출: 2022년 1회 1과목 — Composite vs Decorator 정형 패턴]
**컴포지트(Composite) · 데코레이터(Decorator) 두 구조 패턴**의 차이에 관한 설명 중 **가장 옳지 않은 것**은?

① **Composite** — **부분-전체 계층(Part-Whole Hierarchy)** 을 트리 구조로 표현하며, 단일 객체(Leaf)와 복합 객체(Composite)를 **동일한 인터페이스**로 다룰 수 있게 한다. 파일 시스템(파일 vs 폴더)·UI 컴포넌트 트리·조직도·GUI 위젯 그룹이 대표 예.
② **Decorator** — **객체의 책임(기능)을 동적으로 추가**하는 패턴으로, 서브클래싱 대신 **위임/합성 기반의 감싸기(wrapping)** 로 기능을 확장한다. Java I/O 스트림(`BufferedReader(new FileReader(...))`)·Spring FilterChain·UI 스크롤/테두리 데코레이터 등이 대표 예.
③ **핵심 차이** — Composite는 **여러 객체를 하나로 묶어 트리 구조**로 다룸(같은 인터페이스), Decorator는 **하나의 객체를 감싸 새 기능을 추가**함(같은 인터페이스). 두 패턴 모두 **컴포넌트 인터페이스의 재귀적 합성**을 사용하지만 목적이 다르다.
④ **Composite는 하나의 객체를 감싸 기능을 추가하는 패턴**이고, **Decorator는 여러 객체를 트리 구조로 묶어 부분-전체 계층을 표현하는 패턴**이며, **Java I/O 스트림은 Composite 패턴, 파일-폴더 트리는 Decorator 패턴의 대표 예**이다.

### 문제 7 [유사 기출: 2023년 2회 1과목 — Facade vs Proxy 정형 패턴]
**퍼사드(Facade) · 프록시(Proxy) 두 구조 패턴**의 차이에 관한 설명 중 **가장 옳지 않은 것**은?

① **Facade** — **복잡한 서브시스템**에 대한 **단순화된 통합 인터페이스**를 제공. 서브시스템의 상세를 숨기고 하나의 진입점을 제공하며, 클라이언트가 세부 클래스들과 직접 상호작용하지 않도록 한다. Spring의 `JdbcTemplate`·서비스 계층 API가 대표 예.
② **Proxy** — **다른 객체에 대한 대리자(Surrogate)** 역할을 수행. **동일한 인터페이스**를 유지하면서 **접근 제어·지연 로딩·원격 호출·캐싱·로깅** 등 부가 관심사를 대신 처리한다. Java `Proxy`/CGLIB·Hibernate 지연 로딩·JPA·Spring AOP·JAX-WS/RMI 스텁이 대표 예.
③ **Proxy의 유형** — **Virtual Proxy(지연 초기화)** · **Remote Proxy(원격 객체 대리)** · **Protection Proxy(접근 제어)** · **Smart Reference / Smart Proxy(참조 카운트·캐시·락)** 등으로 분류된다. Facade는 "인터페이스 단순화", Proxy는 "인터페이스 동일 + 부가 기능"이라는 축이 다르다.
④ **Facade는 원본과 동일한 인터페이스로 부가 기능을 추가하는 대리자 패턴**이고, **Proxy는 복잡한 서브시스템을 단순한 통합 인터페이스로 감싸는 진입점 패턴**이며, **Spring AOP·JPA 지연 로딩은 Facade 패턴, `JdbcTemplate`은 Proxy 패턴의 대표 예**이다.

### 문제 8 [유사 기출: 2021년 2회 1과목 — 플라이웨이트(Flyweight) 정형 패턴]
**플라이웨이트(Flyweight) 패턴**에 관한 설명 중 **가장 옳지 않은 것**은?

① **의도** — **동일하거나 유사한 다수의 객체를 공유(share)하여 메모리 사용을 최소화**한다. 다수의 세밀한(fine-grained) 객체를 효율적으로 다루기 위한 구조 패턴.
② **핵심 개념** — **내재적 상태(Intrinsic State · 공유 가능·불변)** 는 플라이웨이트 객체 내부에 저장해 공유하고, **외재적 상태(Extrinsic State · 문맥 의존·가변)** 는 클라이언트가 매 호출마다 전달한다. `FlyweightFactory` 가 인스턴스 풀을 관리하며 요청 시 재사용.
③ **대표 사례** — 워드 프로세서의 **문자(Glyph) 객체**(문자 코드=내재적 / 위치·폰트=외재적) · Java `String.intern()` 문자열 풀 · `Integer.valueOf()` 정수 캐싱(-128~127) · 게임의 다수 동일 그래픽 자원(타일·파티클) · 브라우저 DOM 최적화.
④ **플라이웨이트 패턴은 객체를 절대 공유하지 않고 매번 새로 생성하는 것이 원칙**이며, **내재적 상태와 외재적 상태의 구분은 존재하지 않고**, **`String.intern()`·`Integer.valueOf()` 캐시는 플라이웨이트와 무관한 별개 최적화**이다.

### 문제 9 [유사 기출: 2020년 2회 1과목 — 옵서버(Observer) 정형 패턴]
**옵서버(Observer) 패턴**에 관한 설명 중 **가장 옳지 않은 것**은?

① **의도** — **한 객체의 상태 변화가 있을 때 그에 의존하는 여러 객체들이 자동으로 통보받고 갱신**되도록 하는 **1:N 의존 관계** 를 정의한다. Publish–Subscribe 모델의 원형(intra-process 버전).
② **구성 요소** — **Subject(주체·Publisher)** 는 관찰자 목록(`observers`)을 유지하며 `attach()·detach()·notify()` 메서드 제공, **Observer(관찰자·Subscriber)** 는 `update()` 인터페이스를 구현한다. Push 방식(주체가 데이터 전달) vs Pull 방식(관찰자가 주체를 조회) 두 형태.
③ **대표 사례** — Java `java.util.Observer`/`Observable`(Deprecated 후 `PropertyChangeListener` 권장) · Swing `ActionListener` · MVC 아키텍처의 Model↔View 동기화 · **Reactive Streams(`Publisher`/`Subscriber`)** · Vue/React 반응성 시스템의 개념적 기반.
④ **옵서버 패턴은 1:1 관계에만 적용되며 여러 관찰자에게 통보하는 것은 불가능**하고, **주체의 상태 변화에 반응하지 않는 것이 원칙**이며, **MVC의 Model↔View 동기화·Reactive Streams는 옵서버와 완전히 무관한 별개 개념**이다.

### 문제 10 [유사 기출: 2022년 3회 1과목 — Strategy vs State 정형 패턴]
**전략(Strategy) · 상태(State) 두 행위 패턴**의 차이에 관한 설명 중 **가장 옳지 않은 것**은?

① **Strategy** — **알고리즘군을 정의하고 각각 캡슐화하여 상호 교환 가능**하게 한다. 클라이언트가 **명시적으로 알고리즘(전략)을 선택·주입**하며, 정렬 전략(Bubble/Quick/Merge) · 결제 전략(카드/계좌/포인트) · 압축 전략(Zip/Gzip)이 대표 예.
② **State** — **객체의 내부 상태에 따라 행위가 바뀌도록** 하며, 상태 클래스 자체가 다음 상태로 전이(transition)를 결정. **상태 전이는 컨텍스트나 상태 객체 내부에서 자율적으로 발생**하며, 클라이언트는 상태 존재를 인지하지 못하는 경우가 많다. TCP 연결 상태(LISTEN→SYN-RECV→ESTABLISHED…)·주문 상태(신규→결제→배송→완료)가 대표 예.
③ **구조적 유사성 · 의도 차이** — 두 패턴 모두 컨텍스트가 **인터페이스를 통해 위임된 객체(전략/상태)에 행동을 위임**하는 동일한 UML 구조를 가진다. 그러나 **Strategy는 "동일한 문제를 다른 방식으로"(알고리즘 선택)**, **State는 "상태에 따라 다른 문제를"(상태별 동작)** 이라는 의도 축이 다르다.
④ **Strategy와 State는 완전히 별개의 UML 구조를 가지며 교환 가능하지 않다**. **Strategy는 상태 전이 기반 패턴**이고 **State는 정렬·결제 알고리즘 교체 패턴**이며, **TCP 연결 상태 관리는 Strategy 패턴의 대표 예**이다.

### 문제 11 [유사 기출: 2023년 1회 1과목 — Observer vs Mediator 정형 패턴]
**옵서버(Observer) · 중재자(Mediator) 두 행위 패턴**의 차이에 관한 설명 중 **가장 옳지 않은 것**은?

① **Observer** — **1:N 통보** — 주체(Subject) 하나의 상태 변화가 다수의 관찰자(Observer)에게 자동 전파. 관찰자들은 서로를 몰라도 되며 주체만 안다. **Publish–Subscribe 모델**의 원형.
② **Mediator** — **N:N 상호작용을 중앙 중재자로 캡슐화** — 여러 객체 간의 복잡한 통신을 하나의 중재자 객체가 대신 조율하여, **객체 간 직접 참조를 제거**하고 결합도를 낮춘다. GUI 다이얼로그의 폼 컨트롤 조율·항공 관제·챗룸 서버·Java Swing의 `KeyboardFocusManager` 등이 대표 예.
③ **핵심 차이** — Observer는 **한 방향(주체→관찰자) 자동 통보**에 초점, Mediator는 **다방향(N:N) 조율을 중앙 집중화**에 초점. Mediator는 객체 간 직접 참조를 중재자로 대체해 결합도를 낮추지만, 중재자 자체가 복잡해지는 트레이드오프.
④ **Observer는 여러 객체 간 N:N 통신을 중앙 조율자로 대체하는 패턴**이고, **Mediator는 하나의 주체가 여러 관찰자에게 상태 변화를 자동 통보하는 1:N 패턴**이며, **Publish–Subscribe 모델은 Mediator, Java Swing의 `KeyboardFocusManager`는 Observer 패턴**이다.

### 문제 12 [유사 기출: 2021년 1회 1과목 — 템플릿 메서드(Template Method) · 커맨드(Command) 정형 패턴]
**템플릿 메서드(Template Method) · 커맨드(Command) 두 행위 패턴**에 관한 설명 중 **가장 옳지 않은 것**은?

① **Template Method** — **알고리즘의 골격(뼈대)을 상위 클래스에 정의**하고, 일부 단계(hook·primitive operation)를 **하위 클래스가 오버라이드**하여 채우게 한다. 알고리즘 구조는 변경하지 않고 단계별 세부만 커스터마이즈. **상속 기반**(전략 패턴은 위임 기반)이라는 축이 자주 강조.
② **대표 사례** — Java `AbstractList`/`AbstractSet`의 골격 구현 · Spring `AbstractView`·`JdbcTemplate`(`execute()` 내부에 커넥션 획득·SQL 실행·결과 매핑·자원 해제 골격)의 콜백 훅 · JUnit `TestCase.runBare()` 의 setUp→runTest→tearDown 시퀀스.
③ **Command** — **요청(요구)을 객체로 캡슐화**하여 매개변수화·큐잉·로깅·언두(undo)·리두(redo)·트랜잭션·매크로 등을 가능하게 한다. 실행자(Receiver)·명령(Command)·발행자(Invoker)·클라이언트로 구성. GUI 메뉴/툴바·트랜잭션 로그·매크로 레코더·작업 큐가 대표 예.
④ **Template Method는 요청을 객체로 캡슐화해 undo/redo를 지원하는 패턴**이고, **Command는 알고리즘의 뼈대를 상위 클래스가 정의하고 세부를 하위가 채우는 상속 기반 패턴**이며, **`JdbcTemplate`은 Command 패턴, GUI 메뉴 undo/redo는 Template Method 패턴의 대표 예**이다.

### 문제 13 [유사 기출: 2022년 2회 1과목 — 이터레이터(Iterator) · 방문자(Visitor) · 책임 연쇄(CoR) 정형 패턴]
**이터레이터(Iterator) · 방문자(Visitor) · 책임 연쇄(Chain of Responsibility)** 세 행위 패턴에 관한 설명 중 **가장 옳지 않은 것**은?

① **Iterator** — **집합체(Aggregate)의 내부 구조를 노출하지 않고 순차적으로 원소에 접근**하는 방법을 제공. Java `Iterator`/`Iterable`·`Enumeration`·C++ STL 반복자·Python `__iter__/__next__`가 실무 예. 외부 반복자(External · 클라이언트가 제어) vs 내부 반복자(Internal · 컬렉션이 제어) 분류.
② **Visitor** — **객체 구조(요소들)와 그 객체에 대해 수행할 연산을 분리**. 새 연산을 추가할 때 기존 요소 클래스를 수정하지 않고 새 Visitor를 추가하는 방식. **이중 디스패치(Double Dispatch)** 로 구현. 컴파일러 AST 순회(TypeChecker·CodeGen·PrettyPrinter Visitor)·파일 시스템 크기 계산이 대표 예.
③ **Chain of Responsibility** — **요청을 처리할 수 있는 객체들을 사슬로 연결**해, 요청이 사슬을 따라 전달되며 처리 가능한 객체가 처리하도록 한다. **송신자와 수신자를 분리**하며, 각 핸들러가 처리 여부를 자율 결정. Servlet Filter/`FilterChain`·Spring Security `FilterChainProxy`·로깅 레벨(DEBUG→INFO→WARN→ERROR)·GUI 이벤트 버블링이 대표 예.
④ **Iterator는 요청을 처리할 핸들러 사슬 패턴**이고, **Visitor는 집합체의 순회 방법 제공 패턴**이며, **Chain of Responsibility는 요소 구조와 연산 분리 이중 디스패치 패턴**이고, **Servlet Filter는 Iterator, Java `Iterator`는 Visitor, 컴파일러 AST 순회는 Chain of Responsibility 패턴**이다.

### 문제 14 [유사 기출: 2023년 3회 1과목 — 메멘토(Memento) · 인터프리터(Interpreter) 정형 패턴]
**메멘토(Memento) · 인터프리터(Interpreter) 두 행위 패턴**에 관한 설명 중 **가장 옳지 않은 것**은?

① **Memento** — **객체의 내부 상태를 외부에 노출하지 않고 저장·복원**할 수 있게 하는 패턴. **캡슐화를 깨지 않으면서** 이전 상태로의 롤백(undo/redo)·스냅샷을 지원. 원조(Originator)·기념품(Memento)·관리자(Caretaker) 3역할로 구성. 텍스트 에디터의 undo 스택·게임 세이브 포인트·트랜잭션 롤백이 대표 예.
② **Memento vs Command undo** — Command undo는 **역연산(inverse operation)** 기반, Memento는 **상태 스냅샷 저장** 기반. 상태가 크면 Memento의 메모리 비용이 크고, 역연산이 복잡하면 Command undo가 어렵다. 두 방식은 종종 결합해 사용된다.
③ **Interpreter** — **주어진 언어(문법)의 문법 표현(Grammar)** 을 정의하고 이를 해석해 문장을 평가. 각 문법 규칙을 AST 노드 클래스로 표현하고 `interpret(Context)` 를 재귀 호출. **SQL 표현식·정규 표현식·수식 계산기·룰 엔진·설정 DSL** 등이 대표 예이며, 복잡한 언어에는 부적합해 파서 제너레이터(ANTLR·Yacc)가 대체.
④ **Memento는 문법을 해석해 문장을 평가하는 언어 처리 패턴**이고, **Interpreter는 객체 상태를 캡슐화하며 저장·복원하는 스냅샷 패턴**이며, **텍스트 에디터 undo 스택·트랜잭션 롤백은 Interpreter, SQL 표현식·정규 표현식 해석은 Memento 패턴의 대표 예**이다.

### 문제 15 [유사 기출: 2024년 1회 1과목 — GoF 디자인 패턴 종합 사례 매칭 정형 패턴]
**다음 시나리오와 GoF 디자인 패턴의 매칭** 중 **가장 옳지 않은 것**은?

① **"게임 캐릭터 상태(대기/이동/공격/사망)에 따라 매 프레임 다른 로직이 실행되도록 하되, 상태 전이는 각 상태 클래스가 내부에서 결정하도록 캡슐화한다"** → **State 패턴**.
② **"결제 수단(신용카드·계좌이체·간편결제) 알고리즘을 런타임에 클라이언트가 선택해 주문 서비스에 주입할 수 있게 한다"** → **Strategy 패턴**.
③ **"온라인 문서 편집기에서 여러 사용자의 커서 위치가 변할 때 다른 사용자들의 화면에 자동으로 반영되어야 한다"** → **Observer 패턴** (또는 Publish–Subscribe / Reactive Streams).
④ **"복잡한 서브시스템(주문·결제·재고·배송·통보 모듈)의 세부 API를 클라이언트에게 전부 노출하지 않고, 단순화된 통합 진입점만 제공한다"** → **Adapter 패턴** (또는 Bridge 패턴). **"인터페이스 불일치 문제 해결"이 목적이 아니지만, GoF 분류상 서브시스템 단순화는 어댑터 계열에 속하므로 어댑터 패턴이 유일한 정답 매칭이다**.

## 💡 상세 해설

### 문제 1
- **핵심**: GoF 23 = **생성 5 · 구조 7 · 행위 11** 3대 분류 · 각 분류의 소속 패턴 이름과 개수를 정확히 암기해야 함.
- **정답 ④ 함정**: "3대 분류 없이 생성으로 통합·Adapter/Facade 행위·Observer/Strategy 구조·Singleton 행위"는 4중 뒤바꾸기 함정으로 GoF 분류 원칙을 정면으로 위반. **Adapter·Facade는 구조**, **Observer·Strategy는 행위**, **Singleton은 생성** 이 정답 분류.
- **암기 팁**: 생성 5 = **"AB · FPS"**(Abstract Factory · Builder · Factory Method · Prototype · Singleton) / 구조 7 = **"ABC · DFFP"**(Adapter · Bridge · Composite · Decorator · Facade · Flyweight · Proxy) / 행위 11 = **"CCI · IMM · OSSTV"**(Chain of Responsibility · Command · Interpreter · Iterator · Mediator · Memento · Observer · State · Strategy · Template Method · Visitor).

### 문제 2
- **핵심**: Singleton = **인스턴스 1개 보장 + 전역 접근점 제공**. private 생성자·private static 인스턴스·public static `getInstance()` 3요소 필수.
- **정답 ④ 함정**: "무제한 생성·전역 접근점 금지·public 생성자"는 Singleton의 정의 3요소를 정면으로 뒤집은 함정.
- **실무·시험 포인트**: 스레드 안전 Singleton 5가지 방법(**Eager Initialization · Synchronized Method · DCL + volatile · Bill Pugh Holder · enum Singleton**)과 각각의 트레이드오프를 시험에서 자주 물음. Effective Java(Joshua Bloch)는 **enum Singleton**을 가장 안전한 방법으로 권장(직렬화·리플렉션 안전).

### 문제 3
- **핵심**: Factory Method = **1 팩토리 메서드 → 1 제품 · 상속 기반**, Abstract Factory = **1 팩토리 → 여러 제품군(family) · 합성 기반 + Factory Method 조합**.
- **정답 ④ 함정**: "완전 동일 패턴·구체에서 직접 `new`·제품군 개념 없음"은 두 패턴의 차별점을 3중으로 뭉갠 함정.
- **시험 포인트**: **"제품군(product family)" 키워드가 나오면 무조건 Abstract Factory**, "하위 클래스가 결정" 키워드는 Factory Method. UI 킷 예(Windows/Mac/GTK)에서 버튼·체크박스·스크롤바 세트 전환은 Abstract Factory, 문서 종류(HTML/PDF/Text)에 따라 다른 파서 반환은 Factory Method.

### 문제 4
- **핵심**: Builder = **단계별 조립(step-by-step)·복잡 객체 생성** / Prototype = **원본 복제(clone)·생성 비용 절감**. **두 패턴 모두 생성 패턴**.
- **정답 ④ 함정**: "Builder=복제·Prototype=단계별·구조 패턴·`StringBuilder`/`clone()` 무관"은 정의 뒤바꾸기 + 분류 오류 + 실무 예 부정 3중 함정.
- **암기 팁**: **"객체 조립 → Builder / 객체 복사 → Prototype"**. `Object.clone()`·Cloneable 인터페이스는 Prototype의 Java 언어 지원 예이며, 얕은 복사 함정으로 자주 시험 출제. Lombok `@Builder`는 Builder 패턴의 실무 대표.

### 문제 5
- **핵심**: Adapter = **인터페이스 불일치 해결(사후 통합)** / Bridge = **추상·구현 분리(사전 설계 · 두 축 독립 확장)**. **두 패턴 모두 구조 패턴이지만 목적·시점이 완전 다름**.
- **정답 ④ 함정**: 두 패턴의 정의·의도·시점을 정확히 뒤바꾼 뒤 "동일 패턴"으로 결론 짓는 3중 함정. 시험 최빈출 함정 중 하나.
- **암기 팁**: **"Adapter = 이미 있는 두 인터페이스 사이의 번역기(after) / Bridge = 처음부터 두 계층 분리 설계(before)"**. GoF 원서에서도 "Adapter는 인터페이스 부합을 위한 결과 위주, Bridge는 인터페이스 분리를 위한 설계 원칙 위주"로 명시. 도형×렌더링, OS×GUI 툴킷 같은 2축 확장 → Bridge.

### 문제 6
- **핵심**: Composite = **트리 구조 부분-전체 계층 · Leaf vs Composite 동일 인터페이스** / Decorator = **감싸기(wrapping)로 기능 동적 추가 · 원본과 동일 인터페이스 유지**.
- **정답 ④ 함정**: 두 패턴의 정의를 정확히 뒤바꾸고 대표 예까지 서로 스왑한 정면 함정.
- **암기 팁**: **"트리 = Composite / 감싸기 = Decorator"**. 파일 시스템(파일 vs 폴더) = Composite, Java I/O 스트림(`BufferedReader(new FileReader(...))`) = Decorator. 두 패턴 모두 "컴포넌트 인터페이스를 재귀적으로 합성"한다는 구조적 유사성이 있어 헷갈리기 쉬움. Decorator는 **행동을 추가**, Composite는 **자식들을 관리**.

### 문제 7
- **핵심**: Facade = **복잡 서브시스템의 단순화된 진입점 제공 · 인터페이스가 다름** / Proxy = **원본과 동일한 인터페이스로 부가 기능(접근 제어·지연·원격·캐시) 제공**.
- **정답 ④ 함정**: 두 패턴의 목적·인터페이스 특성·대표 예를 정확히 뒤바꾼 함정.
- **암기 팁**: **"진입점 단순화 = Facade / 대리자 부가 기능 = Proxy"**. Spring `JdbcTemplate` = Facade(JDBC의 커넥션·PreparedStatement·ResultSet·자원 해제를 하나의 진입점으로 단순화), Hibernate 지연 로딩·Spring AOP·JAX-WS/RMI 스텁 = Proxy. Proxy 4대 유형(Virtual · Remote · Protection · Smart)도 시험 빈출.

### 문제 8
- **핵심**: Flyweight = **다수 객체 메모리 절감 · 내재적(공유·불변) + 외재적(문맥 의존·매 호출 전달) 상태 분리 · Factory가 인스턴스 풀 관리**.
- **정답 ④ 함정**: "공유 금지·상태 구분 없음·`String.intern()`/`Integer.valueOf()` 무관"은 Flyweight 정의 3요소를 정면으로 부정한 함정.
- **실무·시험 포인트**: Java `String.intern()`(문자열 리터럴 풀), `Integer.valueOf()`의 **-128~127 캐싱**(`==` 비교 시 캐시 범위 안이면 `true` 반환하는 실무 함정)은 Flyweight의 대표적 언어 지원 예로 시험에 자주 등장. 워드 프로세서의 Glyph 예도 GoF 원서 원안 사례.

### 문제 9
- **핵심**: Observer = **1:N 자동 통보 · Subject/Observer 인터페이스 · Publish-Subscribe 원형 · MVC Model↔View 동기화**.
- **정답 ④ 함정**: "1:1만 적용·상태 변화 무반응·MVC/Reactive Streams 무관"은 Observer의 정의 3요소를 정면 부정한 함정.
- **시험 포인트**: **Java `Observer`/`Observable`는 Deprecated(Java 9+)**·현재는 `PropertyChangeListener` · `Flow.Publisher/Subscriber`(Java 9 Reactive Streams) · Spring `ApplicationEvent`·Vue/React 반응성 시스템의 개념적 기반. Push 방식(주체가 데이터 전달) vs Pull 방식(관찰자가 조회) 구분도 자주 물음.

### 문제 10
- **핵심**: Strategy = **알고리즘 교체(같은 문제 다른 방법) · 클라이언트가 명시적 선택** / State = **상태별 행동 변화 · 상태 전이는 컨텍스트/상태 객체 내부 자율**.
- **정답 ④ 함정**: 두 패턴의 정의·의도·대표 예를 정확히 뒤바꾼 함정. UML 구조는 유사하지만 **의도(intent) 축이 달라 별개 패턴**.
- **암기 팁**: **"알고리즘 선택 = Strategy / 상태 전이 = State"**. TCP 연결 상태 관리·주문 상태 관리 = State, 정렬·결제·압축 알고리즘 교체 = Strategy. GoF 원서에도 "Strategy는 서로 대체 가능한 알고리즘 캡슐화, State는 객체의 상태 변화에 따른 행위 변경"이라고 의도를 명확히 구분.

### 문제 11
- **핵심**: Observer = **1:N 자동 통보(주체→관찰자)** / Mediator = **N:N 상호작용을 중앙 중재자로 캡슐화 · 객체 간 직접 참조 제거**.
- **정답 ④ 함정**: 두 패턴의 방향성·초점·대표 예를 정확히 스왑한 함정.
- **암기 팁**: **"주체→관찰자 1:N = Observer / 여러 객체 ↔ 중재자 N:N = Mediator"**. Publish-Subscribe = Observer 원형, GUI 다이얼로그의 폼 컨트롤 조율·항공 관제·챗룸 서버 = Mediator. Mediator는 각 컴포넌트가 서로를 모르고 중재자만 알며, 중재자에게 이벤트를 통보하고 중재자가 나머지 컴포넌트를 조율하는 구조.

### 문제 12
- **핵심**: Template Method = **알고리즘 골격 상위 클래스 정의 · 하위가 훅 오버라이드 · 상속 기반** / Command = **요청을 객체로 캡슐화 · 큐잉·로깅·undo/redo 지원**.
- **정답 ④ 함정**: 두 패턴의 정의·기반(상속 vs 캡슐화)·대표 예를 정확히 뒤바꾼 함정.
- **암기 팁**: **"뼈대 상속 = Template Method / 요청 객체 캡슐화 = Command"**. Spring `JdbcTemplate`·JUnit `TestCase` = Template Method(콜백 훅 채워넣기), GUI 메뉴 undo/redo·매크로 레코더·트랜잭션 로그·작업 큐 = Command. Strategy는 **위임 기반**(합성)이고 Template Method는 **상속 기반**이라는 축도 자주 시험 출제.

### 문제 13
- **핵심**: Iterator = **집합체 순회 방법 제공** / Visitor = **요소 구조와 연산 분리 · 이중 디스패치** / Chain of Responsibility = **요청을 처리할 핸들러 사슬**.
- **정답 ④ 함정**: 세 패턴의 정의와 대표 예를 완전히 뒤섞은 3중 뒤바꾸기 함정.
- **실무·시험 포인트**: **Java `Iterator`/Python `__iter__` = Iterator**, **컴파일러 AST 순회 = Visitor**, **Servlet Filter/Spring Security `FilterChainProxy` = Chain of Responsibility**. Visitor의 이중 디스패치(방문자·요소 두 방향 오버로드로 실제 실행 메서드 결정)는 정보처리기사 필기에 종종 등장. CoR는 각 핸들러가 처리 여부를 자율 결정(전달 or 소비)한다는 점이 핵심.

### 문제 14
- **핵심**: Memento = **객체 상태 캡슐화 저장·복원 · undo 스택 · Originator/Memento/Caretaker 3역할** / Interpreter = **문법을 AST 노드로 표현 · 재귀 해석 · SQL/정규식/DSL 처리**.
- **정답 ④ 함정**: 두 패턴의 정의·대표 예를 정확히 뒤바꾼 함정.
- **암기 팁**: **"스냅샷 저장·복원 = Memento / 문법 해석·평가 = Interpreter"**. 텍스트 에디터 undo·게임 세이브·트랜잭션 롤백 = Memento, SQL 표현식·정규 표현식·수식 계산기·설정 DSL = Interpreter. Interpreter는 복잡한 언어에는 부적합해 실무에서는 ANTLR·Yacc·Bison 같은 파서 제너레이터로 대체하는 것이 일반적이라는 트레이드오프도 시험 포인트.

### 문제 15
- **핵심**: 서브시스템 단순화 진입점 = **Facade 패턴**(구조). Adapter/Bridge와 구분해야 함.
- **정답 ④ 함정**: "서브시스템 단순화 통합 진입점 → Adapter/Bridge"는 완전 오답. 정답은 **Facade**. Adapter는 **인터페이스 불일치 사후 해결**, Bridge는 **추상·구현 사전 분리**이며, 두 패턴 모두 "복잡 서브시스템 진입점 단순화"의 의도를 가지지 않는다. ①(State)·②(Strategy)·③(Observer) 매칭은 모두 정확.
- **암기 팁**: 시나리오에서 **"세부 API를 감추고 단순 진입점만 제공" 키워드 → Facade** 즉시 매칭. Spring의 `JdbcTemplate`, 결제 서비스의 통합 SDK, 미디어 재생 라이브러리의 단일 진입점 API가 대표 예. Adapter는 "인터페이스 A와 B가 안 맞아 연결", Bridge는 "두 축을 독립 확장 설계"라는 축이 별개.

| # | 트랩 축 | 정답 방향 | 오답 함정 (완전 정반대) |
|:---:|:---|:---|:---|
| 1 | **Adapter vs Bridge** | Adapter = **인터페이스 불일치 사후 해결** / Bridge = **추상·구현 사전 분리(두 축 독립 확장)** | "Adapter가 두 축 분리·Bridge가 인터페이스 변환·완전 동일 패턴" 뒤바꾸기 |
| 2 | **Composite vs Decorator** | Composite = **트리 부분-전체 계층(파일/폴더)** / Decorator = **감싸기 기능 확장(Java I/O)** | "Composite가 감싸기·Decorator가 트리·Java I/O=Composite·파일/폴더=Decorator" 뒤바꾸기 |
| 3 | **Facade vs Proxy** | Facade = **복잡 서브시스템 단순 진입점(인터페이스 다름)** / Proxy = **동일 인터페이스 + 부가 기능(지연/원격/보호)** | "Facade가 부가 기능·Proxy가 진입점 단순화·`JdbcTemplate`=Proxy·Spring AOP=Facade" 뒤바꾸기 |
| 4 | **Strategy vs State** | Strategy = **알고리즘 교체(클라이언트 명시 선택)** / State = **상태 전이(내부 자율)** | "Strategy가 상태 전이·State가 알고리즘 교체·TCP=Strategy·결제=State" 뒤바꾸기 |
| 5 | **Template Method vs Command** | Template Method = **뼈대 상속(하위 훅 채움 · `JdbcTemplate`)** / Command = **요청 객체 캡슐화(undo/redo · 매크로)** | "Template Method가 요청 캡슐화·Command가 뼈대 상속·`JdbcTemplate`=Command·undo=Template Method" 뒤바꾸기 |
