### 문제 1 [출처: 2020년 3회 필기 빈출]
트랜잭션의 특성(ACID)에 대한 설명으로 옳지 않은 것은?
① 원자성(Atomicity): 트랜잭션 연산은 모두 반영되거나 전혀 반영되지 않아야 한다.
② 일관성(Consistency): 트랜잭션이 성공적으로 완료되면 데이터베이스는 일관된 상태를 유지해야 한다.
③ 격리성(Isolation): 둘 이상의 트랜잭션이 동시에 실행될 때 한 트랜잭션의 중간 결과를 다른 트랜잭션이 참조할 수 있다.
④ 지속성(Durability): 성공적으로 완료된 트랜잭션의 결과는 시스템에 영구히 반영되어야 한다.

## 🔍 오답 역추적 개념 발췌

오늘 문제에서 틀릴 가능성이 높은 키워드만 발췌 — 의미는 직접 떠올려 메우세요.
- ACID: 원자성(Atomicity) / 일관성(Consistency) / 격리성(Isolation) / 지속성(Durability)
- 격리성의 정확한 정의: 다른 트랜잭션의 "중간 결과" 참조 ❌
- 트랜잭션 상태 5단계: 활동(Active) → 부분 완료(Partially Committed) → 완료(Committed) / 실패(Failed) → 철회(Aborted)
- 부분 완료 ↔ 완료 차이 (DB에 반영되었나?)
- 로킹 단위 큼 → 로크 수 ↓, 오버헤드 ↓, 병행성 ↓, 데드락 가능성 ↓
- 로킹 단위 작음 → 로크 수 ↑, 오버헤드 ↑, 병행성 ↑, 데드락 가능성 ↑
- 회복 기법: 로그 기반(즉시 갱신·지연 갱신), 그림자 페이지, 검사점(Checkpoint), ARIES
- 동시성 제어 기법: 로킹, 타임스탬프 순서(Timestamp Ordering), 낙관적(Optimistic·검증), 다중버전(MVCC)
- 검사점(Checkpoint) = 회복 기법 (병행 제어 ❌)
- 타임스탬프 = 병행 제어 (회복 ❌)
- 즉시 갱신: COMMIT 전에도 DB 반영 → 장애 시 UNDO + REDO 모두 필요할 수 있음
- 지연 갱신: COMMIT 후에만 DB 반영 → UNDO 불필요, REDO만 필요
- 그림자 페이지: 로그·UNDO·REDO 불필요 (페이지 테이블 복사본 유지)
- 격리수준 4단계: READ UNCOMMITTED < READ COMMITTED < REPEATABLE READ < SERIALIZABLE
- 발생 현상 3종: Dirty Read(오손 읽기) / Non-Repeatable Read(반복 불가 읽기) / Phantom Read(유령 읽기)
- 격리수준 높을수록 → 일관성 ↑, 동시성 ↓
- 2PL(2단계 로킹): 확장 단계(Lock 획득만) → 축소 단계(Lock 해제만)
- 2PL은 직렬 가능성(Serializability) 보장 ⭕ / 데드락 방지 ❌
- 직렬 가능성: 병행 스케줄이 어떤 직렬 스케줄과 동일한 결과를 내면 직렬 가능
- 데드락 4조건: 상호 배제 / 점유와 대기 / 비선점 / 환형 대기

## 💡 상세 해설 (자기 설명과 대조)

### 문제 1 — 정답 ③ 격리성 정의 오류
- 격리성(Isolation)은 "둘 이상의 트랜잭션이 동시에 수행되더라도 **다른 트랜잭션의 중간(부분 완료) 결과를 참조할 수 없다**"는 특성.
- ③은 "참조할 수 있다"로 정반대 진술.
- ①②④ 모두 ACID 정의 정확.

### 문제 2 — 정답 ② 부분 완료(Partially Committed)
- 부분 완료: 마지막 연산까지 메모리에서는 수행 완료, 그러나 DB(저장소)에 아직 영속화 ❌.
- 활동(Active): 트랜잭션 실행 중.
- 완료(Committed): 부분 완료 이후, 데이터베이스에 변경 사항이 영속화된 상태.
- 실패(Failed): 정상 실행 불가 상태 → 철회(Aborted)로 전이.

### 문제 3 — 정답 ④ "로킹 단위가 작으면 병행성이 낮아진다"
- 로킹 단위 작음 → 락을 잡는 영역이 좁음 → 다른 트랜잭션의 진입이 쉬움 → **병행성 ↑**.
- ④는 반대로 진술되어 오답.
- ①②③ 모두 옳음 (작은 단위 = 많은 락 = 오버헤드 ↑ / 큰 단위 = 적은 락 = 오버헤드 ↓·병행성 ↓).

### 문제 4 — 정답 ④ 타임스탬프 순서(Timestamp Ordering)
- 타임스탬프 순서: **동시성 제어(병행 제어) 기법** — 트랜잭션마다 타임스탬프를 부여해 충돌 시 순서대로 처리.
- ①②③: 회복(Recovery) 기법.
  - 즉시 갱신·지연 갱신: 로그 기반 회복.
  - 그림자 페이지: 페이지 테이블 사본을 유지하여 회복.

### 문제 5 — 정답 ④ 체크 포인트(Check Point)
- 체크 포인트: **회복(Recovery) 기법** — 일정 시점의 DB 상태를 안전한 저장소에 기록, 장애 시 가장 가까운 검사점부터 복구.
- ①②③: 동시성 제어 기법.
  - 로킹: 데이터 항목에 락을 걸어 동시 접근 제어.
  - 타임스탬프: 시간 순서로 직렬화.
  - 낙관적(Optimistic·검증): 충돌이 드물다 가정, 트랜잭션 종료 시 검증.

### 문제 6 — 정답 ③ "UNDO + REDO 모두 필요할 수 있다"
- 즉시 갱신: 트랜잭션이 부분 완료된 후 COMMIT 전이라도 변경이 DB에 반영될 수 있음.
- 따라서 장애 발생 시 두 가지 모두 필요:
  - **UNDO**: COMMIT 전에 반영되었으나 트랜잭션이 실패한 경우 → 이전 상태로 되돌림.
  - **REDO**: COMMIT 후 장애 발생한 경우 → 변경을 재반영.
- ① 지연 갱신(Deferred Update)의 설명.
- ② 지연 갱신은 REDO만 필요. 즉시 갱신은 UNDO도 필요.
- ④ 검사점은 회복 효율을 위한 보조 기법으로, 즉시 갱신과 함께 자주 사용됨.

### 문제 7 — 정답 ① READ UNCOMMITTED < READ COMMITTED < REPEATABLE READ < SERIALIZABLE
- 격리수준이 높을수록 일관성 보장 ↑, 동시성 ↓.
- 발생 현상 차단 범위:
  - READ UNCOMMITTED: 아무것도 차단 ❌ (Dirty/Non-Repeatable/Phantom 모두 발생)
  - READ COMMITTED: Dirty Read만 차단
  - REPEATABLE READ: Dirty + Non-Repeatable 차단 (Phantom은 발생 가능)
  - SERIALIZABLE: 셋 모두 차단

### 문제 8 — 정답 ④ "교착 상태(Deadlock)가 절대 발생하지 않음을 보장한다"
- 2단계 로킹 프로토콜(2PL)은 **직렬 가능성(Serializability)은 보장하지만 데드락은 방지하지 못함**.
- 두 트랜잭션이 서로 다른 자원의 락을 점유한 채 상대 자원을 요청하면 데드락 발생 가능.
- ①②③: 2PL의 정의·성질 정확.

## 📚 역추적 참조 (사후 검증용 — 학습 자료 아님)

### 핵심 용어
| 용어 | 정의 |
|---|---|
| 트랜잭션(Transaction) | 데이터베이스의 상태를 변환시키는 논리적 작업 단위, 또는 한꺼번에 모두 수행되어야 할 일련의 연산 |
| ACID | 트랜잭션이 가져야 할 4가지 특성 (Atomicity·Consistency·Isolation·Durability) |
| 부분 완료(Partially Committed) | 트랜잭션의 마지막 연산까지 수행, 그러나 DB에 영속화 전 상태 |
| 완료(Committed) | 모든 연산이 DB에 영속화된 상태 |
| 철회(Aborted) | 실패 후 트랜잭션의 모든 변경이 취소(rollback)된 상태 |
| 회복(Recovery) | 장애로 손상된 DB를 정상 상태로 복원하는 작업 |
| UNDO | 로그 기반 회복에서 이전 값(before image)으로 되돌리는 연산 |
| REDO | 로그 기반 회복에서 변경 값(after image)을 재반영하는 연산 |
| 검사점(Checkpoint) | 일정 시점의 DB 상태를 기록한 지점. 회복 시 시작점 단축 |
| 그림자 페이지(Shadow Paging) | 로그 없이 페이지 테이블 사본(shadow)을 유지하여 회복 |
| 로킹(Locking) | 데이터 항목에 락을 걸어 동시 접근을 제어하는 병행 제어 기법 |
| 로킹 단위(Granularity) | 한 번에 락을 거는 데이터 단위(전체 DB→테이블→페이지→행→속성) |
| 2단계 로킹(2PL) | 확장(Lock 획득) 단계와 축소(Lock 해제) 단계로 분리된 로킹 프로토콜 |
| 직렬 가능성(Serializability) | 병행 실행 스케줄이 어떤 직렬 스케줄과 동일한 결과를 내는 성질 |
| 타임스탬프 순서(Timestamp Ordering) | 트랜잭션에 타임스탬프를 부여해 순서대로 처리하는 병행 제어 기법 |
| 낙관적 제어(Optimistic) | 충돌이 드물다 가정, 트랜잭션 종료 시 검증(Validation) |
| MVCC | 다중 버전 동시성 제어. 데이터 항목의 여러 버전을 유지 |
| 격리수준(Isolation Level) | 동시 트랜잭션 간 상호 영향 범위를 정의한 4단계 |
| Dirty Read | 다른 트랜잭션의 미완료(uncommitted) 데이터를 읽음 |
| Non-Repeatable Read | 같은 행을 다시 읽었을 때 값이 변경되어 있음 |
| Phantom Read | 같은 조건으로 다시 조회했을 때 행이 추가/삭제되어 있음 |
| 데드락(Deadlock) | 서로 다른 자원의 락을 점유한 채 상대 자원을 기다리는 상태 |

### 핵심 개념 (사후 검증용)

**1) ACID**
- Atomicity: All or Nothing (UNDO/COMMIT/ROLLBACK으로 보장)
- Consistency: 트랜잭션 전후 무결성 제약 만족
- Isolation: 동시 실행 시 다른 트랜잭션의 중간 결과 참조 불가
- Durability: 완료된 트랜잭션 결과는 영구 반영 (로그/체크포인트로 보장)

**2) 트랜잭션 상태 전이**
- 활동(Active) → 부분 완료(Partially Committed) → 완료(Committed)
- 활동(Active) → 실패(Failed) → 철회(Aborted)
- 부분 완료(Partially Committed) → 실패(Failed) → 철회(Aborted)

**3) 회복 기법**
- 로그 기반:
  - 즉시 갱신(Immediate Update): COMMIT 전에도 DB 반영. UNDO+REDO 필요.
  - 지연 갱신(Deferred Update): COMMIT 후에만 DB 반영. REDO만 필요.
- 그림자 페이지(Shadow Paging): 로그 없이 페이지 테이블 사본. UNDO/REDO 불필요.
- 검사점(Checkpoint): 로그 기반과 결합. 가까운 체크포인트부터 회복.
- ARIES: 산업 표준 회복 알고리즘 (분석→REDO→UNDO 3단계).

**4) 동시성 제어 기법**
- 로킹(Locking): 공유 락(S-lock, Read) / 배타 락(X-lock, Write).
- 2PL: 확장 단계(Lock만 획득) → 축소 단계(Lock만 해제). 직렬가능성 보장 ⭕, 데드락 방지 ❌.
- 타임스탬프 순서(TO): 트랜잭션 타임스탬프 순서대로 처리.
- 낙관적(Optimistic / Validation): 읽기→검증→쓰기 3단계.
- 다중 버전(MVCC): 데이터 여러 버전 유지, 읽기는 락 없음.

**5) 로킹 단위 트레이드오프**
| 로킹 단위 | 락 수 | 오버헤드 | 병행성 | 데드락 |
|---|---|---|---|---|
| 큼 (DB·테이블) | 적음 | 낮음 | 낮음 | 낮음 |
| 작음 (행·속성) | 많음 | 높음 | 높음 | 높음 |

**6) 격리수준 vs 발생 현상**
| 격리수준 | Dirty Read | Non-Repeatable Read | Phantom Read |
|---|---|---|---|
| READ UNCOMMITTED | 발생 | 발생 | 발생 |
| READ COMMITTED | 차단 | 발생 | 발생 |
| REPEATABLE READ | 차단 | 차단 | 발생 |
| SERIALIZABLE | 차단 | 차단 | 차단 |

**7) 데드락**
- 발생 4조건: 상호 배제 / 점유와 대기 / 비선점 / 환형 대기
- 해결: 예방(Prevention) / 회피(Avoidance, 은행원 알고리즘) / 탐지(Detection, 자원 할당 그래프) / 복구(Recovery)
- 타임스탬프 기반 데드락 방지: Wait-Die, Wound-Wait
