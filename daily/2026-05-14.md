## 🔑 핵심 용어 (10개)

| 용어 | 한 줄 정의 | 핵심 키워드 |
|---|---|---|
| UML(Unified Modeling Language, 통합 모델링 언어) | OMG(Object Management Group)가 표준화한 객체지향 모델링 통합 언어 | 사물·관계·다이어그램 |
| 사물(Things) | UML 모델의 기본 구성 요소 | 구조·행동·그룹·주해 |
| 관계(Relationship) | 사물 간의 연결 | 연관·집합·포함·일반화·의존·실체화 |
| 다이어그램(Diagram) | 사물과 관계를 도식으로 표현한 모델 | 구조 7 + 행위 6 = 13종 |
| 스테레오타입(Stereotype) | UML 기본 표기 외에 의미를 확장하는 도구. **<<>>(guillemet, 길러멧)** 안에 표기 | <<include>>, <<extend>>, <<interface>>, <<entity>>, <<boundary>>, <<control>> |
| 객체(Object) | 속성(데이터)과 메서드(연산)를 캡슐화한 실체. 클래스의 인스턴스 | 속성·메서드·메시지 |
| 클래스(Class) | 공통된 속성과 연산을 가진 객체의 집합/틀(template) | 인스턴스화 |
| 캡슐화(Encapsulation) | 속성과 메서드를 하나로 묶고 외부에는 인터페이스만 공개 | 정보 은닉·재사용성·결합도↓ |
| 상속(Inheritance) | 상위 클래스의 속성·연산을 하위 클래스가 물려받음 | is-a, 재사용 |
| 다형성(Polymorphism) | 같은 메시지에 객체마다 다르게 응답 | 오버로딩·오버라이딩 |

## 📖 핵심 개념 정리

### 1. UML의 3대 구성 요소 (★출제 100%★)

**사물(Things) → 관계(Relationship) → 다이어그램(Diagram)** 순서로 구성된다.

**(1) 사물 4종**
| 종류 | 설명 | 예 |
|---|---|---|
| 구조 사물(Structural) | 시스템의 정적 부분 | 클래스, 인터페이스, 유스케이스, 컴포넌트, 노드 |
| 행동 사물(Behavioral) | 시간·공간에 따른 동적 부분 | 상호작용(Interaction), 상태 머신(State Machine) |
| 그룹 사물(Grouping) | 요소들을 묶는 단위 | 패키지(Package) |
| 주해 사물(Annotation) | 부가 설명 | 노트(Note) |

암기법: **"구·행·그·주"**

**(2) 관계 6종 (★★★최빈출★★★)**

| 관계 | 한글 | 표기 | 의미 (방향: A → B 또는 A-B) |
|---|---|---|---|
| Association | 연관 | ─── (실선) | "사용한다". 화살표로 방향 표시, 양방향이면 생략 |
| Aggregation | 집합/집약 | ◇───(빈 마름모) | "전체-부분", 부분이 독립적으로 존재 가능 (예: 학과-학생) |
| Composition | 포함/합성 | ◆───(채운 마름모) | "전체-부분", 부분이 전체에 종속 (예: 사람-심장) |
| Generalization | 일반화 | ──▷(빈 삼각형) | "is-a, 상속". 자식→부모 방향 |
| Dependency | 의존 | --→(점선 화살표) | "일시적 사용". 매개변수·지역변수 |
| Realization | 실체화 | --▷(점선+빈 삼각형) | "인터페이스 구현, behavior 실체화" |

암기법: **"연-집-포-일-의-실"**

**(3) 다이어그램 13종 — 구조 7 + 행위 6**

🏛 **구조 다이어그램(정적, Static, 7종) — "클객컴배복패합"**
1. **클래스(Class)** 다이어그램 — 시스템 정적 구조의 대표
2. **객체(Object)** 다이어그램 — 특정 시점의 객체 인스턴스
3. **컴포넌트(Component)** 다이어그램 — 물리적 컴포넌트 구성
4. **배치(Deployment)** 다이어그램 — 노드와 컴포넌트 물리 배치
5. **복합체 구조(Composite Structure)** 다이어그램 — 클래스 내부 구조
6. **패키지(Package)** 다이어그램 — 패키지 간 의존
7. **합성(Composite, 신규)** — 일부 교재만 별도 분류

🎬 **행위 다이어그램(동적, Behavioral, 6종) — "유시커상활타"**
1. **유스케이스(Use Case)** — 시스템 기능을 사용자 관점에서
2. **시퀀스(Sequence)** — 시간 흐름에 따른 객체 간 메시지 (Lifeline+활성막대)
3. **커뮤니케이션(Communication)** — 시퀀스와 동일 정보, 객체 관계 중심
4. **상태(State)** — 객체의 상태 전이 (★럼바우 동적 모델링★)
5. **활동(Activity)** — 처리 흐름·분기(swim lane 가능)
6. **타이밍(Timing)** — 시간 제약 표현

> 💡 핵심 함정: **활동(Activity)·시퀀스(Sequence)는 행위 다이어그램**. 클래스·객체·컴포넌트·배치·패키지는 모두 **구조**.

### 2. UML 스테레오타입 (★출제 빈도 ↑)
- 표기: **<< >>** (guillemet, 길러멧/겹화살괄호)
- 종류:
  - `<<include>>` — 유스케이스 포함(필수)
  - `<<extend>>` — 유스케이스 확장(선택)
  - `<<interface>>` — 인터페이스
  - `<<entity>>` — 정보 보관 클래스
  - `<<boundary>>` — 외부와 상호작용 클래스
  - `<<control>>` — 흐름 제어 클래스
  - `<<import>>` — 패키지 import

### 3. 객체지향(Object-Oriented) 5대 특성

| 특성 | 영문 | 핵심 |
|---|---|---|
| 캡슐화 | Encapsulation | 속성+메서드를 묶고 외부 차단. **결합도↓ 응집도↑ 재사용·수정용이** |
| 상속 | Inheritance | 상위 → 하위로 속성·연산 물려받음. "is-a" |
| 다형성 | Polymorphism | 동일 메시지, 다른 응답 (오버로딩·오버라이딩) |
| 추상화 | Abstraction | 공통 속성 추출, 세부 구현 생략 |
| 정보 은닉 | Information Hiding | private 접근자로 내부 감춤 (캡슐화의 결과) |

암기법: **"캡상다추정"**

### 4. 객체지향 분석(OOA, Object-Oriented Analysis) 방법론 (★★★최빈출★★★)

| 방법론 | 제창자 | 특징 / 키워드 |
|---|---|---|
| **럼바우(Rumbaugh) OMT** | James Rumbaugh | **객체(Object) → 동적(Dynamic) → 기능(Functional)** 모델링 3단계 |
| **부치(Booch)** | Grady Booch | **미시적(Micro)·거시적(Macro)** 프로세스. 클래스·객체 분석 |
| **야콥슨(Jacobson)** | Ivar Jacobson | **유스케이스(Use Case) 중심**. OOSE |
| **코드-요든(Coad-Yourdon)** | Coad, Yourdon | **E-R 다이어그램** 사용, 데이터 모델링 중심 |
| **Wirfs-Brock** | Wirfs-Brock | 분석·설계 구분 없음. **책임 기반**(CRC 카드) |

**럼바우 OMT 3단계 (★최빈출★)**
- **객체 모델링** = **객체 다이어그램** (= 클래스 다이어그램)
- **동적 모델링** = **상태 다이어그램** ← 시간 흐름·상호작용·제어
- **기능 모델링** = **자료 흐름도(DFD)** ← 데이터 입력·출력·변환

암기법: **"럼·객·동·기"** / **"럼바우는 객동기로 분석"**

### 5. 객체지향 설계 5원칙 — SOLID (★최빈출★)

| 약자 | 원칙 | 한 줄 정의 | 핵심 키워드 |
|---|---|---|---|
| **S**RP | Single Responsibility Principle (단일 책임) | 클래스는 **단 하나의 책임/변경 이유**만 가져야 한다 | "하나의 책임" |
| **O**CP | Open-Closed Principle (개방-폐쇄) | **확장에는 열려 있고, 수정에는 닫혀** 있어야 한다 | "확장 O, 수정 X" |
| **L**SP | Liskov Substitution Principle (리스코프 치환) | **자식 클래스는 부모 클래스를 대체** 가능해야 한다 | "is-substitutable-for" |
| **I**SP | Interface Segregation Principle (인터페이스 분리) | 클라이언트는 **사용하지 않는 인터페이스에 의존하지 않아야** | "특화된 다수의 인터페이스" |
| **D**IP | Dependency Inversion Principle (의존 역전) | **변화하기 쉬운 것이 아닌 추상**(인터페이스)에 의존 | "추상에 의존" |

**자주 나오는 함정 키워드:**
- "**서브타입은 어디서나 기반타입으로 교체 가능**" → **LSP**
- "**자주 변화하는 것보다 변화하기 어려운 것에 의존**" → **DIP**
- "**기존 코드 수정 없이 기능 추가**" → **OCP**
- "**클라이언트별 인터페이스 분리**" → **ISP**
- "**클래스는 하나의 목적**" → **SRP**

### 6. 객체지향 구성요소

| 구성요소 | 정의 |
|---|---|
| 클래스(Class) | 동일 속성·연산을 가진 객체의 집합 (붕어빵 틀) |
| 객체(Object) | 클래스의 인스턴스. 속성+메서드 묶음 |
| 인스턴스(Instance) | 클래스로부터 생성된 구체적 객체 |
| 메서드(Method) | 객체에 속한 함수/연산 |
| 메시지(Message) | 객체 간 통신 수단. 메서드 호출 |
| 속성(Attribute) | 객체의 상태/데이터 |

## 💡 상세 해설

### 문제 1 해설 — 정답 ④
럼바우 OMT는 **객체-동적-기능 3가지 모델링**만 사용한다. "정적 모델링"이라는 용어는 럼바우 분류에 없으며, 시퀀스 다이어그램은 **동적**이지 정적이 아니다. 또한 동적 모델링은 **상태 다이어그램**을 사용한다.

### 문제 2 해설 — 정답 ②
**기능 모델링(Functional Modeling) = 자료 흐름도(DFD)**. "어떤 데이터가 들어가서 어떤 결과가 나오는가"를 표현. 객체 모델링은 객체 다이어그램, 동적 모델링은 상태 다이어그램을 사용.

### 문제 3 해설 — 정답 ①
럼바우 OMT 절차는 **객체 → 동적 → 기능** 순. 암기법 "**객·동·기**". 분석 대상의 **구조(객체)** 를 먼저 파악한 뒤, **시간 흐름(동적)** 을 보고, 마지막으로 **자료 변환(기능)** 을 정의.

### 문제 4 해설 — 정답 ④
럼바우 분석 기법 = **객체 + 동적 + 기능** 3가지뿐. "블랙박스 분석"은 럼바우 분류에 없다(블랙박스는 테스트 기법 용어).

### 문제 5 해설 — 정답 ③
**LSP(Liskov Substitution Principle, 리스코프 치환 원칙)** — Barbara Liskov 제안. "**서브타입은 그것의 기반 타입으로 치환 가능해야 한다**"는 정의. 부모 자리에 자식을 넣어도 프로그램이 정상 동작해야 함.

### 문제 6 해설 — 정답 ④
**Prototype**은 **생성(Creational) 패턴**(객체 복제). 행위 패턴은 책임·알고리즘에 관여 (Observer/Strategy/Iterator/Command/State/Visitor/Template Method/Chain of Responsibility/Mediator/Memento/Interpreter — 11개).

### 문제 7 해설 — 정답 ③
**Adapter**는 **구조(Structural) 패턴**. 생성 패턴 5개: Singleton, Factory Method, Abstract Factory, Builder, Prototype.

### 문제 8 해설 — 정답 ④
GoF는 **생성 / 구조 / 행위** 3가지로만 분류. "추상 패턴"이라는 분류는 없다.

### 문제 9 해설 — 정답 ②
스테레오타입은 **`<< >>` (guillemet, 길러멧/겹화살괄호)** 안에 표기. 예: `<<include>>`, `<<extend>>`, `<<interface>>`.

### 문제 10 해설 — 정답 ④
**활동(Activity) 다이어그램**은 **행위 다이어그램**. 구조 다이어그램 7종 = 클래스/객체/컴포넌트/배치/복합체구조/패키지/합성.

### 문제 11 해설 — 정답 ③
**시퀀스 다이어그램**이 행위 다이어그램. 패키지·배치·복합체 구조는 모두 구조 다이어그램.

### 문제 12 해설 — 정답 ③
"**더 일반적/더 구체적**" = **일반화(Generalization)** = 상속 관계, "is-a-kind-of". 표기는 빈 삼각형 화살표(▷). 자식 → 부모 방향.

### 문제 13 해설 — 정답 ④
**의존(Dependency)**: 일시적으로 다른 클래스를 사용 (매개변수, 지역변수, 리턴타입). 점선 화살표(--→). 연관은 필드로 보유하는 영속적 관계.

### 문제 14 해설 — 정답 ④
UML 사물 분류 = **구조(Structural) / 행동(Behavioral) / 그룹(Grouping) / 주해(Annotation)**. "의존 사물"은 없으며 의존은 **관계**의 한 종류.

### 문제 15 해설 — 정답 ③
**ISP(Interface Segregation Principle)** — "**클라이언트는 자신이 사용하지 않는 인터페이스에 의존해서는 안 된다**". 하나의 큰 인터페이스보다 여러 작은 인터페이스로 분리.

### 문제 16 해설 — 정답 ②
**OCP(Open-Closed Principle)** — "**확장에는 열려 있고, 변경에는 닫혀 있어야 한다**". 추상화·인터페이스를 통해 새 기능 추가 시 기존 코드는 수정하지 않음.

### 문제 17 해설 — 정답 ③
**다형성(Polymorphism)** — "하나의 메시지(메서드 이름)에 대해 각 객체가 자신만의 방법으로 응답하는 능력". 오버로딩(같은 이름·다른 매개변수)·오버라이딩(상속받은 메서드 재정의)으로 구현.

### 문제 18 해설 — 정답 ④
**메시지(Message)**: 객체 간 통신 수단 = **메서드 호출**. 클래스의 데이터는 **속성(Attribute)** 이다.

### 문제 19 해설 — 정답 ③
**야콥슨(Jacobson) — OOSE(Object-Oriented Software Engineering)** 는 **유스케이스(Use Case)** 를 시작점으로 분석. 사용자 시나리오 → 요구사항 → 시스템 설계.

### 문제 20 해설 — 정답 ②
**부치(Booch) 방법** — 미시적(Micro)·거시적(Macro) 두 가지 개발 프로세스를 동시에 사용. 클래스·객체 분석 → 클래스·객체의 의미 식별 → 관계 식별 → 명세화.

### 문제 21 해설 — 정답 ④
**코드-요든(Coad-Yourdon)** — **E-R 다이어그램**으로 객체 활동을 데이터 모델링. "객체식별 → 구조식별 → 주제정의 → 속성·인스턴스 정의 → 연산·메시지 정의" 5단계.

### 문제 22 해설 — 정답 ④
유스케이스 관계는 **포함(Include) / 확장(Extend) / 일반화(Generalization)** 3종. **합성(Composition)** 은 **클래스 다이어그램**의 관계.

### 문제 23 해설 — 정답 ③
**캡슐화(Encapsulation)** — 데이터(속성)와 행위(연산)를 하나로 묶고, 외부에는 인터페이스만 공개해 내부 정보를 은닉. **결합도↓, 응집도↑, 재사용성↑, 변경 용이성↑**.

### 문제 24 해설 — 정답 ②
**시퀀스(Sequence) 다이어그램** — 객체들이 **시간 순서**에 따라 주고받는 **메시지**를 표현. 세로축이 시간, 가로축이 객체. Lifeline(점선)과 활성막대(Activation bar)로 표기.

## 📌 시험 직전 1분 요약

1. **UML 13종** = 구조 7(클·객·컴·배·복·패·합) + 행위 6(유·시·커·상·활·타). **활동·시퀀스는 행위**.
2. **UML 관계 6종** = **연·집·포·일·의·실**. 일반화 ▷(빈 삼각형, 상속), 실체화 --▷(점선+빈 삼각형, 인터페이스 구현), 의존 --→(점선).
3. **럼바우 OMT** = **객(객체다이어그램) → 동(상태다이어그램) → 기(DFD)**. "정적 모델링"은 럼바우 분류에 없음.
4. **SOLID**: S(단일책임) / O(확장O 수정X) / L(자식으로 부모 치환) / I(인터페이스 분리) / D(추상에 의존).
5. **객체지향 5특성**: 캡슐화·상속·다형성·추상화·정보은닉. **스테레오타입은 `<<>>`(길러멧)**.
