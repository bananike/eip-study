### 문제 13 [유사 기출: 2023년 1회 4과목 — Java 오버로딩(Overloading) 정형 패턴]

```java
class Calc {
    int add(int a, int b) { return a + b; }
    // ①
    double add(int a, int b) { return a + b + 0.5; }
    // ②
    int add(int a, int b, int c) { return a + b + c; }
    // ③
    int add(double a, double b) { return (int)(a + b); }
    // ④
    int add(int b, int a) { return b - a; }
}
```

① ①번 — 반환 타입만 다르고 매개변수 시그니처(int, int)가 완전히 같아 오버로딩 실패
② ②번 — 매개변수 개수가 다르므로 오버로딩 성공
③ ③번 — 매개변수 타입이 다르므로 오버로딩 성공
④ ④번 — 매개변수 이름만 다르지만 타입 시그니처가 같아 오버로딩 성공

### 문제 2
**정답 ② 집합(Aggregation) — 속이 빈 마름모(◇)**. UML 클래스 관계 5종 = **① 연관(Association)** = 단순 실선(개념적 연결) / **② 집합(Aggregation)** = **속이 빈 마름모(◇)** = "**전체가 없어도 부분은 독립적으로 존재**"(예: 회사 ◇— 직원) / **③ 합성(Composition)** = **속이 찬 마름모(◆)** = "**전체가 사라지면 부분도 함께 사라짐**"(예: 집 ◆— 방) / **④ 일반화(Generalization)** = 속이 빈 삼각형(△) = "**is-a 상속**" / **⑤ 실체화(Realization)** = 점선 + 속이 빈 삼각형 = **인터페이스 구현**.
- 함정: **Aggregation(◇) vs Composition(◆) 마름모 색깔 뒤바꾸기 초빈출**. "부분 독립 존재 = ◇ 빈 마름모" / "부분 종속·수명 동일 = ◆ 찬 마름모". 삼각형(△)은 상속/실체화용이지 부분-전체 관계 아님.

### 문제 5
**정답 ③ D E B F C A**. **후위 순회(Post-order) = LRN(Left → Right → Node)** = 왼쪽 서브트리 → 오른쪽 서브트리 → 루트. 트리 구조:
- A(루트): 왼쪽 B, 오른쪽 C
- B의 왼쪽 D, 오른쪽 E
- C의 오른쪽 F

후위 순회 진행: A의 왼쪽 서브트리(B)부터 → B의 왼쪽(D) → B의 오른쪽(E) → B → A의 오른쪽 서브트리(C) → C의 왼쪽(없음) → C의 오른쪽(F) → C → A. 결과 = **D → E → B → F → C → A**.
- 함정: **전위(Pre-order NLR) = A B D E C F** / **중위(In-order LNR) = D B E A C F** / **후위(Post-order LRN) = D E B F C A**. 세 순회 결과 뒤바꾸기 초빈출. **"루트가 어디로 가느냐"** = 전위(맨 앞) / 중위(가운데) / **후위(맨 뒤)** 로 암기.

### 문제 6
**정답 ② 25 → 55**. 배열 `A = [3, 7, 11, 19, 25, 32, 41, 55, 64, 78]`, 길이 10, 인덱스 0~9. 타깃 = **55**.
- **1회**: low=0, high=9, **mid=(0+9)/2=4** → A[4]=**25**. **25 < 55** 이므로 **오른쪽 반**으로 이동 → low = mid+1 = 5.
- **2회**: low=5, high=9, **mid=(5+9)/2=7** → A[7]=**55**. 타깃과 일치 → **탐색 종료**.
- **비교 순서 = A[4]=25 → A[7]=55** = **2회**만에 성공.
- 함정: ① = 첫 시도 성공(mid=4의 A[4]=25이지 55 아님) / ③ = 41을 잘못 방문 삽입 / ④ = mid 계산 오류(A[5]=32는 첫 mid 아님). **이진 탐색 최악 시간복잡도 O(log n)**, n=10 → **최대 4회**. 순차 탐색 O(n)과 뚜렷하게 구분. "이진 탐색이 순차 방문"이라는 정반대 함정도 초빈출.

### 문제 8

### 문제 9
**정답 ④ Union(∪)**. 관계대수 연산자는 **순수(관계) 4종 + 집합 4종** 으로 나뉜다.
- **순수 관계 연산 4종**: ① **Select(σ, 시그마)** = 조건에 맞는 **행(튜플) 추출**·수평 분할 / ② **Project(π, 파이)** = 특정 **열(속성) 추출**·수직 분할 / ③ **Join(⋈, 나비넥타이)** = 두 릴레이션 결합 / ④ **Division(÷)** = "모든 X에 대해" 나눗셈

### 문제 10
**정답 ② S-S: ○, S-X: ×, X-X: ×**. **락 호환성 매트릭스**:

|  | S-Lock | X-Lock |
|---|---|---|
| **S-Lock** | ○ (호환) | × (대기) |
| **X-Lock** | × (대기) | × (대기) |

**공유(S) 락은 여러 트랜잭션이 동시 획득 가능**(읽기 전용). **배타(X) 락은 단독**(쓰기·수정)이며, S든 X든 다른 락과 공존 불가.
- 함정: **S-S만 호환(○)**, 나머지는 모두 대기(×). ① 모두 대기(과잉 보수) · ③ S와 X 병존 허용(무결성 파괴) · ④ 모두 허용(락 무의미) 은 정반대. **락 프로토콜의 목적 = 일관성 보장**을 근본으로 매트릭스 유일.

### 문제 11
**정답 ② After Image**. 회복(Recovery) 기법 2종:
- **UNDO(취소)**: 로그의 **Before Image**(변경 전 값)로 되돌림 → **아직 커밋 안 된 트랜잭션 취소** 시 사용
- **REDO(재실행)**: 로그의 **After Image**(변경 후 값)로 재반영 → **이미 커밋된 트랜잭션이 디스크에 반영 안 됨** 시 사용

체크포인트 이전 커밋 = 완료. 체크포인트 이후 커밋 = **REDO 대상**. 체크포인트 이후 미커밋 = **UNDO 대상**.
- 함정: **UNDO ↔ REDO 방향 뒤바꾸기 초빈출**. ① = UNDO 설명 / ③ = 즉시 갱신은 **UNDO+REDO 모두 필요**·지연 갱신은 REDO만 필요(반대 설명) / ④ = **REDO는 정상·비정상 종료 무관하게 커밋된 트랜잭션이 디스크 미반영이면 실행**. UNDO/REDO 관계: **즉시 갱신 = UNDO+REDO**, **지연 갱신 = REDO만**.

### 문제 12
**정답 ① Read Uncommitted**. SQL 표준 4단계 격리 수준(낮음 → 높음):
- **① Read Uncommitted**(가장 낮음) — Dirty Read/Non-Repeatable Read/Phantom Read **모두 발생 가능**·성능 최고
- **② Read Committed** — Dirty Read 방지, Non-Repeatable/Phantom 발생 가능·PostgreSQL/Oracle 기본
- **③ Repeatable Read** — Dirty/Non-Repeatable 방지, Phantom 발생 가능·MySQL InnoDB 기본
- **④ Serializable**(가장 높음) — 모든 이상 현상 방지·직렬 실행 동등·성능 최저
- 함정: **격리 수준 ↑ = 일관성 ↑ · 성능 ↓ · 락 경합 ↑**. 순서 뒤바꾸기 초빈출. **"Uncommitted가 가장 안전"** 정반대 함정. 3대 이상 현상 **Dirty(미커밋 읽음) / Non-Repeatable(다시 읽으니 변경됨) / Phantom(다시 읽으니 행 추가됨)** 매트릭스도 필수.

- **② 매개변수 개수 다름 (int,int) vs (int,int,int)** → 오버로딩 성공
- **③ 매개변수 타입 다름 (int,int) vs (double,double)** → 오버로딩 성공
- **④ 매개변수 이름만 다르고 타입 시그니처 동일 (int,int) vs (int,int)** → 매개변수 이름 무관, **오버로딩 실패(컴파일 오류)**

정확히는 ①과 ④ 둘 다 컴파일 오류지만, ①은 **"반환 타입만 다르다"** 는 초빈출 함정 패턴 대표. 시험 정형에서는 **"반환 타입만 다른 오버로딩 불가"** 를 첫 번째 확인. **오버라이딩(Overriding)** 은 상속 관계에서 시그니처 동일 + 반환 타입 공변 허용.
- 함정: 오버로딩 vs 오버라이딩 규칙 뒤바꾸기 초빈출. **오버로딩**(같은 클래스, 시그니처 다름) vs **오버라이딩**(상속 관계, 시그니처 동일). **정적 디스패치(오버로딩) vs 동적 디스패치(오버라이딩)** 도 자주 나옴.

### 문제 14
**정답 ③ 24**. 코드 트레이싱:
- `d = {'a':1, 'b':2, 'c':3}` → 초기 딕셔너리
- `d.setdefault('b', 99)` → 'b' **이미 존재** → 기존 값 2 유지(변화 없음)
- `d.setdefault('d', 4)` → 'd' 없음 → **d['d'] = 4** 추가 → `{'a':1,'b':2,'c':3,'d':4}`
- `d.update({'a': 10, 'e': 5})` → 'a' 기존 값 1 → **10으로 덮어씀** + 'e' 없음 → **d['e'] = 5** 추가
- 최종 딕셔너리 = `{'a':10, 'b':2, 'c':3, 'd':4, 'e':5}`
- `sum(d.values()) = 10 + 2 + 3 + 4 + 5 = **24**`

- 함정:
  - **`setdefault(key, default)`** = 키가 **존재하면 기존 값 반환·변경 X**, 없으면 default 삽입 후 반환. "setdefault가 항상 덮어쓴다"는 정반대 함정 → ② 22(99 잘못 채택)
  - **`update(dict)`** = 병합(같은 키는 덮어씀). update 무시 오답 → ① 15(초기 1+2+3 + 4+5)
  - update의 a=10 계산 실수 → ④ 25(a를 11로 착각)
  - 정답 = **③ 24**.

### 문제 16
**정답 ② Preempt(선점)**. 프로세스 상태 5종 + 주요 전이:
- **생성(New) → 준비(Ready)**: Admit
- **준비 → 실행(Running)**: **Dispatch**
- **실행 → 준비**: **Preempt(선점)** = 타임 슬라이스 만료·우선순위 높은 프로세스 도착
- **실행 → 대기(Waiting/Blocked)**: **Block** = I/O 요청·이벤트 대기
- **대기 → 준비**: **Wake-up** = I/O 완료
- **실행 → 종료(Terminated)**: **Exit**

**"실행에서 CPU 빼앗기고 준비로 돌아감" = Preempt(선점)**.
- 함정: **Dispatch(준비→실행)** vs **Preempt(실행→준비)** 뒤바꾸기 초빈출. Block은 실행→대기(I/O), Wake-up은 대기→준비(I/O 완료). **선점형(Preemptive) 스케줄링** = RR·SRT·다단계 큐 등에서만 Preempt 발생.

### 문제 17
**정답 ② 불완전 → 수행 → 관리 → 확립 → 예측 → 최적화**. SPICE(ISO/IEC 15504) 프로세스 성숙도 **6단계 레벨**:
- **레벨 0: 불완전(Incomplete)** — 프로세스가 구현되지 않음·목표 미달
- **레벨 1: 수행(Performed)** — 프로세스 목표 달성
- **레벨 2: 관리(Managed)** — 계획·감시·조정
- **레벨 3: 확립(Established)** — 표준 프로세스 정의·활용
- **레벨 4: 예측(Predictable)** — 정량적 측정·예측 가능
- **레벨 5: 최적화(Optimizing)** — 지속적 개선

- 함정: **CMMI 5단계(초기·관리·정의·정량·최적화)** 와 **SPICE 6단계(불완전·수행·관리·확립·예측·최적화)** 혼동 초빈출. SPICE는 **레벨 0(불완전)** 이 존재하는 것이 CMMI와 결정적 차이. ① CMMI 계열 명칭 혼용 / ③ 존재하지 않는 명칭 / ④ PMBOK 그룹(다른 개념) 함정.

### 문제 18

### 문제 19
**정답 ②** — **Stored XSS(저장형/영구형/Persistent) = 서버 DB·게시판·댓글·프로필 등 영구 저장소에 악성 스크립트가 저장**되어, 이후 페이지를 열람하는 **모든 사용자에게 실행**. 파급력이 가장 크다.
- ① = **Reflected XSS(반사형)** = URL/파라미터에 실려 서버가 응답에 그대로 반사 → 피싱 링크 클릭 시 즉시 실행 · 저장 안 됨
- ③ = **DOM-based XSS** = 서버 응답과 무관, **클라이언트 JS가 URL/입력을 DOM에 삽입**하는 과정에서 실행 · 서버 로그에 안 남음
- ④ = **CSRF(Cross-Site Request Forgery)** = 다른 취약점 · XSS 아님

**방어**: ① **출력 인코딩(HTML Entity Encoding)** — `<`→`&lt;` / ② **CSP(Content Security Policy) 헤더** / ③ **HttpOnly 쿠키** (JS로 쿠키 접근 차단) / ④ **DOMPurify 등 sanitizer** / ⑤ **입력 검증(화이트리스트)**.
- 함정: XSS 3종 정의 뒤바꾸기 초빈출. **"Stored = DB에 영구 저장, 모두에게"** / **"Reflected = URL 반사, 즉시"** / **"DOM = JS만, 서버 로그 없음"**. CSRF와 XSS도 자주 혼동(CSRF는 위조 요청, XSS는 스크립트 실행).

### 문제 20
**정답 ④ OCSP**. PKI 구성 요소:
- **CA(Certificate Authority, 인증기관)**: 인증서 발급·서명 (예: DigiCert, Let's Encrypt, 한국전자인증)
- **RA(Registration Authority, 등록기관)**: CA를 대신해 신원 확인·등록 대행
- **VA(Validation Authority, 검증기관)**: 인증서 유효성 검증
- **CRL(Certificate Revocation List, 폐기 목록)**: 폐기된 인증서 시리얼 번호 목록을 **주기적으로 다운로드**(오프라인·시차 존재)
- **OCSP(Online Certificate Status Protocol)**: **실시간 온라인**으로 인증서 폐기 여부 조회(RFC 6960)
- **OCSP Stapling**: 서버가 CA의 OCSP 응답을 미리 받아 TLS 핸드셰이크에 첨부(성능·프라이버시)

- 함정: **CRL vs OCSP 뒤바꾸기 초빈출**. **CRL = 주기적·오프라인·목록 다운로드(시차)** / **OCSP = 실시간·온라인·개별 조회**. **전자서명 워크플로** = ① 송신자 문서 해시 → ② 송신자 **개인키로 서명** → ③ 문서+서명 전송 → ④ 수신자 문서 해시 재계산 + 송신자 **공개키로 서명 복호** → ⑤ 두 해시 비교. **부인방지·무결성·인증** 3대 효과.

## 📊 자주 틀리는 함정 5가지 (2023년 회독 2/4 정리)

1. **UML Aggregation(◇ 빈 마름모) vs Composition(◆ 찬 마름모) 뒤바꾸기** — "부분이 전체와 수명 무관 → ◇" / "부분이 전체와 수명 동일 → ◆". 삼각형(△)은 상속·실체화용이지 부분-전체 관계 아님. 다이어그램 기호와 개념 매칭 초빈출.
2. **관계대수 순수 4종(σ·π·⋈·÷) vs 집합 4종(∪·∩·−·×) 분류** — **Union(∪)은 집합 연산자**이지 순수 아님. Division(÷)이 순수 연산자라는 사실이 특히 자주 잊혀짐. **"시·파·나·나(시그마·파이·나비넥타이·나눗셈)"** 로 순수 4종 암기.
3. **UNDO(Before Image) vs REDO(After Image) 뒤바꾸기** — UNDO = **커밋 전 트랜잭션 취소**·이전 값 복원 / REDO = **커밋 후 트랜잭션 재실행**·이후 값 반영. 즉시 갱신 = UNDO+REDO 모두 / 지연 갱신 = REDO만. 방향 뒤바꾸기 초빈출.
4. **Java 오버로딩 규칙** — **"반환 타입만 다른 것은 오버로딩 실패"**. 매개변수의 **개수·타입·순서** 중 하나 이상이 달라야 함. 매개변수 이름만 다른 것도 오버로딩 실패. 오버로딩(같은 클래스·시그니처 다름) vs 오버라이딩(상속·시그니처 동일) 개념 혼동 초빈출.
5. **SPICE 6단계(레벨 0 포함) vs CMMI 5단계 혼용** — **SPICE = 불완전(0)·수행(1)·관리(2)·확립(3)·예측(4)·최적화(5)** — 6단계 / **CMMI = 초기(1)·관리(2)·정의(3)·정량(4)·최적화(5)** — 5단계. SPICE의 **레벨 0(불완전)** 존재가 결정적 차이. 명칭·순서·개수 뒤바꾸기 초빈출.
