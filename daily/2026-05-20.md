## 💡 상세 해설 (자기 설명과 대조)

### 문제 1
**정답: ③ 프로그램의 소스 코드**
- PCB는 운영체제가 프로세스를 관리하기 위해 유지하는 자료구조. 식별자(PID), 상태, 프로그램 카운터(PC), 레지스터 값, 우선순위, 스케줄링 정보, 메모리 정보(베이스/한계 레지스터, 페이지 테이블), 입출력 상태, 어카운팅 정보 등을 보관.
- 소스 코드는 PCB가 아닌 디스크의 실행 파일 또는 메모리의 텍스트 영역에 존재. PCB에는 포함되지 않음.
- ① PID, ② 프로세스 상태(생성/준비/실행/대기/종료), ④ 프로그램 카운터는 모두 PCB의 핵심 항목.

### 문제 2
**정답: ④ RR (Round Robin)**
- 비선점: FCFS, SJF, HRN, 우선순위(비선점), 기한부.
- 선점: RR(시간 할당량 단위로 강제 회수), SRT(SJF의 선점 버전), 다단계 큐, 다단계 피드백 큐.
- RR은 시간 할당량 만료 시 강제로 CPU를 회수하므로 대표적인 선점 기법.

### 문제 3
**정답: ④ 8.0 ms**
- SJF는 실행시간 짧은 순으로 정렬 → P2(4) → P3(6) → P4(8) → P1(10).
- 대기시간:
  - P2: 0
  - P3: 4
  - P4: 4+6 = 10
  - P1: 4+6+8 = 18
- 평균 = (0 + 4 + 10 + 18) / 4 = 32 / 4 = **8.0 ms**
- 함정: ① 6.5는 FCFS(도착 순서대로 P1→P2→P3→P4)일 때 자주 등장하는 오답 분포가 아님. SJF는 반드시 실행시간 짧은 순으로 정렬 후 누적 대기 합산.

### 문제 4
**정답: ③ (대기 시간 + 서비스 시간) / 서비스 시간**
- HRN 우선순위 = (대기시간 + 서비스시간) / 서비스시간 = 1 + (대기시간/서비스시간).
- 값이 클수록 우선순위 높음. SJF가 긴 작업을 무한대기시키는 기아(Starvation)를 보완.
- ② 대기시간/서비스시간 — HRN 변형으로 보이지만 표준 정의는 ③.

### 문제 5
**정답: ④ 시간 할당량이 작을수록 문맥 교환 오버헤드가 감소한다.**
- 반대다. 시간 할당량이 **작을수록 문맥 교환이 자주 일어나 오버헤드가 증가**.
- 시간 할당량이 크면 FCFS와 유사, 너무 작으면 문맥 교환 비용 폭증.
- ①, ②, ③은 모두 옳다.

### 문제 6
**정답: ③ 선점(Preemption)**
- 교착상태 4조건: 상호배제, 점유와대기, **비선점(No Preemption)**, 환형대기.
- 선점이 아닌 **비선점**이 조건. ③은 정반대 표현으로 함정.

| 워킹 셋 | 일정 시간 동안 자주 참조되는 페이지 집합 |
| 스레싱 | 페이지 부재가 너무 많아 CPU 효율이 급락하는 현상 |
| 지역성 | 시간 지역성(최근 참조 재참조), 공간 지역성(인접 주소 참조) |

### 핵심 개념 (사후 검증용)

**프로세스 상태 5단계**
생성(New) → 준비(Ready) → 실행(Running) → 대기(Waiting/Blocked) → 종료(Terminated)

**스케줄링 알고리즘 비교 (사후 검증용)**

| 알고리즘 | 선점 | 정렬 기준 | 기아 | 비고 |
|---|---|---|---|---|
| FCFS | × | 도착 순서 | 무 | Convoy 효과 |
| SJF | × | 실행시간 짧은 순 | 유 | 미래 예측 어려움 |
| SRT | ○ | 남은 시간 짧은 순 | 유 | SJF 선점형 |
| HRN | × | (대기+서비스)/서비스 | 무 | SJF 기아 보완 |
| RR | ○ | 시간 할당량 | 무 | 응답시간 짧음 |
| 다단계 피드백 | ○ | 큐별 우선순위 | 무 | 동적 우선순위 |
