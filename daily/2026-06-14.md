### 문제 7 [유사 기출: 정처기 필기 2020·2021·2022 반복 — DFD 구성요소 기호 정형 패턴]
자료 흐름도(DFD; Data Flow Diagram)의 4가지 기본 구성 요소와 기호의 연결이 **틀린** 것은?

① 처리(Process) — **원(○)**
② 자료 흐름(Data Flow) — **화살표(→)**
③ 자료 저장소(Data Store) — **평행선(=)** 또는 **열린 직사각형**

### 문제 8 [유사 기출: 정처기 필기 2020·2022·2023 반복 — 자료사전 표기 기호 정형 패턴]
자료 사전(Data Dictionary)에서 사용하는 표기 기호와 의미의 연결이 **틀린** 것은?

① `=` — 자료의 **정의**(is composed of)
② `+` — 자료의 **연결(AND, 구성)**
③ `{ }` — 자료의 **반복(iteration)**
④ `( )` — 자료의 **선택(택일, OR)**

## 🔍 오답 역추적 개념 발췌

### A. 5대 생명주기 모형 1줄 식별
| 모형 | 제안자/특징 | 핵심 키워드 |
|---|---|---|
| **폭포수(Waterfall)** | Royce(고전적) | 선형 순차·단계 완료 후 다음·문서 중시·요구변경 취약 |
| **프로토타입(Prototype)** | — | 시제품·요구 불명확·고객 피드백·버리고 다시 |
| **나선형(Spiral)** | **Boehm** | **위험 분석** 반복·대규모 프로젝트·점진적 |
| **V 모델(V-Model)** | — | 폭포수 확장·**검증(Verification)·확인(Validation)** 강조·테스트 짝짓기 |
| **RAD(Rapid)** | Martin | 60~90일 단기·CASE 도구·재사용 컴포넌트 |

### B. 나선형 4단계 (매우 빈출 — 순서 통째 암기)
**계획 수립 → 위험 분석 → 개발 및 검증 → 고객 평가** (→ 반복)
> 두음: "**계위개고**" (계 → 위 → 개 → 고)

### C. 애자일 vs 전통 — 4가치
1. **개인과 상호작용** > 프로세스와 도구
2. **작동하는 소프트웨어** > 포괄적 문서
3. **고객과의 협력** > 계약 협상
4. **변화에의 대응** > 계획 고수
> 오른쪽도 가치 있지만 **왼쪽을 더** 중시 (왼쪽으로 완전 대체 X — 함정)

### D. 스크럼 3역할 / 5행사 / 2백로그
| 구분 | 항목 |
|---|---|
| 3역할 | **PO**(Product Owner, 백로그 우선순위) / **SM**(Scrum Master, 코치·장애제거) / **개발팀**(자기조직화 5~9명) |
| 5행사 | ① **스프린트 계획**(Sprint Planning) ② **일일 스크럼**(Daily Scrum, 15분) ③ **스프린트 리뷰**(Sprint Review, **고객 시연**) ④ **스프린트 회고**(Sprint Retrospective, **팀 내부 프로세스 개선**) ⑤ **스프린트**(Sprint, 2~4주 타임박스) |
| 2백로그 | **제품 백로그**(전체) / **스프린트 백로그**(이번 스프린트분) |

### E. XP 5가치 (★ 두음 "**의소단용존**")
**의**사소통(Communication) · **소**단순성(Simplicity) · **단**피드백(Feedback) · **용**용기(Courage) · **존**존중(Respect)
> 5가치에 "**품질·일정·비용·문서**"는 포함 X (함정)
> 12실천법 대표: **Pair Programming · TDD · Refactoring · Continuous Integration · Small Releases · Simple Design · Collective Ownership · Coding Standards**

### F. 기능 vs 비기능 요구사항 (★ 함정 분류)
| 구분 | 정의 | 예시 |
|---|---|---|
| **기능(Functional)** | 시스템이 **무엇을 해야 하는가(What)** | 로그인·결제·주문·검색·CRUD 동작 |
| **비기능(Non-functional)** | 시스템이 **어떻게 동작해야 하는가(How)**, **품질 속성** | **성능·가용성·보안·신뢰성·확장성·유지보수성·이식성·응답시간·동시사용자수** |

### G. 요구공학 5단계 (★ 순서 통째)
**도출 → 분석 → 명세 → 확인 → 관리**
- **도출(Elicitation)**: 인터뷰·설문·관찰·BPM·프로토타이핑
- **분석(Analysis)**: 모델링(DFD·UML)·우선순위·충돌 해결
- **명세(Specification)**: SRS 문서화(정형/비정형)
- **확인(Validation)**: 검토·워크스루·인스펙션·**프로토타입**
- **관리(Management)**: 변경관리·추적성·형상관리

### H. DFD 4구성요소
| 구성요소 | 기호 | 의미 |
|---|---|---|
| Process(처리) | **○ 원 / 둥근 사각형** | 자료 변환 |
| Data Flow(자료 흐름) | **→ 화살표** | 자료 이동 |
| Data Store(자료 저장소) | **= 평행선 / 열린 직사각형** | 자료 보관 |
| External Entity(외부 단말) | **□ 사각형** | 시스템 외부 발생/종착지 |
> **마름모(◇)는 ERD의 관계** (DFD가 아님) — 자주 나오는 함정

### I. 자료사전(Data Dictionary) 7기호
| 기호 | 의미 |
|---|---|
| `=` | **정의** is composed of |
| `+` | **AND, 연결**(구성) |
| `( )` | **생략 가능**(optional) |
| `{ }` | **반복**(iteration) |
| `[ \| ]` | **선택**(택일, OR) |
| `* *` | **주석**(comment) |
| `@` | **키 항목**(key) |

### J. HIPO 3종 도표 (★ 빈출)
- **가시적 도표**(Visual Table of Contents) — 전체 기능 계층 구조 (목차)
- **총체적 도표**(Overview Diagram) — 입력·처리·출력 개요(IPO)
- **세부적 도표**(Detail Diagram) — 각 기능 상세 IPO

### K. 요구사항 검토 기법 3종
- **워크스루(Walkthrough)** — **비공식**·**검토 자료 미리 배포**·**짧은 회의**·**개발자 주도**
- **인스펙션(Inspection)** — **공식**·**체크리스트**·**moderator** 주재·**결함 식별·기록**
- **동료 검토(Peer Review)** — **2~3명 동료**가 설명 듣고 비공식 검토

## 💡 상세 해설

### 문제 1
**정답 ③ 폭포수 모형(Waterfall)**
- "고전적 생명주기 모형", "선형 순차적", "각 단계가 끝나야 다음", "문서 중시" → 모두 **폭포수**의 정형 키워드.
- **① 프로토타입**: 시제품을 만들고 고객 피드백 반복 (요구가 불명확할 때).
- **② 나선형(Boehm)**: 위험 분석(Risk Analysis)을 핵심 단계로 반복.
- **④ RAD**: 60~90일 단기 + CASE 도구 + 재사용으로 빠른 개발.
> 함정: "문서화·결과물 검토가 잘 이루어진다"는 문구를 보고 V 모델로 오인하기 쉬우나, V 모델은 **검증·확인(Verification·Validation) 강조 + 테스트 짝짓기** 키워드가 핵심.

### 문제 2
**정답 ② 계획 수립 → 위험 분석 → 개발 및 검증 → 고객 평가**
- 나선형 4단계 두음 **"계위개고"** — 통째 암기.
- 위험 분석이 **2번째**라는 점이 함정. ③·④처럼 위험 분석을 1번째로 두면 오답.
- ①은 고객 평가가 2번째에 있어 오답 (고객 평가는 **마지막**, 다음 사이클로 이어짐).
> 핵심 키워드: **Boehm 제안 · 위험 분석(Risk Analysis) 명시적 단계화 · 점진적 반복**.

### 문제 3
**정답 ④** (틀린 설명)
- ④는 **Sprint Review**의 설명. **Sprint Retrospective(회고)**는 **팀 내부의 프로세스·협업·도구 개선**을 위한 회의로, **고객은 참여하지 않음**.
- 헷갈리기 쉬운 두 행사 구분:

| 행사 | 대상 | 목적 |
|---|---|---|
| **Sprint Review** | **고객 + 팀** | **제품 증분(Increment) 시연**, 제품 백로그 조정 |
| **Sprint Retrospective** | **팀 내부만** | **프로세스·협업 개선**(Action Item 도출) |

- ①②③은 모두 맞는 설명.
> 함정: "리뷰=Retrospective"라고 단어상 오인하기 쉬우나, **Review=고객 시연(제품 관점)**, **Retrospective=팀 회고(프로세스 관점)**.

### 문제 4
**정답 ③ 품질(Quality) — XP 5가치 아님**
- XP 5가치 두음 **"의소단용존"**:
  - **의**사소통(Communication)
  - **소**단순성(Simplicity)
  - **단**피드백(Feedback)
  - **용**용기(Courage)
  - **존**존중(Respect, XP 2판에서 추가)
- "품질·일정·비용·문서·신속성"은 XP 5가치 **아님**.
> 함정: ④ 용기(Courage)가 "갑자기 코드를 바꾸는 용기"라는 의미라 어색해서 오답으로 고르기 쉬우나, **이것이 정확히 XP의 핵심 가치**.

### 문제 5
**정답 ③** (비기능 요구사항)
- "동시 사용자 1,000명", "응답 시간 3초 이내" → **성능(Performance)**·**확장성(Scalability)** = **품질 속성(How)** = **비기능**.
- ①②④는 모두 시스템이 **무엇을 해야 하는가(What)** — **기능 요구사항**.
- 비기능 7대 분류: **성능·가용성·보안·신뢰성·확장성·유지보수성·이식성** (+ 사용성, 호환성)
> 함정: "사용자가 로그인할 수 있어야 한다" = 기능 / "로그인 응답이 1초 이내여야 한다" = 비기능 — **동작 자체는 기능, 동작의 품질 속성은 비기능**.

### 문제 6
**정답 ② 도출 → 분석 → 명세 → 확인 → 관리**
- 요구공학 5단계 — **순서 통째 암기**.
- 단계별 활동:
  1. **도출(Elicitation)**: 인터뷰·설문·관찰·**프로토타이핑**·BPM
  2. **분석(Analysis)**: 모델링(**DFD·UML**)·우선순위·충돌 해결
  3. **명세(Specification)**: SRS 문서 작성(**정형/비정형** 기법)
  4. **확인(Validation)**: 워크스루·인스펙션·**프로토타입**으로 검증
  5. **관리(Management)**: 변경관리·**추적성(Traceability)**·형상관리
> 함정: ①은 "분석이 도출보다 먼저"라 오답. **도출이 항상 1번째**.

### 문제 7
**정답 ④** (틀린 설명)
- **단말(External Entity)**은 **사각형(□)**으로 표시. **마름모(◇)는 ERD의 관계(Relationship)**.
- DFD 4구성요소 기호 매칭 표는 위 [개념 발췌 H] 참조.
- DFD 발란싱(Balancing) 원칙: 상위·하위 DFD의 **입력·출력 자료 흐름이 일치**해야 함.
> 함정: ◇는 ERD에서 "관계"를, **UML 활동 다이어그램**에서는 "결정 노드(decision)"를 표시 — 다이어그램별 의미가 다르니 주의.

### 문제 8
**정답 ④** (틀린 설명)
- **`( )`는 생략 가능(Optional)**, **선택(택일, OR)은 `[ A | B | C ]`** 표기.
- 자료사전 7기호 통째 암기 ([개념 발췌 I] 표 참조).
- 사용 예: `학생 = 학번 + 이름 + (전화번호) + {수강과목} + [학부생 | 대학원생]`
  - `학번 + 이름`: 필수
  - `(전화번호)`: 생략 가능
  - `{수강과목}`: 0회 이상 반복
  - `[학부생 | 대학원생]`: 둘 중 하나 선택
> 함정: `( )`를 "선택"으로, `{ }`를 "선택"으로 오인하기 쉬움. **( ) = 생략(있어도 없어도 됨), { } = 반복, [ | ] = 택일** — 통째로 외울 것.
