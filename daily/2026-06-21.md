## 🎯 오늘의 기출 8문제

### 문제 1 [유사 기출: 정처기 필기 2020년 3회 · 2022년 1회 · 2024년 1회 반복 — ACID 정의 매칭 정형 패턴]
트랜잭션의 **ACID 속성**에 대한 설명으로 가장 옳지 **않은** 것은?

① **원자성(Atomicity)** — 트랜잭션의 연산은 모두 수행되거나 전혀 수행되지 않아야 한다(All-or-Nothing).
② **일관성(Consistency)** — 트랜잭션 실행 전·후 데이터베이스가 일관된 상태를 유지해야 한다.
③ **격리성(Isolation)** — 동시에 수행되는 트랜잭션들이 서로의 중간 연산 결과를 자유롭게 참조할 수 있어야 한다.
④ **영속성(Durability)** — 성공적으로 완료(COMMIT)된 트랜잭션의 결과는 시스템 장애가 발생해도 영구히 반영되어야 한다.

### 문제 2 [유사 기출: 정처기 필기 2021년 1회 · 2023년 2회 · 2024년 3회 반복 — 트랜잭션 상태 전이 정형 패턴]
다음 중 트랜잭션의 **상태 전이**에 대한 설명으로 가장 옳지 **않은** 것은?

① **활동(Active)** 상태는 트랜잭션이 실행을 시작하여 연산이 진행 중인 상태이다.
② **부분 완료(Partially Committed)** 상태는 마지막 연산을 끝낸 직후로, 아직 변경 사항이 영구 반영되지 않은 상태이다.
③ **완료(Committed)** 상태로 전이된 트랜잭션은 그 결과가 영구적으로 데이터베이스에 반영된다.
④ **실패(Failed)** 상태의 트랜잭션은 자동으로 **완료(Committed)** 상태로 전이되어 결과가 반영된다.

### 문제 3 [유사 기출: 정처기 필기 2021년 2회 · 2023년 1회 · 2025년 1회 반복 — 격리수준 4단계 정형 패턴]
SQL 표준의 **트랜잭션 격리수준(Isolation Level)** 을 동시성 부작용 발생이 **적은 순서(=고립성이 강한 순서)** 부터 나열한 것은?

① Read Uncommitted > Read Committed > Repeatable Read > Serializable
② **Serializable > Repeatable Read > Read Committed > Read Uncommitted**
③ Repeatable Read > Serializable > Read Committed > Read Uncommitted
④ Read Committed > Read Uncommitted > Serializable > Repeatable Read

### 문제 4 [유사 기출: 정처기 필기 2020년 2회 · 2022년 3회 · 2024년 2회 반복 — 동시성 이상현상 정의 정형 패턴]
다음 동시성 이상현상에 대한 설명 중 가장 옳지 **않은** 것은?

① **Dirty Read** — 한 트랜잭션이 다른 트랜잭션의 **아직 커밋되지 않은(uncommitted)** 변경 데이터를 읽는 현상.
② **Non-Repeatable Read** — 동일 트랜잭션 내에서 같은 행을 두 번 조회했을 때, 다른 트랜잭션의 **UPDATE/COMMIT**으로 값이 달라지는 현상.
③ **Phantom Read** — 동일 트랜잭션 내에서 같은 조건으로 범위 조회를 두 번 했을 때, 다른 트랜잭션의 **INSERT/DELETE**로 결과 행 수가 달라지는 현상.
④ **Lost Update**는 위 세 가지 이상현상과 동일한 것으로, **Serializable** 격리수준에서도 항상 발생한다.

### 문제 5 [유사 기출: 정처기 필기 2020년 4회 · 2022년 2회 · 2024년 1회 반복 — 락(Lock) & 2PL 정형 패턴]
데이터베이스 **락(Lock)** 과 **2단계 락 프로토콜(2PL: Two-Phase Locking)** 에 대한 설명으로 가장 옳지 **않은** 것은?

① **공유락(S-Lock)** 은 여러 트랜잭션이 동시에 같은 자원에 대해 가질 수 있으나, 그 자원에 **배타락(X-Lock)** 은 가질 수 없다.
② **배타락(X-Lock)** 은 한 트랜잭션만 보유 가능하며, 다른 트랜잭션은 그 자원에 대해 S-Lock도 X-Lock도 가질 수 없다.
③ 2단계 락 프로토콜은 **확장(Growing) 단계**(락 획득만)와 **축소(Shrinking) 단계**(락 해제만)로 구성되며, **직렬 가능성(Serializability)** 을 보장한다.
④ 2단계 락 프로토콜은 직렬 가능성을 보장하므로 **교착상태(Deadlock) 가 절대 발생하지 않는다.**

### 문제 6 [유사 기출: 정처기 필기 2021년 3회 · 2023년 3회 · 2025년 2회 반복 — 회복 기법 정형 패턴]
데이터베이스 **회복(Recovery) 기법**에 대한 설명으로 가장 옳지 **않은** 것은?

① **UNDO** 는 트랜잭션 실패 시 이전 상태로 되돌리는 연산이고, **REDO** 는 이미 완료된 트랜잭션의 변경을 재실행하는 연산이다.
② **즉시 갱신(Immediate Update)** 기법은 트랜잭션 수행 중 변경 사항을 디스크에 곧바로 반영하므로, 회복 시 **UNDO·REDO 모두 필요**할 수 있다.
③ **지연 갱신(Deferred Update)** 기법은 트랜잭션 완료 시점까지 변경을 디스크에 반영하지 않으므로, 회복 시 **UNDO가 필요 없다(REDO만)**.
④ **검사점(Checkpoint)** 기법은 회복 시 모든 트랜잭션을 처음부터 다시 검사해야 하므로, 회복 시간이 검사점이 없을 때보다 **더 길어진다.**

### 문제 7 [유사 기출: 정처기 필기 2020년 1회 · 2022년 1회 · 2024년 3회 반복 — 분산 DB 투명성 정형 패턴]
**분산 데이터베이스(Distributed Database)** 의 **투명성(Transparency)** 에 해당하지 **않는** 것은?

① **위치(Location) 투명성** — 사용자가 데이터의 물리적 위치를 알 필요 없이 동일한 명령으로 접근할 수 있다.
② **중복(Replication) 투명성** — 동일 데이터가 여러 사이트에 중복 저장되어도 사용자는 하나의 데이터처럼 사용한다.
③ **병행(Concurrency) 투명성** — 여러 트랜잭션이 동시에 수행되어도 결과가 직렬 수행과 동일하다.
④ **확장(Scalability) 투명성** — 시스템에 노드를 추가하거나 제거해도 **성능이 항상 선형으로** 향상된다.

### 문제 8 [유사 기출: 정처기 필기 2021년 2회 · 2023년 2회 · 2025년 1회 반복 — NoSQL 4유형 + CAP 정형 패턴]
**NoSQL** 데이터베이스와 **CAP 정리**에 대한 설명으로 가장 옳지 **않은** 것은?

① NoSQL은 데이터 모델에 따라 **Key-Value, Document, Column-Family, Graph** 의 4가지 유형으로 분류된다.
② **MongoDB**는 JSON·BSON 형태로 데이터를 저장하는 대표적인 **Document** 유형 NoSQL이다.
③ **CAP 정리**는 분산 시스템에서 **Consistency(일관성)·Availability(가용성)·Partition tolerance(분할 내성)** 의 3가지를 **동시에 모두 만족할 수 있다**고 본다.
④ **Neo4j**는 노드와 간선으로 데이터를 표현하는 대표적인 **Graph** 유형 NoSQL이다.

## 🔍 오답 역추적 개념 발췌

### 1. ACID 4속성 — 두음 "원·일·격·영"
| 속성 | 풀네임 | 한 줄 정의 | 위반 시 현상 |
|---|---|---|---|
| **A**tomicity | 원자성 | 모두 수행 or 모두 취소(All-or-Nothing) | 일부만 반영(부분 갱신) |
| **C**onsistency | 일관성 | 트랜잭션 전·후 DB 무결성·제약 유지 | 제약 위반·잘못된 상태 |
| **I**solation | 격리성·고립성 | 동시 트랜잭션은 서로 간섭 X | Dirty/Non-Repeatable/Phantom Read |
| **D**urability | 영속성·지속성 | COMMIT 결과는 장애에도 영구 반영 | COMMIT한 데이터 유실 |

> **함정 단골**: "격리성=중간 결과를 자유롭게 공유 가능" → **X**(정반대). "원자성=일부만 반영해도 됨" → X. ACID 한 글자씩 풀네임 + 한 줄 정의 1:1 매칭.

### 2. 트랜잭션 상태 5종 — 전이 다이어그램
```
[Active 활동] ──정상──▶ [Partially Committed 부분완료] ──반영──▶ [Committed 완료]
      │                          │
      └─실패─▶ [Failed 실패] ──UNDO──▶ [Aborted 철회]
```
- **Active(활동)**: 실행 중. 연산 진행.
- **Partially Committed(부분완료)**: 마지막 연산 끝 직후. 아직 디스크 미반영.
- **Committed(완료)**: COMMIT 성공. 결과 영구 반영(Durability).
- **Failed(실패)**: 오류 발생. 더 이상 진행 불가.
- **Aborted(철회)**: UNDO 완료. 트랜잭션 시작 전 상태로 복귀.

> **함정 단골**: "Failed → Committed로 전이" → **X**(Failed는 Aborted로만). "Partially Committed = 영구 반영 완료" → X(아직 미반영).

### 3. 격리수준 4단계 — 약→강 + 이상현상 발생 매트릭스
| 격리수준 | Dirty Read | Non-Repeatable Read | Phantom Read | 동시성 |
|---|---|---|---|---|
| **Read Uncommitted (RU)** | 발생 | 발생 | 발생 | 최고 |
| **Read Committed (RC)** | 차단 | 발생 | 발생 | 高 |
| **Repeatable Read (RR)** | 차단 | 차단 | 발생 | 中 |
| **Serializable (S)** | 차단 | 차단 | 차단 | 최저 |

> 강할수록 **일관성 ↑ / 동시성 ↓** — 트레이드오프. 시험은 "이상현상 적은(=강한) 순"으로 자주 물음.
> **함정 단골**: 수준 순서 뒤바꿈 / RC가 Phantom 차단 / RR이 모든 이상현상 차단 → 모두 X. 매트릭스 한 줄 외우면 즉답.

### 4. 동시성 이상현상 3종 + Lost Update
| 이상현상 | 정의 | 발생 패턴 |
|---|---|---|
| **Dirty Read** | 미커밋 데이터 읽음 | T1 UPDATE → T2 SELECT → T1 ROLLBACK |
| **Non-Repeatable Read** | 같은 행 재조회 시 값 변경 | T1 SELECT → T2 UPDATE+COMMIT → T1 SELECT |
| **Phantom Read** | 같은 조건 재조회 시 **행 출현·소멸** | T1 SELECT(범위) → T2 INSERT/DELETE+COMMIT → T1 SELECT |
| Lost Update(별도) | 두 트랜잭션 동시 갱신 → 한쪽 덮어씀 | T1 READ → T2 READ → T1 WRITE → T2 WRITE |

> **함정 단골**: "Phantom = 같은 행 값 변경" → X(이는 Non-Repeatable). "Serializable에서도 Lost Update 항상 발생" → X(Serializable이면 거의 모두 차단). Non-Repeatable과 Phantom 구분 = **행 값 변경 vs 행 개수 변경**.

### 5. 락(Lock) 종류 + 호환성 행렬
| 락 종류 | 용도 | 표기 |
|---|---|---|
| **공유락(Shared Lock, S-Lock)** | 읽기 전용 | LOCK ... IN SHARE MODE |
| **배타락(eXclusive Lock, X-Lock)** | 쓰기·갱신 | LOCK ... FOR UPDATE |

**호환성 매트릭스 (Y=동시 보유 가능)**
| 요청 \ 보유 | S | X |
|---|---|---|
| **S** | **Y** | N |
| **X** | N | N |

> S-S만 호환, S-X·X-S·X-X는 비호환(대기). 락 강도: **X-Lock > S-Lock**.

### 6. 2단계 락 프로토콜(2PL) — 직렬 가능성 보장
- **확장(Growing) 단계**: 락을 **획득만**, 해제 불가.
- **축소(Shrinking) 단계**: 락을 **해제만**, 획득 불가.
- 일단 락을 하나라도 해제하면 그때부터 축소 단계 진입.

> **2PL은 직렬 가능성(Serializability) 보장**하지만 **교착상태(Deadlock) 회피 X** — Deadlock 발생 가능. 따라서 **타임아웃·Wait-Die·Wound-Wait** 등 별도 회피·예방 기법 병행 필요.
> 변형: **엄격(Strict) 2PL** = 모든 X-Lock을 트랜잭션 끝까지 보유 → Cascading rollback 방지.

### 7. 회복(Recovery) 기법 — UNDO/REDO 매트릭스
| 기법 | 정의 | 회복 시 |
|---|---|---|
| **즉시 갱신(Immediate Update)** | 트랜잭션 중 변경 곧바로 디스크 반영 | **UNDO 필요** (미커밋 변경 되돌리기) + REDO(커밋 후 장애 시) |
| **지연 갱신(Deferred Update)** | COMMIT 시점까지 디스크 반영 보류 | **UNDO 불필요** (디스크에 안 적혔으므로), **REDO만** |
| **검사점(Checkpoint)** | 일정 시점 모든 변경 디스크 반영 + 로그 기록 | 회복 시 **검사점 이후만** 처리 → 회복 시간 **단축** |
| **그림자 페이징(Shadow Paging)** | 페이지 단위 사본 유지 | 실패 시 그림자 페이지로 즉시 복귀, 로그 불필요 |

> **함정 단골**: "검사점은 회복 시간을 길게 한다" → **X**(반대로 **단축**). "지연 갱신도 UNDO 필요" → X(REDO만). "즉시 갱신은 UNDO만 필요" → X(UNDO·REDO 모두).
> 회복 순서 정형: **로그(WAL: Write-Ahead Log)** → 분석 → REDO → UNDO.

### 8. 분산 DB — 4(+1) 투명성
| 투명성 | 의미 |
|---|---|
| **위치(Location)** | 데이터의 물리적 위치를 사용자는 모름 — 동일 명령으로 접근 |
| **중복(Replication)** | 여러 사이트에 중복 저장돼도 단일 데이터처럼 사용 |
| **병행(Concurrency)** | 다중 트랜잭션 동시 수행 결과가 직렬 수행과 동일 |
| **장애(Failure)** | 일부 사이트·통신 장애에도 트랜잭션이 정확히 수행 |
| 분할(Fragmentation) | 데이터가 분할 저장돼도 사용자는 단일 릴레이션처럼 사용 (4투명성에 포함 시키기도) |

> **함정 단골**: **"확장(Scalability) 투명성"·"성능 투명성"** 같은 가짜 항목 끼워넣기. 4가지(+분할)만 정형, 다른 이름은 거의 오답.

### 9. NoSQL 4유형 + 대표 DBMS
| 유형 | 데이터 모델 | 대표 DBMS |
|---|---|---|
| **Key-Value** | 키 ↔ 값 단순 매핑 | **Redis**, DynamoDB, Riak |
| **Document** | JSON/BSON/XML 문서 | **MongoDB**, CouchDB |
| **Column-Family (Wide-Column)** | 컬럼군 단위 저장 (Sparse 테이블) | **Cassandra**, HBase, Bigtable |
| **Graph** | 노드 + 간선(관계) | **Neo4j**, OrientDB, ArangoDB |

> **함정 단골**: MongoDB를 Key-Value로 / Cassandra를 Document로 분류 → X. 대표 DBMS 1:1 매칭 외워 즉답.

### 10. CAP 정리 — "셋 중 둘만"
- **C**onsistency(일관성): 모든 노드가 같은 시점에 동일한 데이터.
- **A**vailability(가용성): 모든 요청이 응답을 받음(성공 or 실패라도 응답).
- **P**artition tolerance(분할 내성): 네트워크 분단 발생해도 시스템 정상 동작.
- **분산 시스템은 셋 중 2개만 동시 만족**(에릭 브루어의 추측, 길버트·린치 증명).
- 일반적으로 P는 분산에서 거의 필수 → 실제 선택은 **CP** vs **AP**.

| 분류 | 선택 | 예시 |
|---|---|---|
| CP | 일관성+분할내성, 가용성 포기 | HBase, MongoDB(설정), 전통적 RDBMS 분산 |
| AP | 가용성+분할내성, 강한 일관성 포기(결과 일관성) | Cassandra, DynamoDB, CouchDB |
| CA | 일관성+가용성, 분할 X | 단일 노드 RDBMS (분산 X) |

> **함정 단골**: **"CAP는 셋을 동시에 모두 만족 가능"** → **X**(정의상 불가능). 1자 매칭 즉답.

## 💡 상세 해설

### 문제 1
**정답 ③ 격리성(Isolation) — 동시에 수행되는 트랜잭션들이 서로의 중간 연산 결과를 자유롭게 참조할 수 있어야 한다.**
- **격리성(Isolation)의 표준 정의는 정반대** — 동시에 수행되는 트랜잭션은 **서로의 중간 결과를 참조할 수 없어야** 한다(=고립). 미커밋 변경을 다른 트랜잭션이 보면 Dirty Read 발생.
- ① **원자성(Atomicity)** — All-or-Nothing. ✔
- ② **일관성(Consistency)** — 전·후 DB 무결성·제약 유지. ✔
- ④ **영속성(Durability)** — COMMIT 결과는 장애에도 영구 반영(WAL 로그+회복 기법으로 보장). ✔
- 정형 함정: ACID 정의 4개 중 **격리성 정의만 뒤집기** 가 단골. "자유롭게 참조"·"서로 공유"·"중간 결과 사용 가능" 키워드 보이면 즉시 X.

### 문제 2
**정답 ④ 실패(Failed) 상태의 트랜잭션은 자동으로 완료(Committed) 상태로 전이되어 결과가 반영된다.**
- **Failed 상태는 절대 Committed로 전이될 수 없음** — Failed → **UNDO** → **Aborted(철회)** 로만 전이. 변경 사항은 모두 취소되어 트랜잭션 시작 전 상태로 복귀.
- ① **Active**: 실행 중·연산 진행. ✔
- ② **Partially Committed**: 마지막 연산 직후, 디스크 미반영 단계(여기서 실패하면 Failed로 전이 가능). ✔
- ③ **Committed**: COMMIT 성공, 영구 반영(Durability 보장). ✔
- 정형 함정: 전이 화살표 뒤바꿈 — "Aborted → Active 재진입"(X·새 트랜잭션 시작), "Failed → Committed"(X) 단골.

### 문제 3
**정답 ② Serializable > Repeatable Read > Read Committed > Read Uncommitted**
- 동시성 부작용이 **적은 순 = 격리성(고립성)이 강한 순**: **Serializable(S) > Repeatable Read(RR) > Read Committed(RC) > Read Uncommitted(RU)**.
- 강할수록 Dirty/Non-Repeatable/Phantom Read 차단 범위가 넓어지지만 **동시성(throughput)은 떨어짐** — 트레이드오프.
- ①은 정확히 **반대 순서**(약→강), ③·④는 중간 항목 위치가 어긋난 함정.
- 즉답 키워드: **"Serializable이 가장 강함"·"Read Uncommitted가 가장 약함"** 양 끝점 고정 → 중간(RR·RC) 위치 확정.

### 문제 4
**정답 ④ Lost Update는 위 세 가지 이상현상과 동일한 것으로, Serializable 격리수준에서도 항상 발생한다.**
- **Lost Update(갱신 분실)** 은 Dirty/Non-Repeatable/Phantom Read와 **별개 현상** — 두 트랜잭션이 같은 데이터를 동시에 갱신할 때 한쪽 변경이 다른 쪽에 덮여 사라지는 현상.
- **Serializable** 격리수준은 모든 트랜잭션을 직렬 실행과 등가로 보장하므로 **Lost Update를 비롯한 모든 동시성 이상현상을 차단**(이상적인 경우).
- ① **Dirty Read** = 미커밋 데이터 읽기. ✔
- ② **Non-Repeatable Read** = 같은 **행 값 변경**(UPDATE/COMMIT). ✔
- ③ **Phantom Read** = 같은 조건 **행 개수 변경**(INSERT/DELETE/COMMIT). ✔
- 정형 함정: "Serializable에서도 Phantom 발생"·"Phantom = 같은 행 값 변경" 등 매트릭스 항목 뒤바꿈. **"Serializable = 3개 모두 차단" / "Non-Repeatable = 값 변경, Phantom = 행 개수 변경"** 1자 매칭.

### 문제 5
**정답 ④ 2단계 락 프로토콜은 직렬 가능성을 보장하므로 교착상태(Deadlock)가 절대 발생하지 않는다.**
- **2PL은 직렬 가능성(Serializability) 만 보장**하며 **교착상태(Deadlock)는 별개 문제로 여전히 발생 가능**. 따라서 타임아웃·Wait-Die·Wound-Wait·Deadlock detection(자원 대기 그래프) 등 별도 기법 필요.
- ① S-Lock은 여러 트랜잭션이 동시 보유 가능(S-S 호환), 단 X-Lock과는 비호환. ✔
- ② X-Lock은 한 트랜잭션 단독 보유, 다른 트랜잭션은 S·X 모두 대기. ✔
- ③ 2PL: **확장(Growing) 단계**(획득만) → 락 해제 시작과 동시에 **축소(Shrinking) 단계**(해제만) — 직렬 가능성 보장. ✔
- 정형 함정: "**2PL은 Deadlock을 방지한다**" → **X**. 직렬 가능성과 Deadlock 회피는 **별개 보장**. 즉답 키워드.

### 문제 6
**정답 ④ 검사점(Checkpoint) 기법은 회복 시 모든 트랜잭션을 처음부터 다시 검사해야 하므로, 회복 시간이 검사점이 없을 때보다 더 길어진다.**
- **검사점(Checkpoint)** 은 회복 시작점을 표시 → 회복 시 **검사점 이후의 로그·트랜잭션만** 처리하면 됨 → **회복 시간 단축**(반대 진술).
- ① **UNDO**(이전 상태 복원) / **REDO**(완료 트랜잭션 재실행) 정의. ✔
- ② **즉시 갱신** = 트랜잭션 중 디스크에 곧바로 반영 → 실패 시 미커밋 변경 **UNDO 필요** + 커밋 후 장애 시 **REDO 필요** (양쪽 모두). ✔
- ③ **지연 갱신** = COMMIT 시점까지 디스크 반영 보류 → 미커밋 변경은 메모리에만 있음 → **UNDO 불필요**, COMMIT은 했는데 디스크 반영 전 장애 시 **REDO만** 필요. ✔
- 정형 함정: "검사점이 회복 시간을 늘린다" → X (정반대). "지연 갱신도 UNDO 필요" → X. **즉시=UNDO·REDO 둘 다 / 지연=REDO만 / 검사점=회복 시간 단축** 매트릭스 1자 매칭.

### 문제 7
**정답 ④ 확장(Scalability) 투명성 — 시스템에 노드를 추가하거나 제거해도 성능이 항상 선형으로 향상된다.**
- **분산 DB의 표준 투명성 4종**: 위치(Location) / 중복(Replication) / 병행(Concurrency) / 장애(Failure). 분할(Fragmentation) 투명성을 포함해 5종으로 분류하기도 함. **"확장(Scalability) 투명성"은 표준 분류에 없는 가짜 항목.**
- ① **위치 투명성**: 물리적 위치를 몰라도 동일 명령으로 접근. ✔
- ② **중복 투명성**: 동일 데이터 중복 저장돼도 단일 데이터처럼 사용. ✔
- ③ **병행 투명성**: 다중 트랜잭션 동시 수행 결과 = 직렬 수행 결과. ✔
- 정형 함정: **"확장·성능·보안·접근 투명성"** 같은 가짜 명칭 끼워넣기. 표준 4종(+분할) 외 모두 X.

### 문제 8
**정답 ③ CAP 정리는 분산 시스템에서 Consistency·Availability·Partition tolerance의 3가지를 동시에 모두 만족할 수 있다고 본다.**
- **CAP 정리(Brewer's Theorem)** 는 분산 시스템에서 **C·A·P 셋 중 동시에 최대 2가지만 보장 가능**, 셋 모두 동시 만족은 **불가능**하다고 본다(길버트·린치 증명).
- 분산 환경에서 네트워크 분단(P)은 사실상 필수 가정 → 실제 선택은 **CP(일관성+분할내성)** vs **AP(가용성+분할내성)**.
- ① NoSQL 4유형 분류(Key-Value / Document / Column-Family / Graph). ✔
- ② MongoDB = JSON·BSON Document 유형 대표. ✔
- ④ Neo4j = 노드+간선 Graph 유형 대표. ✔
- 정형 함정: **"CAP 셋을 모두 동시에 만족"** → **X**(정의 자체와 정반대). 1자 즉답.
